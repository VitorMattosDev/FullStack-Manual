Capítulo 01

Introdução ao TypeScript

TypeScript é um superconjunto tipado de JavaScript, desenvolvido pela Microsoft. Tudo o que é JavaScript válido é também TypeScript válido — a linguagem adiciona, sobre isso, um sistema de tipos estático opcional que é apagado na compilação, resultando em JavaScript puro que roda em qualquer lugar.

# Por que usar TypeScript?

JavaScript é uma linguagem dinâmica: o mesmo identificador pode conter um número agora e uma função mais tarde. Isso é flexível, mas custa caro em bugs silenciosos e refatorações arriscadas. TypeScript trabalha em tempo de desenvolvimento para:

  • Detectar erros antes do código rodar, não durante uma chamada em produção.
  • Oferecer autocomplete e navegação confiáveis no editor.
  • Documentar contratos entre funções, módulos e equipes.
  • Facilitar refatorações em bases de código grandes.
exemplo.ts
// JavaScript aceita silenciosamente:
function somar(a, b) { return a + b; }
somar("5", 10); // "510" em runtime, sem aviso.

// TypeScript bloqueia o erro antes da execução:
function somarTS(a: number, b: number): number {
  return a + b;
}
somarTS("5", 10); // ❌ Erro: 'string' não é atribuível a 'number'.

# Instalação e setup

TypeScript é distribuído como um pacote npm. A instalação global é útil para experimentos, mas em projetos reais prefira adicionar como dependência de desenvolvimento.

terminal
# Instalação global (opcional)
npm install -g typescript

# Em um projeto novo:
npm init -y
npm install --save-dev typescript @types/node

# Gerar o arquivo de configuração:
npx tsc --init

# Compilar um arquivo:
npx tsc arquivo.ts

# Compilar todo o projeto em modo watch:
npx tsc --watch

Para executar arquivos .ts diretamente, sem gerar .js, existem runners como tsx, ts-node ou o próprio Bun e Deno, que entendem TypeScript nativamente.

terminal
npm install --save-dev tsx
npx tsx arquivo.ts     # Executa direto, sem passo de build

# Ou com Bun (suporte nativo, muito rápido):
bun run arquivo.ts

# Seu primeiro programa

Crie um arquivo ola.ts e experimente a experiência de tipos desde a primeira linha:

ola.ts
interface Pessoa {
  nome: string;
  idade: number;
}

function saudar(p: Pessoa): string {
  return `Olá, ${p.nome}! Você tem ${p.idade} anos.`;
}

const maria: Pessoa = { nome: "Maria", idade: 28 };
console.log(saudar(maria));
Observação
O compilador (tsc) gera JavaScript a partir de TypeScript — ele não verifica tipos em tempo de execução. Toda a tipagem acontece no editor e na hora do build.

# O arquivo tsconfig.json

O tsconfig.json é o coração de um projeto TypeScript. Ele diz ao compilador quais arquivos incluir, qual versão do JavaScript gerar, e quão rigoroso ele deve ser. Um tsconfig moderno e sensato tem mais ou menos esta cara:

tsconfig.json
{
  "compilerOptions": {
    "target": "ES2022",               // versão do JS gerado
    "module": "NodeNext",             // sistema de módulos
    "moduleResolution": "NodeNext",
    "lib": ["ES2022", "DOM"],         // APIs disponíveis
    "outDir": "./dist",               // pasta de saída
    "rootDir": "./src",               // pasta de entrada

    "strict": true,                   // ativa todas checagens rigorosas
    "noImplicitAny": true,            // proíbe 'any' implícito
    "strictNullChecks": true,         // null/undefined são explícitos
    "noUncheckedIndexedAccess": true, // arr[i] pode ser undefined
    "exactOptionalPropertyTypes": true,

    "esModuleInterop": true,          // compatibilidade CommonJS/ESM
    "forceConsistentCasingInFileNames": true,
    "skipLibCheck": true,
    "resolveJsonModule": true,
    "declaration": true,              // gera .d.ts junto com .js
    "sourceMap": true                 // facilita debug
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist"]
}
Dica
Sempre comece com "strict": true. Lutar contra ele depois é muito mais doloroso do que escrever código correto desde o início.
Capítulo 02

Os tipos básicos

TypeScript herda de JavaScript todos os tipos primitivos e adiciona alguns próprios, nascidos da necessidade de descrever situações que o JavaScript não trata. Este capítulo é o vocabulário fundamental da linguagem.

# Primitivos: string, number, boolean

Os três tipos primitivos principais são praticamente os mesmos de JavaScript, agora com rótulos explícitos.

primitivos.ts
let nome: string = "Ana";
let idade: number = 30;
let ativo: boolean = true;

// Template literals funcionam normalmente:
let apresentacao: string = `${nome} tem ${idade} anos`;

// number cobre inteiros, decimais, hex, binário, octal:
let hex: number = 0xff;
let bin: number = 0b1010;
let dec: number = 3.14;

// Além destes, TypeScript também tem:
let grande: bigint = 9007199254740991n;
let id: symbol = Symbol("id");

# Arrays e Tuplas

Arrays têm duas notações equivalentes. A tupla (tuple) é uma invenção do TypeScript: um array de tamanho e tipos fixos por posição.

arrays-tuplas.ts
// Duas formas de declarar arrays:
let numeros: number[] = [1, 2, 3];
let nomes: Array<string> = ["Ana", "Bruno"];

// Array com tipos variados (union):
let mistos: (string | number)[] = ["a", 1, "b", 2];

// TUPLA — ordem e tipo fixos:
let coord: [number, number] = [10, 20];
let registro: [string, number, boolean] = ["item", 10, true];

// Tupla com elementos opcionais (?):
let ponto: [number, number, number?] = [1, 2];

// Tupla com rest (...) — útil para padrões variádicos:
let eventos: [string, ...number[]] = ["click", 10, 20, 30];

// Tupla readonly — imutável:
let rgb: readonly [number, number, number] = [255, 0, 0];
// rgb[0] = 100; // ❌ Erro

# Objetos

Objetos são descritos pelo formato dos seus campos. O TypeScript chama isso de tipagem estrutural: o que importa é o formato, não o nome.

objetos.ts
// Tipo inline em um parâmetro:
function imprimirPonto(p: { x: number; y: number }): void {
  console.log(`(${p.x}, ${p.y})`);
}

// Propriedades opcionais usam '?':
function criarUsuario(info: { nome: string; email?: string }) {
  console.log(info.nome);
  if (info.email) console.log(info.email);
}

// Objeto com campos readonly:
let config: { readonly versao: string; porta: number } = {
  versao: "1.0",
  porta: 8080,
};
// config.versao = "2.0"; // ❌ readonly

# any, unknown, never, void

Estes são os quatro tipos "especiais" de TypeScript. Entender a diferença entre eles é um dos pontos de virada no aprendizado da linguagem.

TipoSignificadoQuando usar
any"Não checar nada"Fuga de emergência. Evite sempre que possível.
unknown"Algo, mas não sei o quê"Valores externos (JSON, API) que você vai validar.
never"Nunca acontece"Funções que lançam erro ou fazem loop infinito.
void"Sem valor de retorno útil"Funções que não retornam nada significativo.
especiais.ts
// ANY — desliga o type checker. Cuidado!
let qualquerCoisa: any = 4;
qualquerCoisa = "texto";
qualquerCoisa.foo.bar; // passa, mas quebra em runtime.

// UNKNOWN — obriga você a verificar antes de usar:
let valor: unknown = JSON.parse('{"x": 1}');
// valor.x; // ❌ Erro: valor é unknown
if (typeof valor === "object" && valor !== null && "x" in valor) {
  console.log(valor.x); // ✅ ok aqui dentro
}

// NEVER — um valor que nunca existe:
function erroFatal(msg: string): never {
  throw new Error(msg);
}
function loopInfinito(): never {
  while (true) {}
}

// VOID — função sem retorno significativo:
function log(msg: string): void {
  console.log(msg);
  // não precisa de 'return'
}
Atenção
Usar any é como remover a tipagem. Se a intenção é "valor desconhecido que vou validar", use unknown. Se a intenção é "qualquer tipo genérico", use generics. any quase sempre é um cheiro de código.

# null e undefined

Com strictNullChecks ativo (o padrão em projetos modernos), null e undefined são tipos distintos e precisam ser tratados explicitamente.

null-undefined.ts
let a: string = null;       // ❌ em strict mode
let b: string | null = null; // ✅

function buscarUsuario(id: number): string | undefined {
  if (id < 0) return undefined;
  return "Ana";
}

const nome = buscarUsuario(1);
// nome.toUpperCase(); // ❌ possivelmente undefined

// Encadeamento opcional (?.) — retorna undefined se null/undefined:
console.log(nome?.toUpperCase());

// Coalescência nula (??) — só usa o fallback em null/undefined:
const tamanho = nome?.length ?? 0;

// Non-null assertion (!) — você jura que não é null:
const nomeCerto = buscarUsuario(1)!;
console.log(nomeCerto.toUpperCase()); // use com cuidado!

# Inferência de tipos

TypeScript é inteligente. Na maior parte do tempo, você não precisa escrever tipos — ele descobre sozinho a partir do valor. O segredo do código TypeScript limpo é escrever tipos só onde acrescentam informação.

inferencia.ts
let total = 10;           // inferido como number
let nome = "Ana";         // inferido como string
let ativos = [1, 2, 3];   // inferido como number[]

// Em const, o tipo é ainda mais estreito (literal):
const cor = "vermelho";   // tipo: "vermelho" (não string)

// Em funções, o retorno é inferido a partir do corpo:
function dobrar(n: number) {
  return n * 2; // TS sabe que retorna number
}

// Em objetos literais o TS também infere:
const pessoa = { nome: "Ana", idade: 30 };
// tipo: { nome: string; idade: number }

# Type assertions: as e satisfies

Às vezes você sabe mais que o compilador. as diz "confie em mim", enquanto satisfies (TypeScript 4.9+) é uma checagem sem mudar o tipo inferido. São ferramentas diferentes para problemas diferentes.

assertions.ts
// AS — afirma um tipo ao compilador:
const elemento = document.getElementById("app") as HTMLInputElement;
elemento.value = "oi"; // acessa propriedade de input

// Dupla asserção (use em último caso):
const x = "123" as unknown as number;

// SATISFIES — valida sem apagar o tipo inferido:
type Cor = "red" | "green" | "blue";

const paleta = {
  primaria: "red",
  secundaria: "blue",
} satisfies Record<string, Cor>;

// paleta.primaria ainda é "red" literal, não 'Cor'.
// E o TS validou que os valores são Cor válidas.

// Compare com 'as', que perderia a inferência:
const paleta2 = {
  primaria: "red",
  secundaria: "blue",
} as Record<string, Cor>;
// aqui paleta2.primaria é Cor, não "red".
Use satisfies
Sempre que você quiser validar que um objeto tem certo formato, mas sem perder a precisão dos tipos literais que o TypeScript infere, prefira satisfies em vez de as.

# Literal types

Você pode usar valores literais como tipos. Isso é a base dos discriminated unions e de muitos padrões avançados.

literal.ts
// Literal types em união formam um "enum leve":
type Alinhamento = "left" | "center" | "right";

function alinhar(texto: string, como: Alinhamento) { /* ... */ }
alinhar("oi", "center"); // ✅
alinhar("oi", "centro"); // ❌ Erro: "centro" não existe

// Também funciona com números e booleanos:
type Dado = 1 | 2 | 3 | 4 | 5 | 6;
type Bit = 0 | 1;

// 'as const' congela literais em um objeto:
const config = {
  modo: "dark",
  versao: 1,
} as const;
// config.modo é "dark", não string
// config agora é readonly

# Enums

Enums são conjuntos nomeados de constantes. TypeScript oferece numeric enums, string enums e const enums. A comunidade moderna tem preferido unions de literais, mas enums ainda aparecem muito em código legado e em APIs de bibliotecas.

enums.ts
// Enum numérico (valores começam em 0 por padrão):
enum Direcao {
  Norte,       // 0
  Leste,       // 1
  Sul,         // 2
  Oeste,       // 3
}

// Você pode customizar:
enum Status {
  Ativo = 1,
  Inativo = 2,
  Banido = 99,
}

// String enum (mais seguro — sem mágica numérica):
enum Cor {
  Vermelho = "RED",
  Verde = "GREEN",
  Azul = "BLUE",
}

// const enum — apagado no build, vira valor literal:
const enum Tamanho { Pequeno = "S", Medio = "M", Grande = "G" }
const t = Tamanho.Medio; // vira "M" no JS final

// Alternativa moderna recomendada — union type:
type StatusModerno = "ativo" | "inativo" | "banido";
Prefira unions
Em código novo, unions de literais ("a" | "b") costumam ser mais previsíveis, mais fáceis de usar com JSON e não deixam artefatos no bundle final. Reserve enum para quando for realmente necessário ou para compatibilidade.
Capítulo 03

Funções tipadas

Funções são cidadãs de primeira classe em JavaScript. Em TypeScript, isso significa que elas têm tipos ricos: parâmetros tipados, retornos tipados, parâmetros opcionais, sobrecargas e até a assinatura do this.

# Assinatura básica

funcoes.ts
// Declaração de função com tipos:
function somar(a: number, b: number): number {
  return a + b;
}

// Arrow function:
const multiplicar = (a: number, b: number): number => a * b;

// Função como valor tipado:
const dividir: (a: number, b: number) => number = (a, b) => a / b;

// Usando um type alias para a assinatura:
type Operacao = (a: number, b: number) => number;
const subtrair: Operacao = (a, b) => a - b;

// Void em retornos — não precisa de return:
function logar(msg: string): void {
  console.log(msg);
}

# Parâmetros opcionais, padrão e rest

params.ts
// Opcional — '?' após o nome:
function saudar(nome: string, titulo?: string): string {
  return titulo ? `${titulo} ${nome}` : nome;
}
saudar("Ana");            // ✅
saudar("Ana", "Dra.");    // ✅

// Padrão — atribui um valor se nada for passado:
function criarConfig(host: string, porta: number = 8080) {
  return `${host}:${porta}`;
}

// Rest — coleta argumentos em um array:
function somarTudo(...numeros: number[]): number {
  return numeros.reduce((acc, n) => acc + n, 0);
}
somarTudo(1, 2, 3, 4);

// Destructuring tipado:
function perfil({ nome, idade }: { nome: string; idade: number }) {
  return `${nome} (${idade})`;
}

# Function overloads

Sobrecarga permite descrever múltiplas formas de chamar a mesma função. Você escreve várias assinaturas seguidas da implementação, que precisa cobrir todos os casos.

overloads.ts
// Assinaturas (visíveis para quem chama):
function pesquisar(id: number): string;
function pesquisar(nome: string): number;
function pesquisar(query: boolean): string[];

// Implementação (não visível externamente):
function pesquisar(query: number | string | boolean): string | number | string[] {
  if (typeof query === "number") return "usuario-" + query;
  if (typeof query === "string") return query.length;
  return [];
}

const a = pesquisar(1);       // string
const b = pesquisar("Ana");   // number
const c = pesquisar(true);    // string[]
Quando usar
Overloads são úteis quando o tipo de retorno depende do tipo de entrada de forma complexa. Em casos mais simples, prefira generics ou união de tipos, que são mais fáceis de manter.

# Tipando o this

Diferente de outras linguagens, this em JavaScript depende de como a função é chamada. TypeScript permite declarar o tipo de this como um primeiro parâmetro "fantasma":

this.ts
interface Contador {
  valor: number;
  incrementar(): void;
}

// 'this: Contador' não aparece em runtime; só vale para o TS:
function incrementar(this: Contador) {
  this.valor++;
}

const c: Contador = { valor: 0, incrementar };
c.incrementar(); // ✅
// incrementar.call({ outra: 1 }); // ❌ 'this' inválido
Capítulo 04

Interfaces e Type aliases

Estas são as duas maneiras primárias de nomear um tipo no TypeScript. Elas parecem a mesma coisa e, na maior parte do tempo, são intercambiáveis — mas cada uma tem suas particularidades.

# Interfaces

interfaces.ts
interface Usuario {
  id: number;
  nome: string;
  email: string;
  ativo: boolean;
}

const u: Usuario = {
  id: 1,
  nome: "Ana",
  email: "ana@exemplo.com",
  ativo: true,
};

// Extensão (herança de interfaces):
interface Admin extends Usuario {
  permissoes: string[];
}

// Múltipla extensão:
interface Timestamped {
  criadoEm: Date;
  atualizadoEm: Date;
}
interface Post extends Usuario, Timestamped {
  titulo: string;
  conteudo: string;
}

// Declaration merging — duas declarações se fundem:
interface Janela { largura: number; }
interface Janela { altura: number; }
// Resultado: Janela tem largura E altura.

# Type aliases

type-alias.ts
// Apelido para tipo de objeto:
type Produto = {
  id: number;
  nome: string;
  preco: number;
};

// Apelido para união:
type ID = number | string;

// Apelido para tupla:
type Coordenada = [number, number];

// Apelido para função:
type Callback<T> = (valor: T) => void;

// Intersection (combinação):
type ProdutoEstocado = Produto & {
  estoque: number;
  localizacao: string;
};

# Interface vs Type — quando usar cada um

Característicainterfacetype
Objetos✅ ideal✅ funciona
Unions❌ não suporta✅ A | B
Tuples❌✅ [a, b]
Primitivos/literals❌✅ "on" | "off"
Extends/herança✅ extends✅ via &
Declaration merging✅ pode fundir❌ duplicação é erro
Mapped/conditional❌✅
Regra prática
Use interface para formas de objeto que podem ser estendidas (especialmente em APIs públicas de bibliotecas, onde merging é útil). Use type para tudo o mais: unions, intersections, tuplas, mapped types, conditional types. Se não souber, type é um bom default.

# Readonly e campos opcionais

readonly-optional.ts
interface Config {
  readonly id: string;       // imutável após criado
  readonly versao: number;
  porta?: number;            // opcional
  debug?: boolean;
}

const cfg: Config = { id: "app", versao: 1 };
// cfg.id = "outro"; // ❌ readonly

// Readonly arrays:
const dias: readonly string[] = ["seg", "ter"];
// dias.push("qua"); // ❌ push não existe em readonly array

// ReadonlyArray<T> é equivalente:
const nums: ReadonlyArray<number> = [1, 2, 3];

# Index Signatures

Quando você não sabe de antemão os nomes das propriedades, mas sabe seu tipo, use uma assinatura de índice.

index-signatures.ts
// Dicionário de strings para números:
interface Placar {
  [jogador: string]: number;
}
const placar: Placar = { ana: 10, bruno: 7 };
placar.carlos = 15; // ✅

// Dicionário misto — com propriedade fixa e índice:
interface ConfigComDefault {
  default: string;
  [chave: string]: string;
}

// Com union em tipo de valor:
interface Cache {
  [url: string]: { data: unknown; ttl: number };
}
Capítulo 05

Unions, Intersections e Narrowing

Poucos recursos expressam tanto quanto as uniões de tipos. Combinadas com o mecanismo de narrowing do TypeScript, você consegue representar e validar cenários do mundo real com precisão cirúrgica.

# Union types (|)

union.ts
// Um valor que pode ser de um de vários tipos:
type ID = string | number;

function identificar(id: ID) {
  // antes do narrowing, só dá para usar métodos comuns:
  console.log(id.toString()); // ✅ (ambos têm toString)
  // console.log(id.toUpperCase()); // ❌ número não tem
}

// Union de objetos:
type Animal =
  | { tipo: "cao"; latidos: number }
  | { tipo: "gato"; miados: number }
  | { tipo: "peixe"; bolhas: number };

# Intersection types (&)

A intersecção combina vários tipos em um só. O resultado precisa satisfazer todos os lados da combinação.

intersection.ts
type ComNome = { nome: string };
type ComIdade = { idade: number };

type Pessoa = ComNome & ComIdade;
// equivalente a: { nome: string; idade: number }

const p: Pessoa = { nome: "Ana", idade: 30 };

// Combinações úteis com generics:
type WithId<T> = T & { id: string };

type UsuarioComId = WithId<{ nome: string; email: string }>;
// { nome: string; email: string; id: string }

# Narrowing e Type Guards

O TypeScript observa o fluxo do seu código e, a cada verificação, estreita (narrow) o tipo automaticamente. Conhecer as formas de guard é essencial.

narrowing.ts
// 1) typeof — para primitivos:
function imprimir(v: string | number) {
  if (typeof v === "string") {
    console.log(v.toUpperCase()); // v é string aqui
  } else {
    console.log(v.toFixed(2));    // v é number aqui
  }
}

// 2) instanceof — para classes:
function lidar(e: Error | string) {
  if (e instanceof Error) {
    console.log(e.stack);
  } else {
    console.log(e);
  }
}

// 3) in — testa a presença de propriedade:
type Carro = { rodas: number };
type Barco = { remos: number };

function mover(v: Carro | Barco) {
  if ("rodas" in v) {
    console.log(v.rodas);  // v é Carro
  } else {
    console.log(v.remos);  // v é Barco
  }
}

// 4) Comparação com literal:
function estado(s: "loading" | "ok" | "erro") {
  if (s === "loading") { /* s é "loading" */ }
}

// 5) Truthiness narrowing:
function tamanho(x: string | null) {
  if (x) {
    return x.length; // x é string aqui
  }
  return 0;
}

// 6) User-defined type guard (predicate):
function ehString(v: unknown): v is string {
  return typeof v === "string";
}

function usar(v: unknown) {
  if (ehString(v)) {
    console.log(v.trim()); // v é string
  }
}

// 7) Assertion function:
function garantirString(v: unknown): asserts v is string {
  if (typeof v !== "string") throw new Error("não é string");
}

function process(v: unknown) {
  garantirString(v);
  console.log(v.toUpperCase()); // v é string após a asserção
}

# Discriminated Unions

O padrão mais poderoso do sistema de tipos TypeScript. Cada variante da união carrega uma propriedade literal única (a "tag"), e o compilador usa essa tag para distinguir os casos.

discriminated.ts
type Requisicao =
  | { status: "loading" }
  | { status: "ok"; dados: string[] }
  | { status: "erro"; mensagem: string };

function renderizar(req: Requisicao) {
  switch (req.status) {
    case "loading":
      return "Carregando...";
    case "ok":
      return `${req.dados.length} itens`; // dados existe aqui
    case "erro":
      return `Falha: ${req.mensagem}`;   // mensagem existe aqui
  }
}

// Exaustividade com 'never' — garante que todos os casos foram tratados:
function renderizarCompleto(req: Requisicao): string {
  switch (req.status) {
    case "loading": return "...";
    case "ok":      return "ok";
    case "erro":    return "erro";
    default:
      const _exhaustive: never = req; // erro se faltar caso
      return _exhaustive;
  }
}
O padrão da discriminated union
Sempre que você tiver "um valor que pode ter formatos diferentes", pense em discriminated union com uma tag literal. É a forma idiomática de representar estados, resultados de API e eventos em TypeScript.

Classes em TypeScript

Classes existem em JavaScript desde ES2015, mas TypeScript adiciona camadas importantes: modificadores de acesso, classes abstratas, implementação de interfaces, membros estáticos tipados e parâmetros de propriedade.

# Classes básicas

class Pessoa {

# Modificadores de acesso

TypeScript adiciona public, private e protected. Eles existem apenas em tempo de compilação — no JavaScript gerado, tudo fica público. Para privacidade em runtime, use os private fields do ECMAScript (#campo).

modificadores.ts
class Conta {
  public titular: string;          // padrão; acessível por todos
  private saldo: number;           // só dentro da classe
  protected numero: string;        // classe + subclasses
  readonly moeda: string = "BRL";  // imutável após o construtor

  constructor(titular: string, saldo: number, numero: string) {
    this.titular = titular;
    this.saldo = saldo;
    this.numero = numero;
  }

  depositar(valor: number) {
    this.saldo += valor;
  }

  consultarSaldo(): number {
    return this.saldo;
  }
}

// PARAMETER PROPERTIES — atalho para declarar+atribuir:
class Cliente {
  constructor(
    public nome: string,
    private cpf: string,
    readonly criadoEm: Date = new Date()
  ) {}
  // equivalente a declarar campos e atribuir no constructor.
}

// Private fields do JS (privacidade REAL em runtime):
class Cofre {
  #segredo: string;
  constructor(s: string) { this.#segredo = s; }

# Getters, setters e estáticos

accessors-static.ts
class Temperatura {
  private _celsius: number = 0;

  get celsius(): number { return this._celsius; }
  set celsius(valor: number) {
    if (valor < -273.15) throw new Error("abaixo do zero absoluto");
    this._celsius = valor;
  }

  get fahrenheit(): number {
    return this._celsius * 9 / 5 + 32;
  }

  // Membros estáticos — pertencem à classe, não à instância:
  static zeroAbsoluto = -273.15;
  static emKelvin(c: number): number {
    return c + 273.15;
  }
}

const t = new Temperatura();

console.log(t.fahrenheit);          // 77
console.log(Temperatura.zeroAbsoluto);
console.log(Temperatura.emKelvin(25));

# Herança, abstração e interfaces

heranca.ts
// Classe ABSTRATA — não pode ser instanciada, serve só como base:
abstract class Forma {
  abstract area(): number;                // filha precisa implementar
  descrever(): string {
    return `Forma com área ${this.area()}`;
  }
}

class Circulo extends Forma {
  constructor(private raio: number) { super(); }
  area(): number { return Math.PI * this.raio ** 2; }
}

class Retangulo extends Forma {
  constructor(private largura: number, private altura: number) { super(); }
  area(): number { return this.largura * this.altura; }
}

// const f = new Forma(); // ❌ não pode instanciar classe abstrata.

// IMPLEMENTS — classe implementa interface:
interface Identificavel {
  id: string;
  igual(outro: Identificavel): boolean;
}

class Produto implements Identificavel {
  constructor(public id: string, public nome: string) {}
  igual(outro: Identificavel): boolean {
    return this.id === outro.id;
  }
}

// Múltiplas interfaces:
interface Serializavel { toJSON(): string; }
class Pedido implements Identificavel, Serializavel {
  constructor(public id: string) {}
  igual(o: Identificavel) { return this.id === o.id; }
  toJSON() { return JSON.stringify(this); }
}
extends vs implements
extends herda comportamento de outra classe (podendo reaproveitar métodos). implements apenas exige que a classe tenha certo formato — não herda implementação nenhuma.

Capítulo 07

Generics

Generics são a ferramenta para escrever código que funciona com múltiplos tipos mantendo total segurança. Em vez de abrir mão da tipagem com any, você parametriza a função, classe ou tipo — o TypeScript preenche o parâmetro conforme o uso.

# Generics em funções

generics.ts
// Sem generics — perde informação:
function primeiroRuim(arr: any[]): any {
  return arr[0];
}
const x = primeiroRuim([1, 2, 3]); // x é any :(

// Com generics — preserva o tipo:
function primeiro<T>(arr: T[]): T | undefined {
  return arr[0];
}

const a = primeiro([1, 2, 3]);      // number | undefined
const b = primeiro(["x", "y"]);     // string | undefined
const c = primeiro<boolean>([true]);// especificando manualmente

// Múltiplos parâmetros de tipo:
function par<A, B>(a: A, b: B): [A, B] {
  return [a, b];
}

const p = par("idade", 30); // [string, number]

# Constraints — restringindo T com extends

Às vezes você precisa que o tipo tenha certas capacidades. extends em um generic significa "T precisa ser compatível com X".

constraints.ts
// T precisa ter length:
function maisLongo<T extends { length: number }>(a: T, b: T): T {
  return a.length >= b.length ? a : b;
}

maisLongo([1, 2, 3], [1]);     // ✅ arrays têm length
maisLongo("abcdef", "xyz");    // ✅ strings têm length
// maisLongo(10, 20);           // ❌ number não tem length

// T restrito a chaves de um objeto:
function pegar<T, K extends keyof T>(obj: T, chave: K): T[K] {
  return obj[chave];
}

const user = { nome: "Ana", idade: 30 };
const nome = pegar(user, "nome");   // string
const idade = pegar(user, "idade"); // number
// pegar(user, "x"); // ❌ "x" não existe em user

# Generics com valores padrão

generics-defaults.ts
// T tem valor padrão string:
interface Resposta<T = string> {
  dados: T;
  sucesso: boolean;
}

const r1: Resposta = { dados: "ok", sucesso: true };             // T = string
const r2: Resposta<number> = { dados: 42, sucesso: true };

// Generics em type aliases:
type Resultado<T, E = Error> =
  | { ok: true; valor: T }
  | { ok: false; erro: E };

function dividir(a: number, b: number): Resultado<number> {
  if (b === 0) return { ok: false, erro: new Error("divisão por zero") };
  return { ok: true, valor: a / b };
}
Capítulo 08

Tipos avançados

A partir daqui, o sistema de tipos deixa de ser "mais um anotador" e começa a ser quase uma linguagem de programação própria. Esta é a parte que torna possível bibliotecas como Zod, tRPC e Prisma produzirem tipagens tão precisas.

# O operador keyof

keyof T produz uma união com as chaves de um tipo.

keyof.ts
interface Pessoa {
  nome: string;
  idade: number;
  email: string;
}

type ChavesDePessoa = keyof Pessoa;
// "nome" | "idade" | "email"

function lerCampo<T, K extends keyof T>(obj: T, chave: K): T[K] {
  return obj[chave];
}

const p: Pessoa = { nome: "Ana", idade: 30, email: "a@x.com" };
const nome = lerCampo(p, "nome");  // string
const idade = lerCampo(p, "idade"); // number

# O operador typeof (em contexto de tipo)

Dentro do contexto de tipos, typeof extrai o tipo de uma variável ou valor existente.

typeof.ts
const config = {
  host: "localhost",
  porta: 8080,
  debug: true,
};

type Config = typeof config;
// { host: string; porta: number; debug: boolean }

// Combinando com 'as const' para precisão:
const ROTAS = {
  HOME: "/",
  LOGIN: "/login",
  PERFIL: "/perfil",
} as const;

type Rota = typeof ROTAS[keyof typeof ROTAS];
// "/" | "/login" | "/perfil"

# Indexed Access Types (T[K])

indexed-access.ts
interface Produto {
  id: number;
  nome: string;
  tags: string[];
  dimensoes: { largura: number; altura: number };
}

type ID = Produto["id"];              // number
type Nome = Produto["nome"];           // string
type Tag = Produto["tags"][number];    // string (elemento do array)
type Largura = Produto["dimensoes"]["largura"]; // number

// Acesso por união de chaves:
type Basico = Produto["id" | "nome"];  // number | string

// Com arrays:
const cores = ["red", "green", "blue"] as const;
type Cor = typeof cores[number];       // "red" | "green" | "blue"

# Conditional Types

Tipos condicionais seguem a mesma ideia do operador ternário do JavaScript, mas no nível de tipos. A forma é T extends U ? X : Y: "se T é compatível com U, então X, senão Y".

conditional.ts
type EhString<T> = T extends string ? "sim" : "nao";

type A = EhString<"ola">;    // "sim"
type B = EhString<42>;       // "nao"

// Distribui automaticamente em unions:
type Envelope<T> = T extends unknown ? { valor: T } : never;

type E = Envelope<string | number>;
// { valor: string } | { valor: number }

// Útil para "filtrar" unions:
type ApenasStrings<T> = T extends string ? T : never;
type Filtrado = ApenasStrings<"a" | 1 | "b" | 2>;
// "a" | "b"

# A palavra-chave infer

infer aparece apenas dentro de conditional types e diz "crie uma variável de tipo com o que estiver nesta posição". É a ferramenta mais poderosa para extrair partes de um tipo.

infer.ts
// Extrair o tipo de retorno de uma função:
type MeuReturnType<T> = T extends (...args: any[]) => infer R ? R : never;

function usuario() { return { nome: "Ana", idade: 30 }; }
type U = MeuReturnType<typeof usuario>;
// { nome: string; idade: number }

// Extrair o tipo dos parâmetros:
type MeuParameters<T> = T extends (...args: infer P) => any ? P : never;

type P = MeuParameters<(a: string, b: number) => void>;
// [string, number]

// Extrair o elemento de um array:
type Elemento<T> = T extends (infer E)[] ? E : never;

type X = Elemento<number[]>;     // number
type Y = Elemento<string[][]>;   // string[]

// Extrair o valor resolvido de uma Promise:
type Desenrolar<T> = T extends Promise<infer V> ? V : T;

type D = Desenrolar<Promise<string>>; // string

# Mapped Types

Mapped types geram novos tipos iterando sobre as chaves de um tipo existente. Essa é a engrenagem por trás de quase todos os utility types.

mapped.ts
// Torna todas as propriedades opcionais:
type MeuPartial<T> = {
  [K in keyof T]?: T[K];
};

// Torna todas readonly:
type MeuReadonly<T> = {
  readonly [K in keyof T]: T[K];
};

// Inverte — remove readonly e opcional:
type Mutavel<T> = {
  -readonly [K in keyof T]-?: T[K];
};

// Renomear chaves com 'as':
type ComPrefixo<T, P extends string> = {
  [K in keyof T as `${P}${Capitalize<string & K>}`]: T[K];
};

type Eventos = { click: MouseEvent; hover: MouseEvent };
type Handlers = ComPrefixo<Eventos, "on">;
// { onClick: MouseEvent; onHover: MouseEvent }

// Filtrar propriedades com 'as ... never':
type SomenteStrings<T> = {
  [K in keyof T as T[K] extends string ? K : never]: T[K];
};

type Pessoa = { nome: string; idade: number; email: string };
type SoStrings = SomenteStrings<Pessoa>;
// { nome: string; email: string }

Templates de string no mundo dos tipos. Permitem descrever padrões textuais com precisão absurda.

template-literal.ts

// "bg-red-100" | "bg-red-500" | ... | "bg-blue-900"

// Útil para CSS, rotas, eventos, etc.

// Utilities de string embutidos:
type A = Uppercase<"hello">;   // "HELLO"
type B = Lowercase<"HELLO">;   // "hello"

type D = Uncapitalize<"Hello">;// "hello"

# Tipos Recursivos

Tipos podem se referir a si mesmos — útil para estruturas aninhadas como JSON, árvores e listas encadeadas.

recursivos.ts
// JSON — um valor pode conter outros valores JSON:
type Json =
  | string
  | number
  | boolean
  | null
  | Json[]
  | { [k: string]: Json };

const dados: Json = {
  nome: "Ana",
  idade: 30,
  hobbies: ["leitura", { tipo: "esporte", nome: "natação" }],

interface No<T> {
  valor: T;
  filhos?: No<T>[];

type DeepReadonly<T> = {
  readonly [K in keyof T]: T[K] extends object

type Config = {
  db: { host: string; porta: number };
  cache: { ttl: number };
};

type ConfigCongelada = DeepReadonly<Config>;

Capítulo 09

Catálogo de Utility Types

TypeScript vem com uma biblioteca embutida de utility types — transformadores de tipo úteis para o dia-a-dia. Todos são implementados usando os recursos do capítulo anterior (keyof, conditional types, mapped types e infer). Conhecê-los é economia de tempo pura.

# Transformações de objeto

utility-objeto.ts
interface Usuario {
  id: number;
  nome: string;
  email: string;
  senha: string;
}

// Partial<T> — todos os campos opcionais:
type UsuarioParcial = Partial<Usuario>;
// { id?: number; nome?: string; email?: string; senha?: string }

// Required<T> — todos os campos obrigatórios (remove ?):
interface Config { host?: string; porta?: number; }
type ConfigCompleta = Required<Config>;

// Readonly<T> — todos os campos imutáveis:
type UsuarioImutavel = Readonly<Usuario>;

// Pick<T, K> — só as chaves escolhidas:
type UsuarioPublico = Pick<Usuario, "id" | "nome" | "email">;

// Omit<T, K> — tudo menos as chaves informadas:
type UsuarioSemSenha = Omit<Usuario, "senha">;

// Record<K, T> — objeto com chaves K e valores T:
type Permissoes = Record<"ler" | "escrever" | "admin", boolean>;
// { ler: boolean; escrever: boolean; admin: boolean }

const p: Permissoes = { ler: true, escrever: false, admin: false };

# Transformações de união

utility-uniao.ts
// Exclude<T, U> — remove membros de U da união T:
type Primitivos = string | number | boolean | null;
type SemNulo = Exclude<Primitivos, null>;
// string | number | boolean

// Extract<T, U> — só os membros que combinam:
type Numericos = Extract<Primitivos, number | boolean>;
// number | boolean

// NonNullable<T> — remove null e undefined:
type Valor = string | null | undefined;
type SemNulos = NonNullable<Valor>; // string

# Transformações de função

utility-funcao.ts

// Parameters<T> — tupla com os tipos dos parâmetros:

// ConstructorParameters<T> — parâmetros do construtor de uma classe:
class Foo { constructor(a: string, b: number) {} }

Módulos

Todo arquivo .ts que contém um import ou export é um módulo — seu escopo é isolado do restante do projeto. TypeScript aceita a sintaxe moderna ESM e também CommonJS, e oferece alguns refinamentos próprios.

# Export e import

matematica.ts
// Named exports:
export const PI = 3.14159;
export function somar(a: number, b: number): number { return a + b; }
export interface Ponto { x: number; y: number; }

// Default export (um por arquivo):
export default class Calculadora { /* ... */ }

// Re-export (atalho para centralizar):
export { outra } from "./outra-coisa";
export * from "./outro-modulo";
app.ts
import Calculadora, { PI, somar, type Ponto } from "./matematica";

// 'type' só importa o tipo — é apagado no build final.
// Equivalente completo:
import type { Ponto } from "./matematica";

// Alias em importação:
import { somar as adicionar } from "./matematica";

// Import como namespace:
import * as Math from "./matematica";
Math.somar(1, 2);
Type-only imports
Use import type ou o prefixo type em imports quando o que você importa é usado apenas em posição de tipo. Isso evita bundlers incluírem código desnecessário e deixa explícito para o leitor o que é tipo e o que é runtime.

# Namespaces

Namespaces são a forma antiga (pré-ESM) de organizar código em TypeScript. Raramente são a melhor escolha em código novo, mas ainda aparecem em bibliotecas de terceiros e em declaration files.

namespace Geometria {

Capítulo 12

Decorators

Decorators são funções que modificam classes, métodos, propriedades, acessores e parâmetros em tempo de declaração. TypeScript 5.0 adotou a sintaxe padrão do ECMAScript (Stage 3) — mais simples e alinhada com o futuro do JavaScript.

# Decorators de classe

decorators-classe.ts
// Um decorator é uma função com assinatura específica:
function logado<T extends new (...args: any[]) => object>(
  classe: T,
  ctx: ClassDecoratorContext
) {
  return class extends classe {
    constructor(...args: any[]) {
      super(...args);
      console.log(`[${String(ctx.name)}] instanciada`);
    }
  };
}

@logado
class Servico {
  constructor(public nome: string) {}
}

new Servico("auth"); // log: "[Servico] instanciada"

# Decorators de método

decorators-metodo.ts
function medirTempo(
  metodoOriginal: any,
  ctx: ClassMethodDecoratorContext
) {
  return function (this: any, ...args: any[]) {
    const inicio = performance.now();
    const resultado = metodoOriginal.apply(this, args);
    const fim = performance.now();
    console.log(`${String(ctx.name)}: ${fim - inicio}ms`);
    return resultado;
  };
}

class API {
  @medirTempo
  buscarDados() {
    // ... operação custosa
    return [1, 2, 3];
  }
}
Habilite com cuidado
Em projetos modernos, a sintaxe nativa de decorators (TS 5.0+) não requer experimentalDecorators. Se você trabalha com frameworks como Angular ou NestJS que dependem da versão antiga, mantenha "experimentalDecorators": true e "emitDecoratorMetadata": true no tsconfig.
Capítulo 13

Tipos assíncronos

Promises e async/await são JavaScript puro, mas TypeScript traz tipagem rica para eles. Saber ler e escrever assinaturas assíncronas é requisito para qualquer código de backend ou de chamadas HTTP.

async.ts
// Funções async sempre retornam Promise<T>:
async function buscarUsuario(id: number): Promise<Usuario> {
  const res = await fetch(`/api/users/${id}`);
  const json = await res.json();
  return json as Usuario;
}

interface Usuario { id: number; nome: string; }

// Promise.all é tipado por tupla:
async function carregar(): Promise<[Usuario, Post[]]> {
  const [u, p] = await Promise.all([
    buscarUsuario(1),
    buscarPosts(),
  ]);
  return [u, p];
}

async function buscarPosts(): Promise<Post[]> { return []; }
interface Post { id: number; titulo: string; }

// Awaited<T> — extrai o tipo resolvido:
type DadosUsuario = Awaited<ReturnType<typeof buscarUsuario>>;
// Usuario

// Result pattern — alternativa a try/catch:
type Resultado<T, E = Error> =
  | { ok: true; valor: T }
  | { ok: false; erro: E };

async function pegarUsuarioSeguro(id: number): Promise<Resultado<Usuario>> {
  try {
    const u = await buscarUsuario(id);
    return { ok: true, valor: u };
  } catch (e) {
    return { ok: false, erro: e as Error };
  }
}
Capítulo 14

Boas práticas

Você aprendeu a mecânica da linguagem. Esta última parte reúne regras não-oficiais que a comunidade TypeScript costuma seguir — o tipo de coisa que faz um código "cheirar" certo.

# Ative o modo strict

Todas as flags strict* ativadas por padrão. Em tsconfig.json, "strict": true é o primeiro passo de um projeto saudável. Também considere:

  • noUncheckedIndexedAccess — acesso a arr[i] devolve T | undefined. Quase sempre o correto.
  • exactOptionalPropertyTypes — diferencia "propriedade ausente" de "propriedade com valor undefined".
  • noImplicitOverride — força override explícito em subclasses.

# Prefira tipos a any

Quando for tentador usar any, pare e pense: seria unknown? Um generic? Uma union? O any só deve aparecer em interop raro com código legado.

# Deixe o TypeScript inferir

Anotações em todo lugar não deixam o código mais seguro — apenas mais ruidoso. Escreva tipos em:

  • Assinaturas públicas (parâmetros e retorno de funções exportadas).
  • APIs de módulos e classes.
  • Casos onde a inferência fica ampla demais (ex.: array vazio).

E confie na inferência em variáveis locais, callbacks inline e destructuring.

# Modelos com discriminated unions

Cada vez que você tem um objeto que "pode ser de três tipos" ou um estado com "várias fases", o padrão correto é quase sempre uma discriminated union. Evite optional chaining intrincado em estruturas que deveriam ser unions.

# Valide entradas externas

TypeScript protege você dentro do seu código. Tudo que vem de fora (JSON de APIs, localStorage, process.env, formulários) deve ser validado em runtime. Use bibliotecas como Zod, Valibot ou ArkType, que fornecem validação e tipos no mesmo gesto.

com-zod.ts
import { z } from "zod";

const UsuarioSchema = z.object({
  id: z.number(),
  nome: z.string(),
  email: z.string().email(),
});

type Usuario = z.infer<typeof UsuarioSchema>;
// { id: number; nome: string; email: string }

async function carregar(): Promise<Usuario> {
  const res = await fetch("/api/me");
  const json = await res.json();
  return UsuarioSchema.parse(json); // valida + tipa em runtime
}

# Nomenclatura

  • PascalCase para tipos, interfaces, classes e enums: Usuario, StatusHttp.
  • camelCase para variáveis, funções e propriedades: buscarUsuario.
  • CONSTANT_CASE para constantes de nível de módulo: MAX_RETRIES.
  • Evite prefixos tipo IUsuario — é cultura do C#, não casa com TypeScript.
  • Prefira nomes concretos a "genéricos demais". T em generics é tradicional, mas TItem, TKey comunicam mais.

# Organize o tsconfig do projeto

Em monorepos, use project references (references no tsconfig) para builds incrementais mais rápidos. Prefira um tsconfig.base.json que é estendido por cada pacote, em vez de duplicar configuração.

# Ferramentas companheiras

  • ESLint com @typescript-eslint para regras de estilo e boas práticas.
  • Prettier para formatação automática.
  • tsx ou Bun para rodar .ts direto em desenvolvimento.
  • vitest ou Jest + ts-jest para testes.
  • zod, valibot para validação em runtime.
Palavra final
TypeScript recompensa quem escreve com intenção. Cada tipo que você declara é uma pequena afirmação sobre o que seu código significa. Tratados com carinho, os tipos são a documentação mais honesta que sua equipe vai ter. Bom código.
