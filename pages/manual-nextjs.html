Capítulo 01

Introdução ao Next.js

Next.js não é só "React com SSR" — é o framework que reorganizou como pensamos em aplicações web em 2026. Server Components, Server Actions, Cache Components, Partial Prerendering — toda essa pilha existe num lugar coerente, e esse lugar é Next.js. Antes de mergulhar em arquivos e diretórios, vale entender por que esse framework dominou — e o que isso muda no seu modelo mental se você vem do React puro ou de versões antigas dele.

# O que é Next.js

o-que-e.tsx
// Next.js é um framework full-stack construído sobre React.
// Ele resolve, num único pacote coerente:

import { db } from "@/lib/db";

// app/produtos/page.tsx — uma rota completa em ~10 linhas
export default async function ProdutosPage() {
  // 1. Renderização — função React, mas executa NO SERVIDOR
  const produtos = await db.produtos.findMany();

  // 2. Roteamento — o caminho do arquivo VIRA a URL
  // 3. Data fetching — async direto no componente, sem useEffect
  // 4. Bundler — compila e otimiza tudo automaticamente
  // 5. Cache, streaming, otimização de imagens, fontes... incluso

  return (
    <ul>
      {produtos.map(p => <li key={p.id}>{p.nome}</li>)}
    </ul>
  );
}

Esse arquivo é uma rota. É HTTP, é React, é SQL, é build. Você não configura webpack, não cria endpoints REST, não monta um sistema de roteamento, não escreve useEffect + fetch. Tudo o que existe no ecossistema React tradicional para "fazer um site" foi consolidado em uma estrutura de pastas previsível e um conjunto de convenções.

# Por que Next.js dominou

O React puro, dos hooks ao fetch, sempre foi um pedaço da aplicação. Você escolhia um router (React Router), um data fetcher (TanStack Query), um bundler (Vite), um framework de SSR (next, remix, gatsby), uma autenticação, um deploy. Cada decisão era um ponto de fricção. Cada projeto era artesanal. Times grandes acabavam construindo internamente o que viraria Next.js — só que com meio das peças faltando.

Next.js venceu porque tomou as decisões por você, e tomou bem. Cada nova versão fechava mais fricções: roteamento por arquivo (Next 9), API routes (Next 9), SSG (Next 10), Image (Next 10), App Router (Next 13), Server Components (Next 13.4), Server Actions (Next 14), Partial Prerendering (Next 14), Cache Components (Next 16). Em vez de lutar contra o framework, você passa a expressar a aplicação através dele.

A consequência cultural
Em 2026, "fazer uma aplicação React fullstack" e "fazer um app Next.js" são, na prática, sinônimos. Existem alternativas legítimas — Remix, TanStack Start, Astro com React — mas elas são refinamentos sobre as ideias que Next.js consolidou. Se você está aprendendo para empregabilidade, Next.js é o caminho mais curto. Se está aprendendo para entender o estado da arte, Next.js também — porque é onde as inovações da React Core Team chegam primeiro.

# A virada — App Router e Server Components

Há dois "Next.js" emocionalmente distintos para quem desenvolve há tempo: o antes e o depois do App Router. O divisor é a versão 13.4 (maio 2023), quando React Server Components saíram do experimental e mudaram fundamentalmente como dados, renderização e interatividade se conectam.

antes-vs-depois.tsx
// ───────────────────────────────────────────────
// ANTES — Pages Router (Next.js ≤ 12)
// ───────────────────────────────────────────────

// pages/produtos.tsx
import { useEffect, useState } from "react";

export default function Produtos() {
  const [produtos, setProdutos] = useState([]);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    fetch("/api/produtos")
      .then(r => r.json())
      .then(data => {
        setProdutos(data);
        setLoading(false);
      });
  }, []);

  if (loading) return <p>Carregando...</p>;
  return <ul>{produtos.map(...)}</ul>;
}

// pages/api/produtos.ts (endpoint separado)
export default async function handler(req, res) {
  const data = await db.produtos.findMany();
  res.json(data);
}

// 1 rota = 2 arquivos. Loading state manual. Bundle JS no cliente
// inclui useState, useEffect, fetch, e a tela de loading.


// ───────────────────────────────────────────────
// DEPOIS — App Router (Next.js 13+)
// ───────────────────────────────────────────────

// app/produtos/page.tsx
export default async function Produtos() {
  const produtos = await db.produtos.findMany();
  return <ul>{produtos.map(...)}</ul>;
}

// 1 rota = 1 arquivo. Sem loading state explícito (Suspense cuida).
// Bundle JS no cliente: ZERO. Esse componente nunca chega ao browser.
// O HTML chega pronto, com os dados já dentro.

Não é apenas menos código — é um modelo mental diferente. No Pages Router, você escrevia React que sabia conversar com APIs. No App Router, você escreve React que é a API quando precisa ser, e que é o cliente quando precisa ser. A fronteira entre frontend e backend não some — ela passa a ser desenhada explicitamente pelo desenvolvedor, no ponto certo da árvore de componentes.

# O quadro mental — quatro camadas

Aplicações Next.js modernas se organizam mentalmente em quatro camadas. Conhecer essas camadas torna a documentação inteira mais legível, porque cada feature do framework "vive" em uma delas.

camadas.txt
# 1. ROTEAMENTO (filesystem-based)
#    - Pasta = segmento de URL
#    - app/sobre/page.tsx → /sobre
#    - app/produtos/[id]/page.tsx → /produtos/123
#    - layouts, loadings, errors são arquivos com nomes especiais

# 2. RENDERIZAÇÃO (Server + Client Components)
#    - Server Components: rodam no servidor, NÃO mandam JS pro cliente
#    - Client Components: rodam no servidor + cliente, têm interatividade
#    - Você marca um Client Component com "use client" no topo
#    - Server é o DEFAULT — você só vira client quando precisa

# 3. DADOS (server-first)
#    - fetch() ou query no DB direto no Server Component
#    - Não há "API entre frontend e backend" — o componente É a API
#    - Para mutações (form submit, click), use Server Actions
#    - Para cliente puxar dados, use Route Handlers (REST/JSON tradicional)

# 4. CACHE (opt-in, granular)
#    - Por padrão, tudo é dinâmico — renderizado em cada request
#    - Você marca o que pode ser cacheado com "use cache"
#    - cacheLife() controla quanto tempo
#    - cacheTag() permite invalidação cirúrgica
#    - Partial Prerendering: shell estático + dinâmico streamado

Os 22 capítulos seguintes destrincham essas quatro camadas em ordem. Se em algum momento você se perder, é útil voltar a esse mapa: a feature que está confundindo você quase sempre vive numa dessas quatro camadas, e a confusão geralmente é sobre o limite entre elas.

# Quando Next.js NÃO é a escolha certa

Como qualquer ferramenta dominante, Next.js virou "default reflexivo" em situações onde não é o melhor encaixe. Vale conhecer os casos limite antes de comprometer um projeto inteiro:

quando-evitar.txt
## NÃO use Next.js quando:

✗ SPA pura, sem SEO ou SSR
  Apps tipo dashboard atrás de login, com 100% client-side rendering,
  ganham pouco com Next.js. Vite + React Router é mais simples.

✗ Site puramente estático e simples
  Landing page de 5 seções? Astro ou um HTML estático servem melhor.
  Next.js é overkill para isso.

✗ Você não pode usar Node.js no servidor
  Embora Next.js possa exportar como estático, suas features mais
  poderosas (Server Components, Server Actions, ISR) exigem runtime
  de servidor — Vercel, Node, Cloudflare Workers, etc.

✗ Vendor lock-in te assusta de verdade
  Next.js NÃO é proprietário (é open source, MIT), mas algumas
  features funcionam melhor na Vercel. Se isso é deal-breaker
  político, considere Remix ou TanStack Start.

✗ Você quer aprender React puro primeiro
  Comece com Vite + React. Next.js abstrai muita coisa — você
  vai entender melhor "por que isso existe" depois de sentir
  a dor de configurar manualmente.

# Quando Next.js é exatamente a escolha certa

quando-usar.txt
## USE Next.js quando:

✓ Você precisa de SEO
  Marketing site, e-commerce, blog, documentação, SaaS público.
  SSG e SSR via App Router são best-in-class.

✓ A app tem partes públicas E privadas
  Landing + dashboard, marketplace, redes sociais.
  A mistura de páginas estáticas (rápidas, cacheáveis) com
  páginas dinâmicas (autenticadas, personalizadas) é o sweet spot.

✓ Você quer parar de manter API separada
  Server Actions deixam mutações triviais sem REST/GraphQL.
  Route Handlers cobrem o que precisa de API tradicional.
  Backend e frontend ficam no mesmo repo, mesma linguagem,
  mesma deploy.

✓ Time pequeno, escopo grande
  Uma equipe de 3 devs full-stack pode entregar mais com Next.js
  do que com 5 devs em arquitetura React-puro + API-Node separada.

✓ Você prioriza Time-to-Market
  Convenções fortes, defaults bons, integração com Vercel
  permitem ir de "git init" a "produção" em horas, não semanas.

# Convenções de arquivo dentro de app/

Dentro do diretório app/, o Next.js reconhece arquivos com nomes específicos que cumprem papéis especiais. Esse é o núcleo do filesystem-based routing.

ArquivoPropósitoObrigatório?
page.tsxUI da rota — torna o segmento acessível como URLPara a rota existir
layout.tsxWrapper que persiste entre navegaçõesObrigatório no root
loading.tsxSkeleton/spinner automático durante streamingNão
error.tsxError boundary para o segmentoNão
not-found.tsxUI quando notFound() é chamadoNão
route.tsRoute handler (API endpoint)Não
template.tsxComo layout, mas remonta a cada navegaçãoNão
default.tsxFallback para Parallel RoutesNão
Colocação (colocation)
Em Next.js, você pode colocar qualquer arquivo dentro de app/ — componentes, hooks, utilitários — e eles não se tornam rotas só porque estão lá. Apenas os arquivos com nomes especiais (page.tsx, layout.tsx, etc.) têm significado de roteamento. Isso permite manter componentes perto das rotas que os usam, sem expô-los como URLs.

const url = process.env.NEXT_PUBLIC_APP_URL;  // ✅ funciona

// Tipando variáveis de ambiente com TypeScript
// env.d.ts na raiz do projeto:
declare namespace NodeJS {
  interface ProcessEnv {
    DATABASE_URL: string;
    AUTH_SECRET: string;
    NEXT_PUBLIC_APP_URL: string;
  }
}
Nunca exponha secrets ao cliente
Qualquer variável de ambiente sem o prefixo NEXT_PUBLIC_ é automaticamente removida do bundle do cliente pelo Next.js. Mas isso não é proteção completa — se você passar um secret como prop para um Client Component, ele vai parar no bundle. A proteção real é arquitetural: manter a lógica que usa secrets em Server Components, Server Actions, ou Route Handlers.

Capítulo 03

App Router — modelo mental

Antes de qualquer arquivo, é preciso entender o paradigma. App Router não é "uma forma diferente de organizar páginas no Next.js" — é uma reformulação completa de como roteamento, renderização e dados se conectam. Cada decisão de design tem uma razão. Quando você entende o porquê, o resto do framework decifra-se sozinho. Este capítulo não tem código novo: tem o mapa que torna todos os outros capítulos legíveis.

# O filesystem como contrato

A primeira ideia central é que o sistema de arquivos é a definição da aplicação. Não há um arquivo de roteamento, não há registro de páginas, não há configuração que mapeie URLs para componentes. A pasta app/ e seus subdiretórios são a estrutura de URLs, e os arquivos especiais (page.tsx, layout.tsx, loading.tsx...) são a definição do que cada parte da árvore faz.

filesystem-contrato.txt
app/
├── layout.tsx              # envolve TUDO
├── page.tsx                # UI de /
├── loading.tsx             # skeleton de / e descendentes
├── error.tsx               # error boundary de / e descendentes
└── dashboard/
    ├── layout.tsx          # envolve /dashboard e descendentes
    ├── page.tsx            # UI de /dashboard
    ├── loading.tsx         # skeleton só desta sub-árvore
    └── relatorios/
        └── page.tsx        # UI de /dashboard/relatorios

# A relação é totalmente derivável da estrutura:
# - Layouts compõem do mais externo para o mais interno
# - Loading e error "protegem" sua sub-árvore
# - Para criar uma rota, basta criar uma pasta com page.tsx dentro
# - Para criar um layout aninhado, adicione layout.tsx em qualquer nível

Isso pode parecer "mais um router por arquivos" se você usou frameworks como Remix ou SvelteKit. A diferença que importa é o que cada arquivo significa. page.tsx não é "a página"; é especificamente o segmento folha que torna a rota navegável. Se você cria uma pasta dashboard/ sem page.tsx dentro, /dashboard não existe — mas /dashboard/relatorios pode existir se houver dashboard/relatorios/page.tsx. O roteador não enxerga "diretórios" — enxerga uma árvore de segmentos onde só folhas com page.tsx são URLs reais.

# A árvore — pais envolvendo filhos

O App Router pensa a aplicação como uma árvore onde cada nível pode contribuir com layout, loading e error boundary próprios. Quando você navega para /dashboard/relatorios, o que renderiza não é "o componente de /dashboard/relatorios" — é uma composição:

composicao-arvore.tsx
// O que efetivamente renderiza para /dashboard/relatorios:

<RootLayout>                       // app/layout.tsx
  <DashboardLayout>                // app/dashboard/layout.tsx
    <RelatoriosPage />             // app/dashboard/relatorios/page.tsx
  </DashboardLayout>
</RootLayout>

// E se houver erro em RelatoriosPage:

<RootLayout>
  <DashboardLayout>
    <DashboardError />             // app/dashboard/error.tsx (mais próximo)
  </DashboardLayout>
</RootLayout>

// Note: DashboardLayout PERMANECE renderizado.
// Apenas o filho dele troca para o error boundary.
// Isso preserva sidebar, navegação, contexto da seção.

Essa decisão tem uma implicação prática enorme para UX: quando você clica em um link dentro do dashboard e a página de destino dá erro, o usuário continua vendo a sidebar, o header, o menu — só a área de conteúdo é substituída pelo error boundary. O mesmo vale para loading: o esqueleto de loading aparece apenas onde está faltando dado, não cobre a tela inteira. Esse comportamento é gratuito — você não escreve nada para que aconteça, ele emerge da estrutura.

# Server-first como mudança de paradigma

A segunda ideia central — e a mais difícil de internalizar para quem vem de React tradicional — é que tudo dentro de app/ é um Server Component por padrão. Você precisa marcar explicitamente um componente como Client Component (com "use client") quando quer que ele rode no browser.

Isso é o oposto do React que dominou de 2015 a 2023. No React clássico, todo componente roda no browser. Server-side rendering era uma técnica de otimização que produzia HTML inicial, mas o componente em si era client-side. useState, useEffect, event handlers — tudo funcionava porque tudo era cliente.

server-first.tsx
// app/produtos/page.tsx — Server Component (default)
export default async function Produtos() {
  // Roda APENAS no servidor, na hora do request.
  // Pode usar await em qualquer coisa.
  const produtos = await db.produtos.findMany();

  // NÃO PODE: useState, useEffect, onClick, window, document
  // Porque nada disso existe no servidor.

  return <ListaProdutos items={produtos} />;
}

// app/produtos/lista-produtos.tsx — também Server por default
export function ListaProdutos({ items }) {
  // Renderiza no servidor.
  // Quando o request termina, ZERO bytes de JS deste componente
  // chegaram ao cliente. Apenas o HTML produzido por ele.
  return <ul>{items.map(p => <li>{p.nome}</li>)}</ul>;
}
client-when-needed.tsx
"use client";  // declaração no topo do arquivo

import { useState } from "react";

// Este componente E TODOS os componentes que ele importar
// rodam no cliente. Você ganha:
// - useState, useEffect, useReducer, etc.
// - Event handlers (onClick, onChange)
// - Acesso a window, document, localStorage
// - Hooks de bibliotecas client-side (TanStack Query, etc.)
//
// Em troca, perde:
// - async/await direto no componente
// - Acesso a banco de dados, secrets, sistema de arquivos
// - Tudo que precisa de runtime de servidor

export function Contador() {
  const [n, setN] = useState(0);
  return <button onClick={() => setN(n + 1)}>{n}</button>;
}
A boundary é o ponto de virada
A diretiva "use client" não significa "este componente é cliente". Significa "deste componente para baixo na árvore de imports, tudo é cliente". É uma fronteira, não uma marca individual. Se um Server Component importa um Client Component, OK — a fronteira é cruzada nesse import. Mas um Client Component não pode importar diretamente um Server Component (você passaria um pelo outro como prop ou children, mas nunca como import). O capítulo 8 detalha isso. Por ora, basta saber: o servidor é o default; você opta por cliente quando precisa de interatividade.

# Por que isso é uma vitória

"Mas eu já escrevia React no servidor com getServerSideProps no Pages Router". Esse argumento ignora o que mudou. No Pages Router, o componente era client-side, e getServerSideProps era uma função paralela que rodava no servidor para fornecer dados. Você tinha duas execuções: uma no servidor (props) e outra no cliente (componente).

No App Router com Server Components, o componente em si roda no servidor. O HTML produzido por ele é enviado ao browser. Os imports que ele faz não vão para o bundle do cliente — porque nunca precisaram. Se você importa Lodash em um Server Component, Lodash não viaja para o navegador. Se você usa um cliente Postgres, o cliente Postgres não viaja. Toda a árvore de dependência server-only some do bundle.

# A boundary entre servidor e cliente

O ponto sutil — e que confunde muita gente nas primeiras semanas — é que servidor e cliente coexistem na mesma árvore. Você pode ter um Server Component que renderiza um Client Component que renderiza um Server Component. Cada um roda onde precisa, e a comunicação entre eles é via props serializáveis.

arvore-mista.tsx
// app/produto/[id]/page.tsx — Server Component
import { BotaoComprar } from "./botao-comprar";  // Client Component
import { Avaliacoes } from "./avaliacoes";      // Server Component

export default async function ProdutoPage({ params }) {
  const { id } = await params;
  const produto = await db.produtos.findById(id);

  return (
    <article>
      <h1>{produto.nome}</h1>
      <p>{produto.descricao}</p>

      {/* BotaoComprar é Client — precisa de useState para o carrinho */}
      {/* Recebe dados serializados como props normais */}
      <BotaoComprar produtoId={produto.id} preco={produto.preco} />

      {/* Avaliacoes é Server — busca no DB e renderiza */}
      {/* Recebe o id e faz query própria */}
      <Avaliacoes produtoId={produto.id} />
    </article>
  );
}

Essa página produz: HTML estático com nome, descrição, e a árvore de avaliações já renderizada (também HTML); um pequeno bundle JS que hidrata o BotaoComprar (e só ele); zero JS para o resto. O usuário vê tudo instantaneamente; pode interagir com o botão de comprar assim que o JS hidrata. As avaliações já estão lá — não precisam carregar.

# Composição em vez de "passagem de dados"

Um corolário importante: você não precisa "passar dados pra baixo" como em React tradicional. Cada Server Component pode buscar os dados de que precisa, no nível em que precisa. Não há prop drilling, não há context provider para state global do servidor — porque o servidor não tem state, só execução.

composicao-direta.tsx
// app/dashboard/page.tsx
export default async function Dashboard() {
  return (
    <div>
      <CardReceita />      {/* busca a própria query */}
      <CardClientes />     {/* busca a própria query */}
      <UltimasVendas />    {/* busca a própria query */}
    </div>
  );
}

async function CardReceita() {
  const total = await db.vendas.somaTotal();
  return <div>R$ {total}</div>;
}

async function CardClientes() {
  const count = await db.clientes.count();
  return <div>{count} clientes</div>;
}

// Three queries diferentes. Three componentes independentes.
// Composição limpa. E como veremos no cap. 9, com Suspense
// elas podem até ser streamed em paralelo.

Compare isso com o que seria necessário no Pages Router: uma getServerSideProps central que busca todos os dados do dashboard, devolve um objeto enorme, propaga via props até cada componente. Em algum momento o dev pensa "vou usar SWR para cada componente buscar o que precisa" — mas aí volta-se ao client-side fetching com loading states. Server Components resolvem essa tensão: cada componente busca o que precisa, mas no servidor, sem JavaScript extra no cliente.

# Pages Router vs App Router — onde estamos

Next.js ainda suporta Pages Router. Em projetos legados ou em times que ainda não migraram, é normal encontrá-lo. Para projetos novos em 2026, a recomendação oficial e da comunidade é unânime: use App Router. Pages Router está em modo manutenção — sem novas features, sem evolução paralela.

# O que isso muda na sua cabeça

Uma vez que você internaliza essas duas ideias — filesystem como contrato, server-first — quase tudo no resto do framework decifra-se sozinho. Loading states automáticos? Existe porque o servidor demora para buscar dados, e Suspense dá o ponto natural para mostrar fallback. Server Actions? Existem porque mutações (form submit, click) também podem ser server-first, sem precisar de API REST intermediária. Cache Components? Existem porque, num mundo onde tudo roda no servidor, faz sentido cachear o resultado granularmente, em vez de "página estática vs página dinâmica".

O App Router é uma resposta coerente a uma pergunta: e se quase tudo na app rodasse no servidor, e o cliente fosse minoria — usado apenas onde interatividade exige? Os 19 capítulos restantes são detalhamento das consequências dessa pergunta.

As três coisas que mais valem internalizar agora
✓ Server por default, client por exceção. Você não precisa de "use client" em todo arquivo. Use só onde precisar de interatividade ou hooks de cliente.
✓ Layouts e loadings emergem da estrutura. Você não "configura" — você cria os arquivos certos nos lugares certos, e o framework compõe.
✓ Cada componente busca seus próprios dados. Pare de pensar em "fetch no topo, propaga pra baixo". O servidor pode buscar onde quiser, sem custo de bundle.

Os capítulos 4 a 7 cobrem o roteamento na prática. O cap. 8 aprofunda a fronteira Server/Client. Quando você sentir confusão em qualquer ponto adiante, volte mentalmente para essas três coisas — quase sempre é uma delas que está sendo violada.

# params — parâmetros de URL

Em rotas dinâmicas (com [colchetes]), o page recebe um prop especial chamado params contendo os valores capturados da URL. Em Next.js 15+, params é uma Promise e precisa ser aguardada com await.

params.tsx
// app/blog/[slug]/page.tsx → URL: /blog/meu-post

type Props = {
  params: Promise<{ slug: string }>;
};

export default async function PostPage({ params }: Props) {
  const { slug } = await params;  // ⚠️ await obrigatório em Next.js 15+
  const post = await db.posts.findBySlug(slug);

    <article>
      <h1>{post.titulo}</h1>
      <div dangerouslySetInnerHTML={{ __html: post.conteudo }} />
    </article>
  );
}

  params: Promise<{ categoria: string; produto: string }>;

  const { categoria, produto } = await params;
  // /loja/eletronicos/notebook → categoria="eletronicos", produto="notebook"

Antes do Next.js 15, params era um objeto síncrono. Você usava params.slug direto. ✓ params e searchParams são Promises. Sempre await antes de usar.
✓ Layouts não re-renderizam quando params mudam. Se precisa, use template.tsx ou mova lógica para a page.
✓ Grupos () organizam sem afetar URL. Padrão para separar marketing e app.
✓ Metadata fica no próprio arquivo. Estática como const; dinâmica via generateMetadata.
Capítulo 05

Rotas dinâmicas

Nem toda URL é fixa. Posts de blog, perfis de usuário, produtos, categorias — tudo isso precisa de rotas que aceitam valores variáveis. Este capítulo cobre os três tipos: dinâmicas simples ([param]), catch-all ([...param]), e opcionais ([[...param]]) — além de geração estática com generateStaticParams.

Catch-all é greedy
Uma rota catch-all [...slug] "engole" qualquer URL que comece com seu prefixo. Se você tem /docs/[...slug] e quer um /docs/api com layout diferente, precisa criar /docs/api/page.tsx — rotas estáticas têm prioridade sobre dinâmicas. O Next.js sempre prefere o match mais específico.

# generateStaticParams — pre-renderização

Se você sabe os valores possíveis de uma rota dinâmica em build-time (lista finita de slugs, IDs conhecidos), pode pre-renderizar todas elas como HTML estático. Resultado: páginas servidas instantaneamente, sem hit no banco a cada request.

app/blog/[slug]/page.tsx
// Função especial — Next.js chama em build-time
export async function generateStaticParams() {
  const posts = await db.posts.findAll();

  // Retorna array de objetos com os params
  return posts.map(post => ({
    slug: post.slug,
  }));
}

// Para múltiplos params, retorne objetos com todas as chaves:
export async function generateStaticParams() {
  const produtos = await db.produtos.findAll();
  return produtos.map(p => ({
    categoria: p.categoria,
    produto: p.slug,
  }));
}

// Para catch-all, slug deve ser um array:
export async function generateStaticParams() {
  return [
    { slug: ["intro"] },
    { slug: ["guia", "setup"] },
    { slug: ["api", "routes"] },
  ];
}

Decisões em rotas dinâmicas
✓ 1 segmento: use [param]. Cobre 90% dos casos (perfis, posts, produtos).
✓ Profundidade variável: [...slug] para docs, breadcrumbs, file-system-like apps.
✓ Rota raiz + nested: [[...slug]] quando precisa do mesmo handler para /docs e /docs/qualquer/coisa.
✓ Conteúdo conhecido em build: generateStaticParams + (opcionalmente) dynamicParams = false. Páginas servidas instantaneamente.
✓ Conteúdo dinâmico: sem generateStaticParams — Next.js renderiza sob demanda e cacheia conforme você configurar (caps. 13–14).
✓ Slugs inválidos: sempre cheque if (!post) notFound() — dispara not-found.tsx (próximo capítulo).
Capítulo 06

Loading, error e not-found

Todo fluxo real de usuário precisa responder a três situações além do happy path: dados ainda carregando, algo deu errado, e recurso não existe. No App Router, cada uma tem um arquivo dedicado que o Next.js compõe automaticamente na posição certa da árvore — sem você precisar escrever try/catch, lógica de estado, ou wrappers manuais.

# loading.tsx — skeleton automático

Quando um Server Component está aguardando dados (uma query async, um fetch), o Next.js mostra o conteúdo de loading.tsx automaticamente enquanto espera. Por baixo dos panos, ele envolve o page.tsx num <Suspense> — você não precisa escrever o Suspense manualmente.

app/dashboard/loading.tsx

  return (
    <div className="space-y-4 p-6">
      <div className="h-8 w-48 animate-pulse rounded bg-zinc-200" />
      <div className="grid grid-cols-3 gap-4">
        {[1, 2, 3].map(i => (
          <div
            key={i}
            className="h-32 animate-pulse rounded-xl bg-zinc-200"
          />
        ))}
      </div>
    </div>
  );
}
o-que-acontece-por-baixo.tsx
// O Next.js converte automaticamente:

// app/dashboard/layout.tsx

  return <div>{children}</div>;
}

// Em algo equivalente a:
import { Suspense } from "react";

loading.tsx mostra imediatamente
Diferente do estado de carregamento manual (if (loading) return <Spinner>), o loading.tsx é enviado ao browser antes dos dados chegarem — o HTML do skeleton viaja na resposta inicial. O usuário vê o esqueleto instantaneamente, sem esperar o JavaScript hidratar. Isso melhora o Largest Contentful Paint percebido significativamente em conexões lentas.

# error.tsx — error boundary de rota

error.tsx funciona como um React Error Boundary automático para o segmento. Se o page.tsx (ou qualquer Server Component filho) lançar um erro, error.tsx é renderizado no lugar — sem derrubar o layout pai.

error.tsx é sempre um Client Component
React Error Boundaries precisam ser Client Components — essa é uma limitação do React, não do Next.js. Portanto, error.tsx deve ter "use client" no topo. Se você esquecer, o Next.js vai avisar em build.
app/dashboard/error.tsx
"use client";  // obrigatório

import { useEffect } from "react";

export default function DashboardError({
  error,
  reset,
}: {
  error: Error & { digest?: string };  // digest = ID do erro no servidor
  reset: () => void;                  // tenta renderizar novamente
}) {
  useEffect(() => {
    // Log do erro para seu serviço de monitoring
    console.error(error);
  }, [error]);

  return (
    <div className="flex flex-col items-center gap-4 p-12">
      <h2 className="text-xl font-semibold">Algo deu errado</h2>
      <p className="text-zinc-500">{error.message}</p>
      <button
        onClick={reset}
        className="rounded-lg bg-blue-500 px-4 py-2 text-white"
      >
        Tentar novamente
      </button>
    </div>
  );
}

# Diferença entre notFound() e throw

notfound-vs-throw.tsx
import { notFound } from "next/navigation";

export default async function Page({ params }) {
  const { id } = await params;
  const item = await db.items.findById(id);

  // ✅ notFound() — recurso simplesmente não existe (HTTP 404)
  //    Renderiza not-found.tsx, retorna status 404
  if (!item) notFound();

  // ✅ throw — algo inesperado aconteceu (HTTP 500)
  //    Renderiza error.tsx, retorna status 500
  if (item.corrupted) throw new Error("Dados corrompidos");

  return <div>{item.nome}</div>;
}

// notFound() é para "não encontramos o que você pediu"
// throw é para "algo quebrou enquanto processávamos seu pedido"
// A distinção importa para SEO, monitoramento e UX.

Navegação

Navegar em Next.js vai muito além de um simples link HTML. Este capítulo cobre todas as formas de mover o usuário pela aplicação — e as implicações de cada abordagem.

# Link — navegação declarativa

Quando um <Link> entra no viewport, o Next.js automaticamente prefetcha o código e os dados da rota de destino. Na maioria dos casos, quando o usuário clica, a transição é instantânea — os dados já estão disponíveis.

// usePathname retorna a URL atual — /sobre, /blog/meu-post, etc.
// Para rotas aninhadas, "startsWith" é mais adequado que "===":
// pathname.startsWith("/dashboard") — ativo em /dashboard e /dashboard/vendas

# useRouter — navegação programática

useRouter é necessário quando a navegação acontece como resultado de uma ação — submit de form, clique em botão, timeout — em vez de um link clicável. Como usa hooks, é exclusivo de Client Components.

use-router.tsx
"use client";

import { useRouter } from "next/navigation";

export function FormCadastro() {
  const router = useRouter();

  async function handleSubmit(e: React.FormEvent) {
    e.preventDefault();
    await criarConta(dados);

    // Navegar após ação bem-sucedida
    router.push("/dashboard");
  }

  return <form onSubmit={handleSubmit}>...</form>;
}
router-metodos.tsx
const router = useRouter();

// prefetch — prefetcha uma rota programaticamente

Em Server Components, Server Actions e Route Handlers, não há hooks — use redirect() e permanentRedirect() do next/navigation.

# Atualizar searchParams sem reload

Filtros, ordenação, paginação — tudo que vive na query string pode ser atualizado programaticamente sem recarregar a página, preservando o estado dos Client Components e atualizando os Server Components que dependem de searchParams.

update-search-params.tsx
"use client";

import { useRouter, usePathname, useSearchParams } from "next/navigation";

export function FiltroBusca() {
  const router = useRouter();
  const pathname = usePathname();
  const searchParams = useSearchParams();

  function setFiltro(chave: string, valor: string) {
    // Cria nova URLSearchParams baseada nos atuais
    const params = new URLSearchParams(searchParams.toString());
    params.set(chave, valor);

    // Atualiza a URL sem reload — re-renderiza Server Components
    router.replace(`${pathname}?${params.toString()}`);
  }

  return (
    <div className="flex gap-2">
      <button onClick={() => setFiltro("ordem", "recente")}>Mais recentes</button>
      <button onClick={() => setFiltro("ordem", "popular")}>Mais populares</button>
    </div>
  );
}

// Quando o URL muda via router.replace():
// - Server Components que usam searchParams re-renderizam com os novos valores
// - Client Components mantêm seu estado (inputs, dropdowns abertos, etc.)
// - Não há reload completo da página

Capítulo 08

Server vs Client Components

Este é o capítulo que separa quem usa Next.js de quem domina Next.js. A fronteira entre Server e Client Components não é "diferentes formas de renderizar" — é uma reorganização fundamental do que código faz, onde, e quando. Os erros mais comuns no App Router não são bugs de sintaxe; são confusões sobre essa fronteira. Investir tempo aqui economiza dezenas de horas adiante.

# O default server-first

Toda função React que você escreve dentro de app/ começa como Server Component. Não há diretiva, marcação, ou exportação especial. Você só precisa marcar — com "use client" no topo do arquivo — quando precisa que o componente rode no cliente.

default-server.tsx
// app/produtos/page.tsx — Server Component (default)
import { db } from "@/lib/db";

export default async function ProdutosPage() {
  const produtos = await db.produtos.findMany();
  return <Lista items={produtos} />;
}

// O que esse componente PODE fazer:
//   ✓ async/await direto na função
//   ✓ ler banco de dados
//   ✓ ler variáveis de ambiente (incluindo secrets)
//   ✓ ler arquivos do filesystem
//   ✓ chamar APIs internas com credenciais
//   ✓ importar bibliotecas pesadas (não vão para o cliente)
//
// O que esse componente NÃO PODE fazer:
//   ✗ useState, useEffect, useReducer, useContext
//   ✗ event handlers (onClick, onChange...)
//   ✗ acessar window, document, localStorage
//   ✗ usar APIs do browser (geolocation, IndexedDB...)
virando-cliente.tsx
"use client";  // declaração no topo do arquivo

import { useState } from "react";

export function Contador() {
  const [n, setN] = useState(0);
  return <button onClick={() => setN(n + 1)}>{n}</button>;
}

// O que esse componente PODE fazer:
//   ✓ tudo o que React clássico permite — useState, useEffect, hooks
//   ✓ event handlers
//   ✓ window, document, localStorage
//   ✓ bibliotecas client-only (TanStack Query, Framer Motion, ...)
//
// O que esse componente NÃO PODE fazer:
//   ✗ async direto na função (componente cliente não pode ser async)
//   ✗ acessar banco de dados
//   ✗ ler secrets (process.env.SECRET é undefined no cliente)
//   ✗ importar arquivos server-only sem causar erro

# "use client" é uma fronteira, não um marcador

O detalhe que mais confunde: "use client" não significa "este componente é cliente". Significa "deste arquivo para baixo na árvore de imports, tudo é cliente". É uma fronteira que você desenha — uma vez cruzada, todos os filhos importados a partir daquele ponto rodam no cliente também.

fronteira.tsx
// app/produto/[id]/botao-comprar.tsx
"use client";

import { useState } from "react";
import { Toast } from "./toast";          // implicitamente client
import { formatarMoeda } from "@/lib/utils";  // implicitamente client

export function BotaoComprar({ preco }) {
  const [aberto, setAberto] = useState(false);
  return (
    <>
      <button onClick={() => setAberto(true)}>
        Comprar por {formatarMoeda(preco)}
      </button>
      {aberto && <Toast />}
    </>
  );
}

// Toast.tsx NÃO precisa ter "use client".
// formatarMoeda em utils.ts NÃO precisa ter "use client".
// Como BotaoComprar (que tem "use client") os importa,
// eles AUTOMATICAMENTE viram client neste contexto.

// Você só precisa de "use client" no PONTO DE ENTRADA do "lado cliente".
A regra de ouro
Coloque "use client" o mais profundo possível na árvore. Se você adiciona ao layout raiz, todo o app vira cliente — perde Server Components inteiramente. Se adiciona em uma página, a página inteira vira cliente. O ideal é marcar apenas os componentes folha que de fato precisam de interatividade (botões, formulários, dropdowns), deixando o resto da árvore como Server. Esse é o padrão "ilhas de interatividade".

# A direção do import importa

Há uma regra que o Next.js impõe e que muita gente não percebe: Server Components podem importar Client Components, mas Client Components não podem importar Server Components. A direção é unidirecional, do servidor para o cliente.

direcao-import.tsx
// ✅ CERTO — Server importa Client

// app/page.tsx (Server)
import { Contador } from "./contador";  // Client

export default async function Page() {
  const dados = await db.findAll();
  return (
    <div>
      <h1>{dados.titulo}</h1>
      <Contador inicial={dados.contagem} />  {/* ok */}
    </div>
  );
}


// ❌ ERRADO — Client tenta importar Server

// contador.tsx
"use client";
import { ListaProdutos } from "./lista";  // ListaProdutos é Server (faz query no DB)

export function Contador() {
  return <ListaProdutos />;  // 💥 ListaProdutos vira client e quebra
}

# O padrão de composição — children e props

Quando você precisa renderizar um Server Component "dentro" de um Client Component (cenário comum: layout cliente com dados do servidor), a solução é composição via props, não import direto. O Server Component pai monta a árvore, passa o filho server como children/prop, e o Client renderiza-o no slot certo.

padrao-composicao.tsx
// modal-cliente.tsx — Client Component, controla abrir/fechar
"use client";

import { useState } from "react";

export function Modal({ children }) {
  const [aberto, setAberto] = useState(false);
  return (
    <>
      <button onClick={() => setAberto(true)}>Abrir</button>
      {aberto && (
        <div className="modal">
          {children}     {/* slot — não importa o tipo do filho */}
          <button onClick={() => setAberto(false)}>Fechar</button>
        </div>
      )}
    </>
  );
}

// app/page.tsx — Server Component compõe
import { Modal } from "./modal-cliente";
import { DetalhesUsuario } from "./detalhes-usuario";  // Server

export default async function Page() {
  return (
    <Modal>
      <DetalhesUsuario />   {/* Server Component como children — ok! */}
    </Modal>
  );
}

// O que acontece tecnicamente:
// 1. Server Component pai renderiza DetalhesUsuario (faz query no DB)
// 2. O HTML/RSC payload de DetalhesUsuario é serializado
// 3. Esse payload é passado como prop "children" para Modal
// 4. Modal (cliente) recebe e renderiza no DOM
//
// DetalhesUsuario nunca virou cliente. Ele rodou no servidor,
// produziu HTML, e o HTML viajou como dado para o Modal.

Essa técnica é tão importante que vale repetir a regra mental: o que você "importa" em um Client Component vira cliente; o que você "recebe" como children/prop preserva sua natureza original. Bibliotecas de UI bem desenhadas (shadcn/ui é o exemplo canônico) seguem esse padrão religiosamente — primitivos como Dialog, DropdownMenu, Sheet são clientes que aceitam children, permitindo que o conteúdo seja servidor.

# Props serializáveis — o contrato do boundary

Quando um Server Component passa props para um Client Component, essas props precisam atravessar a fronteira processual entre servidor e cliente. Por trás dos panos, são serializadas em JSON-like. Logo, há restrições sobre o que pode ser passado.

props-serializaveis.tsx
// ✅ PROPS QUE FUNCIONAM
<ClienteComp
  texto="olá"                          // strings, números, booleans
  numero={42}
  ativo={true}
  data={new Date()}                  // Date
  items={[1, 2, 3]}                // arrays
  objeto={{ a: 1, b: "x" }}          // objetos planos
  map={new Map([["k", 1]])}           // Map, Set
  children={<ServerComp />}            // JSX (incluindo Server Comps)
/>

// ❌ PROPS QUE NÃO FUNCIONAM
<ClienteComp
  funcao={() => salvar()}                   // funções (exceto Server Actions)
  classe={new UsuarioModel()}             // instâncias com métodos
  prisma={db.produtos}                    // objetos com closures
  handler={fetchHandler}                  // callbacks arbitrários
/>

// 💥 Erro em runtime: "Cannot serialize Function"
//    "Cannot serialize bigint" / etc.

# Server Actions — a exceção da regra

Há uma exceção elegante à regra "funções não podem ser passadas": Server Actions. Uma função marcada com "use server" pode ser passada como prop para Client Components — porque o que viaja não é a função em si, mas uma referência opaca que o cliente usa para invocar a função no servidor remotamente.

server-action-as-prop.tsx
// app/produtos/page.tsx — Server Component
async function deletarProduto(id: string) {
  "use server";
  await db.produtos.delete(id);
}

export default async function Page() {
  const produtos = await db.produtos.findMany();
  return (
    <ListaProdutos items={produtos} onDelete={deletarProduto} />
  );
}

// lista-produtos.tsx — Client Component pode chamar a action
"use client";

export function ListaProdutos({ items, onDelete }) {
  return items.map(p => (
    <li key={p.id}>
      {p.nome}
      <button onClick={() => onDelete(p.id)}>Deletar</button>
    </li>
  ));
}

// O onClick chama "onDelete", que é uma referência ao Server Action.
// O cliente envia uma requisição POST com o id; o servidor executa
// deletarProduto. Sem REST endpoint, sem fetch manual.

# Mapa mental de decisão

decisao.txt
## "Devo marcar este componente como cliente?"

PRECISO de useState, useEffect, useReducer, useContext?
  → SIM, cliente.

PRECISO de event handlers (onClick, onChange, onSubmit)?
  → SIM, cliente.

PRECISO acessar window, document, localStorage, navigator?
  → SIM, cliente.

  → SIM, cliente — ou crie wrapper "use client" e reexporte.

PRECISO renderizar texto/HTML com dados que vêm do servidor?
  → NÃO — Server Component.

PRECISO buscar dados (DB, fetch interno, leitura de arquivo)?
  → NÃO — Server Component (pode ser async).

PRECISO ler secrets (DB url, API keys privadas)?
  → NÃO — Server Component (e use server-only).

## Quando em dúvida:
  Comece como Server. Mude para cliente quando o erro aparecer.
  É mais fácil descobrir o que precisa de cliente do que descobrir
  que algo poderia ser servidor.
As cinco verdades a guardar
✓ Server é default; Client é exceção marcada com "use client".
✓ "use client" é uma fronteira, não um marcador. Tudo importado a partir dali vira cliente.
✓ Server Components podem importar Client, mas Client não pode importar Server. Use composição via children/props para passar Server dentro de Client.
✓ Props que cruzam a fronteira precisam ser serializáveis. Funções não passam — exceto Server Actions, que são a exceção desenhada.
✓ Marque "use client" o mais profundo possível na árvore. O padrão correto é "ilhas de interatividade" — paginas e layouts no servidor, botões e formulários no cliente.

Capítulo 09

Streaming e Suspense

No modelo tradicional de SSR, o servidor renderiza toda a página, espera todos os dados, e envia o HTML completo de uma vez. Com streaming, o servidor começa a enviar HTML enquanto ainda busca dados — pedaços da página chegam ao browser progressivamente. O React Suspense é a API que torna isso possível de forma declarativa: você define onde mostrar fallbacks, e o framework cuida do resto.

# O problema do "tudo ou nada"

problema-waterfall.tsx
// ❌ Sem streaming — servidor espera TUDO antes de enviar qualquer HTML

export default async function Dashboard() {
  const receita = await db.somaReceita();        // 200ms
  const clientes = await db.contarClientes();   // 150ms
  const vendas = await db.ultimasVendas();      // 800ms ← gargalo

  // O usuário espera 200+150+800 = 1150ms antes de VER qualquer coisa
  return (
    <div>
      <CardReceita valor={receita} />
      <CardClientes total={clientes} />
      <TabelaVendas dados={vendas} />
    </div>
  );
}

# Suspense — a solução declarativa

com-suspense.tsx
import { Suspense } from "react";

export default function Dashboard() {
  // Não é mais async! Os filhos é que são async.
  return (
    <div>
      {/* Cada seção streama independentemente */}
      <Suspense fallback={<SkeletonCard />}>
        <CardReceita />        {/* async, 200ms */}
      </Suspense>

      <Suspense fallback={<SkeletonCard />}>
        <CardClientes />       {/* async, 150ms */}
      </Suspense>

      <Suspense fallback={<SkeletonTabela />}>
        <TabelaVendas />       {/* async, 800ms */}
      </Suspense>
    </div>
  );
}

// O que acontece agora:
// t=0ms     → HTML com os 3 skeletons chega ao browser INSTANTANEAMENTE
// t=150ms   → CardClientes pronto → skeleton substituído
// t=200ms   → CardReceita pronto → skeleton substituído
// t=800ms   → TabelaVendas pronta → skeleton substituído
//
// O usuário VÊ a estrutura da página imediatamente.
// Cada pedaço aparece assim que está pronto, sem esperar os outros.
card-receita.tsx
// Cada componente async busca seus próprios dados
async function CardReceita() {
  const receita = await db.somaReceita();
  return (
    <div className="rounded-xl border p-6">
      <p className="text-sm text-zinc-500">Receita total</p>
      <p className="text-3xl font-bold">R$ {receita.toLocaleString()}</p>
    </div>
  );
}

// É um componente React normal — mas async.
// O Suspense pai envolve e mostra o fallback enquanto resolve.
Suspense vs loading.tsx — qual usar?
loading.tsx cria um Suspense automático na granularidade de rota — envolve a page inteira. <Suspense> manual permite granularidade fina — seções específicas da page. Use loading.tsx para o fallback da rota inteira (navegação entre páginas). Use <Suspense> manual para streaming de seções dentro de uma página.

# Streaming paralelo vs sequencial

paralelo-vs-sequencial.tsx
// ❌ SEQUENCIAL — cada await espera o anterior
async function Page() {
  const a = await buscarA();   // 300ms
  const b = await buscarB();   // 400ms — começa DEPOIS de A
  // Total: 700ms
}

// ✅ PARALELO COM Promise.all — ambos começam juntos
async function Page() {
  const [a, b] = await Promise.all([buscarA(), buscarB()]);
  // Total: ~400ms (mais lento dos dois)
  // Porém, NADA renderiza até ambos terminarem
}

// ✅✅ MELHOR — componentes async separados com Suspense
function Page() {
  return (
    <>
      <Suspense fallback={<SkeletonA />}>
        <ComponenteA />   {/* busca A internamente */}
      </Suspense>
      <Suspense fallback={<SkeletonB />}>
        <ComponenteB />   {/* busca B internamente */}
      </Suspense>
    </>
  );
}
// Total: A aparece em 300ms, B em 400ms — AMBOS paralelos
// E cada um aparece individualmente assim que pronto.
// O melhor de dois mundos: paralelismo + streaming progressivo.
A regra prática do streaming
Se dois componentes fazem queries independentes (um não depende do resultado do outro), coloque cada um em seu próprio <Suspense>. Eles executam em paralelo automaticamente e cada um streama assim que resolvido. Reserve Promise.all para quando precisar de ambos os dados antes de renderizar qualquer parte da seção.

# Streaming e SEO

SEO funciona com streaming
Crawlers modernos (Googlebot, Bingbot) entendem streaming e esperam a resposta completa. O HTML final que chega ao crawler inclui todo o conteúdo — não os skeletons. Isso significa que streaming não sacrifica SEO. Na prática, melhora: a TTFB é menor (o servidor começa a enviar mais cedo), e o LCP é melhor (conteúdo aparece progressivamente). Desde 2024, o Googlebot processa streaming HTTP nativamente.

Capítulo 10

Cache Components e PPR

Esta é a mudança mais profunda do Next.js 16 — e uma das mais sutis de absorver. Por anos, o modelo mental foi "rotas são estáticas por default; declare dinâmico quando precisar". Agora é o contrário: rotas são dinâmicas por default; declare cache quando quiser. Junto disso vem o PPR (Partial Prerendering): uma única página pode ter um shell estático que chega instantaneamente do CDN, com buracos dinâmicos que streamam depois. Não é mais "SSG vs SSR" — é uma terceira coisa, e ela só faz sentido quando você entende o porquê.

# A mudança de paradigma

No Next.js 13–14, o framework adivinhava: se o componente não usava cookies(), headers(), ou fetch sem cache, ele assumia que a rota era estática e fazia prerender no build. Se você usasse qualquer dessas APIs, a rota inteira virava dinâmica. Esse modelo era opt-out de cache — e produziu uma classe inteira de bugs em que uma chamada inocente quebrava o cache de uma página inteira sem aviso.

Em Next.js 16, com Cache Components habilitado, o default inverte. Tudo é dinâmico até você escrever 'use cache'. Não há mais adivinhação. A rota é dinâmica, ponto. Se você quer estático, marca explicitamente.

paradigma-antes-depois.tsx
// ❌ Next.js 13–14 (modelo antigo, "static by default")
// O framework analisa o componente e decide:
// - Sem cookies/headers/fetch dinâmico? → STATIC, prerender no build
// - Usa qualquer API dinâmica?         → DYNAMIC, render por request
//
// Problema: comportamento "magicamente" muda quando você adiciona
// uma chamada nova. Você acha que está cacheando, mas não está.

export default async function Page() {
  const dados = await fetch("https://api.exemplo.com/posts");
  return <PostList data={dados} />;
  // Era estático por default — surpresa quando alguém adicionava
  // uma feature de "usuário logado" e quebrava o cache.
}


// ✅ Next.js 16 (modelo novo, "dynamic by default")
// Sem ambiguidade: dynamic, sempre — a menos que você diga o contrário.

export default async function Page() {
  const dados = await fetch("https://api.exemplo.com/posts");
  return <PostList data={dados} />;
  // Dynamic. Sem surpresas. Para cachear: adicione 'use cache' no componente.
}
Cache Components (antes "ppr") — agora estável
Em Next.js 15 isso era experimental.ppr. Em 16 virou cacheComponents e é estável. O nome mudou para refletir o que a feature realmente é: não é só "renderização parcial", é uma reorganização completa do modelo de cache. Você opta em via flag de configuração — projetos novos ainda começam sem essa flag por compatibilidade. Quando ela está ligada, o comportamento default da aplicação inteira muda.

# Habilitando Cache Components

next.config.ts
import type { NextConfig } from "next";

const config: NextConfig = {
  // Liga o paradigma "dynamic by default" + PPR + 'use cache' nativo
  cacheComponents: true,
};

export default config;

Com a flag ligada, três coisas mudam ao mesmo tempo:

  • Default invertido — toda rota é dynamic até prova em contrário.
  • 'use cache' habilitado — você pode marcar funções, componentes ou rotas inteiras como cacheáveis.
  • PPR ativo — partes estáticas e dinâmicas da mesma página coexistem; o shell estático é servido do CDN, e os buracos dinâmicos streamam.

# O que é PPR — anatomia de uma página híbrida

Partial Prerendering é melhor entendido com uma imagem mental: a página tem um esqueleto estático que foi prerenderizado no build e está cacheado num CDN edge. Esse esqueleto contém o que não muda por usuário — header, navegação, layout, conteúdo público. Dentro desse esqueleto há "buracos" demarcados por <Suspense>: o carrinho do usuário, suas notificações, o feed personalizado. O servidor preenche esses buracos por request, streamando o conteúdo conforme fica pronto.

visualizacao-ppr.txt

│   │  vem do CDN        │  │  streama em ~150ms  │   │   dinâmico
│   └────────────────────┘  └─────────────────────┘   │
│                                                       │
│   ┌──────────────────────────────────────────────┐   │
│   │  FEED PERSONALIZADO                          │   │ ← outro

│                                                       │

└──────────────────────────────────────────────────────┘

  ┌─ 0ms     → Shell estático chega do CDN (header, nav, layout)
  ├─ 150ms   → CARRINHO chega via stream (server resolveu user.cart)
  └─ 400ms   → FEED chega via stream (server resolveu recommendations)

  Time-to-first-byte: ~30ms (CDN edge)
  Largest Contentful Paint: depende do conteúdo crítico (geralmente <500ms)

O resultado prático: páginas que antes tinham que escolher entre "estático e mentiroso" (mostra dados velhos) ou "dinâmico e lento" (espera tudo no servidor) agora podem ser "estático onde dá, dinâmico onde precisa". O CDN entrega o esqueleto em ~30ms; o servidor cuida só dos pedaços que realmente variam.

# connection() — o sinalizador de "isto é dinâmico"

Quando Cache Components está ligado, o framework precisa saber, em build time, quais partes da árvore podem ser prerenderizadas e quais não. A convenção é: qualquer <Suspense> cujo filho async não tem 'use cache' e tem uma chamada a connection() (ou usa cookies(), headers(), searchParams, etc.) marca aquele subárvore como dinâmica — e ela vira um buraco no shell.

app/page.tsx
import { Suspense } from "react";
import { connection } from "next/server";

// Componente STATIC — sem connection(), sem cookies, sem headers.
// Será prerenderizado no build e servido do CDN.
async function Produtos() {
  const produtos = await db.produtos.list();
  return <ProdutosGrid items={produtos} />;
}

// Componente DYNAMIC — connection() força o componente para o lado dinâmico.
// Vira um "buraco" no shell, é resolvido por request, streama via Suspense.
async function Carrinho() {
  await connection();  // "este componente é dinâmico"
  const itens = await getCarrinhoUsuario();
  return <CarrinhoView items={itens} />;
}

export default function Page() {
  return (
    <div className="grid grid-cols-2 gap-8">
      {/* Estático — vai pro shell prerenderizado */}
      <Produtos />

      {/* Dinâmico — Suspense é o "limite" do buraco */}
      <Suspense fallback={<CarrinhoSkeleton />}>
        <Carrinho />
      </Suspense>
    </div>
  );
}
Por que connection() existe?
Em build time, o framework não pode simplesmente tentar renderizar um componente que depende de cookies — isso quebraria o build. Ele precisa de um sinal claro de "este componente requer um request real". connection() é esse sinal: ela retorna uma Promise que só resolve em runtime, com a request ativa. No build, o componente não tenta executar; vira um placeholder. Em runtime, ela resolve, o componente roda e streama. APIs como cookies(), headers() e searchParams têm o mesmo efeito automaticamente — connection() é o gancho explícito quando você quer marcar dinâmico sem usar nenhuma delas.

# 'use cache' — o oposto de connection()

Se connection() diz "isto é dinâmico", 'use cache' diz "isto é estático e pode ser memorizado". A diretiva pode ir em três lugares: no topo de uma função, no topo de um componente, ou no topo de um arquivo de rota inteiro.

use-cache-overview.tsx
// 1. 'use cache' em FUNÇÃO — qualquer chamada com mesmos args retorna cache
async function getProduto(id: string) {
  "use cache";
  const produto = await db.produto.find(id);
  return produto;
}


// 2. 'use cache' em COMPONENTE — toda a árvore renderizada vai pro cache
async function ProdutoCard({ id }: { id: string }) {
  "use cache";
  const produto = await getProduto(id);
  return <Card data={produto} />;
}


// 3. 'use cache' em ARQUIVO DE ROTA — a página inteira é cacheada
//    (volta ao comportamento "estático" do Next 13–14, mas explícito)
"use cache";

export default async function Page() {
  const posts = await getPosts();
  return <PostList posts={posts} />;
}
A simetria mental
Default = dinâmico. Você não escreve nada e a coisa é dinâmica.
'use cache' = vira estático. Marca explícita de "memoriza isso".
connection() dentro de 'use cache' = erro de build. Você não pode pedir cache de algo que depende do request — o framework reclama na sua cara.
A simetria é tão limpa que muita gente reclamava do modelo antigo só por não ter ela.

Data Fetching no App Router

Com o App Router, buscar dados é surpreendentemente simples: você escreve async/await diretamente no componente, sem hooks, sem stores, sem boilerplate. O framework cuida de deduplicação, paralelismo e integração com o sistema de cache. Este capítulo cobre os padrões práticos que você vai usar toda semana — de um fetch simples a múltiplas fontes paralelas — e as opções que controlam o comportamento de cache do fetch nativo.

# fetch em Server Components

O padrão fundamental é direto: torne o componente async e use await normalmente. O Next.js estende o fetch global com opções extras de cache.

app/posts/page.tsx
// Server Component — async por padrão
export default async function PostsPage() {

  if (!res.ok) {

  }

  const posts: Post[] = await res.json();

  return (
    <ul>

// Não precisa de try/catch se você prefere o boundary declarativo.

Acesso direto ao banco de dados

Como Server Components rodam no servidor, você pode chamar seu ORM, pool de SQL ou qualquer cliente de banco diretamente — sem expor credenciais ao cliente, sem criar uma rota de API intermediária.

app/usuarios/page.tsx
import { db } from "@/lib/db";  // Prisma, Drizzle, postgres.js...

export default async function UsuariosPage() {
  // Query direta — sem API, sem serialização desnecessária
  const usuarios = await db.usuario.findMany({
    orderBy: { nome: "asc" },
    take: 20,

Server Component vs Route Handler para dados
Use Server Component quando os dados servem apenas para renderizar aquela página/componente. Use Route Handler (app/api/.../route.ts) quando você precisa de um endpoint HTTP reutilizável — para consumir de um Client Component com fetch, de um app mobile, ou de um serviço externo. Não crie route handlers "por hábito" só para passar dados a um Server Component; isso adiciona latência de rede desnecessária.

# Opções de cache do fetch

O fetch no Next.js aceita um segundo parâmetro com a chave next para controlar cache e revalidação, além das opções nativas da Fetch API para cache.

opcoes-fetch.ts
// 1. Dynamic — sem cache (default com cacheComponents: true)
//    Roda a cada request, nunca armazena.
const res = await fetch(url);


// 2. Cache permanente — "static", igual ao getStaticProps de antes
//    Armazenado indefinidamente (até revalidação manual).
const res = await fetch(url, { cache: "force-cache" });


// 3. ISR — revalida por tempo (em segundos)
//    Cache serve por N segundos, depois roda de novo em background.
const res = await fetch(url, { next: { revalidate: 60 } });


// 4. On-demand — marca com tag para invalidar depois
//    Serve do cache até alguém chamar revalidateTag("posts").
const res = await fetch(url, { next: { tags: ["posts"] } });


// 5. No-store — nunca armazena, nem no Router Cache
//    Útil para dados de autenticação, dados super-frescos.
const res = await fetch(url, { cache: "no-store" });
fetch com cacheComponents ligado
Com cacheComponents: true, o fetch sem opção é dinâmico por default (equivalente a cache: "no-store"). Para cachear, você precisa ser explícito com force-cache, next.revalidate, ou — de preferência — colocar a função que chama o fetch dentro de 'use cache'. Nos projetos sem a flag (legado), o default ainda é force-cache. Sempre verifique qual modo está ativo antes de assumir comportamento.

# Request memoization

O React automaticamente deduplica chamadas fetch idênticas na mesma árvore de renderização de um request. Isso significa que você pode chamar a mesma função de busca em vários componentes sem preocupação com múltiplas requisições de rede — o React executa uma só e compartilha o resultado.

memoizacao-request.tsx
// lib/queries.ts — função de query centralizada
async function getUsuario(id: string) {
  const res = await fetch(`https://api.exemplo.com/usuarios/${id}`);
  return res.json();
}

// Componente A chama getUsuario("42")
async function Avatar({ userId }: { userId: string }) {
  const user = await getUsuario(userId);
  return <img src={user.avatarUrl} alt={user.nome} />;
}

// Componente B chama getUsuario("42") — mesma URL, mesmo request
async function HeaderUsuario({ userId }: { userId: string }) {
  const user = await getUsuario(userId);
  return <span>Olá, {user.nome}</span>;
}

// O React executa o fetch UMA VEZ para o mesmo request HTTP.
// Avatar e HeaderUsuario recebem o mesmo objeto — zero requisições duplicadas.
//
// Escopo: por render tree / por request. Não persiste entre requests.
// Para cache persistente, use 'use cache' ou fetch com next.revalidate.
Memoização com clientes de banco (Prisma, Drizzle)
A memoização automática só funciona com fetch. Para queries de banco, use React.cache() para embrulhar a função — isso dá a mesma garantia de deduplicação para qualquer função assíncrona, não só fetch.

import { cache } from "react";
import { db } from "@/lib/db";

// React.cache() deduplica por argumentos, dentro do mesmo request
export const getUsuario = cache(async (id: string) => {
  return db.usuario.findUnique({ where: { id } });
});

export const getPosts = cache(async (autorId: string) => {
  return db.post.findMany({
    where: { autorId },
    orderBy: { criadoEm: "desc" },
  });
});

// Múltiplos componentes chamam getUsuario("42") no mesmo request?
// Prisma executa UMA consulta. React.cache cuida da deduplicação.

# Paralelo vs sequencial

O erro mais comum em data fetching com async/await é criar waterfalls involuntários: cada await bloqueia o próximo, mesmo quando as chamadas são independentes. A regra é simples: se duas chamadas não dependem uma da outra, dispare-as em paralelo.

paralelo-vs-sequencial.tsx
// ❌ Sequencial — waterfall desnecessário
async function DashboardSequencial() {
  const usuario = await getUsuario();   // 100ms — espera
  const posts   = await getPosts();    // 200ms — espera o de cima
  const stats   = await getStats();    // 150ms — espera os dois
  // Total: 450ms 😞

  return <div>...</div>;
}


// ✅ Paralelo — Promise.all para chamadas independentes
async function DashboardParalelo() {
  const [usuario, posts, stats] = await Promise.all([
    getUsuario(),
    getPosts(),
    getStats(),
  ]);
  // Total: ~200ms (o mais lento dos três) ✅

  return <div>...</div>;
}


// ✅ Sequencial quando há dependência real
async function PostComAutor({ postId }: { postId: string }) {
  const post = await getPost(postId);           // precisa do post primeiro
  const autor = await getUsuario(post.autorId);  // depende do post.autorId

  return <Article post={post} autor={autor} />;
}

Paralelo com Suspense — o padrão mais escalável

Uma alternativa ao Promise.all: cada componente busca seus próprios dados, e você os coloca em <Suspense> paralelos. Cada um começa a buscar imediatamente quando renderiza, sem esperar os outros. Isso também distribui os skeletons de forma granular.

paralelo-com-suspense.tsx
import { Suspense } from "react";

// Cada componente async busca os próprios dados

}

  return <PostList items={posts} />;
}

async function GraficoStats() {
  const stats = await getStats();   // 150ms

}

// Página pai — NÃO é async, não faz await de nada
export default function Dashboard() {
  return (
    <div className="grid grid-cols-3 gap-6">
      {/* Os três fetches começam em paralelo quando React renderiza os filhos */}
      <Suspense fallback={<Skeleton />}><CardUsuario /></Suspense>

//            t=100ms → CardUsuario aparece
//            t=150ms → GraficoStats aparece
//            t=200ms → ListaPosts aparece
// Cada seção tem seu próprio skeleton, some individualmente.

Capítulo 12

Server Actions

Server Actions são, para mutações, o que Server Components são para leitura: código de servidor escrito como função normal, chamado diretamente do cliente, sem rota HTTP intermediária visível. Você define uma função, marca com 'use server', e ela vira um endpoint POST automático que pode ser passado como prop, atribuído a um <form action={…}> ou chamado de um event handler. Isso elimina três coisas de uma vez: a route handler de mutação, o fetch client-side com manipulação de estado, e a invalidação manual de cache. Mas também muda o modelo mental — funções viram superfície de ataque, e dá para errar feio se você não souber.

# O que Server Actions substituem

Compare o fluxo de "criar um post" no React tradicional com o equivalente Server Action. A coisa antiga tinha quatro peças móveis: o componente cliente com useState e fetch, o route handler de API, a lógica de revalidação manual, e o gerenciamento de loading/erro. Server Actions colapsam isso em uma função.

antes-vs-depois.tsx
// ❌ ANTES (React clássico) — quatro arquivos, três modelos mentais

// 1. app/api/posts/route.ts — handler de API
export async function POST(req: Request) {
  const body = await req.json();
  const post = await db.post.create({ data: body });
  return Response.json(post);
}

// 2. components/post-form.tsx — formulário cliente com toda parafernália
"use client";
function PostForm() {
  const [titulo, setTitulo] = useState("");
  const [loading, setLoading] = useState(false);

    finally { setLoading(false); }
  }
  // ... 20 linhas de JSX
}

# A diretiva 'use server' — duas formas

A diretiva pode ir em dois lugares, e a diferença é mais conceitual do que sintática. No topo de um arquivo, todas as exportações async daquele arquivo viram Server Actions. Dentro de uma função (Server Component), torna aquela função específica uma Server Action.

duas-formas.tsx
// FORMA 1: arquivo dedicado de actions
// app/posts/actions.ts
"use server";

export async function criarPost(data: FormData) { /* ... */ }
export async function deletarPost(id: string) { /* ... */ }
export async function editarPost(id: string, data: FormData) { /* ... */ }

// Todas as três viram Server Actions. Convenção comum: agrupar por feature.


// FORMA 2: ação inline em Server Component
// app/posts/page.tsx
export default async function PostsPage() {
  const posts = await getPosts();

  async function deletar(formData: FormData) {
    "use server";  // dentro da função, não no topo do arquivo
    const id = formData.get("id") as string;
    await db.post.delete({ where: { id } });
    revalidatePath("/posts");
  }

  return (
    <ul>
      {posts.map((p) => (
        <li key={p.id}>
          {p.titulo}
          <form action={deletar}>
            <input type="hidden" name="id" value={p.id} />
            <button>Deletar</button>
          </form>
        </li>
      ))}
    </ul>
  );
}
Toda Server Action é um endpoint público
Quando você marca uma função com 'use server', o framework gera um endpoint POST acessível por qualquer pessoa que conheça (ou descubra) o ID gerado dela. Não confie em "ah, mas o botão só aparece para admin" — o endpoint existe independentemente da UI. Toda Server Action deve validar input e checar autorização internamente, como se fosse uma rota de API exposta. Mais sobre isso ao final do capítulo.

# Padrão 1 — Form action (progressive enhancement)

Atribuir uma Server Action ao action de um <form> é o padrão mais elegante: o form funciona mesmo sem JavaScript. O React intercepta o submit no cliente quando o JS está disponível (sem reload de página), e cai num POST tradicional quando não está.

form-action-basico.tsx
// app/comentarios/actions.ts
"use server";
import { revalidatePath } from "next/cache";

export async function adicionarComentario(postId: string, formData: FormData) {
  const texto = formData.get("texto") as string;
  if (!texto.trim()) return;

  await db.comentario.create({ data: { postId, texto } });
  revalidatePath(`/posts/${postId}`);
}


// app/posts/[id]/page.tsx — passa argumentos extras com .bind()
import { adicionarComentario } from "@/app/comentarios/actions";

export default async function PostPage({ params }: Props) {
  const { id } = await params;
  const post = await getPost(id);

  // .bind() pré-aplica o postId — o form só envia o texto
  const adicionarAoPost = adicionarComentario.bind(null, id);

  return (
    <article>
      <h1>{post.titulo}</h1>
      <ListaComentarios postId={id} />

      <form action={adicionarAoPost}>
        <textarea name="texto" required />
        <button>Comentar</button>
      </form>
    </article>
  );
}

// O atributo action recebe a função diretamente — React serializa
// a referência para um endpoint POST. O navegador chama esse endpoint
// no submit. Sem JS: form HTML normal. Com JS: refresh sem reload.

# Padrão 2 — Chamada programática

Server Actions não precisam estar atreladas a forms. Você pode chamá-las de qualquer event handler em Client Components — útil para deletes, toggles, ações que disparam de cliques em ícones, etc.

programatico.tsx
"use client";
import { deletarPost } from "./actions";
import { useTransition } from "react";

export function BotaoDeletar({ id }: { id: string }) {
  const [pending, startTransition] = useTransition();

  async function handleDelete() {
    if (!confirm("Tem certeza?")) return;

    // startTransition mantém a UI responsiva e ativa o estado pending

      await deletarPost(id);
    });
  }

  return (
    <button onClick={handleDelete} disabled={pending}>
      {pending ? "Deletando..." : "Deletar"}
    </button>
  );
}

# Validação com Zod

Validar input em Server Actions não é opcional — é a primeira linha de defesa. Zod é a escolha quase universal: ele valida e infere tipos no mesmo passo, e tem ergonomia limpa para extrair erros de campo.

app/posts/actions.ts
"use server";
import { z } from "zod";
import { revalidatePath } from "next/cache";
import { redirect } from "next/navigation";

const PostSchema = z.object({
  titulo: z.string().min(3, "Mínimo 3 caracteres").max(120),
  conteudo: z.string().min(10, "Conte mais que isso"),
  publicado: z.coerce.boolean().default(false),
});

export type FormState = {
  erro?: string;
  erros?: Record<string, string[]>;
  sucesso?: boolean;
};

export async function criarPost(
  estadoAnterior: FormState,
  formData: FormData,
): Promise<FormState> {
  // 1. Autorização — sempre primeiro
  const sessao = await getSessao();
  if (!sessao) return { erro: "Faça login" };

  // 2. Validação
  const parsed = PostSchema.safeParse({
    titulo: formData.get("titulo"),
    conteudo: formData.get("conteudo"),
    publicado: formData.get("publicado"),
  });

  if (!parsed.success) {
    return { erros: parsed.error.flatten().fieldErrors };
  }

  // 3. Mutação
  const post = await db.post.create({
    data: { ...parsed.data, autorId: sessao.userId },
  });

  // 4. Invalidação + redirect
  revalidatePath("/posts");
  redirect(`/posts/${post.id}`);  // não retorna — redirect lança erro especial
}
redirect() funciona por exception
Tanto redirect() quanto notFound() não retornam — eles lançam um erro especial que o framework intercepta. Por isso você não precisa de return redirect(...). Mas nunca coloque redirect() dentro de um try/catch sem rethrow: o catch vai engolir a exception e quebrar o redirecionamento. A regra é: chame fora de try/catch, ou faça catch (e) {`{`} if (e instanceof RedirectError) throw e; ... {`}`}.

# useOptimistic — UI otimista

Para ações que devem parecer instantâneas (curtir um post, marcar um todo como feito), useOptimistic permite atualizar a UI antes da resposta do servidor. Se a action falhar, o React reverte automaticamente.

components/like-button.tsx
"use client";
import { useOptimistic, startTransition } from "react";
import { toggleLike } from "./actions";

type Props = {
  postId: string;
  likes: number;
  curtido: boolean;
};

export function LikeButton({ postId, likes, curtido }: Props) {
  const [otim, setOtim] = useOptimistic(
    { likes, curtido },
    (state) => ({
      curtido: !state.curtido,
      likes: state.likes + (state.curtido ? -1 : 1),
    }),
  );

  async function handleClick() {
    startTransition(async () => {
      setOtim(null);          // aplica o update otimista
      await toggleLike(postId);  // se falhar, React reverte
    });
  }

  return (
    <button onClick={handleClick} aria-pressed={otim.curtido}>
      {otim.curtido ? "❤️" : "🤍"} {otim.likes}
    </button>
  );
}

// Clique → coração muda imediatamente, contador soma 1.
// Servidor responde em 300ms → estado real bate com o otimista, nada muda.
// Servidor falha → React reverte para o estado original automaticamente.

# revalidatePath e revalidateTag

Após mutar dados, o cache da página afetada está obsoleto. Você diz ao Next o que invalidar — ele cuida do resto na próxima navegação. revalidatePath invalida por rota; revalidateTag invalida por tag aplicada a fetches/queries em 'use cache'.

revalidacao.ts
"use server";
import { revalidatePath, revalidateTag } from "next/cache";

export async function atualizarPost(id: string, data: FormData) {
  await db.post.update({ where: { id }, data });

  // Invalida uma rota específica
  revalidatePath(`/posts/${id}`);

  // Invalida um layout (e tudo aninhado nele)
  revalidatePath("/posts", "layout");

  // Invalida tudo marcado com a tag "posts" — independente de rota
  revalidateTag("posts");
}

Cap. 14 trata de revalidação a fundo. Por ora, a regra prática: toda Server Action que muta dados deve terminar com pelo menos um revalidate* — caso contrário, o usuário vê dados velhos na próxima navegação.

# Segurança — o que não dispensar

Checklist de segurança para toda Server Action
1. Autenticação primeiro. A primeira coisa dentro da action é checar se há sessão válida. Sem sessão, retorne erro ou jogue exception.
2. Autorização por recurso. Não basta ser logado — o usuário tem permissão nesse recurso? if (post.autorId !== sessao.userId) throw...
3. Validação de input com Zod. Tipos do TypeScript não existem em runtime. FormData traz strings ou Files. Valide tudo.
4. Não confie em hidden inputs. Um <input type="hidden" name="userId" value={user.id} /> pode ser modificado pelo navegador. Pegue o user da sessão, sempre.
5. Não jogue stack traces para o cliente. Em produção, retorne mensagens genéricas. Logue o erro real no servidor.
6. Rate limiting em ações sensíveis (login, criação de conteúdo) — use Upstash Ratelimit ou similar.
7. Considere CSRF. Server Actions têm proteção via origin check por default. Se você expor a action a outro domínio, pense bem.

Capítulo 13

Caching com 'use cache'

Se o capítulo 10 introduziu o paradigma "dynamic-by-default", este é o capítulo do opt-in explícito. 'use cache' é como você reconquista cache no novo modelo — uma diretiva que transforma uma função, um componente, ou um arquivo inteiro numa unidade memorizável. Mas há nuances que diferenciam um sistema de cache que funciona de um que vaza dados de um usuário para outro: o que vira cache key, quando o cache expira, como invalidar seletivamente, e o que nunca deve estar dentro da fronteira do cache. Esses são os pontos em que vale dedicar atenção.

# O contrato da diretiva

Quando você marca uma função com 'use cache', você está prometendo ao framework: "o output desta função depende exclusivamente dos argumentos que ela recebe". Em troca, o framework promete: "chamadas com os mesmos argumentos retornarão o mesmo resultado memorizado, sem reexecutar". Esse contrato é determinístico e referencial — e quebrá-lo silenciosamente é a fonte da maioria dos bugs com cache.

contrato.ts
// ✅ Função pura — output depende só dos argumentos
async function getProduto(id: string) {
  "use cache";
  return db.produto.findUnique({ where: { id } });
}
// id é o único input → id vira a cache key.
// getProduto("42") sempre retorna o mesmo resultado (até invalidar).


// ❌ Função impura — depende de coisa externa não declarada
async function getPostsRecentes() {
  "use cache";
  const agora = new Date();  // ← timestamp do MOMENTO da primeira chamada
  return db.post.findMany({
    where: { criadoEm: { gte: new Date(agora.getTime() - 3600_000) } },
  });
}
// "Recentes" continua mostrando os mesmos posts pra sempre.
// O cache memorizou o resultado da PRIMEIRA chamada com aquele "agora".
// Quem chama hoje vê posts de uma hora atrás de uma semana atrás.


// ✅ Versão correta — passe a "data corte" como argumento
async function getPostsDesde(desde: Date) {
  "use cache";
  return db.post.findMany({ where: { criadoEm: { gte: desde } } });

// Quem chama escolhe o corte, e o argumento entra na cache key.

# Os três escopos — função, componente, rota

tres-escopos.tsx
// 1️⃣ ESCOPO FUNÇÃO — granular, ideal para queries reutilizadas

  "use cache";

}

  "use cache";
  const categoria = await getCategoria(slug);
  const destaque   = await getProdutoDestaque(slug);

      <h3>{categoria.nome}</h3>
      <ProdutoMini item={destaque} />
    </Card>
  );
}
// O HTML renderizado (não os dados, o JSX serializado) vira cache.
// Mesmo slug → mesmo HTML reaproveitado, sem renderizar de novo.


// 3️⃣ ESCOPO ROTA — página inteira como página estática
// app/blog/page.tsx

  const posts = await getPosts();
  return <PostList posts={posts} />;

// Equivalente ao "static" de Next 13: a página inteira é gerada uma vez,
// servida do CDN para todo mundo. Use para landing, docs, blog público.
Granularidade — qual escopo escolher
A regra empírica: função para queries reutilizadas em vários lugares (cabeçalhos compartilhados, lookups por ID); componente quando a unidade lógica de cache é a árvore renderizada (cards, listas, widgets); rota inteira só para páginas que são genuinamente as mesmas para todo mundo (blog, docs, landing). Quando em dúvida, comece com função — é o escopo mais composável e mais fácil de invalidar.

# cacheLife() — duração e revalidação

Por default, 'use cache' usa o perfil "default" — boa para a maioria dos casos. Para controle fino, importe cacheLife e chame dentro da função (depois da diretiva). Você pode passar um perfil pré-definido ("minutes", "hours", "days", "weeks", "max") ou um objeto custom.

cache-life.ts
import { unstable_cacheLife as cacheLife } from "next/cache";

// Perfis pré-definidos (cobrem 90% dos casos)
async function getCotacao() {
  "use cache";
  cacheLife("minutes");  // dados financeiros — alguns minutos
  return fetchCotacaoUSD();
}

async function getNoticiasDoDia() {
  "use cache";
  cacheLife("hours");     // notícias — algumas horas
  return fetchNoticias();
}

async function getCategoriasMenu() {
  "use cache";
  cacheLife("days");      // estrutura de menu — dias
  return db.categoria.findMany();
}

async function getDocumentacao(versao: string) {
  "use cache";
  cacheLife("max");       // docs de versão fechada — pra sempre
  return fetchDocs(versao);
}


// Perfil customizado — três janelas independentes
async function getInventario(sku: string) {
  "use cache";
  cacheLife({
    stale:      60,      // 60s servindo do cache do cliente sem checar
    revalidate: 300,     // 5min até refazer em background
    expire:     3600,    // 1h hard expiry — depois disso, espera nova
  });
  return fetchInventario(sku);
}
Os três tempos de cacheLife — vale entender
stale — janela em que o cache é servido como está, sem qualquer trabalho. Resposta máxima velocidade.
revalidate — depois de stale, o cache ainda é servido, mas o framework dispara revalidação em background. Ninguém espera.
expire — depois de expire, o cache é considerado morto. A próxima request espera a regeneração antes de ver resposta.
Pense numa cebola: cliente CDN (stale) → cache do servidor com background refresh (revalidate) → cache totalmente expirado (expire).

# cacheTag() — invalidação seletiva

Tags são rótulos arbitrários atrelados a uma entrada de cache. Mais tarde, revalidateTag(tag) invalida todas as entradas com aquela tag, em qualquer função, em qualquer escopo. É o mecanismo central para invalidação dirigida — sem precisar saber onde o dado está cacheado.

cache-tag.ts
import { unstable_cacheTag as cacheTag } from "next/cache";

export async function getPost(id: string) {
  "use cache";
  cacheLife("hours");
  // Tag genérica + tag específica do post
  cacheTag("posts", `post:${id}`);

  return db.post.findUnique({ where: { id } });
}

export async function getPostsDoAutor(autorId: string) {
  "use cache";
  cacheLife("hours");
  cacheTag("posts", `autor:${autorId}`);

  return db.post.findMany({ where: { autorId } });
}


// Em uma Server Action de update:
"use server";
import { revalidateTag } from "next/cache";

export async function atualizarPost(id: string, data: FormData) {
  const post = await db.post.update({ where: { id }, data });

  // Invalida só ESTE post — getPost("42") refaz, getPost("43") continua cacheada
  revalidateTag(`post:${id}`);

  // Invalida lista do autor — porque a lista contém este post
  revalidateTag(`autor:${post.autorId}`);
}

// Para invalidar tudo: revalidateTag("posts") — usar com moderação.
Estratégia de tags — pense em "famílias"
Tags funcionam melhor quando você as desenha em camadas: uma tag geral ("posts") e uma tag específica (`post:${`{id}`}`). Updates pontuais invalidam a tag específica; rebuilds em massa (ex: limpar cache após mudança de schema) invalidam a geral. Inclua a relação de "pai" também: ao atualizar um comentário, invalide tanto comment:42 quanto post:7:comments — porque a página do post mostra a lista de comentários.

# O que NUNCA pode estar dentro de 'use cache'

O contrato de cache fica óbvio quando você lista o que o quebra. Se uma função cacheada lê algo que varia fora dos seus argumentos, o resultado é vazamento de dados — o que era para um usuário acaba retornando para outro. O framework bloqueia esses casos em build, mas vale entender o porquê.

erros-bloqueados.tsx
// ❌ ERRO — cookies() é dinâmico, depende do request
async function getDashboardUsuario() {
  "use cache";
  const sessao = (await cookies()).get("sessao");  // 💥 Build error
  return fetchDashboard(sessao);
}
// "Cannot use cookies() inside 'use cache' boundary"


// ❌ ERRO — connection() declara dinâmico, conflita com cache
async function getQualquerCoisa() {
  "use cache";
  await connection();  // 💥 Build error — não dá para ser dinâmico E cacheado
  return fetchAlgo();
}


// ❌ ERRO — headers() depende do request
async function getConteudoLocalizado() {
  "use cache";
  const aceita = (await headers()).get("accept-language");  // 💥
  return fetchConteudo(aceita);
}


// ✅ Padrão correto: passe o que varia como argumento
async function getConteudoLocalizado(idioma: string) {
  "use cache";
  cacheTag(`conteudo:${idioma}`);
  return fetchConteudo(idioma);
}

// E o caller, fora do cache, lê o que precisa do request:
async function PaginaInicial() {
  const aceita = (await headers()).get("accept-language") ?? "pt";
  const idioma = aceita.slice(0, 2);
  const conteudo = await getConteudoLocalizado(idioma);
  return <Conteudo data={conteudo} />;
}

# Argumentos como cache key — regras de serialização

Cada combinação única de argumentos vira uma entrada separada no cache. O framework serializa os argumentos para gerar a chave — o que impõe restrições sobre o que pode ser passado.

argumentos-validos.ts
// ✅ Permitido — primitivos, objetos planos, arrays, Date
async function buscar(
  id: string,
  filtros: { categoria?: string; precoMax?: number },
  desde: Date,
) {
  "use cache";
  /* ... */
}


// ❌ Proibido — funções, classes complexas, símbolos, instâncias de DB clients
async function buscarMal(
  comparador: (a, b) => number,  // 💥 não serializável
  client: PrismaClient,             // 💥 não serializável
) {
  "use cache";
  /* ... */
}
// "Cannot serialize argument of type 'function' as cache key"


// ⚠️ Cuidado — ordem das chaves importa para a chave gerada
await buscar(id, { categoria: "x", precoMax: 100 }, agora);
await buscar(id, { precoMax: 100, categoria: "x" }, agora);
// Em teoria são equivalentes, mas a serialização canônica do Next.js
// resolve isso ordenando chaves alfabeticamente. Ainda assim, prefira
// formatos consistentes nos call sites para reduzir confusão.

Erros comuns que custam horas
✗ Capturar new Date(), Math.random() ou variáveis de closure dentro de 'use cache' — fica congelado no primeiro valor.
✗ Esquecer cacheTag e tentar invalidar depois — sem tag, só dá para invalidar por revalidatePath ou esperar expire.
✗ Usar a mesma tag para coisas não relacionadas — uma invalidação derruba mais do que devia. Tags devem ser específicas.
✗ Pôr autenticação dentro de 'use cache' — risco de vazar dados entre usuários. Auth fica fora; o que entra na função são IDs/escopos como argumentos.
✗ Cachear queries que dependem do "tempo agora" sem passar a janela como argumento — ver primeiro exemplo do capítulo.
Heurística de adoção
✓ Comece sem 'use cache' em lugar nenhum. Tudo dinâmico.
✓ Identifique queries que aparecem em vários lugares e mudam pouco — adicione 'use cache' nelas primeiro (escopo função).
✓ Adicione cacheLife baseado em quão fresca a informação precisa ser. Default é seguro para a maioria.
✓ Adicione cacheTag assim que houver Server Action que muta esse dado — invalidação dirigida vence delay.
✓ Suba para escopo componente quando perceber que o JSX é mais caro de renderizar do que de buscar dados.
✓ Suba para escopo rota só em páginas 100% públicas e raramente alteradas.
✓ Em produção, monitore taxa de cache hit. Páginas com baixa taxa de hit podem não merecer estar cacheadas — talvez sejam dinâmicas demais.
Capítulo 14

Revalidação de cache

Cachear dados resolve latência. Revalidar dados mantém o cache honesto. O Next.js oferece dois modelos ortogonais: time-based (revalida após N segundos, sem intervenção manual) e on-demand (você decide quando algo está obsoleto e dispara a invalidação). Entender quando usar cada um — e como os dois se compõem — é o que separa um sistema de cache que "funciona na demo" de um que funciona em produção com dados reais mutando o tempo todo.

# Os dois modelos — visão geral

modelos.txt
TIME-BASED (passivo)
────────────────────────────────────────────────────
  Cache serve a resposta.
  Após N segundos (revalidate), refaz em background.
  Após M segundos (expire), força esperar nova geração.

  Configurado em:
    cacheLife({ revalidate: N, expire: M })      ← 'use cache'
    fetch(url, { next: { revalidate: N } })      ← fetch puro
    export const revalidate = N                  ← rota inteira

  Bom para: dados que mudam com frequência previsível,
  feeds de terceiros, preços que atualizam a cada hora, etc.


ON-DEMAND (ativo)
────────────────────────────────────────────────────
  Cache serve a resposta por tempo indeterminado.
  Você chama revalidateTag() ou revalidatePath()
  quando sabe que o dado mudou.

  Chamado em:
    Server Actions (após mutação de dados)
    Route Handlers com webhook de CMS/plataforma
    scripts de admin / migração

  Bom para: dados que mudam por ação do usuário,
  publicação de conteúdo, deploys com dado novo, etc.


COMBINADOS
────────────────────────────────────────────────────
  Nada impede usar os dois juntos:
  revalidateTag("posts") invalida agora,
  E cacheLife({ revalidate: 3600 }) garante
  que mesmo sem invalidação explícita o cache
  não fica mais de 1h obsoleto.

# revalidatePath

revalidatePath invalida o cache de uma ou mais rotas pelo caminho. Pode invalidar uma página específica, um layout inteiro (que engloba todos os segmentos abaixo dele), ou todas as rotas de um tipo.

revalidate-path.ts
import { revalidatePath } from "next/cache";

// 1. Invalidar uma página específica
revalidatePath("/blog/post-sobre-nextjs");
// Apenas /blog/post-sobre-nextjs é invalidada.

// 2. Invalidar um segmento dinâmico específico
revalidatePath(`/produtos/${id}`);
// Apenas /produtos/42 é invalidada (se id === "42").

// 3. Invalidar um layout — afeta o layout E tudo aninhado nele
revalidatePath("/blog", "layout");
// /blog, /blog/[slug], /blog/categoria/[cat] — todos invalidados.
// Use quando o dado afeta o layout (ex: contador no header).

// 4. Invalidar uma rota de página específica (default quando não passa tipo)
revalidatePath("/blog", "page");
// Apenas a page.tsx de /blog. Layouts e filhos continuam.

// 5. Invalidar TUDO (uso extremo — prefira tags)
revalidatePath("/", "layout");
// Invalida o root layout → toda a aplicação.
// Útil em deploys com mudanças de schema global.
revalidatePath e rotas dinâmicas
revalidatePath("/produtos/[id]") não funciona com o padrão [id] literal — você precisa do valor real: revalidatePath(`/produtos/${id}`). Para invalidar todos os produtos de uma vez, use revalidateTag("produtos") em vez de tentar iterar por IDs. Tags são a ferramenta certa para invalidação em massa.

# revalidateTag

revalidateTag invalida todas as entradas de cache marcadas com uma tag, independentemente de onde estão — funções, componentes, layouts, rotas. É a ferramenta preferida porque é cirúrgica e não precisa conhecer a estrutura de URLs.

revalidate-tag.ts
import { revalidateTag } from "next/cache";

// Invalidar uma tag específica de instância
revalidateTag(`post:${id}`);

// Invalidar todos os posts de um autor
revalidateTag(`autor:${autorId}`);

// Invalidar a coleção inteira
revalidateTag("posts");


// Funciona também com tags no fetch (para APIs externas)
async function getArtigoCMS(slug: string) {
  "use cache";
  cacheTag("artigos-cms", `artigo:${slug}`);
  const res = await fetch(`https://cms.exemplo.com/api/artigos/${slug}`);
  return res.json();
}

// Webhook do CMS chama revalidateTag("artigos-cms") ao publicar.
// Todos os artigos pendentes são refrescados na próxima request.

# On-demand via Route Handler (webhooks)

Quando o conteúdo vive num CMS externo (Contentful, Sanity, Strapi), o padrão mais comum é um Route Handler que recebe o webhook de publicação e dispara a revalidação. Proteja o endpoint com um token secreto.

app/api/revalidate/route.ts
import { revalidateTag } from "next/cache";
import { NextRequest } from "next/server";

export async function POST(req: NextRequest) {
  // 1. Validar token secreto compartilhado com o CMS
  const token = req.headers.get("x-revalidate-token");
  if (token !== process.env.REVALIDATE_SECRET) {
    return Response.json({ error: "Não autorizado" }, { status: 401 });
  }

  // 2. Ler o payload do CMS
  const payload = await req.json();
  const { tipo, slug } = payload;

  // 3. Revalidar de forma dirigida
  if (tipo === "artigo") {
    revalidateTag(`artigo:${slug}`);
    revalidateTag("artigos-cms");   // invalida listas que incluem o artigo
  }

  if (tipo === "pagina") {
    revalidatePath(`/${slug}`);
  }

  return Response.json({
    revalidated: true,
    timestamp: Date.now(),
  });
}

# ISR no App Router

ISR (Incremental Static Regeneration) existe no App Router, mas funciona diferente do Pages Router. Em vez de controlar tudo pelo getStaticProps, você compõe 'use cache' com cacheLife, ou usa o export const revalidate de rota. O fluxo stale-while-revalidate é o mesmo: servir do cache enquanto refaz em background.

isr-app-router.tsx
// ISR clássico no App Router — geração de rotas dinâmicas no build
// app/blog/[slug]/page.tsx

export const revalidate = 86400;  // revalida cada post a cada 24h

// generateStaticParams: quais slugs pré-gerar no build
export async function generateStaticParams() {
  const posts = await db.post.findMany({
    select: { slug: true },
    take: 100,   // os 100 mais recentes no build
  });
  return posts.map((p) => ({ slug: p.slug }));
}

export default async function PostPage({
  params,
}: { params: Promise<{ slug: string }> }) {
  const { slug } = await params;
  const post   = await getPost(slug);

  if (!post) notFound();
  return <Article post={post} />;
}

// Slugs fora dos 100 gerados no build?
// Next gera on-demand na primeira request e armazena — ISR clássico.
// Após 86400s, o próximo visitante recebe o stale enquanto refaz.
dynamicParams — o que acontece com slugs não pré-gerados
Por default, dynamicParams = true: slugs não listados em generateStaticParams são gerados on-demand na primeira request. Se você quiser que apenas os slugs pré-gerados existam (404 para o resto), exporte export const dynamicParams = false. Útil quando o conjunto de URLs é fechado — ex: landing pages de campanhas.

# Fluxo stale-while-revalidate

fluxo-swr.txt
  Request 1 (t=0s)
  ─────────────────────────────────────────────
  Cache vazio → executa função → armazena resultado
  Resposta: fresh (gerado agora)


  Request 2 (t=30s) — dentro do stale window
  ─────────────────────────────────────────────
  Cache hit → serve stale imediatamente
  Sem revalidação. Resposta: stale (mas dentro do prazo)


  Request 3 (t=90s) — após revalidate (60s), antes de expire
  ─────────────────────────────────────────────
  Cache hit → serve stale imediatamente (sem esperar)
  Background: dispara reexecução da função
  Próxima request: recebe o dado novo
  Resposta: stale (usuário não percebe, mas dado está refrescando)


  Request 4 (t=400s) — após expire (300s)
  ─────────────────────────────────────────────
  Cache expirado → AGUARDA nova geração
  Resposta: fresh (mas houve latência)

  cacheLife({ stale: 60, revalidate: 60, expire: 300 })
     stale=revalidate → serve e já agenda revalidação ao mesmo tempo

✗ Usar revalidatePath("/", "layout") em produção para tudo — invalida a aplicação inteira, incluindo caches caros de recompor.
✗ Colocar revalidateTag antes da mutação — invalide depois de confirmar que a mutação foi bem-sucedida.
✗ Tag muito genérica + invalidação muito frequente — se toda ação chama revalidateTag("tudo"), você está pagando custo de cache sem colher benefício.
Checklist de revalidação por feature
Ao implementar qualquer mutação, percorra:
1. Quais caches ficam obsoletos? (o item em si, as listas que o contêm, contadores no layout)
2. Qual o custo de ficar stale? (financeiro, segurança, UX) → define se on-demand é necessário
3. Há um tempo máximo aceitável de staleness? → define o expire como rede de segurança
4. As tags estão declaradas nas funções cacheadas? → sem cacheTag, revalidateTag não tem efeito
Capítulo 15

Route Handlers

Route Handlers são os endpoints HTTP do App Router — o equivalente ao pages/api de antes, mas integrados à estrutura de diretórios e com acesso ao mesmo sistema de cache. Você os usa quando precisa de um endpoint real: para um app mobile consumir, para um webhook receber, para retornar dados a um Client Component via fetch, ou para operações que precisam de controle total sobre o response (CORS, streaming, cabeçalhos customizados). Para mutações disparadas por componentes React, prefira Server Actions — Route Handlers são para quando você precisa de uma URL HTTP de verdade.

# Convenção de arquivo

Um Route Handler é qualquer arquivo route.ts (ou route.js) dentro de app/. Ele coexiste com page.tsx na mesma pasta apenas se estiverem em segmentos diferentes — você não pode ter app/posts/page.tsx e app/posts/route.ts na mesma pasta, pois ambos respondem ao mesmo URL.

export async function POST(req: NextRequest) {
  const tipo = req.headers.get("content-type") ?? "";

  if (tipo.includes("application/json")) {
    const data = await req.json();         // body como objeto JS
    return NextResponse.json({ data });
  }

  if (tipo.includes("multipart/form-data")) {
    const form = await req.formData();    // upload de arquivo, form
    const file = form.get("arquivo") as File;
    return NextResponse.json({ nome: file.name, tamanho: file.size });
  }

  const texto = await req.text();           // body como string bruta
  return NextResponse.json({ texto });
}

Capítulo 16

Proxy e Cookies

Em Next.js 16, o arquivo middleware.ts foi renomeado para proxy.ts — uma mudança de nome que explicita o que ele sempre foi: um interceptador que roda antes de qualquer rota, no edge, e pode redirecionar, reescrever ou modificar a request/response em voo. O nome "proxy" reflete melhor a responsabilidade: inspecionar e transformar tráfego, não executar lógica de negócio. Entender o que cabe (e o que não cabe) nesse arquivo é o que separa um proxy limpo e rápido de um gargalo silencioso.

# O arquivo proxy.ts

O arquivo fica na raiz do projeto (ao lado de app/), exporta uma função proxy por default, e opcionalmente um objeto config com o matcher para delimitar em quais rotas ele roda.

proxy.ts
import { NextRequest, NextResponse } from "next/server";

export function proxy(req: NextRequest) {
  // req.nextUrl é um URL mutável (clone de req.url)
  // req.cookies, req.headers — somente leitura aqui
  // NextResponse.next()         — deixa passar
  // NextResponse.redirect(url)  — redireciona
  // NextResponse.rewrite(url)   — reescreve (URL do browser não muda)

  return NextResponse.next();
}

export const config = {
  matcher: [
    /*
     * Roda em tudo, exceto:
     * - _next/static  (assets estáticos)
     * - _next/image   (otimização de imagens)
     * - favicon.ico
     */
    "/((?!_next/static|_next/image|favicon.ico).*)",
  ],
};

# Redirect e rewrite

proxy-redirect-rewrite.ts
import { NextRequest, NextResponse } from "next/server";

export function proxy(req: NextRequest) {
  const url = req.nextUrl.clone();

  // ─── REDIRECT (URL do browser muda) ────────────────────

  // Forçar HTTPS em produção
  if (
    process.env.NODE_ENV === "production" &&
    req.nextUrl.protocol === "http:"
  ) {
    url.protocol = "https:";
    return NextResponse.redirect(url, { status: 301 });
  }

  // Trailing slash — /sobre/ → /sobre
  if (url.pathname.endsWith("/") && url.pathname !== "/") {
    url.pathname = url.pathname.slice(0, -1);
    return NextResponse.redirect(url, { status: 308 });
  }


  // ─── REWRITE (URL do browser NÃO muda) ─────────────────

  // Multi-tenant: empresa.app.com → /tenants/empresa
  const hostname = req.headers.get("host") ?? "";
  const empresa  = hostname.split(".")[0];

  if (hostname.endsWith(".app.com") && empresa !== "www") {
    url.pathname = `/tenants/${empresa}${url.pathname}`;
    return NextResponse.rewrite(url);
    // Browser ainda mostra "empresa.app.com/dashboard"
    // Next serve /tenants/empresa/dashboard
  }

  return NextResponse.next();
}

# Padrão 1 — Proteção de rotas com auth

O uso mais comum do proxy é checar se o usuário está autenticado antes de permitir acesso a rotas protegidas. A regra de ouro: faça a verificação no proxy usando apenas o cookie/token — não chame o banco de dados aqui.

proxy.ts
import { NextRequest, NextResponse } from "next/server";
import { verificarJWT } from "@/lib/jwt";  // só parse do token, sem DB

const ROTAS_PROTEGIDAS = ["/dashboard", "/admin", "/configuracoes"];
const ROTAS_AUTH      = ["/login", "/cadastro"];

export async function proxy(req: NextRequest) {
  const pathname  = req.nextUrl.pathname;
  const token     = req.cookies.get("sessao")?.value;
  const payload   = token ? await verificarJWT(token) : null;
  const autenticado = !!payload;

  const eRotaProtegida = ROTAS_PROTEGIDAS.some((r) => pathname.startsWith(r));
  const eRotaAuth      = ROTAS_AUTH.some((r) => pathname.startsWith(r));

  // Rota protegida sem sessão → /login?de=/dashboard
  if (eRotaProtegida && !autenticado) {
    const url = req.nextUrl.clone();
    url.pathname = "/login";
    url.searchParams.set("de", pathname);  // redirect pós-login
    return NextResponse.redirect(url);
  }

  // Já autenticado tentando acessar /login → /dashboard
  if (eRotaAuth && autenticado) {
    const url = req.nextUrl.clone();
    url.pathname = "/dashboard";
    return NextResponse.redirect(url);
  }

  // Injetar o userId nos headers para Server Components lerem sem re-validar
  if (payload) {
    const requestHeaders = new Headers(req.headers);
    requestHeaders.set("x-user-id", payload.sub);
    requestHeaders.set("x-user-role", payload.role);
    return NextResponse.next({ request: { headers: requestHeaders } });
  }

  return NextResponse.next();
}

# Padrão 2 — Internacionalização (i18n)

proxy-i18n.ts
import { NextRequest, NextResponse } from "next/server";

const LOCALES   = ["pt", "en", "es"] as const;
const DEFAULT   = "pt";

export function proxy(req: NextRequest) {
  const pathname = req.nextUrl.pathname;

  // Já tem locale no path? Deixa passar
  const jaTemLocale = LOCALES.some(
    (loc) => pathname.startsWith(`/${loc}/`) || pathname === `/${loc}`,
  );
  if (jaTemLocale) return NextResponse.next();

  // Detectar locale preferido: cookie → Accept-Language → default
  const cookieLocale = req.cookies.get("NEXT_LOCALE")?.value;
  const headerLocale = req.headers
    .get("accept-language")
    ?.split(",")[0]
    ?.slice(0, 2);

  const locale = LOCALES.includes((cookieLocale ?? "") as never)
    ? cookieLocale!
    : LOCALES.includes((headerLocale ?? "") as never)
    ? headerLocale!
    : DEFAULT;

  // Rewrite para /pt/sobre (URL do browser continua /sobre)
  const url = req.nextUrl.clone();
  url.pathname = `/${locale}${pathname}`;
  return NextResponse.rewrite(url);
}

# O que NÃO fazer no proxy

O proxy roda no edge — respeite as limitações
✗ Não chame o banco de dados. O runtime do proxy é o Edge Runtime — sem Node.js completo, sem drivers TCP nativos. Prisma, pg, mysql2 não funcionam aqui. Use somente JWT (parse sem rede), cookies e lógica pura.
✗ Não faça lógica de negócio pesada. O proxy roda em toda request — 1ms de overhead vira latência global. Se precisa de mais, mova para um Server Component ou Route Handler.
✗ Não use APIs Node.js nativas. Sem fs, path, crypto completo, Buffer (apenas Uint8Array). Use a Web Crypto API.
✗ Não implemente autorização por recurso aqui. "É admin?" pode vir do JWT. "Tem permissão para este documento específico?" precisa do banco — faça isso no Server Component ou Route Handler.
✓ Use para: verificar token JWT (sem DB), redirecionar, reescrever, setar/ler cookies, injetar headers, A/B testing, locale detection.

# Proxy — proteção por rota

A combinação mais robusta é: proxy garante que usuários sem sessão nunca chegam às rotas protegidas, e o Server Component faz a autorização por recurso. O proxy usa auth() exposto para o contexto de proxy.

proxy.ts
import { auth } from "@/auth";
import { NextResponse } from "next/server";

// Auth.js v5 exporta o proxy diretamente — use o wrapper dele
export default auth((req) => {
  const sessao  = req.auth;   // auth fica em req.auth no wrapper
  const pathname = req.nextUrl.pathname;

  // Proteger todo /dashboard e /admin
  if (!sessao &&
    (pathname.startsWith("/dashboard") || pathname.startsWith("/admin"))
  ) {
    const url = req.nextUrl.clone();
    url.pathname = "/login";
    url.searchParams.set("de", pathname);
    return NextResponse.redirect(url);
  }

  // Admin — checar role (do JWT callback)
  if (pathname.startsWith("/admin") && (sessao?.user as any)?.role !== "admin") {
    return NextResponse.redirect(new URL("/403", req.url));
  }

  return NextResponse.next();
});

Turbopack e React Compiler

Dois motores que mudaram a experiência de desenvolvimento no Next.js 16: o Turbopack substituiu o Webpack como bundler default em dev e em build, trazendo partida a frio e HMR significativamente mais rápidos em projetos grandes. O React Compiler 1.0, agora estável, elimina a necessidade de escrever useMemo, useCallback e React.memo manualmente — o compilador insere memoização automaticamente onde ela beneficia performance. Este capítulo cobre o que mudou, como configurar, o que verificar ao migrar, e o impacto prático em cada cenário.

# Turbopack — o que é e o que mudou

Turbopack é um bundler escrito em Rust, com arquitetura de cache incremental: ele só recompila os módulos que mudaram, e mantém o grafo de dependências em memória entre recarregamentos. A diferença é perceptível principalmente em projetos com centenas de rotas e dependências — projetos pequenos não notam muito.

# React Compiler 1.0 — o que ele faz

O React Compiler analisa seu código em build time e insere memoização automaticamente — nos lugares exatos onde ela beneficia performance, sem os erros que humanos cometem ao usar useMemo e useCallback manualmente (deps array errado, memoização de coisas que mudam sempre, etc.).

antes-depois-compiler.tsx
// ANTES do Compiler — memoização manual, propensa a erro
function ListaProdutos({ filtros, onSelecionar }: Props) {
  // deps array: esqueceu de colocar filtros.categoria?
  const produtosFiltrados = useMemo(
    () => produtos.filter((p) => p.categoria === filtros.categoria),
    [produtos, filtros.categoria],
  );

  // onSelecionar precisa ser estável para não re-renderizar filhos
  const handleClick = useCallback(
    (id: string) => onSelecionar(id, filtros.categoria),
    [onSelecionar, filtros.categoria],
  );

  return <Grid items={produtosFiltrados} onItemClick={handleClick} />;
}


// COM React Compiler — sem useMemo, sem useCallback, sem deps arrays
function ListaProdutos({ filtros, onSelecionar }: Props) {
  // O Compiler analisa o código e insere a memoização correto
  const produtosFiltrados = produtos.filter(
    (p) => p.categoria === filtros.categoria,
  );

  const handleClick = (id: string) => onSelecionar(id, filtros.categoria);

  return <Grid items={produtosFiltrados} onItemClick={handleClick} />;
}

# O que o Compiler exige — Regras do React

O React Compiler só otimiza código que segue as Regras do React: sem mutação de props ou state, sem efeitos colaterais fora de hooks, sem leituras não determinísticas. Código que viola essas regras é simplesmente ignorado pelo Compiler (não otimizado, não quebrado).

regras-do-react.tsx
// ❌ Viola Regras do React — o Compiler IGNORA este componente
function ComponenteMutante({ items }: { items: Item[] }) {
  // Mutação direta de prop — undefined behavior
  items.push({ id: "novo" });
  return <Lista items={items} />;
}


// ❌ Viola — leitura não determinística fora de hook
function ComponenteRandom() {
  const id = Math.random();  // valor diferente a cada render
  return <div id={String(id)}>...</div>;
}


// ✅ Seguro — imutável, determinístico
function ComponenteLimpo({ items }: { items: Item[] }) {
  const comNovo = [...items, { id: "novo" }];  // novo array, sem mutação
  return <Lista items={comNovo} />;
}

// O Compiler nunca quebra código que viola as Regras — ele apenas não otimiza.
// Para inspecionar o que foi/não foi compilado: React DevTools → ícone de faísca ⚡

# Opt-out seletivo

Se um componente específico não deve ser compilado (temporariamente, durante migração, ou porque tem efeitos especiais intencionais), use a diretiva 'use no memo':

opt-out.tsx
function ComponenteEspecial() {
  "use no memo";  // o Compiler ignora este componente
  // comportamento original preservado
  return <div>...</div>;
}

// Use 'use no memo' temporariamente durante migração.
// Não é para ficar no código de produção — é uma saída de emergência.
Posso remover todos os useMemo e useCallback?
Com o Compiler ativo, você pode parar de escrever useMemo e useCallback para otimização — o Compiler faz isso melhor. Porém, se você tem useMemo existente que segue as Regras do React, o Compiler o respeita e pode otimizá-lo ainda mais. Remover código legado funcional não é urgente. O ganho real é: não escrever novo. Para limpeza gradual, o eslint-plugin-react-compiler aponta onde hooks de memoização são desnecessários.

# Dynamic imports — adiar o que não é crítico

next/dynamic divide o componente em um chunk separado que só é baixado quando necessário. Use para: componentes pesados abaixo da dobra, modais/drawers que abrem raramente, bibliotecas de gráficos, editores de texto rico, widgets de mapa.

dynamic-imports.tsx
import dynamic from "next/dynamic";

// Componente pesado — carregado apenas quando renderizado
const EditorRico = dynamic(() => import("@/components/editor-rico"), {
  loading: () => <div className="skeleton h-48" />,
});

// Componente com biblioteca pesada — MUI, Chart.js, etc.
const GraficoVendas = dynamic(
  () => import("@/components/grafico-vendas"),
  {
    loading: () => <GraficoSkeleton />,
    ssr: false,   // sem renderização no servidor (bibliotecas que usam window)
  },
);

// Modal — só carrega quando o usuário abre
const ModalConfirmacao = dynamic(
  () => import("@/components/modal-confirmacao"),
);

export function Dashboard() {
  const [modalAberto, setModalAberto] = useState(false);

  return (
    <div>
      <GraficoVendas />                    {/* carrega ao renderizar */}
      {modalAberto && <ModalConfirmacao />}  {/* carrega ao abrir */}
    </div>
  );
}

# Tree shaking — importações que valem

Tree shaking remove exports não usados do bundle. Para ele funcionar, o código precisa usar imports estáticos (não dinâmicos em runtime) e as bibliotecas precisam ser marcadas como side-effect-free. Na prática, o problema costuma ser importações erradas.

tree-shaking.ts
// ❌ Importação total — traz TUDO do lodash para o bundle (~70kB)
import _ from "lodash";
const resultado = _.groupBy(lista, "categoria");


// ✅ Importação por módulo — só groupBy entra no bundle (~2kB)
import groupBy from "lodash/groupBy";
const resultado = groupBy(lista, "categoria");


// ✅ Para bibliotecas com tree shaking nativo (ES modules), import named funciona
import { format, parseISO } from "date-fns";
// date-fns é ES module — só format e parseISO entram


// ❌ Importar ícone de barrel export de biblioteca grande
import { FiUser } from "react-icons/fi";
// react-icons suporta tree shaking — isso é OK

import { FiUser } from "react-icons";
// ❌ barrel de todos os ícones de todos os pacotes — traz tudo

modularizeImports — tree shaking de barrels automático

Para libs que não suportam tree shaking nativamente mas têm módulos granulares, o modularizeImports do Next.js transforma as importações automaticamente em build:

next.config.ts
const config: NextConfig = {
  modularizeImports: {
    // Transforma: import { groupBy } from "lodash"
    // Em:         import groupBy from "lodash/groupBy"
    lodash: {
      transform: "lodash/{{member}}",
    },
    // Material UI — importação granular automática
    "@mui/material": {
      transform: "@mui/material/{{member}}",
    },
  },
};

# Separação de código — code splitting automático

O Next.js já divide o código por rota automaticamente: cada page.tsx gera seu próprio chunk, e componentes compartilhados vão para chunks comuns. O que você controla é o que fica dentro do chunk de cada rota.

code-splitting.tsx
// ✅ Server Components pesados não entram no bundle JS do cliente
// Toda lógica de renderização fica no servidor
async function TabelaRelatorio() {
  const dados = await db.relatorio.findMany({ take: 1000 });
  // Renderiza HTML no servidor — zero JS no cliente para este componente
  return ...
;
}

// Evite importar bibliotecas só usadas em Server Components dentro de Client Components
// ❌ Isto traz marked para o bundle do cliente
"use client";
import { marked } from "marked";  // biblioteca de Markdown, ~50kB

function MarkdownRenderer({ md }: { md: string }) {
  return <div dangerouslySetInnerHTML={{ __html: marked(md) }} />;
}

// ✅ Mova para Server Component — marked fica só no servidor
async function MarkdownRenderer({ md }: { md: string }) {
  const html = await marked.parse(md);
  return <div dangerouslySetInnerHTML={{ __html: html }} />;
  // marked nunca entra no bundle do cliente
}

# A pergunta que define a estrutura

Antes de pasta nenhuma, escolha entre dois eixos: type-based (organizar por o que o arquivo é) ou feature-based (organizar por o que o arquivo serve). Type-based agrupa todos os componentes em components/, todas as queries em queries/, todas as actions em actions/. Feature-based cria pastas como posts/ com tudo relacionado a posts dentro. A regra prática: type-based escala melhor em apps pequenas, feature-based escala melhor em apps grandes — e a transição doi sempre.

A regra de uma pasta
Se um arquivo é usado por mais de uma feature, ele não pertence à pasta da feature — sobe para components/ ou lib/. Se um arquivo é usado por uma feature só, ele nunca deveria estar em pasta global; vive dentro da própria feature. Esta regra simples evita o "cemitério de utils" que toda app desenvolve com o tempo.

# Data Access Layer — uma camada que vale ouro

Espalhe acesso a banco de dados pelo código e você vai pagar por isso em três frentes: violações de cache (queries que deveriam estar memoizadas mas não estão), inconsistências de autorização (regras diferentes para o mesmo recurso em locais diferentes), e dificuldade de migração (mudar de ORM vira ano de trabalho). A solução é simples: todo acesso a banco passa por funções nomeadas dentro de features/*/queries.ts. Componentes nunca chamam db.modelo.findX diretamente.

features/posts/queries.ts
import {
  unstable_cacheLife as cacheLife,
  unstable_cacheTag as cacheTag,
} from "next/cache";
import { db } from "@/lib/db";
import { auth } from "@/lib/auth";

// ─── Queries públicas — cacheadas, sem dependência de sessão ──────
export async function getPostPublico(slug: string) {
  "use cache";
  cacheLife("hours");
  cacheTag("posts", `post:${slug}`);

  return db.post.findUnique({
    where: { slug, publicado: true },
    include: { autor: { select: { nome: true, avatarUrl: true } } },
  });
}

// ─── Queries autorizadas — checam sessão DENTRO da query ──────────
// Esta é a chave: a autorização vive na camada de dados, não no componente.
// Qualquer caller que receber o resultado já tem garantia de permissão.
export async function getMeuPost(id: string) {
  const sessao = await auth();
  if (!sessao?.user) throw new Error("Não autenticado");

  const post = await db.post.findUnique({ where: { id } });
  if (!post) return null;

  // Autorização por recurso
  if (post.autorId !== sessao.user.userId) {
    throw new Error("Não autorizado");
  }
  return post;
}

// ─── Lista paginada do usuário — paginação sempre por argumento ───
export async function listarMeusPosts(pagina = 1) {
  const sessao = await auth();
  if (!sessao?.user) throw new Error("Não autenticado");

  return db.post.findMany({
    where:   { autorId: sessao.user.userId },
    orderBy: { criadoEm: "desc" },
    skip: (pagina - 1) * 20,
    take: 20,
  });
}
Por que isto é tão importante
Quando autorização vive no componente, basta um Server Component novo esquecer a checagem para o vazamento acontecer — silenciosamente, sem erro de TypeScript, sem teste falhando. Quando autorização vive na query, o vazamento exige um esforço deliberado: você teria que criar uma nova query sem proteção. A diferença entre "fácil de errar" e "fácil de acertar" não é estilística — é a diferença entre uma app segura e uma app com CVE.

# Anti-padrões — os erros que mais doem

1. 'use client' espalhado sem necessidade

use-client-anti.tsx
// ❌ Página inteira virou cliente porque "tem um botão"
"use client";

export default function PostPage({ params }) {
  const [post, setPost] = useState(null);

  useEffect(() => {
    fetch(`/api/posts/${params.id}`).then((r) => r.json()).then(setPost);
  }, [params.id]);

  if (!post) return <Spinner />;

  return (
    <article>
      <h1>{post.titulo}</h1>
      <p>{post.conteudo}</p>
      <BotaoCurtir postId={post.id} />
    </article>
  );
}

// ✅ Server Component renderiza o conteúdo; só o botão é cliente
export default async function PostPage({ params }: Props) {
  const { id } = await params;
  const post   = await getPostPublico(id);
  if (!post) notFound();

  return (
    <article>
      <h1>{post.titulo}</h1>
      <p>{post.conteudo}</p>
      <BotaoCurtir postId={post.id} />  {/* só este é client */}
    </article>
  );
}

2. Server Action sem checagem de autorização

action-anti.ts
// ❌ "Mas o botão de deletar só aparece para o dono!"
//    O endpoint POST gerado pela Server Action é público.
//    Qualquer um com curl pode chamar — sem verificar dono = vulnerabilidade.
"use server";

  await db.post.delete({ where: { id } });
  revalidatePath("/posts");
}


// ✅ Sempre: auth → autorização por recurso → mutação → invalidação
"use server";
export async function deletarPost(id: string) {
  const sessao = await auth();
  if (!sessao?.user) return { erro: "Não autenticado" };

  const post = await db.post.findUnique({ where: { id } });
  if (!post) return { erro: "Não encontrado" };
  if (post.autorId !== sessao.user.userId) {
    return { erro: "Sem permissão" };
  }

  await db.post.delete({ where: { id } });
  revalidateTag(`post:${id}`);
  revalidateTag("posts");
}

4. Cache invalidation que derruba o mundo

// ❌ Toda mutação invalida tudo — cache vira ornamento

  await db.comentario.update({ where: { id }, data });
  revalidatePath("/", "layout");  // ← invalida a app inteira
}


// ✅ Tags específicas: só invalida o que realmente ficou stale
export async function atualizarComentario(id: string, data: FormData) {
  const c = await db.comentario.update({
    where: { id },
    data,
    include: { post: { select: { id: true } } },
  });
  revalidateTag(`comentario:${id}`);
  revalidateTag(`post:${c.postId}:comentarios`);
}

5. Outros anti-padrões frequentes

  • Hidden inputs como autoridade. <input type="hidden" name="userId" value={user.id} /> pode ser alterado pelo browser. Sempre pegue o user da sessão dentro da action.
  • Layout raiz com 7 providers aninhados. Cada provider que usa Context obriga o filho mais próximo a ser Client Component, contaminando a árvore. Use providers só onde o estado é necessário; mantenha o layout raiz puro.
  • Importar bibliotecas pesadas em Client Component. Se a lib só é necessária para renderizar (não para interatividade), mova o componente para Server Component. marked, shiki, marked-react ficam só no servidor.
  • Reaching for libraries antes de checar o platform. Antes de instalar React Query, lembre que Server Components + cache nativo cobrem 80% dos casos. Antes de Redux, lembre que useReducer + Server Actions cobrem 90%. Antes de SWR, considere se a query nem precisa ser client-side.
  • Estrutura de pastas que mudou três vezes em seis meses. A causa raiz quase nunca é a estrutura — é falta de regras claras sobre o que é compartilhado e o que é local. Decida cedo a regra de "uma pasta", e deixe ela sozinha.

# Mapas mentais para internalizar

Onde colocar cada coisa — pergunta única para decidir
Componente: "ele precisa de interatividade ou state?" Sim → Client. Não → Server.
Endpoint: "vou consumir só do meu app, ou clientes externos?" Só interno → Server Action. Externo → Route Handler.
Cache: "muda por ação do usuário ou pelo relógio?" Ação → on-demand (revalidateTag). Relógio → time-based (cacheLife). Os dois → ambos.
Auth: "preciso parar antes da rota ou checar dentro do dado?" Antes → proxy. Por recurso → query/action.
Dado: "público ou específico do usuário?" Público → estático com cache. Específico → dinâmico (connection() ou via cookies/headers fora de cache).
