Capítulo 01

Introdução ao utility-first

Tailwind CSS é a biblioteca de estilização mais influente da última década — e também a mais mal-entendida. Para quem está chegando, parece "CSS inline com nome diferente". Para quem usa há anos, é a forma mais produtiva de estilizar interfaces. A diferença entre essas duas percepções é exatamente o que este capítulo explora. Antes de sintaxe, antes de classes, antes de qualquer utilidade — o que você precisa entender é o porquê da abordagem.

# O que é Tailwind

o-que-e.html
<!-- CSS tradicional — escrita uma vez, usada via classe -->
<div class="card">
  <h2 class="card-title">Título</h2>
  <p class="card-body">Conteúdo</p>
</div>

/* No arquivo .css: */
.card { padding: 24px; background: white; border-radius: 8px; }
.card-title { font-size: 20px; font-weight: 600; }


<!-- Tailwind — utilitários compostos diretamente no HTML -->
<div class="p-6 bg-white rounded-lg">
  <h2 class="text-xl font-semibold">Título</h2>
  <p class="text-gray-700">Conteúdo</p>
</div>

/* Nenhum CSS para escrever. Cada classe corresponde a uma propriedade. */

Cada classe do Tailwind faz uma única coisa: p-6 aplica padding: 1.5rem, bg-white aplica background: white, rounded-lg aplica border-radius: 0.5rem. São centenas dessas — espaçamento, cor, tipografia, layout, animação — e você compõe interfaces juntando-as.

# "Mas isso não é só CSS inline?"

É a primeira pergunta que todo mundo faz ao ver Tailwind. E é uma boa pergunta. A resposta é: não, e a diferença importa. Vale entender por quê.

inline-vs-utility.html
<!-- CSS inline — pelagem livre, sem freios -->
<div style="padding: 23px; color: #f0a832; background: rgb(15, 13, 11);">
  <!-- valores arbitrários, fora de qualquer sistema -->
</div>

<!-- Tailwind — utilitários partem de um sistema -->
<div class="p-6 text-amber-500 bg-zinc-950">
  <!-- p-6 = 24px (do scale 4px), amber-500 = cor do design system,
       bg-zinc-950 = cor da paleta neutra. Tudo coerente. -->
</div>

A diferença é a restrição. CSS inline aceita qualquer valor — você pode escrever padding: 23px, color: #f0a832, ou qualquer combinação aleatória. Tailwind te dá um conjunto pré-definido: p-1, p-2, p-3... saltando em incrementos coerentes. amber-500 faz parte de uma paleta de 11 tons de âmbar, harmonizada com 22 outras paletas.

O constraint é a feature. Em vez de pensar "quanto exatamente vai aqui?", você pensa "qual nível dessa escala?". Decisões mais rápidas, resultados mais consistentes, menos drift visual entre componentes.

A escala como design system
A paleta de cores, a escala de espaçamento, os tamanhos de fonte do Tailwind formam um design system pronto — pensado por designers, refinado ao longo de anos. Ao usar text-zinc-700 num parágrafo e border-zinc-200 numa borda, você está consumindo decisões de design já tomadas. Para 90% dos projetos, é melhor que reinventar tokens próprios.

# Os outros pontos que diferenciam de inline

recursos.html
<!-- 1. Estados — impossível em inline, trivial em Tailwind -->
<button class="bg-blue-500 hover:bg-blue-600 active:bg-blue-700 disabled:opacity-50">
  Clique
</button>

<!-- 2. Responsivo — breakpoints como prefixo -->
<div class="grid grid-cols-1 md:grid-cols-2 lg:grid-cols-3">
  <!-- 1 coluna mobile, 2 em tablet, 3 em desktop -->
</div>

<!-- 3. Dark mode — outra dimensão ortogonal -->
<div class="bg-white text-black dark:bg-zinc-900 dark:text-white">

<!-- 4. Pseudo-elementos — antes era impossível em qualquer lugar -->
<p class="first-letter:text-4xl first-letter:text-amber-500">
  Texto com primeira letra grande...
</p>

<!-- 5. Animações compostas, group/peer, container queries... -->

CSS inline (style="...") só consegue declarar propriedades estáticas. Não tem :hover, não tem @media, não tem dark mode, não tem ::before. Tailwind faz tudo isso — porque por baixo ele gera CSS real, com todos os recursos da linguagem.

# Por que isso vence ao longo do tempo

O argumento mais sutil de Tailwind não é técnico — é sobre como o código envelhece. Em projetos CSS tradicionais, você acumula classes ao longo dos meses: .card, .card-large, .card-featured, .card-sidebar... cada uma com seu CSS específico. Com o tempo, ninguém sabe mais o que cada classe afeta, mover um div quebra estilos em outro lugar, e o CSS final tem 200KB que ninguém ousa tocar.

envelhecimento.css
/* O CSS típico de um projeto de 3 anos */

.btn { /* base */ }
.btn-primary { /* azul */ }
.btn-primary-large { /* azul, maior */ }
.btn-primary-large-icon { /* azul, maior, com ícone */ }
.btn-primary-large-icon-disabled { /* ... */ }
.btn--alt { /* sobrescreve algumas coisas */ }
.dashboard .btn-primary { /* override específico */ }
.legacy-page .btn { /* override de override */ }

/* 600 classes, 4 desenvolvedores que escreveram, 0 que entendem hoje. */

Em projetos Tailwind, o CSS final tem o tamanho de uma única paleta de utilidades — e é gerado automaticamente. Você não acumula classes ao longo do tempo; só compõe utilidades que já existem. Quando deleta um componente, nenhum CSS órfão fica. Quando muda um estilo, está mudando no HTML — onde você está olhando — não em algum arquivo distante.

A localidade da decisão
Em Tailwind, os estilos vivem onde a marcação vive. Quando você lê <button class="px-4 py-2 bg-blue-500 rounded">, sabe imediatamente como aquele botão se parece — sem abrir nenhum outro arquivo, sem grep no projeto. Essa "localidade" é o que faz Tailwind escalar em equipes grandes e codebases longas. É o oposto de action at a distance.

# A objeção do "HTML feio"

html-comprido.html
<!-- "Mas as classes ficam enormes!" — sim, ficam -->
<button class="inline-flex items-center gap-2 px-4 py-2 bg-blue-500 hover:bg-blue-600 active:bg-blue-700 disabled:opacity-50 disabled:cursor-not-allowed text-white font-medium rounded-lg shadow-sm transition-colors">
  Clique aqui
</button>

<!-- A primeira reação é estética. Mas considere: -->
<!-- 1. Esse botão tem ESTADOS, RESPONSIVIDADE, ANIMAÇÃO. -->
<!-- 2. Em CSS tradicional, essas regras existem em outro arquivo. -->
<!-- 3. Aqui, elas estão visíveis no ponto de uso. -->
<!-- 4. Em frameworks (React, Vue), você cria o componente UMA vez. -->

<!-- React resolve via componentização: -->
<Button variant="primary">Clique</Button>
<!-- O HTML "feio" fica encapsulado dentro do componente. -->

A reclamação de "HTML feio" é legítima — e tem solução: você não usa Tailwind cru no HTML em projetos sérios. Você cria componentes (em React, Vue, Svelte, ou mesmo via classes @apply em projetos sem framework) que encapsulam as classes longas. O HTML do dia a dia volta a ser limpo. As classes só ficam visíveis dentro da definição do componente — onde você quer vê-las.

# Quando Tailwind brilha — e quando não

quando-usar.txt
# BRILHA QUANDO:

✓ Você usa um framework de componentes (React, Vue, Svelte, etc.)
   Componentes encapsulam as classes — HTML do dia fica limpo

✓ A equipe valoriza VELOCIDADE de iteração
   Sem nomear classes, sem alternar arquivos, sem decidir convenções

✓ O design tem variações granulares
   Botões com 8 variantes, cards com 5 estados — utility scales bem

✓ Você quer um design system PRONTO
   Paleta, tipografia e spacing pensados — você não precisa criar do zero

✓ Múltiplos desenvolvedores tocam o mesmo CSS
   Não há "regra global" para alguém quebrar sem perceber


# NÃO BRILHA QUANDO:

✗ O time inteiro odeia ver classes no HTML
   Adoção forçada nunca dá certo. Se a estética é decisiva, escolha outra.

✗ O projeto é puramente estático sem build step
   Tailwind PRECISA de PostCSS/build. Se é só HTML cru servido,
   você terá que carregar o CSS completo (gigante) ou usar a CDN play
   (sem otimizações). É possível, mas não é o terreno natural.

✗ Você tem um design highly custom que não cabe em utilidades
   Sites de portfolio artístico, landing pages experimentais —
   às vezes vale escrever CSS direto. Tailwind permite isso, mas
   se 80% do projeto é assim, o overhead não compensa.

✗ A equipe já tem um design system maduro próprio
   Se você tem tokens, componentes, e disciplina — Tailwind
   pode adicionar duplicação em vez de simplificação.

# O panorama em 2025 — v3 e v4

Em janeiro de 2025, o Tailwind lançou a versão 4 — uma reescrita completa do engine, com mudanças na arquitetura da configuração. Os dois mundos coexistem em projetos reais. Este manual cobre principalmente o que é comum aos dois, com seções específicas para v4 onde a diferença importa.

Sobre escolher entre v3 e v4
Para projeto novo em 2025, comece com v4 — é o caminho oficial e tem performance significativamente melhor. Para projeto existente, ficar em v3 é perfeitamente válido — o suporte continua, e a migração tem cuidados a tomar (tema CSS-first, baseline de browsers, ajustes em alguns nomes de utility). Vamos cobrir migração no capítulo 20.

Capítulo 02

Setup e instalação

Tailwind precisa de um passo de build — ele analisa seu código, gera apenas as classes que você usa, e produz um CSS minificado. Este capítulo cobre os três cenários mais comuns: projetos com Vite, projetos com Next.js, e a opção CDN para protótipos rápidos sem build.

# Vite + Tailwind v3

terminal
# Criar projeto Vite (se ainda não tem)
npm create vite@latest meu-projeto -- --template react-ts
cd meu-projeto

# Instalar Tailwind, PostCSS e Autoprefixer
npm install -D tailwindcss postcss autoprefixer

# Gerar arquivos de configuração
npx tailwindcss init -p
tailwind.config.js
/** @type {import('tailwindcss').Config} */
export default {
  content: [
    "./index.html",
    "./src/**/*.{js,ts,jsx,tsx}",  // onde estão seus arquivos
  ],
  theme: {
    extend: {},    // customizações ficam aqui (cap. 18)
  },
  plugins: [],
};
src/index.css
/* As três diretivas base — adicione no CSS global */
@tailwind base;
@tailwind components;
@tailwind utilities;
src/main.tsx
import "./index.css";  // importar o CSS global
import { StrictMode } from "react";
import { createRoot } from "react-dom/client";
import App from "./App";

createRoot(document.getElementById("root")!).render(
  <StrictMode><App /></StrictMode>
);

# Vite + Tailwind v4

No v4, o plugin nativo para Vite substitui o PostCSS — mais rápido, menos configuração:

terminal
# Tailwind v4 com Vite — plugin dedicado
npm install -D tailwindcss @tailwindcss/vite
vite.config.ts
import { defineConfig } from "vite";
import tailwindcss from "@tailwindcss/vite";
import react from "@vitejs/plugin-react";

export default defineConfig({
  plugins: [
    tailwindcss(),   // Tailwind antes do React
    react(),
  ],
});
src/index.css
/* v4 — importação única substitui as três diretivas */
@import "tailwindcss";

/* Customizações ficam aqui (CSS-first — cap. 20) */
@theme {
  --color-brand: #f0a830;
}
v4 — sem tailwind.config.js
No v4, não existe tailwind.config.js. A configuração de tema acontece diretamente no CSS via @theme. O Tailwind detecta automaticamente seus arquivos de conteúdo — não precisa configurar o array content. Isso reduz o setup a dois passos: instalar o plugin e adicionar o @import.

# Configurar o content corretamente

No v3, o array content é o mais crítico do tailwind.config.js — ele diz ao Tailwind onde procurar classes para incluir no CSS final. Se um arquivo não está coberto, as classes dentro dele serão removidas do bundle.

content-exemplos.js
content: [
  // Padrão seguro para React/Vite
  "./index.html",
  "./src/**/*.{js,ts,jsx,tsx}",

  // Incluir arquivos de configuração que podem ter classes
  "./src/**/*.{json}",  // se tiver classes em JSON de config

  // Bibliotecas externas que usam classes Tailwind
  "./node_modules/@minha-lib/**/*.{js,jsx}",

  // Monorepo — pacotes internos
  "../../packages/ui/src/**/*.{tsx}",
]

// ❌ Não funciona — Tailwind não "entende" valores dinâmicos
const cor = `text-${tipo}-500`          // classe NÃO gerada — string interpolada
const cor = ["text-red-500", "text-green-500"][n]  // ✅ gerada — strings completas
Nunca construa classes com interpolação de string
O Tailwind analisa seu código como texto — ele não executa JavaScript. Se você escrever `text-${cor}-500`, a classe text-red-500 nunca aparece inteira no código, então o Tailwind não a inclui. Sempre use nomes de classe completos: "text-red-500", "text-green-500". Você pode mapear via objeto: const cores = {'{"erro": "text-red-500"}'}.

Capítulo 03

Como o Tailwind funciona

Antes de mergulhar nas categorias de utilidades, vale entender o que acontece quando você escreve class="p-6 bg-blue-500" num arquivo. Tailwind não é mágica — é um pipeline bem definido: scan, geração, otimização. Entender esse pipeline explica por que certas coisas funcionam, por que outras silenciosamente não, e por que o comportamento do v4 é diferente do v3 em pontos sutis.

# O modelo mental — escaneamento de strings

como-encontra.html
<!-- Quando você escreve isto: -->
<div class="p-6 bg-blue-500 text-white rounded-lg">
  Olá
</div>

<!-- O Tailwind, durante o build, faz algo conceitualmente assim: -->
// 1. Lê todos os arquivos no array 'content'
// 2. Passa um regex pra encontrar candidatos a classe
//    (qualquer "palavra" que se pareça com utility)
// 3. Para cada candidato, verifica se corresponde a uma utility válida
// 4. Para cada match, gera o CSS correspondente
// 5. Junta tudo num arquivo final, deduplica, ordena
// 6. Em produção, passa por minificação e otimização

// O CSS final gerado para o div acima:
.p-6 { padding: 1.5rem; }
.bg-blue-500 { background-color: rgb(59 130 246); }
.text-white { color: rgb(255 255 255); }
.rounded-lg { border-radius: 0.5rem; }

O ponto crítico: o Tailwind nunca executa seu código. Ele só lê o texto dos arquivos. Isso parece um detalhe técnico, mas é a única coisa que você precisa internalizar para entender 90% dos comportamentos surpreendentes do framework.

JIT — Just-In-Time
No v3, esse mecanismo de scan se chama JIT engine. Em v4, o engine foi reescrito em Rust (Lightning CSS) e ficou ~10x mais rápido, mas o modelo conceitual é o mesmo: scan textual de arquivos, geração sob demanda. A diferença em v4 é que ele detecta arquivos automaticamente — você não precisa configurar content manualmente. O modelo mental, porém, não muda: scan textual.

# Por que interpolação não funciona

Já mencionamos no capítulo anterior — agora aprofundamos. A regra é consequência direta de "scan textual":

interpolacao-detalhe.tsx
// Imagine o Tailwind "lendo" estes três arquivos:

// Arquivo 1 — funciona
<div className="text-red-500">...
// Tailwind lê: "text-red-500" → reconhece → gera CSS ✓

// Arquivo 2 — NÃO funciona
const cor = "red";
<div className={`text-${cor}-500`}>...
// Tailwind lê: "text-${cor}-500" como TEXTO LITERAL
// Procura por "text-${cor}-500" no dicionário → não existe → ignora ✗
// Em runtime, a string vira "text-red-500", mas o CSS já foi gerado SEM ela

// Arquivo 3 — funciona (via mapa de strings completas)
const classes = {
  erro: "text-red-500",     // string completa no fonte
  sucesso: "text-green-500",  // string completa no fonte
};
<div className={classes[tipo]}>...
// Tailwind lê o arquivo, vê "text-red-500" e "text-green-500"
// Reconhece ambas, gera CSS para as duas ✓

# O CSS que NÃO está lá

apenas-usadas.css
/* O Tailwind tem MILHARES de utilidades possíveis: */
/*   text-red-500, text-blue-500, text-green-500... 22 paletas × 11 tons = 242 cores
     × 6 propriedades (text, bg, border, ring, divide, ...) = 1452 só para cores
     × 5 estados (hover, focus, active, disabled, group-hover...) = 7260
     × 6 breakpoints (sm, md, lg, xl, 2xl, default) = 43560
     ... e isso é só para cores. */

/* Mas o seu CSS final só tem o que você USOU. */
/* Se no projeto inteiro só apareceu text-red-500 e bg-zinc-900, */
/* o CSS final tem APENAS essas duas regras. */

.text-red-500 { color: rgb(239 68 68); }
.bg-zinc-900 { background-color: rgb(24 24 27); }

/* Em produção, o CSS típico fica entre 5-50KB — minúsculo. */

O resultado é elegante: por padrão você tem acesso a todo o universo de utilidades, mas paga apenas pelo que usa. É o oposto de bibliotecas tradicionais (Bootstrap, Materialize) que enviam um CSS gigante com tudo, e você usa 5%.

# Valores arbitrários — quebrando o sistema (com ordem)

arbitrarios.html
<!-- Às vezes você precisa de um valor que NÃO está no design system.
     Tailwind permite isso via colchetes — mas com semântica. -->

<!-- Padding arbitrário em pixels específicos -->
<div class="p-[17px]">...
/* Gera: padding: 17px; */

<!-- Cor arbitrária em hex -->
<div class="bg-[#1a2b3c]">...
/* Gera: background-color: #1a2b3c; */

<!-- Grid template específico -->
<div class="grid-cols-[200px_1fr_auto]">...
/* Gera: grid-template-columns: 200px 1fr auto; */

<!-- Fórmula CSS calc -->
<div class="w-[calc(100vw-280px)]">...
/* Espaços em valores precisam ser sublinhados ou ausentes */

<!-- Variável CSS -->
<div class="text-[var(--minha-cor)]">...

Os colchetes resolvem a tensão entre "ter um sistema" e "às vezes precisar fugir dele". Você fica dentro do design system 95% do tempo, e tem uma porta de saída clara — visível no próprio código — para os 5% restantes.

Use arbitrários como exceção, não regra
Quando você se pega usando p-[17px] em vários lugares, a pergunta certa é: esse valor deveria estar no design system?. Talvez sim — adicione no tema (cap. 18). Talvez você esteja usando o número errado e p-4 (16px) resolve. Arbitrários são uma ferramenta, não um padrão. Quando viram regra, você está nadando contra o sistema do framework.

# Modificadores — outra camada do scan

modificadores.html
<!-- Modificadores aparecem como prefixos separados por : -->

<button class="bg-blue-500 hover:bg-blue-600">
  <!-- bg-blue-500       → estado normal -->
  <!-- hover:bg-blue-600 → ao passar mouse -->
</button>

<!-- O CSS gerado: -->
.bg-blue-500 { background-color: #3b82f6; }
.hover\:bg-blue-600:hover { background-color: #2563eb; }
/* O \: é o : escapado — Tailwind transforma cada modificador em pseudo-classe ou media query */

<!-- Modificadores se EMPILHAM -->
<div class="md:dark:hover:bg-zinc-800">
  <!-- Aplica APENAS quando: tela >= 768px E dark mode E hover -->
</div>

<!-- O CSS gerado para isso fica dentro de @media + .dark + :hover -->
@media (min-width: 768px) {
  .dark .md\:dark\:hover\:bg-zinc-800:hover {
    background-color: #27272a;
  }
}

O Tailwind tem dezenas de modificadores: hover:, focus:, active:, disabled:, dark:, md:, lg:, group-hover:, peer-checked:, first:, last:, odd:... cada um vira um prefixo. Eles compõem livremente — uma classe pode ter 4-5 modificadores empilhados quando necessário. Vamos cobrir cada categoria nos capítulos 14-16.

# Especificidade e ordem

ordem.html
<!-- A ordem das classes no atributo NÃO importa.
     A ordem no CSS gerado importa — e é controlada pelo Tailwind. -->

<div class="bg-red-500 bg-blue-500"> <!-- Qual cor ganha? -->
<div class="bg-blue-500 bg-red-500"> <!-- E aqui? -->

// Resposta: ambos terão a MESMA cor — a que aparece DEPOIS no CSS gerado.
// O Tailwind ordena utilities por categoria, então o resultado é determinístico.
// Para "vencer" outra utility, você usa a versão posterior do CSS, não a ordem no class.

// ❌ Pensar que ordem das classes resolve conflitos
<div class="p-4 p-6">  // não decide qual ganha — depende do CSS

// ✅ Eliminar conflitos no código fonte
<div class="p-6">  // uma classe por propriedade. simples.

// ✅ Para sobrescrever condicionalmente, use bibliotecas como `tailwind-merge`
import { twMerge } from "tailwind-merge";
twMerge("p-4", "p-6") // → "p-6" — resolve conflitos automaticamente
tailwind-merge é essencial em projetos React
Quando você cria componentes que aceitam classes adicionais via prop (className), conflitos viram inevitáveis. tailwind-merge resolve isso programaticamente — a função entende o sistema de utilities e mantém apenas a última de cada categoria. É o que permite o pattern cn() do shadcn/ui funcionar bem. Vamos usar isso constantemente daqui para frente.

# O fluxo completo do build

pipeline.txt
# O que acontece quando você roda `npm run build`:

1. SCAN
   Tailwind lê todos os arquivos no array `content`
   (ou auto-detecta no v4)

2. EXTRAÇÃO DE CANDIDATOS
   Regex pega "palavras" que parecem com classes:
   /[a-z][\w-]*(?:\:[\w-]+)*/g (simplificado)

3. VALIDAÇÃO
   Para cada candidato, checa se corresponde a uma utility válida
   ("xyz-foo" → não, ignora; "p-4" → sim, mantém)

4. GERAÇÃO DE CSS
   Para cada utility válida, escreve a regra CSS correspondente
   Aplica modificadores (hover:, md:, etc.) como wrappers

5. ORDENAÇÃO
   Ordena por categoria (layout antes de cor antes de tipografia)
   Garante que `dark:` vem depois do default, `md:` depois do default
   Cascade fica determinística

6. PROCESSAMENTO
   Roda Autoprefixer (vendor prefixes — v3 com PostCSS)
   Roda Lightning CSS (otimizações — v4)

7. MINIFICAÇÃO
   Em produção, remove espaços, comentários, quebras
   CSS final tipicamente entre 5-50KB

# Em desenvolvimento, isso roda incrementalmente — só re-gera o que mudou.
# Em v3, levava ~100ms; em v4, <10ms na maioria dos casos.

# O que muda do v3 para o v4

Diferenças no engine
✓ Detecção automática de conteúdo: v4 não precisa do array content — escaneia projeto inteiro inteligentemente.
✓ Build <10ms: reescrita em Rust faz primeiros builds em ~100ms e rebuilds em milissegundos.
✓ Sem PostCSS no Vite: plugin nativo (@tailwindcss/vite) elimina a camada PostCSS.
✓ Cores em OKLCH: nova paleta nativamente em OKLCH — gama de cores maior, mistura mais natural.
✓ Container queries nativos: sem precisar do plugin @tailwindcss/container-queries.
✓ Tema CSS-first: configuração via @theme dentro do CSS, não em JS.

O modelo mental, porém, é o mesmo. Tudo neste capítulo se aplica aos dois.

# Resumo — o que internalizar

O modelo mental para o resto do manual
✓ Tailwind lê seus arquivos como texto, busca padrões de utility, gera CSS apenas para o que encontra.
✓ Nunca executa código — interpolação `text-${cor}` não funciona; use mapas de strings completas.
✓ Valores arbitrários com [colchetes] permitem fugir do design system pontualmente.
✓ Modificadores (hover:, md:, dark:) compõem livremente como prefixos.
✓ A ordem das classes no atributo não importa; o Tailwind ordena o CSS gerado.
✓ Para conflitos em componentes React, use tailwind-merge.
✓ safelist força inclusão de classes vindas de fontes externas — exceção, não regra.

# Cores no v4 — OKLCH

Paleta em OKLCH no v4
No Tailwind v4, a paleta padrão foi reescrita em OKLCH — um espaço de cor perceptualmente uniforme. Na prática isso significa dois ganhos: os tons são mais consistentes visualmente entre cores diferentes, e as cores em displays P3 (iPhone, MacBook) ficam mais vibrantes. A API não muda — você ainda escreve bg-blue-500. O que muda é o valor CSS gerado: oklch(0.623 0.214 259.8) em vez de rgb(59 130 246). Para customizar no v4, você define cores em OKLCH no bloco @theme (cap. 20).

Espaçamento

Espaçamento é a categoria que você vai usar mais — padding, margin, gap, e a escala que governa todos eles. Tailwind define uma escala numérica única baseada em incrementos de 4 pixels. # Gap — espaço entre filhos

O gap aplica espaço entre filhos diretos de um container flex ou grid — sem precisar mexer em margins individuais. É a forma moderna e idiomática de espaçar elementos.

gap.html
<!-- Gap em flex -->
<div class="flex gap-4">
  <div>Item 1</div>
  <div>Item 2</div>
  <div>Item 3</div>
  <!-- 1rem entre cada item, automático -->
</div>

<!-- Gap em grid -->
<div class="grid grid-cols-3 gap-6">

<!-- Gap por eixo -->
<div class="grid grid-cols-3 gap-x-4 gap-y-8">
  <!-- 1rem entre colunas, 2rem entre linhas -->
</div>
Prefira gap a margin entre filhos
Antes de gap ser amplamente suportado, espaçava-se filhos com space-x-* e space-y-*, ou com margins específicas. Hoje, gap funciona em todos os browsers modernos (incluindo Safari desde 14.1) e é mais limpo: o espaço é responsabilidade do container, não dos filhos. Resultado: você pode adicionar/remover/reordenar filhos sem ajustar margins.

<!-- flex-basis: tamanho inicial antes de grow/shrink -->
<div class="basis-0">     <!-- flex-basis: 0 -->
<div class="basis-auto">  <!-- flex-basis: auto -->
<div class="basis-1/2">   <!-- flex-basis: 50% -->
<div class="basis-64">    <!-- flex-basis: 16rem -->

<!-- Atalhos mais usados -->
<div class="flex-1">      <!-- flex: 1 1 0% — cresce e encolhe livremente -->
<div class="flex-auto">   <!-- flex: 1 1 auto — respeita o tamanho natural -->
<div class="flex-none">   <!-- flex: none — tamanho fixo, não cresce nem encolhe -->
<div class="flex-initial"><!-- flex: 0 1 auto — encolhe mas não cresce (padrão) -->
flex-1 vs flex-auto vs flex-none
flex-1 é o mais usado — distribui espaço igualmente entre itens que o têm. flex-none é para itens com largura fixa (ícones, avatares, labels). flex-auto respeita o tamanho natural do conteúdo antes de distribuir o espaço restante. Na prática, 80% dos casos são flex-1 nos itens que crescem e flex-none nos que precisam ter tamanho fixo.

Grid vs Flexbox — a decisão rápida
Use Flexbox quando:
✓ O layout é em uma dimensão (só linha ou só coluna).
✓ Os itens têm tamanhos variados e você quer distribuição natural.
✓ É um componente (botão, navbar, chip).

Use Grid quando:
✓ O layout é em duas dimensões (linhas e colunas).
✓ Você quer controle preciso sobre posições e spans.
✓ É estrutura de página (dashboard, formulário, galeria).

Na prática, a maioria dos UIs usa os dois: grid no macro layout da página, flexbox nos componentes internos.

ring usa box-shadow para criar um anel ao redor do elemento — sem interferir no layout. É o mecanismo preferido para indicadores de foco e estados de seleção no Tailwind.

focus vs focus-visible
focus: dispara em qualquer foco — inclusive cliques com mouse (o que cria rings visuais desnecessários em botões clicados). focus-visible: dispara apenas quando o usuário está navegando por teclado — o comportamento correto para indicators de foco. Sempre prefira focus-visible:ring-* em botões e links. Mantenha focus:border-* em inputs de formulário, onde feedback visual no clique é desejado.

Ring vs Border vs Outline — quando usar cada um
border: delimitação permanente de elementos (cards, inputs, tabelas). Faz parte do box model — ocupa espaço.

outline: acessibilidade nativa do browser. Remova com outline-none apenas se substituir por ring ou border equivalente. Nunca remove sem alternativa.

<div class="backdrop-blur-lg">   <!-- blur: 16px -->
<div class="backdrop-blur-xl">   <!-- blur: 24px -->
<div class="backdrop-blur-3xl">  <!-- blur: 64px -->

<div class="backdrop-grayscale">       <!-- fundo em preto e branco -->
<div class="backdrop-saturate-150">    <!-- fundo mais saturado -->
<div class="backdrop-invert">          <!-- fundo invertido -->

  backdrop-blur-md
  bg-white/10
  border border-white/20
  rounded-2xl
  shadow-xl
  p-6
">
  Vidro fosco sobre fundo colorido
</div>

backdrop-blur precisa de fundo semitransparente
backdrop-blur só tem efeito visível quando o elemento tem alguma transparência (bg-white/70, bg-zinc-900/60). A fórmula do glassmorphism é sempre: backdrop-blur-* + bg-{cor}/{opacidade} + border border-white/{opacidade}.

Use com moderação
Sombras e filtros são poderosos — e por isso perigosos. Algumas diretrizes práticas:
✓ Uma sombra por nível de hierarquia (base, card, modal, tooltip) — não misture shadow-lg em tudo.
✓ shadow-sm para elementos sutis, shadow-md para cards, shadow-xl para modais.
✓ Sombras coloridas em elementos de destaque único — botão primário, hero card.
✓ backdrop-blur com bg/70 para headers sticky — padrão consagrado.
✓ Filtros em imagens (grayscale, hover saturate) para galerias e portfólios.
✗ Evite múltiplas sombras pesadas na mesma tela — cria confusão de profundidade.

Estados e modificadores

Este é o capítulo onde Tailwind realmente se diferencia de "CSS inline". Modificadores como hover:, focus:, group-*, peer-* e has-* permitem estilos condicionais que só seriam possíveis em CSS tradicional via seletores complexos. Aqui está a parte em que utility-first deixa de parecer apenas conveniência e começa a parecer poder real.

# Os estados básicos — hover, focus, active

estados-basicos.html
<!-- Cada estado é um prefixo. Combina com qualquer utility. -->
<button class="
  bg-blue-500
  hover:bg-blue-600         /* :hover */
  focus:ring-2              /* :focus (qualquer foco) */
  focus-visible:ring-blue-500 /* :focus-visible (só teclado) */
  active:bg-blue-700        /* :active (durante o clique) */
  disabled:opacity-50       /* :disabled */
  disabled:cursor-not-allowed
">
  Botão completo
</button>

Empilhar modificadores funciona naturalmente. md:hover:bg-blue-600 aplica apenas em telas md+ e durante hover. dark:hover:bg-blue-700 aplica apenas em dark mode durante hover. A ordem dos modificadores não importa — Tailwind compila tudo para CSS válido.

# focus vs focus-visible — uma distinção crítica

focus-distincao.html
<!-- focus dispara em QUALQUER foco — incluindo cliques de mouse -->
<button class="focus:ring-2 focus:ring-blue-500">
  <!-- ❌ Ring aparece após CLIQUE também — feio em UI de mouse -->
</button>

<!-- focus-visible dispara apenas com TECLADO (Tab navigation) -->
<button class="focus-visible:ring-2 focus-visible:ring-blue-500">
  <!-- ✅ Ring aparece SÓ ao navegar por teclado — comportamento correto -->
</button>

<!-- Padrão correto para botões e links: -->
<button class="
  px-4 py-2 bg-blue-500 text-white rounded-lg
  focus-visible:outline-none
  focus-visible:ring-2 focus-visible:ring-blue-500 focus-visible:ring-offset-2
">
  Botão acessível
</button>

<!-- INPUTS são exceção — use focus normal:
     o usuário sempre quer feedback visual ao focar um campo,
     mesmo clicando com mouse -->
<input class="
  border border-zinc-300
  focus:outline-none
  focus:border-blue-500 focus:ring-2 focus:ring-blue-500/30
" />
A regra prática
focus-visible: em botões e links — onde o ring após clique fica feio. focus: em inputs e elementos de formulário — onde feedback ao clique é desejado. Esta distinção foi um problema que durou anos no CSS — :focus-visible resolve, mas só funciona se você usar a utility correta. Vale memorizar.

# Group — estilizar filhos baseado no pai

O modificador group resolve um problema clássico do CSS: como aplicar um estilo a um filho quando o pai recebe hover/focus?. Em CSS você precisaria de .parent:hover .child. Em Tailwind você marca o pai como group e usa group-hover: nos filhos.

group-basico.html
<!-- O pai recebe a classe `group` -->
<a href="#" class="group flex items-center gap-2">
  <!-- Filhos usam group-hover: para reagir ao hover do pai -->
  <span class="group-hover:text-blue-500">Texto</span>
  <Icon class="group-hover:translate-x-1 transition-transform" />
</a>

<!-- Quando o usuário passa o mouse sobre o <a>, ambos filhos reagem:
     - texto fica azul
     - ícone se move 4px para a direita
     Isso seria CSS chato com .group:hover .child -->

<!-- Funciona com qualquer estado, não só hover -->
<div class="group">
  <span class="group-focus:bg-blue-100">...      <!-- pai focado -->
  <span class="group-active:scale-95">...      <!-- pai sendo clicado -->
  <span class="group-disabled:opacity-50">...   <!-- pai desabilitado -->
</div>

# Peer — estilizar irmão baseado em irmão anterior

Se group resolve "filho reage ao pai", peer resolve "irmão reage ao irmão anterior". É especialmente útil para inputs e seus labels — sem JavaScript.

peer-basico.html
<!-- O elemento que QUEREMOS OBSERVAR recebe `peer` -->
<!-- O elemento IRMÃO POSTERIOR usa peer-* para reagir -->

<div>
  <input type="checkbox" class="peer" />
  <span class="peer-checked:line-through peer-checked:text-zinc-400">
    <!-- texto risca quando o checkbox é marcado, sem JavaScript -->
    Comprar leite
  </span>
</div>

<!-- Floating label com peer -->
<div class="relative">
  <input
    id="email"
    type="email"
    placeholder=" "
    class="peer w-full border-b-2 border-zinc-300 pt-6 pb-2 outline-none
          focus:border-blue-500"
  />
  <label
    for="email"
    class="absolute left-0 top-4 text-zinc-500 transition-all
          peer-placeholder-shown:top-4 peer-placeholder-shown:text-base
          peer-focus:top-0 peer-focus:text-xs peer-focus:text-blue-500
          peer-[:not(:placeholder-shown)]:top-0 peer-[:not(:placeholder-shown)]:text-xs"
  >
    Email
  </label>
</div>

<!-- Quando input está vazio: label centralizado
     Quando input tem foco ou texto: label sobe e fica pequeno
     Tudo CSS puro. -->
peer só funciona em irmãos POSTERIORES
A limitação técnica vem do CSS: o seletor ~ (irmão geral) só olha para frente. peer pode estar antes do elemento que reage, mas nunca depois. Por isso, em formulários, o input deve vir antes do label se você quiser que o label use peer-* baseado no estado do input. Se a ordem visual desejada for diferente, use Flexbox com order para reorganizar visualmente sem mudar a ordem do HTML.

# has-* — estilizar pai baseado em filho

O modificador has-* implementa o seletor :has() do CSS — permite estilizar um elemento baseado no que tem dentro. É o inverso semântico de group.

has.html
<!-- Estilizar o <label> quando o <input> dentro está marcado -->
<label class="
  flex items-center gap-2 p-3 rounded-lg border
  has-[:checked]:border-blue-500
  has-[:checked]:bg-blue-50
">
  <input type="radio" name="plano" class="accent-blue-500" />
  Plano Pro
</label>

<!-- Estilizar form quando contém um input inválido -->
<form class="has-[input:invalid]:border-red-500">
  ...
</form>

<!-- Estilizar card quando tem uma imagem dentro -->
<article class="p-6 has-[img]:p-0 has-[img]:overflow-hidden">
  ...
</article>

<!-- Card de seleção que reage ao radio interno -->
<label class="
  block cursor-pointer rounded-xl border-2 border-zinc-200 p-6
  has-[:checked]:border-blue-500
  has-[:checked]:ring-2 has-[:checked]:ring-blue-500/20
  hover:border-zinc-300
  transition-all
">
  <input type="radio" name="opcao" class="sr-only" />
  <h3>Plano Pro</h3>
  <p>R$ 49/mês</p>
</label>
has-* é a "view-from-above" do CSS
Por décadas, CSS só conseguia descer na árvore (.pai .filho) ou andar lateralmente (.irmao + .vizinho). Subir era impossível — uma das maiores limitações da linguagem. :has() finalmente resolve isso, e has-* é a interface idiomática do Tailwind. Ainda é relativamente novo (suportado em todos os browsers modernos desde 2023), mas já é o padrão de muitos componentes em design systems modernos.

# O modelo mental dos modificadores

modelo-mental.txt
# Cinco perguntas para escolher o modificador certo:

1. O estado é DESTE elemento?
   → hover:, focus:, active:, disabled:, invalid:

2. O estado é do PAI?
   → group + group-hover:, group-focus:, etc.

3. O estado é de um IRMÃO ANTERIOR?
   → peer + peer-checked:, peer-focus:, peer-invalid:, etc.

4. O estado é de um FILHO?
   → has-[seletor]: no pai

5. O estado vem de ATRIBUTO HTML?
   → aria-*: para acessibilidade
   → data-*: para bibliotecas de componentes

# Quando empilhar múltiplos modificadores:

Ordem natural de leitura:
{contexto}:{trigger}:{utility}

md:dark:hover:bg-zinc-800
└─ tela md+
   └─ E dark mode
      └─ E hover
         └─ bg-zinc-800

group-hover:peer-checked:rotate-90
└─ pai em hover
   └─ E peer marcado
      └─ rotacionar 90°

# A ordem que você escreve não importa para o CSS gerado, mas
# uma ordem consistente facilita leitura. Convenção sugerida:
# breakpoint → tema → estado de pai/peer → estado próprio → pseudo-elemento
Onde Tailwind realmente vence
Este capítulo é o argumento mais forte de utility-first. Em CSS tradicional, criar um card com 5 efeitos coordenados ao hover (imagem zoom, título de cor, ícone deslizando, sombra surgindo, borda mudando) requer:
✓ Nomear classes (.card-image, .card-title, .card-icon...).
✓ Escrever seletores aninhados (.card:hover .card-image).
✓ Manter os arquivos CSS sincronizados com a marcação.

Em Tailwind, é um group no card pai e group-hover: nos filhos. Tudo no mesmo arquivo, tudo visível ao ler o JSX. Esta é a parte que faz Tailwind escalar em codebases grandes — não a economia de teclas digitadas. É a redução da complexidade conceitual.

Responsivo

Tailwind é mobile-first por padrão — você escreve estilos para mobile sem prefixo, e usa breakpoints como modificadores para sobrescrever em telas maiores. O sistema é simples: cinco breakpoints padrão (sm, md, lg, xl, 2xl), aplicados como prefixo a qualquer utility. Container queries (v3.4+ via plugin, nativo no v4) adicionam outra dimensão: estilos baseados no tamanho do container, não da viewport.

# Mobile-first — a lógica

mobile-first.html
<!-- Sem prefixo = base (mobile) — sempre aplica -->
<!-- Prefixo = aplica APENAS naquele breakpoint OU MAIOR -->

<div class="text-base md:text-lg lg:text-xl">
  <!-- mobile: text-base
       md+:    text-lg
       lg+:    text-xl -->
</div>

<!-- ❌ ANTI-PATTERN: pensar como max-width (desktop-first) -->
<div class="text-xl md:text-base">
  <!-- ❌ Em mobile pequeno usa text-xl, em md+ usa text-base
       Funciona, mas vai contra o paradigma do Tailwind. -->
</div>

<!-- ✅ Pense de mobile para cima:
     "Comece pequeno. O que muda quando a tela cresce?" -->

<!-- Layout que muda gradualmente -->
<div class="
  flex flex-col gap-4
  md:flex-row md:gap-6
  lg:gap-8
">
  <!-- mobile: coluna empilhada com gap-4
       tablet+: linha lado-a-lado com gap-6
       desktop+: gap maior (8) -->
</div>
Por que mobile-first vence
Mobile-first não é só "filosofia" — é estratégia técnica. Mobile tem mais constraints (tela pequena, dedos imprecisos, conexão limitada). Se você começa pelo desktop e depois "adapta", quase sempre acaba com layouts mobile que parecem versões espremidas. Começando do mobile e expandindo, cada decisão é tomada com restrições reais. O resultado: mobile que respira e desktop que aproveita o espaço extra.

# max-* — direção inversa

Às vezes você precisa de "mobile-only" — aplicar até certo ponto. Tailwind permite com prefixos max-* que invertem a lógica.

max-prefix.html
<!-- max-* aplica APENAS em telas MENORES que o breakpoint -->
<div class="max-md:hidden">
  <!-- Escondido em telas < 768px (mobile only)
       Aparece em md+ -->
</div>

<div class="max-lg:flex-col">
  <!-- Coluna em telas < 1024px
       Layout normal em lg+ -->
</div>

<!-- Comparação: -->
md:flex      /* aplica em md+ (≥768px) */
max-md:flex  /* aplica em max-md (<768px) */

<!-- Quando usar max-* vs sem prefixo:
     - "sem prefixo" = base mobile, mas aplica em TUDO
     - "max-md:" = APENAS em mobile, some em md+

     Use max-* quando o estilo é EXCLUSIVO de mobile,
     e o desktop tem comportamento totalmente diferente. -->

# Container queries

Container queries são a evolução do responsivo: estilos baseados no tamanho do container, não da viewport. Permite componentes verdadeiramente reutilizáveis — o mesmo card adapta seu layout em sidebar (estreito) e em hero section (largo).

container-queries.html
<!-- v3 — instalar plugin -->
npm install @tailwindcss/container-queries

// tailwind.config.js
plugins: [require("@tailwindcss/container-queries")]

<!-- v4 — nativo, sem plugin -->

<!-- Marcar elemento como container -->
<div class="@container">
  <!-- Filhos podem usar @md:, @lg:, etc.
       Esses prefixos são baseados no TAMANHO DO @container,
       NÃO da viewport! -->
  <div class="flex flex-col @md:flex-row @lg:gap-8">
    <!-- Quando o container tem ≥ 28rem: vira linha
         Quando o container tem ≥ 32rem: gap maior -->
  </div>
</div>

<!-- Breakpoints @ disponíveis: -->
@xs   16rem   /* 256px */
@sm   24rem   /* 384px */
@md   28rem   /* 448px */
@lg   32rem   /* 512px */
@xl   36rem   /* 576px */
@2xl  42rem   /* 672px */
@3xl  48rem   /* 768px */
@4xl  56rem   /* 896px */

<!-- Card que se adapta ao container, não à viewport -->
<div class="@container">
  <article class="flex flex-col @md:flex-row gap-4 p-4 border rounded-lg">
    <img class="w-full @md:w-32 aspect-square" />
    <div class="flex-1">
      <h3 class="@md:text-lg font-semibold">Título</h3>
      <p>Em sidebar (estreito): vira coluna
       Em hero section (largo): vira linha lado a lado</p>
    </div>
  </article>
</div>
v4 — container queries nativos
No Tailwind v4, container queries são nativas — não precisa instalar plugin. A sintaxe é a mesma: @container no pai, @md:, @lg:, etc. nos filhos. Suporte de browser para @container CSS é universal desde 2023, então pode usar com tranquilidade em projetos novos.

# Estratégia class — controle manual

// tailwind.config.js

  darkMode: "class",  // dark: só ativa quando <html class="dark">
  // ...
}

// Agora você controla o tema adicionando/removendo .dark no <html>:
document.documentElement.classList.add("dark")     // ativa dark
document.documentElement.classList.remove("dark")  // ativa light
document.documentElement.classList.toggle("dark")  // alterna

document.documentElement.classList.toggle("dark", tema === "dark")
class + system — o melhor dos dois

# Toggle de tema — implementação React

theme-toggle.tsx
import { useState, useEffect } from "react";

type Tema = "light" | "dark" | "system";

function useTema() {
  const [tema, setTemaState] = useState<Tema>(() => {
    if (typeof window === "undefined") return "system";
    return (localStorage.getItem("tema") as Tema) ?? "system";
  });

  useEffect(() => {
    const root = document.documentElement;
    const escuro =
      tema === "dark" ||
      (tema === "system" &&
        window.matchMedia("(prefers-color-scheme: dark)").matches);

    root.classList.toggle("dark", escuro);
    localStorage.setItem("tema", tema);
  }, [tema]);

  return { tema, setTema: setTemaState };
}

function BotaoTema() {
  const { tema, setTema } = useTema();

  return (
    <button
      onClick={() => setTema(t => t === "dark" ? "light" : "dark")}
      className="p-2 rounded-lg hover:bg-zinc-100 dark:hover:bg-zinc-800"
    >
      {tema === "dark" ? "☀️" : "🌙"}
    </button>
  );
}

# Animações em v4

v4 — keyframes direto no CSS
No Tailwind v4, keyframes e animações customizadas ficam no próprio arquivo CSS usando @theme, sem precisar do tailwind.config.js:

@theme { --animate-fade-up: fade-up 0.4s ease-out both; }
@keyframes fade-up { from { opacity: 0; transform: translateY(16px); } to { opacity: 1; transform: translateY(0); } }

Depois use animate-fade-up normalmente. A integração CSS-first do v4 torna customizações de animação mais naturais — você vê as regras CSS diretamente, sem camada de JavaScript intermediária.
Regras de animação que envelhecem bem
✓ Menos é mais. Uma animação bem escolhida tem mais impacto que cinco medianas.
✓ Hover transitions são obrigatórios. transition-colors duration-150 em botões e links não é opção — é polimento mínimo.
✓ Use motion-safe: em qualquer animação que não seja crítica para o funcionamento.
✓ ease-out para entradas (parece natural), ease-in para saídas, ease-in-out para toggles.
✓ Evite transition-all em produção — use o modificador específico (transition-colors, transition-transform).
✓ animate-pulse para skeleton e animate-spin para loaders — são as duas mais usadas do dia a dia.

Customização

A paleta padrão do Tailwind cobre 80% dos projetos — mas o que faz Tailwind viável em design systems sérios é a facilidade de customizar. Este capítulo cobre como estender o tema, criar utilities próprias, integrar tokens de design, e a diferença mental entre os dois modelos: extend (adicionar) vs override (substituir). Entender essa distinção é o que separa quem usa Tailwind como ferramenta de quem usa como base de design system.

# Anatomia do tailwind.config.js

config-anatomy.js
/** @type {import('tailwindcss').Config} */
export default {
  // 1. Onde escanear classes (cap. 2)
  content: ["./src/**/*.{ts,tsx}"],

  // 2. Modo dark (cap. 16)
  darkMode: "class",

  // 3. Tema — onde a customização acontece
  theme: {
    // 3a. SUBSTITUI completamente o que é definido aqui
    screens: {
      tablet: "640px",
      desktop: "1024px",
    },
    // ⚠️ Acima: substitui sm/md/lg/xl/2xl pelos seus próprios breakpoints

    // 3b. EXTENDS adiciona sem remover o padrão
    extend: {
      colors: { brand: "#f0a830" },
      fontFamily: { display: ["Fraunces"] },
    },
  },

  // 4. Plugins — utilities customizadas (mais adiante)
  plugins: [],
};

O ponto crítico — e a fonte mais comum de confusão em projetos Tailwind — é a diferença entre escrever uma chave diretamente em theme versus dentro de theme.extend. A primeira substitui tudo da configuração padrão. A segunda adiciona sem remover. Esta distinção tem implicações profundas.

extend-vs-override.js
theme: {
  // ❌ SUBSTITUI a paleta inteira — agora só tem 2 cores!
  // bg-zinc-500, text-blue-600 etc. param de funcionar.
  colors: {
    primary: "#f0a830",
    secondary: "#1a1410",
  },
}

theme: {
  extend: {
    // ✅ ADICIONA — paleta padrão continua disponível
    // bg-zinc-500 funciona, e bg-primary também.
    colors: {
      primary: "#f0a830",
      secondary: "#1a1410",
    },
  },
}
Quando você QUER substituir
Em design systems maduros, faz sentido substituir o tema inteiro — você não quer que bg-amber-500 funcione se sua marca não usa âmbar. Isso disciplina o time a só usar tokens aprovados. Mas a regra prática é: na maioria dos projetos, sempre use extend. Substituir o tema é uma decisão arquitetural deliberada, não algo que se faz "porque parece mais limpo".

## Princípios das boas escalas:

- 50/100: backgrounds sutis, badges, hover de items
- 200/300: bordas, separadores
- 400: ícones secundários, placeholder
- 500: tom "padrão" — o "azul" da marca
- 600: hover do 500
- 700/800: texto sobre fundos claros
- 900/950: dark mode background, alta hierarquia

## Anti-padrões:

- Usar saturação igual em todos os tons (escala fica plana)
- Tons claros muito saturados (parecem "pastéis falsos")
- Tons escuros sem matiz (vira "sujo" em vez de profundo)

# Plugins — utilities customizadas

Quando customizar valores não basta — você precisa de uma utility nova — entram os plugins. Um plugin é uma função que recebe helpers do Tailwind e adiciona regras CSS, components, ou utilities ao bundle final.

plugin-basico.js
import plugin from "tailwindcss/plugin";

export default {
  // ...
  plugins: [
    plugin(function ({ addUtilities, addComponents, theme }) {

      // addUtilities — adiciona utilities atômicas
      addUtilities({
        ".text-balance": {
          "text-wrap": "balance",
        },
        ".text-pretty": {
          "text-wrap": "pretty",
        },
        // uso: <h1 class="text-balance">...</h1>
      });

      // addComponents — para padrões compostos
      addComponents({
        ".btn-primary": {
          "background-color": theme("colors.blue.500"),
          "color": theme("colors.white"),
          "padding": `${theme("spacing.2")} ${theme("spacing.4")}`,
          "border-radius": theme("borderRadius.lg"),
          "&:hover": {
            "background-color": theme("colors.blue.600"),
          },
        },
      });
    }),
  ],
};
Quando criar plugin vs quando usar componente
addComponents tenta resolver o que React/Vue/Svelte já resolvem: encapsular padrões. Em projetos com framework de componentes, prefira criar um componente <Button variant="primary"> em vez de uma classe .btn-primary via plugin. Plugins são úteis para utilities atômicas que faltam (text-balance, scroll-snap, container queries) e para projetos sem framework (HTML estático, server-rendered tradicional).

# Tokens de design — a mentalidade

Customizar Tailwind com gosto não é sobre criar 50 cores próprias. É sobre traduzir as decisões de design em tokens reutilizáveis. Tokens são as variáveis fundamentais — cores, espaçamentos, tipografia, sombras — das quais todas as outras decisões derivam.

tokens-mentalidade.js
// ❌ Tokens de "implementação" — descrevem como, não o quê
colors: {
  "red-500": "#ef4444",      // só está renomeando uma cor
  "sidebar-bg": "#f5f5f4",    // se mudar de sidebar pra topbar, o nome desabona
}

// ✅ Tokens semânticos — descrevem o significado
colors: {
  // Tokens primitivos (a paleta crua)
  slate: { /* paleta */ },
  blue:  { /* paleta */ },

  // Tokens semânticos (o vocabulário do produto)
  background:  "hsl(var(--background))",
  foreground:  "hsl(var(--foreground))",
  muted:       {
    DEFAULT:    "hsl(var(--muted))",
    foreground: "hsl(var(--muted-foreground))",
  },
  primary:     {
    DEFAULT:    "hsl(var(--primary))",
    foreground: "hsl(var(--primary-foreground))",
  },
  destructive: {
    DEFAULT:    "hsl(var(--destructive))",
    foreground: "hsl(var(--destructive-foreground))",
  },
}

O padrão de duas camadas (primitivos + semânticos) é o que sustenta design systems sérios. Os primitivos são os "ingredientes"; os semânticos são as "receitas". Componentes só consomem semânticos — se você precisar trocar a paleta inteira, muda os primitivos e o restante segue. É a mesma ideia das CSS Variables, integrada ao Tailwind.

# Modo CSS-first do v4

v4 — sem tailwind.config.js
No Tailwind v4, customização migra para o CSS via @theme. Tokens semânticos, paletas, fontes — tudo declarado em CSS. Isso une CSS variables nativas com o sistema de tokens do Tailwind, eliminando a duplicação anterior (definir no JS, copiar para :root). Cobriremos a sintaxe CSS-first no capítulo 20.
v4-customization.css
/* v4: tudo no CSS, sem tailwind.config.js */
@import "tailwindcss";

@theme {
  /* Cores — gera bg-brand, text-brand, border-brand */
  --color-brand-50:  #fef9ec;
  --color-brand-500: #f0a830;
  --color-brand-900: #7a4a10;

  /* Fontes */
  --font-display: "Fraunces", "Georgia", serif;

  /* Spacing custom */
  --spacing-13: 3.25rem;

  /* Breakpoints */
  --breakpoint-3xl: 1920px;
}

# Quando NÃO customizar

Sinais de over-customization
Customizar Tailwind tem custo. Você assume responsabilidade pela coerência da escala, atualizações futuras, onboarding de devs novos. Sinais de que você está customizando em excesso:

✗ Tokens semânticos demais (--bg-card-hover-active-disabled). Se você nomeia 5 estados de uma cor, talvez seja hora de aceitar que o sistema é um sistema, não uma especificação infinita.
✗ Substituir paletas oficiais sem necessidade. A paleta padrão do Tailwind foi refinada por anos. Use-a até realmente colidir com a marca.
✗ Spacing em escala não-relacionada (spacing.7 = 19px). Se você quebra o sistema múltiplos-de-4, perde a previsibilidade que faz Tailwind valer a pena.
✗ Plugins reinventando componentes. Se a equipe usa React, classes .btn-primary via plugin geralmente são pior do que <Button variant="primary">.

A regra das três camadas
✓ Camada 1 — Primitivos: use a paleta padrão do Tailwind (zinc, blue, etc.). Não reinvente.
✓ Camada 2 — Semânticos: defina primary, background, foreground, muted, destructive, border. Use CSS vars.
✓ Camada 3 — Componentes: escreva componentes (React/Vue) que usam APENAS tokens semânticos. Nunca bg-blue-500 direto.

Esta separação parece overhead no início — e é, em projetos de uma página. Em projetos que vão crescer e potencialmente ter dark mode, multi-tenant, ou rebranding futuro, ela paga sozinha em poucas semanas.

Diretivas e funções

Tailwind não é só utilities — é também um conjunto de diretivas CSS e funções que estendem a linguagem. @layer organiza onde seu CSS vai; @apply permite compor utilities em classes próprias; theme() dá acesso aos tokens do tema dentro do CSS; @config resolve casos de múltiplas configurações. Este capítulo cobre todas elas — e, mais importante, quando usar cada uma. Diretivas mal usadas viram a anti-receita do utility-first.

# @tailwind — as três camadas (v3)

tailwind-layers.css
/* Em v3, todo arquivo CSS principal começa com: */
@tailwind base;        /* Preflight + estilos base */
@tailwind components;  /* Camada de componentes */
@tailwind utilities;   /* Utility classes (a maioria) */

/* Cada diretiva injeta uma "camada" do CSS gerado:

   base       → reset (Preflight) + tipografia padrão + estilos de elemento
   components → padrões reutilizáveis (botões, cards) — geralmente vazia
   utilities  → bg-*, text-*, p-*, m-*, todas as utilities

   ORDEM IMPORTA: utilities devem vir DEPOIS de components,
   para que classes como `text-red-500` sobrescrevam `.btn` se necessário. */

/* Em v4, isso vira uma única importação: */
@import "tailwindcss";
/* As três camadas ainda existem internamente, mas a sintaxe simplificou. */

# @layer — onde meu CSS vai?

Quando você escreve CSS próprio em um projeto Tailwind, esse CSS precisa "saber" em qual camada se encaixa. @layer resolve isso, garantindo a ordem correta de cascade.

layer.css
@tailwind base;
@tailwind components;
@tailwind utilities;

/* @layer base — estilos de ELEMENTO (h1, body, a) */
@layer base {
  html {
    scroll-behavior: smooth;
  }
  body {
    font-feature-settings: "cv11", "ss01";  /* features OpenType */
  }
  h1, h2, h3 {
    text-wrap: balance;          /* títulos com quebra balanceada */
  }
  [type="text"], textarea {
    @apply px-3 py-2 border rounded-lg;  /* defaults globais para inputs */
  }
}

/* @layer components — padrões reutilizáveis NOMEADOS */
@layer components {
  .card {
    @apply rounded-xl border border-zinc-200 bg-white p-6 shadow-sm;
  }
  .btn-primary {
    @apply px-4 py-2 bg-blue-500 text-white rounded-lg hover:bg-blue-600 transition;
  }
}

/* @layer utilities — utilities adicionais que não existem no Tailwind */
@layer utilities {
  .text-balance {
    text-wrap: balance;
  }
  .scrollbar-thin {
    scrollbar-width: thin;
  }
}

A diferença entre as três camadas não é puramente organizacional — afeta especificidade e cascade. Estilos de base têm a menor prioridade (html { ... } é fácil de sobrescrever). Estilos de components têm prioridade média. Utilities têm a maior prioridade — por isso class="bg-red-500" sempre vence sobre uma classe customizada de component. Sem @layer, sua classe customizada poderia "vencer" utilities que tentam sobrescrevê-la, criando comportamento errático.

Por que CSS sem @layer dá problema
Imagine que você define .btn-primary em um arquivo CSS qualquer (sem @layer) e depois escreve <button class="btn-primary bg-red-500"> querendo sobrescrever o fundo. Resultado: a cor do .btn-primary ganha porque utilities são definidas antes dele no arquivo final. Com @layer components, o .btn-primary entra na camada certa, e bg-red-500 (que está em utilities) sobrescreve corretamente. Sempre coloque seu CSS customizado dentro de uma @layer.

# @apply — compondo utilities

apply.css
@layer components {
  /* @apply "expande" utilities Tailwind dentro de uma classe própria */
  .btn {
    @apply px-4 py-2 rounded-lg font-medium transition-colors;
  }
  .btn-primary {
    @apply btn bg-blue-500 text-white hover:bg-blue-600;
    /* @apply pode incluir outras classes customizadas (.btn) */
  }

  /* Modificadores funcionam dentro de @apply: */
  .input {
    @apply border-zinc-300 focus:border-blue-500 focus:ring-2 focus:ring-blue-500/20;
  }

  /* Dark mode também: */
  .card {
    @apply bg-white dark:bg-zinc-900 border border-zinc-200 dark:border-zinc-800;
  }

  /* IMPORTANTE — @apply gera CSS REAL em build time.
     Se você nunca usa .btn no HTML, ela ainda fica no CSS final
     (diferente de utilities, que são tree-shaken). */
}

# Quando @apply é certo — e quando é uma armadilha

apply-quando-usar.css
/* ✅ USOS LEGÍTIMOS de @apply */

/* 1. Estilizando elementos HTML que você não controla diretamente */
/* (markdown renderizado, conteúdo de CMS, prosa) */
@layer base {
  .markdown h2 {
    @apply text-2xl font-semibold mt-8 mb-4;
  }
}

/* 2. Resets de elemento (estilos globais base) */

  [type="checkbox"] {
    @apply size-4 rounded text-blue-500;
  }
}

/* 3. Projetos sem framework de componentes (HTML estático) */
/* — caso onde você precisa de classes nomeadas para reuso */

/* ❌ ANTI-PADRÕES de @apply */

/* 1. ⚠️ Recriando "todos os componentes" como classes (.btn-primary, .card-header)
   — Isso é "voltar ao CSS tradicional" usando Tailwind como pré-processador.
   Em projetos React/Vue/Svelte, prefira COMPONENTES: */
/* RUIM: */

/* Uso: <button class="btn-primary">...</button> */

/* BOM: */

}

/* 2. ⚠️ Usar @apply para "limpar HTML longo"
   — A solução é componentização, não recompactação CSS.

/* 3. ⚠️ @apply para um único uso
   — Se .my-class só aparece uma vez no projeto, escreva inline. */
A regra mental do @apply
A pergunta certa antes de usar @apply não é "isso fica mais limpo?", mas "isto é HTML que eu não controlo via componentes?". Se a resposta for "sim" (markdown de CMS, biblioteca de terceiros injetando HTML, conteúdo legacy), @apply é a ferramenta certa. Se a resposta for "não, é meu próprio React" — crie um componente. Componentes encapsulam comportamento; classes CSS encapsulam apenas aparência. A primeira é mais poderosa em arquiteturas modernas.

# CSS variables como tokens semânticos

O padrão mais robusto para temas em Tailwind v3 é definir tokens semânticos como CSS variables, e depois mapeá-las no tailwind.config. Isso permite trocar tema inteiro mudando valores no :root — sem mexer em classes ou componentes.

vars-tokens.css
@tailwind base;
@tailwind components;
@tailwind utilities;

@layer base {
  :root {
    /* Tokens semânticos — light theme */
    --background:             0 0% 100%;           /* HSL sem hsl() */
    --foreground:             240 10% 3.9%;
    --muted:                  240 5% 96%;
    --muted-foreground:       240 4% 46%;
    --primary:                217 91% 60%;
    --primary-foreground:     0 0% 100%;
    --border:                 240 6% 90%;
    --ring:                   217 91% 60%;
    --radius:                 0.5rem;
  }

  .dark {
    /* Mesmos tokens, valores diferentes */
    --background:             240 10% 3.9%;
    --foreground:             0 0% 98%;
    --muted:                  240 3.7% 15.9%;
    --muted-foreground:       240 5% 64.9%;
    --primary:                217 91% 60%;
    --border:                 240 3.7% 15.9%;
  }

  /* Estilos base que consomem os tokens */
  * { @apply border-border; }
  body { @apply bg-background text-foreground; }
}

/* No tailwind.config (já visto no cap. 18): */
// colors: {
//   background: "hsl(var(--background))",
//   primary: { DEFAULT: "hsl(var(--primary))", ... },
// }

/* RESULTADO:
   - <div class="bg-primary text-primary-foreground"> funciona em ambos os temas
   - Trocar de tema = mudar uma classe no <html> (.dark / sem .dark)
   - Multi-tenant fica trivial: cada tenant tem seu próprio :root */

Este é exatamente o padrão usado pelo shadcn/ui e adotado como referência por design systems modernos. As CSS vars são "buracos" que o tema preenche; classes Tailwind apontam para esses buracos via hsl(var(--primary)); e o tema muda alterando os valores no :root ou em uma classe estratégica como .dark.

# Por que HSL sem hsl() nas CSS vars

hsl-trick.css
/* ❌ Token guarda valor "completo" */
:root { --primary: hsl(217 91% 60%); }

/* Problema: usar com opacidade fica difícil */
<div class="bg-primary/50">...
/* O Tailwind precisa decompor a cor para aplicar opacidade.
   Com hsl() embutido, ele não consegue. */

/* ✅ Token guarda só os COMPONENTES */
:root { --primary: 217 91% 60%; }

/* No config, monta o hsl(): */
colors: {
  primary: "hsl(var(--primary) / <alpha-value>)",
}

/* Agora bg-primary, bg-primary/50, bg-primary/[0.7] funcionam todos. */

/* ALTERNATIVA — em v4 ou cores OKLCH, padrão é diferente: */
:root {
  --color-primary: oklch(0.62 0.21 259);
  /* v4 lida com opacidade nativamente, sem o truque do alpha-value */
}

@variant hocus (&:hover, &:focus);
/* Permite usar hocus:bg-blue-500 — combina hover e focus */

@variant dark (&:where(.dark, .dark *));
/* Customiza o seletor do dark mode */
v4 — diretivas mais poderosas
No v4, a fronteira entre "config Tailwind" e "CSS" praticamente desaparece. @theme, @utility, @variant permitem definir tokens, criar utilities e modificadores diretamente no CSS, sem JavaScript intermediário. Isso simplifica o mental model significativamente: seu CSS é a fonte da verdade. O capítulo 20 cobre o impacto disso na arquitetura de projetos novos.

A regra mental para diretivas
✓ @layer base — quando você precisa estilizar elementos HTML diretamente (body, h1, input) ou definir CSS variables.
✓ @layer utilities — quando uma utility falta no Tailwind e seria útil em vários lugares (scrollbar-thin, bg-grid, text-balance).
✓ @layer components — só para projetos sem framework de componentes (HTML estático). Em React/Vue, evite — use componentes.
✓ @apply — para HTML que você não controla (markdown, conteúdo de CMS) ou estilos de elemento globais. Não para "limpar" classes longas — isso é trabalho de componentes.
✓ theme() — quando precisa de valores do tema dentro de CSS customizado (gradientes, fundos complexos, cálculos).
✓ CSS variables semânticas — para temas dinâmicos (light/dark/multi-tenant). Padrão moderno consagrado.
Capítulo 20

Tailwind v4

A versão 4 do Tailwind, lançada em janeiro de 2025, é a maior reformulação do framework desde sua criação. Não é uma evolução incremental — é uma reescrita do engine, da arquitetura de configuração, e do modelo mental de customização. Este capítulo cobre o que mudou, por que importa, e como decidir entre v3 e v4 em projetos novos. Para quem aprendeu Tailwind antes de 2025, este capítulo é tanto sobre aprender o novo quanto desaprender o velho.

# O que mudou — visão geral

# Mudanças categóricas no Tailwind v4:

ENGINE
  v3: JavaScript (PostCSS plugin)
  v4: Rust (Lightning CSS) — ~10× mais rápido

CONFIGURAÇÃO
  v3: tailwind.config.js — arquivo JS com objeto exportado

IMPORTAÇÃO

  v4: @tailwindcss/vite — nativo, sem PostCSS

CONTAINER QUERIES
  v3: @tailwindcss/container-queries (plugin)
  v4: nativo

  v3: rgb(...) por padrão, paleta atualizada periodicamente
  v4: oklch(...) nativamente — gama maior, mistura mais natural

# CSS-first — a mudança fundamental

A maior mudança conceitual do v4 não é técnica — é filosófica. Em v3, sua configuração vivia em JavaScript: um arquivo tailwind.config.js exportava um objeto que descrevia o tema, e o Tailwind processava isso em build. Em v4, sua configuração vive em CSS, usando a diretiva @theme. CSS volta a ser a fonte da verdade.

v3-config.js
// v3 — configuração em JavaScript
// tailwind.config.js
export default {
  content: ["./src/**/*.{ts,tsx}"],
  theme: {
    extend: {
      colors: {
        brand: {
          500: "#f0a830",
          900: "#7a4a10",
        },
      },
      fontFamily: {
        display: ["Fraunces", "Georgia"],
      },
    },
  },
};

// + globals.css separado:
// @tailwind base;
// @tailwind components;
// @tailwind utilities;
// + :root { --primary: ...; } separado também

// = TRÊS lugares para configurar uma cor
v4-config.css
/* v4 — TUDO no CSS */
@import "tailwindcss";

@theme {
  /* Paletas — gera bg-brand-500, text-brand-500, etc. */
  --color-brand-500: #f0a830;
  --color-brand-900: #7a4a10;

  /* Fontes — gera font-display */
  --font-display: "Fraunces", "Georgia", serif;
}

/* É isso. Um arquivo. Uma fonte da verdade. */

A elegância dessa mudança vai além de "menos arquivos". Em v3, havia uma tradução contínua — você definia algo em JS, o Tailwind gerava CSS. Para usar em CSS customizado, precisava da função theme(). Para usar em runtime, precisava de CSS variables paralelas. Em v4, @theme declara CSS variables reais que tanto o Tailwind quanto seu CSS customizado consomem diretamente. A duplicação some.

@theme gera CSS vars automaticamente
Quando você escreve --color-brand-500: #f0a830 dentro de @theme, o Tailwind faz duas coisas: (1) gera as utilities bg-brand-500, text-brand-500, etc., e (2) expõe a CSS variable --color-brand-500 no :root do documento, acessível via var(--color-brand-500) em qualquer CSS. Isso elimina a necessidade de declarar variáveis paralelas — o tema do Tailwind é o conjunto de CSS variables.

# Cores em OKLCH — por que importa

oklch.txt
## OKLCH vs RGB — a diferença prática

A paleta padrão do v4 está em OKLCH (não RGB). É uma mudança técnica
com benefícios reais que aparecem na tela:

1. CONSISTÊNCIA PERCEPTUAL
   Em RGB, blue-500 e red-500 podem ter "luminância" diferente
   apesar do número idêntico. Em OKLCH, tons iguais parecem iguais.

2. GAMA AMPLIADA
   OKLCH suporta wide-gamut (P3, Rec.2020) — cores mais vivas
   em displays modernos (iPhone, MacBook, monitores OLED).

3. INTERPOLAÇÃO MAIS NATURAL
   Gradientes ficam mais suaves — sem aquele "cinza no meio"
   que aparecia em gradientes RGB entre cores complementares.

A SINTAXE é parecida com HSL:
   oklch(L C H)
   - L: lightness (0 a 1)
   - C: chroma (0 a ~0.4 — saturação)
   - H: hue (0 a 360 — matiz)

Exemplos:
   oklch(0.74 0.16 68)    /* amarelo dourado */
   oklch(0.62 0.21 259.8)  /* azul (blue-500) */
   oklch(0.65 0.25 27)    /* vermelho (red-500) */

API IDÊNTICA À DE V3:
   bg-blue-500, text-zinc-700, etc. — todos funcionam igual.
   A diferença é puramente visual quando renderizado em telas modernas.

## Por que isso importa

Em projetos pequenos (1-3 páginas), 100ms vs 10ms não é notado.
Mas em projetos com:
- Hot Module Replacement frequente
- Storybook com 100+ stories
- Monorepos com múltiplos pacotes
- CI/CD com builds frequentes

A diferença vira hora-trabalho. Em times grandes, 9× mais rápido em
dev = ciclos mais curtos = mais iteração. É o tipo de melhoria que
você não pede mas, depois de ter, não consegue largar.

# Migrando de v3 para v4

migracao.txt
## Caminho de migração — projeto existente

### 1. Use a CLI oficial de migração

npx @tailwindcss/upgrade@latest

  Esta CLI:
  - Atualiza package.json com dependências corretas
  - Converte tailwind.config.js → @theme em CSS
  - Atualiza @tailwind base/components/utilities → @import "tailwindcss"
  - Remove o autoprefixer (não é mais necessário)
  - Identifica utilities renomeadas/depreciadas

### 2. Verifique mudanças quebradoras

a) bg-opacity-*, text-opacity-*, etc.
   v3: ainda funcionavam (deprecated)
   v4: removidos. Use bg-blue-500/50 (modificador /).

b) shadow-{cor} sem espessura
   v3: shadow-blue-500 herdava do shadow base
   v4: você precisa especificar shadow-lg shadow-blue-500/30

c) flex-grow → grow, flex-shrink → shrink
   Renomeados (mais curtos). Os antigos foram removidos.

d) Cores em RGB → OKLCH
   API igual. Visual levemente diferente em wide-gamut.
   Em sRGB normal, praticamente indistinguível.

### 3. Browsers suportados — checar baseline

v4 requer baseline mais moderna:
  - Safari 16.4+ (março 2023)
  - Chrome 111+ (março 2023)
  - Firefox 128+ (julho 2024)

Se você precisa dar suporte a browsers anteriores, FIQUE EM V3.
Para a maioria dos projetos, esse baseline não é problema.

### 4. Plugins descontinuados

@tailwindcss/aspect-ratio   → nativo (aspect-* utilities)
@tailwindcss/container-queries → nativo (@container)
@tailwindcss/line-clamp     → nativo (line-clamp-*)

@tailwindcss/typography (prose) — ainda existe, ainda funciona.
@tailwindcss/forms — ainda existe, ainda funciona.

# Quando ficar em v3

Razões legítimas para não migrar
v3 continua suportado e recebe atualizações de segurança. Não migre se:

✗ Você dá suporte a browsers antigos. Sites e-commerce, governo, sistemas internos legados frequentemente precisam de Safari 14+ ou IE/Edge legacy. v4 não vai funcionar.
✗ Sua codebase tem 50+ plugins próprios. A arquitetura de plugins mudou. Reescrever todos pode levar dias.
✗ O projeto está em manutenção. Se ninguém vai tocar nele por anos, migrar agrega risco sem benefício.
✗ Você usa pacotes externos que ainda dependem de v3. Verifique antes (shadcn/ui, Headless UI, Radix, etc. já são v4-ready em 2025).

A migração não é obrigatória. v3 vai continuar viável por anos. Migre quando o ROI fizer sentido — não por moda.

# Palavras finais

O que ficar deste manual
Vinte capítulos depois, o que importa não é decorar nomes de classes — é internalizar o modelo mental que faz Tailwind valer a pena.

✓ Utility-first é localidade. O estilo vive onde a marcação vive. Você lê o que vê.
✓ Constraints são features. A escala limitada é o que faz o sistema funcionar — não a paleta enorme.
✓ group, peer, has-* são o diferencial. Não bg-blue-500. É a composição condicional que CSS inline nunca teve.
✓ Tokens semânticos sobrevivem ao tempo. Trocar cores em um :root é pouco. Trocar 200 ocorrências de bg-blue-500 é muito.
✓ Componentes encapsulam, classes não. Em React/Vue/Svelte, evite @apply. Crie componentes com utilities inline.
✓ v4 é o novo normal. CSS-first, OKLCH, performance Rust. Aprenda a versão atual; o resto se adapta.

Tailwind é uma ferramenta — boa e influente, mas só ferramenta. UI excelente vem de pensar bem sobre o usuário, design system maduro, e código sustentável. Tailwind acelera a parte mecânica. O resto é com você.
