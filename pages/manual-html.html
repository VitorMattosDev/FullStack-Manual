Capítulo 01

Por que semântica importa

HTML é a linguagem mais fácil de escrever errado sem perceber. O navegador renderiza quase qualquer coisa — <div>s aninhadas viram página, e a aparência final fica correta. Mas por trás da tela, há uma camada inteira de significado que o HTML bem escrito fornece e o HTML ruim esconde. Este é o capítulo que explica o porquê antes de qualquer como.

# O que "semântica" significa na prática

Semântica é a diferença entre dizer o que algo é e dizer como algo se parece. Compare:

ruim.html
<!-- Sem semântica — tudo é uma div -->
<div class="header">
  <div class="logo">Minha Empresa</div>
  <div class="menu">
    <div class="item">Home</div>
    <div class="item">Sobre</div>
  </div>
</div>
<div class="content">
  <div class="title">Bem-vindo</div>
  <div class="text">Lorem ipsum...</div>
  <div class="button" onclick="...">Clique</div>
</div>
bom.html
<!-- Com semântica — cada elemento declara o seu papel -->
<header>
  <a href="/" class="logo">Minha Empresa</a>
  <nav>
    <a href="/">Home</a>
    <a href="/sobre">Sobre</a>
  </nav>
</header>
<main>
  <h1>Bem-vindo</h1>
  <p>Lorem ipsum...</p>
  <button type="button">Clique</button>
</main>

Os dois exemplos produzem páginas visualmente idênticas com CSS. Mas só o segundo funciona para:

  • Leitores de tela — que anunciam "navegação", "cabeçalho", "botão", e não "div".
  • Buscadores — que entendem o que é título, o que é navegação, o que é conteúdo principal.
  • Navegação por teclado — <button> já é focável; <div onclick> não é.
  • Modo de leitura do navegador — depende de <main>, <article>, <h1> para extrair conteúdo.
  • Extensões e ferramentas — desde tradutores até modo de alto contraste.

# Os três públicos do HTML

Quando você escreve HTML, está escrevendo para três leitores simultâneos:

LeitorO que precisaComo responde
NavegadorEstrutura correta para renderizarPinta a tela
MáquinasSignificado (buscadores, screen readers, agentes de IA)Indexa, anuncia, interpreta
HumanosCódigo legível para manterEdita com segurança

HTML semântico é o único que atende os três ao mesmo tempo. Uma página toda feita de <div> funciona para o primeiro, decepciona o segundo e pune o terceiro.

# Acessibilidade não é opcional

Cerca de 15% da população mundial vive com alguma deficiência permanente. Some a isso as deficiências temporárias (braço quebrado, mouse sem bateria, tendinite) e situacionais (celular no sol, ambiente barulhento, uma mão ocupada com o bebê), e o número de pessoas que se beneficiam de uma página acessível chega perto de 100%.

Acessibilidade
Boa notícia: HTML semântico é acessível por padrão. Você ganha a maior parte da acessibilidade simplesmente usando as tags certas. Quase tudo que este manual ensina sobre semântica é também uma lição de acessibilidade.

# HTML Living Standard

O HTML não tem mais versões numeradas. Desde ~2014, a especificação é mantida como um Living Standard pelo WHATWG — um documento vivo, atualizado continuamente. Quando alguém diz "HTML5", geralmente está se referindo a esse HTML moderno, mas tecnicamente a versão correta é simplesmente HTML.

A fonte canônica é html.spec.whatwg.org. Para consulta amigável, o MDN Web Docs (developer.mozilla.org) é a referência de fato da indústria.

Observação
Ao contrário do que acontece com frameworks, o HTML moderno é retrocompatível para sempre. Código escrito em 1998 ainda funciona. Isso significa que boas práticas de hoje provavelmente ainda serão boas em dez anos — seu investimento em aprender HTML bem não expira.
Capítulo 02

Anatomia de um documento

Antes de mergulhar em tags específicas, é preciso ter em mente o esqueleto que todo documento HTML compartilha — o que é obrigatório, o que é convenção, e o que é simplesmente um bom hábito. Um HTML bem montado desde a primeira linha previne uma categoria inteira de bugs silenciosos.

# O documento mínimo válido

index.html
<!DOCTYPE html>
<html lang="pt-BR">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Título da página</title>
</head>
<body>
  <h1>Olá, mundo</h1>
</body>
</html>

Esse é o piso mínimo de um documento profissional. Cada linha tem um motivo:

LinhaFunção
<!DOCTYPE html>Ativa o modo padrão do navegador. Sem isso, ele entra em "quirks mode" — compatibilidade com bugs dos anos 90.
lang="pt-BR"Informa o idioma aos screen readers, buscadores e corretores ortográficos.
meta charsetDefine a codificação. UTF-8 é o único valor correto hoje.
meta viewportFaz a página se adaptar a telas móveis. Sem isso, celular mostra versão desktop encolhida.
titleAparece na aba do navegador, resultados de busca, e é o primeiro texto lido por screen readers.

HTML é feito de elementos, formados por tags e atributos. A anatomia é consistente:

<!--↑    ↑          ↑               ↑         ↑   -->
<!--tag  atributos  atributo        texto     tag-->
<!--abre                            interno   fecha-->

<!-- Elemento vazio (void) — não tem conteúdo nem tag de fechamento -->

<!-- Atributos booleanos — presença basta -->

# Entidades HTML

Alguns caracteres têm significado especial em HTML e precisam ser escapados quando aparecem como texto literal:

CaractereEntidadeQuando usar
<&lt;Sempre que for texto literal (ex: mostrar código)
>&gt;Em atributos e dentro de código
&&amp;Em URLs com parâmetros, texto literal
"&quot;Dentro de atributos com aspas duplas
espaço fixo&nbsp;Evitar quebra de linha entre palavras
Na prática
Frameworks modernos (React, Vue, Svelte) fazem esse escape automaticamente para você — é por isso que raramente você escreve &amp; à mão. Mas saber que o escape existe é essencial quando você escreve HTML cru, lida com segurança (XSS) ou debuga dados vindos de APIs.

# Validação

HTML moderno é tolerante a quase tudo — tags sem fechamento, aninhamento estranho, atributos inventados. Essa tolerância é boa para retrocompatibilidade, mas péssima para quem quer escrever código correto. Use o W3C Nu HTML Checker (validator.w3.org/nu) ou extensões do editor para pegar erros cedo.

Capítulo 03

Meta tags e o <head>

O <head> é o que o usuário não vê, mas é o que o resto do mundo vê — buscadores, redes sociais, bookmarks, PWA installers, screen readers. Uma página bem preparada aqui aparece corretamente no Google, em previews de WhatsApp, como ícone na tela inicial do celular e em modo escuro. Este capítulo cobre o que é obrigatório, o que é fortemente recomendado e o que é supérfluo.

# O essencial absoluto

head-minimo.html
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Título específico da página — Nome do site</title>
  <meta name="description" content="Uma descrição curta (150–160 caracteres) que aparece nos resultados de busca.">
</head>
Sobre o title
Cada página deve ter um <title> único e descritivo. O padrão comum é "Tópico específico — Nome do site". Evite genéricos como "Home" ou "Página 1" — tanto buscadores quanto usuários com várias abas abertas agradecem.

# Favicons e ícones

icones.html
<!-- Favicon tradicional (tamanho legado) -->
<link rel="icon" href="/favicon.ico" sizes="32x32">

<!-- Versão vetorial (SVG) — escala perfeita em qualquer tamanho -->
<link rel="icon" href="/icon.svg" type="image/svg+xml">

<!-- Ícone para iOS quando salvo na tela inicial -->
<link rel="apple-touch-icon" href="/apple-touch-icon.png">

<!-- Manifest para PWA (inclui ícones para Android, temas, etc) -->
<link rel="manifest" href="/manifest.webmanifest">

Um conjunto moderno e enxuto: um .ico (compatibilidade), um .svg (navegadores modernos), um apple-touch-icon.png (iOS) e um manifest (PWA). Gerar tudo isso à mão é tedioso — use ferramentas como realfavicongenerator.net.

# Open Graph — previews em redes sociais

Quando alguém cola o link do seu site no WhatsApp, Twitter, Discord ou LinkedIn, essas plataformas leem Open Graph (og:*) para montar o preview. Sem isso, aparece um link cru.

og.html
<!-- Open Graph — padrão do Facebook, adotado por quase todo mundo -->
<meta property="og:title" content="Título que aparece no preview">
<meta property="og:description" content="Descrição curta.">
<meta property="og:image" content="https://site.com/preview.jpg">
<meta property="og:image:alt" content="Texto alternativo da imagem de preview">
<meta property="og:url" content="https://site.com/pagina">
<meta property="og:type" content="website">
<meta property="og:site_name" content="Nome do Site">
<meta property="og:locale" content="pt_BR">

<!-- Twitter Cards — o Twitter usa og:* como fallback,
     mas dá para customizar com twitter:* -->
<meta name="twitter:card" content="summary_large_image">
<meta name="twitter:site" content="@seu_usuario">
Dimensões recomendadas
A imagem ideal para og:image é 1200×630 pixels. Use URL absoluta (com https:// completo), não relativa. Teste com o debugger do Facebook (developers.facebook.com/tools/debug) ou cards validator do Twitter.

# Canonical, robots e SEO

seo.html
<!-- URL canônica — evita conteúdo duplicado.
     Ex: /produto?ref=abc e /produto apontam para o mesmo recurso,
     e o canonical informa qual é o "oficial" para o Google indexar. -->
<link rel="canonical" href="https://site.com/produto">

<!-- Controlar indexação -->
<meta name="robots" content="index, follow">     <!-- padrão -->
<meta name="robots" content="noindex, nofollow">  <!-- não indexa, não segue links -->
<meta name="robots" content="index, follow, max-image-preview:large">

<!-- Versões alternativas em outros idiomas -->
<link rel="alternate" hreflang="en" href="https://site.com/en/produto">
<link rel="alternate" hreflang="es" href="https://site.com/es/producto">
<link rel="alternate" hreflang="x-default" href="https://site.com/produto">

# Estilos e scripts

assets.html
<!-- CSS externo (no <head> para não haver "flash" de conteúdo sem estilo) -->
<link rel="stylesheet" href="/estilo.css">

<!-- CSS inline (para critical CSS de primeira renderização) -->
<style>
  body { font-family: system-ui; }
</style>

<!-- JS externo — prefira 'defer' no <head> -->
<script defer src="/app.js"></script>

<!-- JS como módulo ESM (sempre deferred por padrão) -->
<script type="module" src="/app.js"></script>

<!-- JS inline -->
<script>
  console.log("oi");
</script>
defer vs async vs nada
Sem atributo, o script bloqueia o parsing do HTML — pior para performance. async baixa em paralelo e executa assim que possível (ordem imprevisível). defer baixa em paralelo e executa depois do HTML carregado, na ordem escrita. Na dúvida, use defer.

Capítulo 04

Landmarks e estrutura

Landmarks são as grandes regiões de uma página — cabeçalho, navegação, conteúdo principal, rodapé. Antes de HTML5, tudo isso era <div class="header">. Hoje temos tags dedicadas que anunciam a estrutura da página para tecnologias assistivas e dão a ela um mapa navegável.

# As sete landmarks principais

TagPapelRegra prática
<header>banner (quando filho direto de body)Logo, título principal, nav global
<nav>navigationGrupos de links de navegação
<main>mainO conteúdo único e principal da página
<article>articleConteúdo independente (post, card)
<section>region (quando tem nome)Seção temática dentro de um documento
<aside>complementaryConteúdo tangencial (sidebar, related)
<footer>contentinfo (quando filho direto de body)Rodapé — copyright, links secundários

        <li><a href="/produtos">Produtos</a></li>
        <li><a href="/sobre">Sobre</a></li>
      </ul>
    </nav>
  </header>

  <main>
    <h1>Título principal da página</h1>
    <p>Conteúdo principal...</p>

    <section aria-labelledby="sec-destaques">

      <!-- ... -->
    </section>

Acessibilidade
Screen readers oferecem um atalho de teclado para navegar entre landmarks — é como um índice invisível da página. Uma página sem landmarks obriga o usuário a ouvir tudo desde o começo. Com landmarks corretas, ele pula direto para o conteúdo principal, menu ou rodapé.

# section vs article vs div

Esta é a dúvida mais comum do HTML moderno. A diferença ficou mais clara com a seguinte regra prática:

  • <article> — um conteúdo independente, que faria sentido isolado. Um post de blog, um card de produto, um comentário, uma notícia. Se você colasse esse trecho em outro site, ele continuaria fazendo sentido? Então é <article>.
  • <section> — uma seção temática dentro de um documento, tipicamente com título próprio. "Sobre nós", "Perguntas frequentes", "Depoimentos". Geralmente precisa de <h2>+ dentro.
  • <div> — quando não há significado semântico, só precisa agrupar por motivos de estilo ou layout. Nenhuma vergonha em usar.
section-vs-article.html
<main>
  <h1>Blog da empresa</h1>

  <!-- Cada post é independente → article -->
  <article>
    <h2>Como escolher um notebook</h2>
    <p>...</p>

    <!-- Dentro do post, uma seção com título próprio → section -->
    <section>
      <h3>Processador</h3>
      <p>...</p>
    </section>

    <section>
      <h3>Memória RAM</h3>
      <p>...</p>
    </section>

    <!-- Comentários também são independentes → cada um é article -->
    <section aria-labelledby="coment">
      <h3 id="coment">Comentários</h3>
      <article>
        <h4>Maria disse:</h4>
        <p>Ótimo post!</p>
      </article>
    </section>
  </article>
</main>

# Padrões e armadilhas comuns

Use <main> uma vez só

Só deve existir um <main> visível por página. É o "aqui começa o conteúdo relevante desta página".

<header> e <footer> podem aninhar

header-article.html
<article>
  <header>
    <h2>Título do post</h2>
    <p>Por Ana · 15 de março</p>
  </header>

  <p>Corpo do post...</p>

  <footer>
    <p>Tags: #javascript #html</p>
  </footer>
</article>

<header> e <footer> só se tornam landmarks (banner/contentinfo) quando são filhos diretos de <body>. Dentro de <article>, <section>, <aside>, eles são só blocos genéricos de cabeçalho/rodapé do pai.

Múltiplas <nav> precisam de rótulo

nav-rotulos.html
<nav aria-label="Principal">...</nav>
<nav aria-label="Breadcrumb">...</nav>
<nav aria-label="Paginação">...</nav>

<!-- Sem aria-label, o screen reader anuncia "navigation"
     para as três — e o usuário fica perdido. -->

Skip link — o primeiro foco da página

Um link logo no início do <body> que permite pular a navegação repetitiva e ir direto ao conteúdo. É invisível até receber foco pelo teclado.

skip-link.html
<body>
  <a href="#conteudo" class="skip-link">Pular para o conteúdo</a>

  <header>...</header>
  <nav>...</nav>

  <main id="conteudo" tabindex="-1">
    <h1>Bem-vindo</h1>
  </main>
</body>

<style>
  .skip-link {
    position: absolute;
    left: -9999px;  /* fora da tela */
  }
  .skip-link:focus {
    left: 8px;      /* aparece ao focar */
    top: 8px;
    z-index: 100;
  }
</style>
Acessibilidade
Skip links são recurso padrão em sites acessíveis sérios — GitHub, gov.uk, grandes jornais. Sem eles, um usuário de teclado que chega numa página com 50 links no menu precisa pressionar Tab 51 vezes para chegar ao conteúdo.
Capítulo 05

Cabeçalhos e hierarquia

Os cabeçalhos <h1> a <h6> são o esqueleto do seu documento — a espinha dorsal que define o sumário da página. Usados corretamente, criam um mapa navegável para usuários de screen reader, melhoram SEO e dão ao texto uma lógica visual. Usados errado, geram confusão e tornam a página inutilizável para muita gente.

# Os seis níveis

hierarquia.html
<h1>Título principal da página (um só!)</h1>

<h2>Seção principal</h2>
  <h3>Subseção</h3>
    <h4>Tópico dentro da subseção</h4>
    <h4>Outro tópico</h4>
  <h3>Outra subseção</h3>

<h2>Próxima seção principal</h2>
  <h3>Subseção</h3>

<!-- h5 e h6 existem, mas raramente são necessários.
     Se você está em h5, talvez a estrutura esteja profunda demais. -->

# As três regras essenciais

1. Um <h1> por página

O <h1> representa o tópico principal da página — não do componente, não da seção, da página inteira. Ter múltiplos <h1> é permitido pela especificação, mas confunde screen readers e diminui clareza.

2. Não pule níveis

pulos-hierarquia.html
<!-- ❌ Ruim — pula de h1 direto para h3 -->
<h1>Blog</h1>
<h3>Artigo recente</h3>

<!-- ✅ Bom — passa por h2 -->
<h1>Blog</h1>
<h2>Artigos recentes</h2>
<h3>Como escolher um notebook</h3>

Pular para cima (voltar de h4 para h2) é normal e esperado — você encerrou uma subárvore e começou outra. Pular para baixo (h1 direto para h3) quebra a sequência e deixa buracos no sumário.

3. A ordem reflete o significado, não a aparência

Se um <h2> "parece grande demais" no design, mude o CSS, não a tag. Escolha o nível pela posição na hierarquia, não pelo tamanho da fonte.

estilo-vs-semantica.html
<!-- ❌ Ruim — usa h1 só porque o designer quer texto grande -->
<div class="card">
  <h1>Produto X</h1>
  <p>R$ 99</p>
</div>

<!-- ✅ Bom — h3 combina com a estrutura; CSS cuida do tamanho -->
<div class="card">
  <h3 class="card-title">Produto X</h3>
  <p>R$ 99</p>
</div>

<style>
  .card-title { font-size: 2rem; } /* grande, mas ainda h3 */
</style>
Tamanho ≠ nível
O navegador dá tamanhos padrão decrescentes a h1–h6, mas isso é só um default. Você pode (e deve) ajustar no CSS. A tag comunica importância estrutural; o CSS comunica hierarquia visual. Os dois não precisam ser proporcionais.

# Outlining — o sumário invisível

Screen readers geram um "outline" (sumário) a partir dos cabeçalhos. O usuário pode listar todos os h1–h6 da página e navegar direto para qualquer um — é o equivalente assistivo de um índice.

Para ver como sua página aparece nesse modo, use a extensão HeadingsMap (Chrome/Firefox) ou o comando Rotor no VoiceOver (macOS/iOS). Se o outline não faz sentido sem olhar a página, os cabeçalhos estão errados.

# Cabeçalhos em componentes

Componentes reutilizáveis (cards, modais, tabs) têm um desafio: não sabem em que nível da hierarquia serão inseridos. Duas abordagens:

Abordagem 1 — prop de nível

card.jsx
// Componente recebe o nível como prop
function Card({ titulo, nivel = 3, children }) {
  const Tag = `h${nivel}`;
  return (
    <div className="card">
      <Tag className="card-title">{titulo}</Tag>
      {children}
    </div>
  );
}

<Card titulo="Produto" nivel={2} />

Abordagem 2 — começar um novo outline com <section>

Um <h2> dentro de outro <section> funciona como "filho" daquela section para fins de estrutura, mesmo que não seja semanticamente um h3. É uma forma pragmática de componentizar sem fazer contas de nível.

hgroup.html

  <p>O cavaleiro da triste figura</p>

<!-- Um único 'heading' com legenda. Só o h1 entra no outline;
     o <p> é tratado como subtítulo. -->
Alternativas simples
Se <hgroup> parecer complexidade desnecessária, nada impede você de fazer <h1>Dom Quixote <small>O cavaleiro da triste figura</small></h1> ou simplesmente usar um <p> visualmente próximo com classe. Qualquer uma das três é aceitável.

Capítulo 07

Imagens

A tag <img> parece trivial — src, alt, e pronto. Na prática, imagens são responsáveis por boa parte do peso de qualquer página, do tempo de carregamento percebido e de uma fatia enorme da acessibilidade. Este capítulo cobre desde alt escrito corretamente até srcset, picture, lazy loading e formatos modernos.

# O básico — src, alt, dimensões

img-basico.html
<!-- Imagem decorativa ou informativa -->
<img src="/fotos/gato.jpg"
     alt="Gato preto dormindo em um sofá marrom"
     width="800"
     height="600">

<!-- Imagem puramente decorativa (não agrega informação):
     alt="" (vazio, mas presente!) faz o screen reader IGNORAR a imagem -->
<img src="/ornamento.svg" alt="">

<!-- Imagem que faz parte de um link — alt descreve o DESTINO -->
<a href="/perfil">
  <img src="/icones/user.svg" alt="Meu perfil">
</a>
Sempre declare width e height
Mesmo que a imagem seja responsiva no CSS, declare as dimensões em pixels no HTML. O navegador usa esses números para reservar o espaço antes do download — evitando que o conteúdo "pule" quando a imagem carregar. Isso melhora o Cumulative Layout Shift (CLS), uma das métricas centrais de Core Web Vitals.

# Como escrever um bom alt

O atributo alt é a coisa mais importante de uma imagem. É o que screen readers leem, o que aparece quando a imagem falha em carregar, o que buscadores usam para entender o conteúdo visual.

SituaçãoComo escrever o alt
InformativaDescreva o que é relevante no contexto
Decorativaalt="" (vazio, mas obrigatório)
Dentro de linkDescreva o DESTINO, não a imagem
Logo da empresaNome da empresa (ex: alt="Apple")
Funcional (ícone de ação)Descreva a AÇÃO (ex: alt="Fechar")
Gráfico/diagrama complexoDescrição curta no alt + detalhada no texto próximo
alt-exemplos.html
<!-- ❌ Ruim — genérico e redundante -->
<img src="cachorro.jpg" alt="imagem de um cachorro">
<img src="logo.png" alt="logo">

<!-- ✅ Bom — específico e útil -->
<img src="cachorro.jpg" alt="Golden retriever correndo na praia ao pôr do sol">
<img src="logo.png" alt="Anthropic">

<!-- ❌ Não comece com "imagem de" — screen reader já anuncia "imagem" -->

<!-- Em artigo de blog: a foto ilustra o texto → alt curto -->
<article>
  <h2>A importância do sono</h2>
  <img src="gato.jpg" alt="Gato dormindo">
  <p>Dormir bem é essencial...</p>
</article>

<!-- Se a imagem É o conteúdo (galeria de fotos) → alt descritivo longo -->
<figure>
  <img src="obra.jpg"
       alt="Pintura abstrata com grandes blocos verticais em vermelho, laranja e amarelo sobre fundo preto. Obra 'Nº 14' de Mark Rothko, 1960.">
  <figcaption>Rothko, Nº 14 (1960)</figcaption>
</figure>
Acessibilidade
Toda <img> deve ter alt — nem que seja alt="". Sem o atributo, alguns screen readers leem a URL do arquivo (horrível). Com alt="", eles pulam a imagem silenciosamente (correto para decoração).

# Imagens responsivas com srcset

Uma imagem única não serve para todos os cenários. Um notebook Retina 4K precisa de mais pixels que um celular antigo. srcset deixa o navegador escolher a melhor versão.

srcset-densidade.html
<!-- Resolução adaptativa — 1x, 2x, 3x (densidade de pixels) -->
<img src="foto.jpg"
     srcset="foto.jpg 1x, foto@2x.jpg 2x, foto@3x.jpg 3x"
     alt="..."
     width="400" height="300">

<!-- Tamanhos diferentes + sizes (para imagens responsivas por largura) -->
<img src="foto-800.jpg"
     srcset="foto-400.jpg 400w,
             foto-800.jpg 800w,
             foto-1200.jpg 1200w,
             foto-2000.jpg 2000w"
     sizes="(max-width: 600px) 100vw,
            (max-width: 1200px) 50vw,
            800px"
     alt="..."
     width="800" height="600">

<!-- Como ler o sizes acima:
     - Em telas até 600px, a imagem ocupa 100% da largura (100vw)
     - Entre 600px e 1200px, ocupa 50% da largura
     - Acima disso, tem 800px fixos
     Com base nisso, o navegador escolhe o srcset ideal. -->

# picture — art direction e formatos

Quando você precisa servir imagens diferentes (não só tamanhos) por condição — diferentes recortes, diferentes formatos —, use <picture>.

picture.html
<!-- Art direction: recorte quadrado no mobile, widescreen no desktop -->
<picture>
  <source media="(max-width: 600px)" srcset="foto-mobile.jpg">
  <source media="(min-width: 601px)" srcset="foto-desktop.jpg">
  <img src="foto-desktop.jpg" alt="..." width="1200" height="600">
</picture>

<!-- Formatos modernos com fallback — servir AVIF, depois WebP, depois JPEG -->
<picture>
  <source type="image/avif" srcset="foto.avif">
  <source type="image/webp" srcset="foto.webp">
  <img src="foto.jpg" alt="..." width="800" height="600">
</picture>

<!-- Dark mode adaptativo -->
<picture>
  <source media="(prefers-color-scheme: dark)" srcset="logo-dark.svg">
  <img src="logo-light.svg" alt="Nome da empresa">
</picture>
Formatos modernos
Em 2025, AVIF e WebP oferecem 30–50% menos peso que JPEG/PNG na mesma qualidade. Use <picture> com fallback JPEG/PNG para navegadores antigos. Ferramentas como Squoosh, Sharp (Node) e serviços como Cloudinary/Imgix geram essas variações automaticamente.

# Lazy loading e priorização

lazy.html
<!-- Lazy loading nativo — só baixa quando perto do viewport.
     Ideal para imagens abaixo da primeira dobra. -->
<img src="foto.jpg" alt="..." loading="lazy" width="800" height="600">

<!-- Para a imagem principal da primeira dobra (hero, LCP),
     NÃO use lazy — e considere priorizar: -->
<img src="hero.jpg" alt="..."
     fetchpriority="high"
     width="1200" height="600">

<!-- decoding="async" permite ao navegador decodificar sem bloquear o render.
     Útil para imagens fora do fluxo crítico. -->
<img src="foto.jpg" alt="..." loading="lazy" decoding="async">
Não use lazy na imagem principal
A imagem do topo (hero, banner) frequentemente é o Largest Contentful Paint — métrica crítica de performance. Usar loading="lazy" nela é contraproducente: o navegador atrasa o download, piorando a métrica. Reserve o lazy para imagens abaixo da primeira dobra.

Para vídeos hospedados em plataformas externas (YouTube, Vimeo), você usa <iframe>. Há alguns atributos que poucos conhecem mas que fazem grande diferença:

# O atributo scope

É o atributo mais importante para acessibilidade em tabelas. Ele diz ao screen reader que cabeçalho se aplica a que célula. Sem scope, o leitor precisa adivinhar — e frequentemente erra.

  • scope="col" — o <th> é cabeçalho da coluna.
  • scope="row" — o <th> é cabeçalho da linha.
  • scope="colgroup" — cabeçalho de um grupo de colunas.
  • scope="rowgroup" — cabeçalho de um grupo de linhas.
Acessibilidade
Ao navegar em uma tabela com screen reader, o usuário pula célula por célula com as setas. Com scope correto, o leitor anuncia tanto o cabeçalho da coluna quanto o da linha — por exemplo, "Fevereiro, Receita, R$ 58.000". Sem isso, ele anuncia só o valor, e o usuário perde o contexto.

# Quando NÃO usar tabelas

Nunca use tabelas para layout
Tabelas são para dados tabulares — algo que naturalmente teria linhas e colunas de significado. Uma grade de cards, um menu em duas colunas, um layout de página: tudo isso é trabalho do CSS Grid ou Flexbox. Usar <table> para layout quebra acessibilidade, responsividade e SEO.

Para saber se vale a pena usar tabela, pergunte: "esta informação seria naturalmente exportada como planilha?" Se sim, é tabela. Se é apenas uma organização visual que não faz sentido fora da tela, não é.

# target e segurança

target.html
<!-- Abrir em nova aba -->
<a href="https://exemplo.com" target="_blank" rel="noopener noreferrer">
  Abre em nova aba
</a>

<!-- Por que rel="noopener noreferrer" com target="_blank"?
     - noopener: impede que a página aberta acesse window.opener
       (ataque 'tabnabbing' — a nova aba poderia modificar sua página)
     - noreferrer: também não envia o header Referer (privacidade)

     Navegadores modernos já aplicam 'noopener' por padrão,
     mas escrever explícito é mais seguro e portável. -->

<!-- Outros valores de target -->
<a href="..." target="_self">Mesma aba (padrão)</a>
<a href="..." target="_parent">Frame pai</a>
<a href="..." target="_top">Topo da janela</a>
<a href="..." target="janela-X">Nomeada — reutiliza se existir</a>
Não abuse de target="_blank"
Abrir em nova aba sem o usuário pedir é frustrante — ele pode abrir em nova aba com Ctrl+Clique se quiser. Use _blank apenas quando fizer sentido real: links para outros sites em um formulário que o usuário não terminou, documentação aberta durante um tutorial, etc. E sempre avise (ícone ou texto).
Acessibilidade
Quando usar target="_blank", avise o usuário que o link abre em nova janela. Isso pode ser um ícone com aria-label ou texto complementar: <a target="_blank">Docs <span class="sr-only">(abre em nova aba)</span></a>. Uma nova aba abrindo sem aviso confunde usuários de screen reader.

# Link vs Button — a diferença fundamental

A regra ouro
Link navega. Botão age. Se o elemento muda a URL ou leva a outro lugar, é <a href>. Se ele faz algo na página atual (abre um menu, envia um form, remove um item), é <button>. Confundir os dois é um dos erros mais comuns e prejudica teclado, screen readers e comportamento padrão do navegador.
link-vs-button.html
<!-- ❌ Ruim — link para "javascript:" ou href="#" com onclick -->
<a href="#" onclick="abrirMenu()">Menu</a>
<a href="javascript:void(0)" onclick="...">Clique</a>

<!-- ✅ Bom — use button quando a ação é na página -->
<button type="button" onclick="abrirMenu()">Menu</button>

<!-- ❌ Ruim — button para navegar -->
<button onclick="location.href='/sobre'">Sobre</button>

<!-- ✅ Bom — link de verdade -->
<a href="/sobre">Sobre</a>

A diferença se manifesta em dezenas de comportamentos sutis:

  • Link: botão direito mostra "Abrir em nova aba", "Copiar endereço". Button não.
  • Link é anunciado como "link" pelo screen reader; button como "botão".
  • Ctrl+Clique em link abre em nova aba; em button não faz nada.
  • Link sem href não é focável; button é focável por padrão.
  • Buscadores seguem links; não seguem buttons.

# Texto de link — evite "clique aqui"

O texto visível de um link deve fazer sentido fora de contexto. Usuários de screen reader podem listar todos os links da página de uma vez — imagine uma lista com 30 itens dizendo "clique aqui", "clique aqui", "leia mais", "leia mais".

Tipos de input

O atributo type de <input> define tudo: qual teclado aparece no mobile, que validação é aplicada, que UI o navegador mostra. Escolher o tipo certo é meio caminho andado — cada um traz, de graça, comportamentos que você teria que implementar à mão.

Validação nativa

O navegador valida formulários para você — de graça, em tempo real, integrado com acessibilidade. Muitos desenvolvedores reimplementam tudo em JavaScript por desconhecer o que já vem pronto. # Controlando a validação com JavaScript

A validação nativa expõe uma API rica. Você raramente precisa substituí-la — apenas complementá-la ou customizar a UI.

constraint-api.js
const form = document.querySelector("form");
const email = document.querySelector("#email");

// Checar se o campo é válido
email.checkValidity();           // true/false, SEM mostrar UI
email.reportValidity();          // true/false, COM mostrar UI de erro

// Inspecionar validações específicas (ValidityState)
email.validity.valueMissing;     // required não preenchido
email.validity.typeMismatch;     // formato errado (email, url)
email.validity.tooShort;         // abaixo de minlength
email.validity.tooLong;          // acima de maxlength
email.validity.rangeUnderflow;   // abaixo de min
email.validity.rangeOverflow;    // acima de max
email.validity.stepMismatch;     // fora do step
email.validity.patternMismatch;  // regex não casou
email.validity.customError;      // setCustomValidity foi usado
email.validity.valid;            // true se tudo ok

// Customizar mensagem de erro
email.addEventListener("invalid", () => {
  if (email.validity.valueMissing) {
    email.setCustomValidity("Precisamos do seu e-mail para continuar.");
  } else if (email.validity.typeMismatch) {
    email.setCustomValidity("Hmm, esse e-mail não parece válido.");
  } else {
    email.setCustomValidity(""); // limpa
  }
});

// Importante: limpar a mensagem customizada quando o usuário digita
email.addEventListener("input", () => email.setCustomValidity(""));

// Validar ao enviar (interceptar se inválido)
form.addEventListener("submit", (e) => {
  if (!form.checkValidity()) {
    e.preventDefault();
    form.reportValidity();
  }
});

# Padrão completo — erro customizado acessível

A UI nativa de erro é feia e pouco customizável. Este padrão mantém a API de validação e mostra as mensagens do seu jeito, mantendo acessibilidade:

validacao-custom.html
<form novalidate>
  <div class="form-group">
    <label for="email">E-mail</label>
    <input id="email"
           type="email"
           required
           aria-describedby="email-erro"
           aria-invalid="false">
    <p id="email-erro" class="erro" role="alert" hidden></p>
  </div>

  <button type="submit">Enviar</button>
</form>

<script>
  const form = document.querySelector("form");
  const email = document.querySelector("#email");
  const erro = document.querySelector("#email-erro");

  function mostrarErro(msg) {
    email.setAttribute("aria-invalid", "true");
    erro.textContent = msg;
    erro.hidden = false;
  }
  function limparErro() {
    email.setAttribute("aria-invalid", "false");
    erro.hidden = true;
    erro.textContent = "";
  }

  email.addEventListener("blur", () => {
    if (!email.checkValidity()) {
      if (email.validity.valueMissing) mostrarErro("Informe seu e-mail.");
      else if (email.validity.typeMismatch) mostrarErro("E-mail inválido.");
    }
  });

  email.addEventListener("input", limparErro);

  form.addEventListener("submit", (e) => {
    if (!form.checkValidity()) {
      e.preventDefault();
      email.dispatchEvent(new Event("blur"));
      email.focus();
    }
  });
</script>
Acessibilidade
Pontos-chave do padrão acima: novalidate no form (desliga a UI padrão, mas mantém a API); aria-describedby liga o campo à mensagem; aria-invalid anuncia o estado; role="alert" faz o screen reader ler a mensagem assim que aparece; ao submeter com erro, o foco vai para o primeiro campo inválido.

# Quando ir além da validação nativa

A validação nativa cobre muito, mas não cobre tudo. Você vai precisar de JavaScript quando:

  • Validação depende de outros campos (confirmar senha, data de início < data de fim).
  • Validação precisa consultar o servidor (nome de usuário já existe?).
  • Validação de formato complexo que regex não cobre bem (CPF, CNPJ com dígito verificador).
  • Mensagens condicionais ou multi-etapa.
Validação server-side
Validação em JavaScript é para UX, não para segurança. Qualquer pessoa pode desligar o JS, ou enviar requisições direto à sua API. Toda validação crítica (formato, obrigatoriedade, permissões, unicidade) precisa acontecer também no servidor. Bibliotecas como Zod, Valibot e ArkType permitem você definir o schema uma vez e usar dos dois lados.

# output — resultado calculado

O <output> é um elemento semanticamente "resultado de um cálculo" — perfeito para mostrar valores computados a partir de inputs.

Capítulo 15

Dialog, details e popover

Durante anos, componentes como modais, menus dropdown e tooltips eram construídos à mão — e raramente bem. Gerenciar foco, backdrop, tecla Esc, acessibilidade... era muito código para algo comum. Hoje, HTML tem três elementos nativos que resolvem isso: <dialog>, <details> e o atributo popover.

# <dialog> — modais nativos

O <dialog> é a forma moderna de criar modais. Ele gerencia foco, trava scroll ao abrir como modal, fecha com Esc, tem backdrop estilizável — tudo sem uma linha de JS além de .showModal().

dialog.html
<button type="button" id="abrir">Abrir modal</button>

<dialog id="modal">
  <h2>Confirmar exclusão</h2>
  <p>Essa ação não pode ser desfeita.</p>

  <form method="dialog">
    <button value="cancelar">Cancelar</button>
    <button value="confirmar">Confirmar</button>
  </form>
</dialog>

<script>
  const modal = document.getElementById("modal");
  const abrir = document.getElementById("abrir");

  // Modal "de verdade" — bloqueia interação com o resto da página
  abrir.addEventListener("click", () => modal.showModal());

  // Fechar com o valor do botão clicado (via form method="dialog")
  modal.addEventListener("close", () => {
    console.log("Retornou:", modal.returnValue); // "cancelar" ou "confirmar"
  });
</script>

Os três modos de abertura:

  • dialog.show() — abre como "não-modal" (janela comum, não bloqueia a página).
  • dialog.showModal() — abre como modal (trava scroll, captura foco, tem backdrop).
  • dialog.close(retorno?) — fecha e dispara o evento close.

# Fechar ao clicar fora

Comportamento esperado mas não-padrão: o dialog não fecha se você clicar fora. Implementação:

click-outside.js
modal.addEventListener("click", (e) => {
  // Como o backdrop é o próprio dialog (evento "bolha" vem dele),
  // basta checar se o alvo é o dialog em si (não um filho).
  if (e.target === modal) {
    modal.close();
  }
});
Acessibilidade do dialog
Boas notícias: <dialog> já tem role="dialog", gerencia foco (move para o primeiro elemento focável ao abrir, devolve ao botão de origem ao fechar) e fecha com Esc quando aberto como modal. Só resta garantir um <h2> ou aria-labelledby para dar título ao modal — screen readers anunciam esse título ao entrar.

# popover — tooltips, menus, pop-ups

O atributo popover (suportado em todos navegadores modernos) resolve uma classe enorme de problemas: tooltips, menus de contexto, dropdowns. Ele gerencia o top layer do navegador (o mesmo usado por <dialog>), escape por Esc e "light dismiss" (clique fora fecha).

  Texto do popover aqui.
</div>

</div>

# Posicionamento com Anchor Positioning

Popovers precisam ser posicionados relativamente ao botão que os abre. A CSS Anchor Positioning API (suporte crescente em 2025) resolve isso de forma nativa:

anchor.css
/* Botão é o "âncora" */
#botao {
  anchor-name: --meu-botao;
}

/* Popover se posiciona relativo ao âncora */
#popover {
  position-anchor: --meu-botao;
  top: anchor(bottom);
  left: anchor(left);
  margin-top: 4px;
}

/* Fallback: se não couber embaixo, põe em cima */
#popover {
  position-try-fallbacks: --top, --bottom;
}
@position-try --top {
  top: auto;
  bottom: anchor(top);
}
popover vs dialog
Use <dialog> quando a UI é uma interrupção — o usuário precisa responder antes de continuar (confirmação de exclusão, login modal). Use popover quando é uma extensão do contexto — um menu, uma dica, uma lista que pode ser ignorada. Popover é "leve"; dialog é "sério".

Princípios WCAG

WCAG — Web Content Accessibility Guidelines — é o padrão internacional de acessibilidade web, mantido pelo W3C. Se você vai escrever HTML profissionalmente, precisa conhecer ao menos os princípios fundamentais. Não é um documento enorme assustador como parece — em essência, são quatro palavras.

# Os quatro princípios — POUR

Todas as regras da WCAG derivam de quatro princípios. Um site acessível precisa ser:

PrincípioSignificadoExemplo prático
PerceivableO conteúdo precisa ser percebido por qualquer sentidoToda imagem tem alt; vídeos têm legendas
OperableA interface precisa ser operada por qualquer dispositivoTudo funciona por teclado; sem armadilhas de foco
UnderstandableO conteúdo e a operação devem ser compreensíveisTexto claro; erros explicados; comportamento previsível
RobustCompatível com tecnologias atuais e futurasHTML válido; ARIA apenas quando necessário
Mnemônico
Memorize POUR — Perceivable, Operable, Understandable, Robust. Quase qualquer problema de acessibilidade se encaixa em um desses. Ao revisar uma tela, pergunte: "alguém percebe? alguém opera? alguém entende? funciona em qualquer lugar?"

# Níveis de conformidade — A, AA, AAA

WCAG define três níveis de exigência. Não se trata de "fácil, médio, difícil" — cada nível é um conjunto específico de critérios:

  • Nível A — mínimo absoluto. Sem isso, o site é inutilizável para muita gente. Exemplos: toda imagem funcional tem alt, tudo é operável por teclado, formulários têm labels.
  • Nível AA — o padrão da indústria. É o nível exigido por leis na maioria dos países (ADA nos EUA, EAA na Europa, LBI no Brasil). Exemplos: contraste mínimo 4.5:1, ampliação até 200% sem quebra, erros identificados com texto (não só cor).
  • Nível AAA — excelência. Inclui coisas como contraste 7:1, linguagem simples, sem parallax ou paralaxe agressiva. Nem todos os sites conseguem ser AAA em tudo.
O alvo real
Empresas profissionais miram em WCAG 2.2 nível AA. É o nível que satisfaz a maior parte das legislações, e os critérios são alcançáveis sem comprometer o design. Ir além de AA é louvável em casos específicos (governo, saúde, educação) mas raramente viável em todo o site.

# As situações que a WCAG cobre

Além de deficiências permanentes, a acessibilidade ajuda em situações temporárias e contextuais — o que significa que todo mundo é beneficiado:

  • Temporárias: braço quebrado, catarata pré-cirurgia, enxaqueca.
  • Situacionais: celular no sol forte (baixo contraste perceptual), ambiente ruidoso (precisa de legendas), bebê no colo (uma mão só).
  • Conexão ruim: site precisa funcionar com pouca largura de banda.
  • Dispositivos inesperados: relógios, TVs, carros, refrigeradores.
O argumento decisivo
Não existe "versão acessível" de um site. Ou o site é acessível, ou ele exclui pessoas — permanentemente para algumas, temporariamente para quase todas. Designers e desenvolvedores seniores entendem que escrever HTML semântico desde o início é mais barato do que consertar depois, e abre o produto para milhões de pessoas que, de outra forma, não poderiam usá-lo.

# Legislação — por que isso virou negócio sério

Acessibilidade deixou de ser "boa prática" e passou a ser obrigação legal na maioria dos países ocidentais:

  • Brasil: Lei Brasileira de Inclusão (Lei 13.146/2015) exige acessibilidade digital. eMAG é o padrão governamental.
  • EUA: ADA e Section 508 — processos contra sites inacessíveis passam dos milhares por ano.
  • União Europeia: European Accessibility Act (EAA) entrou em vigor em junho de 2025. Todo e-commerce, banco, transporte e telecomunicação precisa ser acessível, ou paga multa.
  • Reino Unido: Equality Act 2010.

ARIA — quando e como usar

ARIA (Accessible Rich Internet Applications) é um conjunto de atributos que adicionam semântica de acessibilidade ao HTML. Ele existe para cobrir o que o HTML não cobre — e essa é a parte mais importante da história. A primeira regra de ARIA é literalmente "não use ARIA se não precisar".

# As cinco regras de ARIA

Publicadas pela W3C, são as regras que todo dev deveria ter tatuadas:

  1. Se você pode usar um elemento HTML nativo com a semântica que precisa, use-o. Um <button> é sempre melhor que <div role="button">.
  2. Não mude a semântica nativa a menos que seja estritamente necessário. Não faça <h1 role="button">.
  3. Todos os controles interativos devem ser operáveis por teclado. ARIA não faz isso por você — é trabalho do tabindex e event handlers.
  4. Não use role="presentation" ou aria-hidden="true" em elementos focáveis visíveis. Você cria elementos "fantasma".
  5. Todos os controles interativos devem ter um nome acessível. Um botão sem texto visível precisa de aria-label.
ARIA ruim é pior que sem ARIA
Estudos do WebAIM mostram que páginas com ARIA têm mais erros de acessibilidade do que páginas sem ARIA. Não porque ARIA seja ruim, mas porque é frequentemente mal usado — adicionado sem entender, copiado de Stack Overflow, aplicado "por via das dúvidas". O resultado é semântica conflitante que confunde screen readers.

# Roles — o "que este elemento é"

A forma mais conhecida (e perigosa) de ARIA são os role. Eles dizem ao screen reader o papel do elemento — útil quando o HTML não tem tag equivalente.

roles.html
<!-- ❌ QUASE SEMPRE ERRADO — reinventar a roda -->
<div role="button" tabindex="0" onclick="...">Clique</div>

<!-- ✅ CORRETO -->
<button type="button" onclick="...">Clique</button>

<!-- ❌ ERRADO — nav já é navigation -->
<nav role="navigation">...</nav>

<!-- ✅ CORRETO — nav sozinho basta -->
<nav>...</nav>

<!-- ✅ VÁLIDO — quando o HTML não tem equivalente -->
<div role="tablist">
  <button role="tab" aria-selected="true">Perfil</button>
  <button role="tab" aria-selected="false">Configurações</button>
</div>

<div role="alert">Senha incorreta</div>        <!-- anúncio urgente -->
<div role="status">Salvando...</div>            <!-- anúncio suave -->
<div role="tooltip" id="t1">Dica aqui</div>      <!-- tooltip -->

# Nomeação — aria-label e aria-labelledby

Todo elemento interativo precisa de um nome acessível — algo que o screen reader anuncia. Em ordem de preferência:

  1. Conteúdo visível do elemento (texto do botão, texto do link).
  2. aria-labelledby — aponta para outro elemento que já tem o rótulo.
  3. aria-label — string explícita, como último recurso.

<!-- 3) aria-label — quando não há texto visível -->

# Live regions — anunciar mudanças dinâmicas

Quando um conteúdo muda sem o usuário interagir (uma notificação, um carrinho atualizado, um erro de form), o screen reader não percebe por padrão. Live regions resolvem isso.

live-regions.html
<!-- aria-live="polite" — anuncia quando o leitor terminar o que está falando.
     Use para: mudanças não-críticas (contador, status, "salvo") -->
<div aria-live="polite">
  <p id="status"></p>
</div>

<!-- aria-live="assertive" — INTERROMPE o leitor para anunciar imediatamente.
     Use com MUITO cuidado. Só para: erros críticos, avisos urgentes -->
<div aria-live="assertive">
  <p id="erro-critico"></p>
</div>

<!-- Roles pré-configurados (equivalências) -->
<div role="status">...</div>     <!-- = aria-live="polite" -->
<div role="alert">...</div>      <!-- = aria-live="assertive" -->
<div role="log">...</div>        <!-- logs, chat -->

<script>
  // Ao mudar o texto, o screen reader anuncia automaticamente
  document.getElementById("status").textContent = "Carrinho atualizado (3 itens)";
</script>
Regras para live regions funcionarem
(1) O elemento com aria-live precisa existir no DOM desde o carregamento — adicionar o elemento e o conteúdo junto não funciona. (2) Apenas o texto filho que muda é anunciado. (3) Evite mensagens muito longas em assertive — é agressivo. (4) Não abuse: se tudo for live, nada chama atenção de verdade.

# Elementos focáveis por padrão

Alguns elementos HTML recebem foco automaticamente com Tab; outros não. Conhecer essa lista ajuda a evitar reinventar a roda.

  • Focáveis por padrão: <a href>, <button>, <input>, <select>, <textarea>, <audio controls>, <video controls>, <iframe>, <details>/<summary>, elementos com contenteditable.
  • Não-focáveis por padrão: <div>, <span>, <p>, <a> sem href, <img>.
A regra de ouro
Se você está adicionando onclick em uma <div>, pare. Use <button> (se é ação) ou <a href> (se é navegação). Isso te dá foco, Enter/Espaço, semântica e acessibilidade de graça. A única razão legítima para um <div onclick> é quando você está construindo um widget complexo (drag-and-drop, canvas) — e aí precisa de tabindex + handlers de teclado.

# tabindex — controlando a ordem de foco

     Útil para destinos de foco (modal que acaba de abrir, resultado de busca) -->

<!-- tabindex="1", "2", "3"... — define ordem ESPECÍFICA de Tab.
     ⚠️ Quase sempre uma má ideia. Cria confusão, quebra expectativas

Nunca use tabindex > 0
Valores positivos de tabindex criam uma ordem customizada que precede a ordem do DOM. Isso soa útil, mas é uma armadilha: você agora tem duas ordens de foco (custom e natural) que interagem de formas confusas, e qualquer novo elemento bagunça tudo. Se a ordem de tab não está boa, conserte a ordem do HTML no DOM.

# Foco visível — nunca esconda o outline

O anel de foco padrão do navegador (focus ring) é como a luz da frente da bicicleta — feio talvez, mas vital. Designers frequentemente pedem para remover. Não remova. Estilize.

focus.css
/* ❌ NUNCA */
*:focus { outline: none; }

/* ❌ Também ruim — esconde foco para TODO mundo */
button:focus { outline: none; }

/* ✅ Bom — outline customizado, consistente, visível */
button:focus-visible {
  outline: 2px solid var(--accent);
  outline-offset: 2px;
}

/* ✅ Melhor — use box-shadow para formas orgânicas (botões redondos) */
button:focus-visible {
  outline: none;
  box-shadow: 0 0 0 2px var(--bg),
              0 0 0 4px var(--accent);
}

:focus vs :focus-visible

A pseudoclasse :focus-visible ativa apenas quando o navegador acha útil mostrar o anel — tipicamente ao navegar por teclado, não ao clicar com mouse. Isso resolve a eterna tensão entre "designer não quer ver o anel" e "usuário de teclado precisa do anel".

focus-visible.css
/* :focus — sempre que o elemento tem foco (inclui cliques com mouse) */ button:focus { outline: 2px solid blue; } /* :focus-visible — só quando o foco veio do teclado */ button:focus-visible { outline: 2px solid blue; } /* Padrão moderno recomendado */ *:focus { outline: none; } /* remove padrão feio */ *:focus-visible { /* re-adiciona só quando útil */ outline: 2px solid currentColor; outline-offset: 2px; }
Teste rápido
Tire o mouse da mão e tente usar sua aplicação só com Tab, Shift+Tab, Enter, Espaço e setas. Se algo é inalcançável ou você não sabe onde está o foco, há trabalho a fazer. Este é o teste mais barato e revelador de acessibilidade que existe.

# Armadilhas de foco — focus traps

Há dois tipos, um ruim e um bom:

Armadilha ruim (bug)

O usuário entra em um componente (widget de autocomplete, iframe) e não consegue sair com Tab. Aprisionamento acidental. Sempre cause aborto imediato de carreira em code review.

Armadilha intencional (modais)

Em um modal, o foco deve ficar preso dentro — Tab no último elemento volta ao primeiro, Shift+Tab no primeiro vai ao último. Caso contrário o usuário sai do modal sem perceber.

focus-trap.js
function prenderFoco(container) {
  const focaveis = container.querySelectorAll(
    'a[href], button:not([disabled]), input:not([disabled]), ' +
    'select:not([disabled]), textarea:not([disabled]), ' +
    '[tabindex]:not([tabindex="-1"])'
  );
  const primeiro = focaveis[0];
  const ultimo = focaveis[focaveis.length - 1];

  container.addEventListener("keydown", (e) => {
    if (e.key !== "Tab") return;

    if (e.shiftKey && document.activeElement === primeiro) {
      e.preventDefault();
      ultimo.focus();
    } else if (!e.shiftKey && document.activeElement === ultimo) {
      e.preventDefault();
      primeiro.focus();
    }
  });
}
Use o nativo
O elemento <dialog> aberto com .showModal() faz focus trap automaticamente — é mais um motivo para preferir ele a modais customizados. Para componentes que exigem implementação manual (comboboxes, menus ricos), bibliotecas como focus-trap (npm) cobrem todos os casos extremos.

  if (invalido) invalido.focus();
}

// Após navegação em SPA, mover foco para o título
function aposNavegar() {
  const h1 = document.querySelector("h1");
  if (h1) {
    h1.setAttribute("tabindex", "-1");
    h1.focus();
  }
}

# Ordem visual vs ordem do DOM

O foco segue a ordem do DOM, não a ordem visual. Se o CSS (com flex-direction: row-reverse, order ou position: absolute) muda a aparência, o Tab pode parecer "pular" pela tela.

CSS pode confundir o foco
Se o seu layout depende de reordenar visualmente com CSS, garanta que a ordem faça sentido em ambas as modalidades. A regra geral: escreva o HTML na ordem em que faz sentido ser lido/navegado; use CSS para ajustar a apresentação visual dentro disso — não para reorganizar radicalmente a estrutura lógica.

# Como o screen reader "lê" uma página

Screen readers não leem de forma linear como um humano. Eles constroem uma árvore de acessibilidade (baseada no DOM + semântica + ARIA) e oferecem dois modos de navegação:

  • Modo de leitura: setas navegam palavra por palavra, linha por linha, como um livro.
  • Modo de foco: Tab pula entre elementos interativos — formulários, botões, links.
  • Atalhos estruturais: pular entre headings (H), landmarks (D), listas (L), tabelas (T), links (K), formulários (F).

Isso significa que a estrutura do HTML é o índice da página. Uma página sem cabeçalhos, sem landmarks, com divs no lugar de listas — é uma página sem índice. O usuário é obrigado a ouvir tudo desde o começo.

# Texto visualmente escondido, mas disponível para leitor

Às vezes você precisa de texto que só existe para screen readers. O padrão universal é a classe sr-only (ou visually-hidden):

sr-only.css
.sr-only {
  position: absolute;
  width: 1px;
  height: 1px;
  padding: 0;
  margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  white-space: nowrap;
  border: 0;
}

/* Variante focável — volta a ser visível quando recebe foco
   (útil para skip links) */
.sr-only-focusable:focus {
  position: static;
  width: auto;
  height: auto;
  margin: 0;
  overflow: visible;
  clip: auto;
  white-space: normal;
}
uso-sr-only.html
<!-- Complementar um ícone -->
<button>
  <svg aria-hidden="true">...</svg>
  <span class="sr-only">Fechar menu</span>
</button>

<!-- Indicar página atual em menu -->
<a href="/produtos" aria-current="page">
  Produtos
  <span class="sr-only">(página atual)</span>
</a>

<!-- Dar contexto a um "leia mais" -->
<a href="/post-1">
  Leia mais
  <span class="sr-only">sobre suculentas</span>
</a>

<!-- Skip link -->
<a href="#main" class="sr-only sr-only-focusable">
  Pular para o conteúdo
</a>
Não confunda esconder técnicas
display: none — ninguém vê (nem olho, nem screen reader). Para esconder totalmente.
visibility: hidden — ninguém vê; ocupa espaço no layout.
opacity: 0 — invisível para o olho, mas lido pelo screen reader. Quase sempre é bug.
.sr-only (CSS acima) — só screen reader lê; ocupa 1px na tela (invisível).
aria-hidden="true" — só olho vê; screen reader pula. Bom para ícones decorativos.
hidden attr HTML — equivale a display: none (ninguém vê).

Pronúncia

  • Use lang="pt-BR" no HTML (ativa o motor português).
  • Para trechos em outro idioma dentro de texto português: <span lang="en">meetup</span>.
  • Siglas: <abbr title="Instituto Brasileiro de Geografia e Estatística">IBGE</abbr> — nem todos os screen readers leem o title, mas alguns leem.

# Anúncios para mudanças dinâmicas

Já vimos aria-live no capítulo de ARIA. Aqui vai um padrão prático para toasts e notificações:

     role="region"
     aria-label="Notificações"

  function toast(msg, tipo = "polite") {

Contraste, movimento e preferências

Até aqui, o foco foi semântica — o que os elementos são. Este capítulo cobre as questões visuais da acessibilidade: contraste de cores, respeito a preferências do sistema, tamanhos mínimos, e o cuidado com movimento. São questões que separam interfaces bonitas de interfaces realmente usáveis.

# Contraste — a métrica que você não pode ignorar

WCAG define requisitos mínimos de contraste entre texto e fundo. Não é opinião: são números, medidos em uma escala de 1 (sem contraste) a 21 (preto puro em branco puro).

SituaçãoWCAG AAWCAG AAA
Texto normal (< 18pt)4.5:17:1
Texto grande (≥ 18pt, ou 14pt bold)3:14.5:1
Componentes e estados de foco3:1—
Ícones informativos3:1—
Cuidado com cinzas "elegantes"
Texto cinza-claro sobre fundo branco — um clichê de design moderno — quase sempre falha no contraste. Cinzas como #999, #ccc, #ddd ficam muito abaixo do mínimo 4.5:1. Use ferramentas para checar antes de aprovar a paleta.

# Ferramentas para medir contraste

  • DevTools do Chrome/Firefox — no inspetor, a cor do texto mostra o contraste com o fundo e indica AA/AAA.
  • WebAIM Contrast Checker (webaim.org/resources/contrastchecker) — teste duas cores manualmente.
  • axe DevTools — escaneia a página toda e sinaliza contrastes insuficientes.
  • APCA (apcacontrast.com) — nova métrica experimental, mais precisa que o algoritmo atual da WCAG. Em teste para WCAG 3.0.

# Cor nunca é o único indicador

Um erro clássico: marcar campos obrigatórios só com borda vermelha. Ou indicar "fora de estoque" só com texto cinza. Daltonismo afeta cerca de 8% dos homens e 0.5% das mulheres — sem contar pessoas com monitor ruim ou em condições de iluminação difíceis.

  <p id="email-erro">
    <svg aria-hidden="true">...</svg> E-mail inválido
  </p>
</div>

<span class="dot-green"></span> Online

<!-- ✅ Cor + rótulo + semântica -->

# Áreas de clique — tamanho mínimo

WCAG 2.2 introduziu um critério novo (AA): elementos interativos precisam ter área mínima de 24×24 pixels CSS. AAA pede 44×44. É especialmente importante para pessoas com limitações motoras finas ou usando touchscreen.

tamanho-alvo.css
/* Botão de ícone — garanta o mínimo */
.icon-button {
  min-width: 44px;
  min-height: 44px;
  padding: 10px;
}

/* Links em menu mobile — padding generoso */
.menu-mobile a {
  padding: 12px 16px;  /* aumenta a área clicável */
  display: block;
}

/* Ícones pequenos podem ter área clicável maior via pseudo-elementos */
.small-icon {
  position: relative;
  width: 16px;
  height: 16px;
}
.small-icon::before {
  content: "";
  position: absolute;
  inset: -14px;   /* expande 14px em todos os lados = 44px total */
}

# Respeitando preferências do sistema

O sistema operacional do usuário expõe várias preferências via media queries de CSS. Usá-las é questão de respeito básico — e, em alguns casos, de acessibilidade obrigatória.

prefers-color-scheme — dark mode automático

dark-mode.css
/* Padrão moderno — variáveis que mudam conforme preferência */
:root {
  --bg: #ffffff;
  --text: #0f0f0f;
  color-scheme: light;
}

@media (prefers-color-scheme: dark) {
  :root {
    --bg: #0f0f0f;
    --text: #f0f0f0;
    color-scheme: dark;
  }
}

body {
  background: var(--bg);
  color: var(--text);
}

/* O 'color-scheme' no :root ajuda o navegador a colorir
   elementos nativos (scrollbars, form controls, etc) de forma coerente */

prefers-reduced-motion — respeite quem tem sensibilidade

reduced-motion.css
/* Animação padrão */
.card {
  transition: transform 0.3s ease;
}
.card:hover {
  transform: translateY(-4px);
}

/* Quando o usuário pede menos movimento, desligue animações */
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}

/* Ou só desligue as animações "agressivas" — micro-interações podem ficar */
@media (prefers-reduced-motion: reduce) {
  .parallax, .autoplay-video, .bounce {
    animation: none;
    transform: none;
  }
}
Por que reduzir movimento
Parallax, scroll infinito com animação, vídeos em autoplay e transições grandes podem causar náusea, enxaqueca ou crises em pessoas com distúrbios vestibulares. A preferência prefers-reduced-motion é como o usuário diz "me poupe". Ignorar é literalmente fazer mal à saúde de alguém.

# Zoom e redimensionamento

WCAG AA exige que a página continue funcional em zoom até 200%, e AAA até 400%. Isso significa:

  • Não use maximum-scale ou user-scalable=no no viewport — é pura agressão a quem precisa ampliar.
  • Use unidades relativas (rem, em, %) para tamanhos de fonte, não px fixos.
  • Evite texto em imagens (vira pixelado com zoom). Quando usar (logos), ofereça alternativa.
  • Teste: pressione Ctrl/Cmd ++ algumas vezes. Algum conteúdo ficou cortado? Alguma interação quebrou?
viewport-certo.html
<!-- ❌ Bloqueia zoom — inacessível -->
<meta name="viewport" content="width=device-width, initial-scale=1, maximum-scale=1, user-scalable=no">

<!-- ✅ Permite zoom do usuário -->
<meta name="viewport" content="width=device-width, initial-scale=1">

# Atenção com flashes e piscadas

WCAG proíbe qualquer conteúdo que pisque mais de 3 vezes por segundo — pode desencadear crises em pessoas com epilepsia fotossensível. Isso inclui gifs, animações CSS agressivas e vídeos com edições rápidas.

Teste seu conteúdo
Se tem vídeo com cortes rápidos ou animações repetitivas, use o PEAT (Photosensitive Epilepsy Analysis Tool) da Trace Center. É grátis e detecta padrões potencialmente perigosos.

SEO técnico

SEO não é só escrever bem ou conseguir links. A camada técnica — como o HTML comunica a estrutura do seu conteúdo aos buscadores — é o que separa páginas que aparecem bem no Google de páginas que ninguém acha. A boa notícia: HTML semântico já resolve 70% do problema. Este capítulo cobre o resto.

# Semântica = SEO

Bots de busca leem a mesma árvore de acessibilidade que screen readers. Quando você escreve HTML semântico, está simultaneamente fazendo acessibilidade e SEO. Um site com <h1> correto, <main> identificado, links descritivos e imagens com alt — já está bem servido antes de qualquer otimização extra.

  • Um <h1> por página, descrevendo o tópico central.
  • Hierarquia de headings respeitada (não pule de h1 para h3).
  • <main> envolvendo o conteúdo principal.
  • Links com texto descritivo (não "clique aqui").
  • Imagens com alt relevante ao conteúdo.
O atalho honesto
Se você fez os capítulos 2 a 10 bem feitos, seu SEO técnico está 80% resolvido. O resto deste capítulo é sobre os 20% finais.

# Title e meta description — revisitados

Estes dois campos aparecem nos resultados de busca. São sua "vitrine" no Google. Vale um capricho que, surpreendentemente, poucos sites têm.

  • Title: 50–60 caracteres. Comece com a palavra-chave principal. Termine com o nome do site se couber.
  • Description: 150–160 caracteres. Descreva o conteúdo específico. Escreva para humanos — o Google já não usa muito para ranqueamento, mas usa para mostrar o snippet, que afeta o clique.
  • Único por página: páginas com mesmo title/description competem entre si.
title-desc.html
<!-- ❌ Genérico — não diferencia -->
<title>Home</title>
<meta name="description" content="Bem-vindo ao nosso site.">

<!-- ✅ Específico, com palavra-chave, call to action -->
<title>Como escolher um notebook em 2025 — Guia completo | TechBlog</title>
<meta name="description" content="Comparamos 15 modelos de notebook em 2025: processador, memória, bateria e preço. Descubra qual é o melhor para seu uso.">

Princípios:

  • Hifens em vez de underscores (o Google vê "-" como separador de palavras)
  • Palavras reais, não IDs
  • Hierarquia reflete a estrutura do site
  • Sem extensões desnecessárias (.php, .html) quando possível
  • Curtas mas descritivas

# Redirects 301 vs 302

Quando você muda a URL de uma página, o servidor precisa sinalizar isso ao buscador. Caso contrário, você perde autoridade acumulada e cria páginas duplicadas.

  • 301 — permanente. Transfere toda a autoridade SEO para a nova URL. Use para mudanças definitivas.
  • 302 — temporário. Mantém a autoridade na URL antiga. Use para testes A/B, manutenções curtas.
  • 308, 307 — versões modernas que preservam o método HTTP (GET/POST).
Nunca redirecione em JavaScript sem pensar
Redirects feitos com window.location.href = ... ou meta refresh são ruins para SEO — buscadores podem demorar a segui-los ou não seguir. Sempre que possível, faça redirect no servidor (Apache, Nginx, CDN) ou via configuração do framework (ex: redirects em next.config.js).

Performance do HTML

Performance na web é frequentemente associada a otimizações em JavaScript ou CSS. Mas o HTML tem um papel silencioso — e enorme — na velocidade percebida. Resource hints, priorização, lazy loading, ordem de carregamento: decisões pequenas no HTML determinam se sua página aparece em 200ms ou em 3 segundos. Este capítulo cobre as alavancas que estão ao alcance de quem escreve HTML.

# Resource hints — avisando o navegador antes da hora

Resource hints são tags que instruem o navegador a começar a trabalhar em recursos antes de ele descobrir que precisa deles. São sutis, mas o impacto em tempo de carregamento pode ser grande.

hints.html
<!-- preconnect — abre conexão TCP+TLS antecipadamente.
     Use para domínios de terceiros que você SABE que vai pedir algo
     (fonts, CDN, analytics). -->
<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>

<!-- dns-prefetch — só resolve o DNS, mais leve que preconnect.
     Use para domínios com baixa probabilidade mas ainda possíveis. -->
<link rel="dns-prefetch" href="https://analytics.example.com">

<!-- preload — baixa recurso crítico imediatamente com prioridade alta.
     Use APENAS para recursos descobertos tarde (fonts, CSS crítico).
     O 'as' é OBRIGATÓRIO. -->
<link rel="preload" href="/fonts/principal.woff2" as="font" type="font/woff2" crossorigin>
<link rel="preload" href="/hero.jpg" as="image" fetchpriority="high">

<!-- modulepreload — preload específico para módulos ES -->
<link rel="modulepreload" href="/app.js">

<!-- prefetch — baixa em baixa prioridade, para a próxima navegação.
     Útil em listas de artigos, paginação. -->
<link rel="prefetch" href="/proxima-pagina">
preload em excesso atrapalha
Preload é poderoso, mas fácil de abusar. Cada preload rouba banda de algo que o navegador já estava baixando. A regra: só preload recursos realmente críticos que o parser HTML demoraria a descobrir sozinho (font, CSS crítico inline, imagem hero). Nunca faça preload de scripts e estilos que já estão na cascata normal.

# Priority hints — fetchpriority

Um atributo relativamente novo que permite ao HTML sugerir qual recurso é mais urgente. Valores: high, low, auto (padrão).

priority.html
<!-- Imagem principal (LCP) — prioridade alta -->
<img src="/hero.jpg"
     alt="..."
     fetchpriority="high"
     width="1200"
     height="600">

<!-- Imagens "above-the-fold" secundárias — deixa o default -->
<img src="/foto-2.jpg" alt="...">

<!-- Imagens abaixo da dobra — baixa prioridade + lazy -->
<img src="/foto-3.jpg"
     alt="..."
     loading="lazy"
     fetchpriority="low">

<!-- Script analytics — não é crítico -->
<script src="/analytics.js" async fetchpriority="low"></script>

# CSS e o caminho crítico de renderização

CSS é "render-blocking": o navegador não pinta nada até baixar e processar todo o CSS declarado no <head>. Isso afeta diretamente o LCP.

css-estrategias.html
<!-- ❌ Um CSS enorme que bloqueia a renderização -->
<link rel="stylesheet" href="/tudo.css">

<!-- ✅ Inline o CSS crítico (only-above-the-fold) -->
<style>
  /* Critical CSS: tipografia, cores, layout da primeira dobra.
     Gerado por ferramentas como Critters, Critical. */
  body { font-family: sans-serif; margin: 0; }
  header { background: #111; padding: 1rem; }
  /* ... */
</style>

<!-- E carregue o resto de forma não-bloqueante -->
<link rel="preload"
      href="/estilos.css"
      as="style"
      onload="this.onload=null;this.rel='stylesheet'">
<noscript><link rel="stylesheet" href="/estilos.css"></noscript>

<!-- Em CSS específico para media, o navegador ainda baixa
     mas NÃO bloqueia a renderização inicial -->
<link rel="stylesheet" href="/print.css" media="print">

# Evitando layout shift (CLS)

Layout shift acontece quando elementos "pulam" após o carregamento — imagens sem dimensões, fonts swap, banners injetados. As causas principais e como evitar:

cls.html
<!-- 1) SEMPRE declare width/height em imagens -->
<img src="..." alt="..." width="800" height="450">
<!-- O navegador reserva a proporção 16:9, nada pula. -->

<!-- 2) Para iframes idem -->
<iframe src="..." width="560" height="315"></iframe>

<!-- 3) Banners e anúncios — reserve espaço mínimo via CSS -->
<div class="ad-slot"></div>
<style>
  .ad-slot { min-height: 250px; }
</style>

<!-- 4) Fonts — use font-display: swap + preload para evitar FOIT -->
<link rel="preload" href="/fonte.woff2" as="font" type="font/woff2" crossorigin>
<style>
  @font-face {
    font-family: "Minha";
    src: url("/fonte.woff2") format("woff2");
    font-display: swap;  /* mostra texto com fallback enquanto baixa */
  }
</style>

<!-- 5) Use CSS aspect-ratio para containers responsivos -->
<style>
  .video-wrap {
    aspect-ratio: 16 / 9;  /* reserva proporção sem height fixo */
  }
</style>

# Speculative loading — prefetch ao passar o mouse

Um padrão moderno: o navegador começa a baixar a próxima página quando o usuário passa o mouse sobre o link, tornando a navegação quase instantânea. Bibliotecas como Quicklink (Google) ou Instant.page automatizam isso. Next.js e Astro já fazem por padrão.

speculation.html
<!-- Speculation Rules API (Chrome moderno) -->
<script type="speculationrules">
{
  "prerender": [{
    "source": "document",
    "where": { "href_matches": "/*" },
    "eagerness": "moderate"
  }]
}
</script>

<!-- O navegador pré-renderiza páginas prováveis em background.
     Clicar fica praticamente instantâneo. -->
Medindo o que importa
Use Lighthouse (Chrome DevTools) para audit local, PageSpeed Insights (pagespeed.web.dev) para dados do mundo real vindos de usuários do Chrome, e WebPageTest para análise detalhada. As três métricas Core Web Vitals são públicas e reais — não adianta "passar no teste" se o usuário real sofre.

# O princípio geral

Performance em 5 frases
(1) Envie menos HTML, CSS e JS — o que não existe é instantâneo.
(2) Priorize o caminho crítico: CSS crítico inline, resto deferred.
(3) Lazy load tudo que estiver abaixo da dobra.
(4) Declare dimensões para reservar espaço e evitar CLS.
(5) Meça no mundo real, não só no laptop do desenvolvedor.

Todos os capítulos anteriores ensinam ferramentas. Este fecha com os princípios: o que diferencia alguém que escreve HTML correto de alguém que escreve HTML que envelhece bem. É um compilado de convenções que a comunidade acumulou — não por regra, mas porque funciona.

# Valide seu HTML

HTML moderno é tolerante demais para o próprio bem. Páginas com erros graves renderizam assim mesmo, escondendo bugs que aparecem no futuro (ou em outros navegadores). Validar é trivial e deveria ser etapa de build.

  • W3C Nu HTML Checker (validator.w3.org/nu) — o validador oficial.
  • HTMLHint e html-validate — linters integráveis ao seu editor e CI.
  • axe DevTools — valida acessibilidade (que é parte da "correção").
  • Lighthouse — dá uma nota geral de melhores práticas.

# Data attributes — estendendo HTML corretamente

Quando você precisa anexar dados a elementos (para JS), use data-*. É a forma oficial, validada pela especificação, e acessível via dataset.

data-attrs.html
<button data-user-id="42"
        data-action="delete"
        data-confirm="Tem certeza?">
  Excluir
</button>

<script>
  const btn = document.querySelector("button");
  btn.dataset.userId;    // "42" (camelCase automático!)
  btn.dataset.action;    // "delete"
  btn.dataset.confirm;   // "Tem certeza?"

  // Atribuir também funciona:
  btn.dataset.status = "loading";  // vira data-status="loading"
</script>

<!-- CSS também acessa -->
<style>
  [data-status="loading"] {
    opacity: 0.5;
    cursor: wait;
  }
</style>

# Escolhas semânticas — um checklist mental

Antes de escrever uma tag, pergunte:

  • Isso é texto com ênfase? → <em> ou <strong>
  • Isso é uma lista? → <ul> ou <ol>
  • Isso é um dado tabular? → <table>
  • Isso é uma ação? → <button>
  • Isso navega para outro lugar? → <a href>
  • Isso é um conteúdo independente? → <article>
  • Isso é uma seção temática com título? → <section>
  • Isso é conteúdo tangencial? → <aside>
  • Isso é o conteúdo principal? → <main>
  • É um grupo de links de navegação? → <nav>
  • É um formulário? → <form> com <label> e <fieldset>
  • Nenhuma das opções acima se aplica, mas preciso agrupar? → <div> (ou <span> inline) sem vergonha

# Progressive enhancement

O princípio mais duradouro da web: construa para o menor denominador comum, enriqueça a experiência para quem pode. HTML semântico no núcleo, CSS para aparência, JavaScript para interatividade enriquecida.

progressive.html
<!-- Formulário que funciona sem JS -->
<form action="/buscar" method="get">
  <label for="q">Buscar</label>
  <input id="q" name="q" type="search" required>
  <button type="submit">Ir</button>
</form>

<!-- JS melhora com autocomplete dinâmico, mas o form
     continua funcionando mesmo se o JS falhar. -->

<!-- Link que funciona sem JS -->
<a href="/produtos" data-turbo>Ver produtos</a>

<!-- JS intercepta para transição SPA, mas clique com Ctrl
     abre em nova aba, Ctrl+C copia URL, Shift+Click abre janela,
     SEO funciona, botão de voltar funciona... tudo de graça. -->
Não é 2010
O argumento "ninguém desliga JavaScript" não é mais o ponto. A questão é: JS pode falhar — conexão lenta, script quebrado, adblocker bloqueando CDN, timeout. Cada vez que o form é um botão com onclick e não um <form action>, você está criando um ponto de falha a mais. Progressive enhancement é robustez.

# Separação de responsabilidades

  • HTML — estrutura e significado do conteúdo.
  • CSS — apresentação visual.
  • JavaScript — interatividade e comportamento dinâmico.

Essa separação é menos rígida hoje do que era (frameworks como Tailwind misturam estilo com HTML, componentes encapsulam os três). Mas o princípio permanece: o HTML deve fazer sentido sozinho, mesmo se o CSS falhar ou o JS quebrar.

HTML é, de longe, a linguagem mais estável da web. O código que você escreve hoje vai funcionar daqui a dez anos. Frameworks vêm e vão — jQuery, Backbone, Angular 1, depois Angular 2+, React, Vue, Svelte, e os próximos. Mas <h1>, <button>, <form> continuam exatamente onde sempre estiveram.
