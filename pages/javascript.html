Capítulo 01

O que é o JavaScript

JavaScript é a linguagem da web — e, cada vez mais, de tudo o que é digital. Ela começou em 1995 como um script de 10 dias dentro do Netscape, e hoje executa desde interfaces de usuário em navegadores até servidores, drones, apps de TV e impressoras 3D. Este capítulo é o mapa mental antes do tutorial propriamente dito.

# Uma linguagem de múltiplos paradigmas

JavaScript não é "só orientada a objetos" nem "só funcional". Ela aceita — e se beneficia de — misturar estilos. Você pode escrever:

  • Imperativo: sequências de comandos que mudam estado.
  • Orientado a objetos: classes, herança, encapsulamento.
  • Funcional: funções puras, composição, imutabilidade.
  • Orientado a eventos: reagir a cliques, timers, mensagens.

Conhecer todos os quatro estilos — e saber quando cada um cabe melhor — é o que separa uma pessoa que usa JavaScript de alguém que sabe JavaScript.

# Strict mode

No começo da linguagem, o JavaScript permitia várias coisas estranhas em silêncio — variáveis globais acidentais, atribuições a propriedades read-only, números octais implícitos. O strict mode, introduzido em 2009, desliga esses comportamentos e impõe regras mais sensatas.

strict.js
"use strict";

// Sem declaração → erro (no modo não-strict, criaria variável global):
// nome = "Ana"; // ❌ ReferenceError

// Atribuição a variável readonly vira erro:
const PI = 3.14;
// PI = 3; // ❌ TypeError

// Duplicar parâmetros é proibido:
// function f(a, a) {} // ❌ SyntaxError

// 'this' em função comum é undefined, não window:
function f() { return this; }
f(); // undefined (não window/global)
Observação
Em módulos ES (qualquer arquivo .mjs ou que tenha import/export), o strict mode é implícito e obrigatório. Você não precisa escrever "use strict". O mesmo vale dentro de class.

# Interpretado, compilado ou ambos?

JavaScript moderno usa JIT (Just-In-Time) compilation: o código é interpretado inicialmente e, conforme é executado, partes "quentes" (funções chamadas muitas vezes) são compiladas para código de máquina otimizado. Isso explica por que código idiomático e consistente costuma ser mais rápido do que truques aparentemente "otimizados".

Para o desenvolvedor, o efeito prático é: escreva código limpo; confie no motor. Otimize apenas quando medir um problema real.

Capítulo 02

Variáveis e escopos

JavaScript tem três palavras-chave para declarar variáveis — var, let e const — e a diferença entre elas não é só estilo. Elas obedecem a regras de escopo e inicialização radicalmente diferentes, e confundi-las é fonte de bugs sutis há mais de duas décadas.

# const, let e var

declaracoes.js
// const — vínculo imutável (não pode ser reatribuído):
const PI = 3.14159;
// PI = 3; // ❌ TypeError

// Mas atenção: o conteúdo de um objeto const pode mudar!
const config = { tema: "dark" };
config.tema = "light"; // ✅ permitido — não mudou a referência

// let — reatribuível, escopo de bloco:
let contador = 0;
contador = 1; // ✅

// var — reatribuível, escopo de FUNÇÃO (não de bloco):
var velho = "evite";

# Escopo de bloco vs escopo de função

Esta é a diferença mais importante entre var e let/const. Um bloco é qualquer coisa entre chaves { } — um if, um for, um bloco solto.

escopos.js
// let/const respeitam o bloco:
{
  let x = 10;
  const y = 20;
}
// console.log(x); // ❌ ReferenceError — x não existe aqui

// var ignora o bloco — vaza para a função inteira:
function exemplo() {
  if (true) {
    var z = 30;
  }
  console.log(z); // 30 — var vaza!
}

// O caso clássico do loop com var:
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);
}
// Imprime: 3, 3, 3 — todos compartilham o mesmo i!

// Com let, cada iteração tem seu próprio i:
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);
}
// Imprime: 0, 1, 2

# Hoisting e Temporal Dead Zone

Hoisting é o comportamento pelo qual declarações são "içadas" para o topo do seu escopo antes da execução. As três formas sofrem hoisting, mas de maneiras diferentes.

hoisting.js
// VAR — içada e inicializada como undefined:
console.log(a); // undefined (não erro!)
var a = 10;

// LET e CONST — içadas, MAS não inicializadas.
// A zona entre o início do bloco e a declaração é a
// Temporal Dead Zone (TDZ).
console.log(b); // ❌ ReferenceError: Cannot access 'b' before initialization
let b = 20;

// Funções declaradas são totalmente içadas:
dizerOi(); // ✅ funciona
function dizerOi() { console.log("oi"); }

// Mas funções em expressões NÃO são içadas:
// dizerTchau(); // ❌ TypeError
const dizerTchau = function() { console.log("tchau"); };
Regra de ouro
Use const como padrão. Troque para let quando precisar reatribuir. Só use var em código legado ou quando você precisa explicitamente do escopo de função (o que é quase nunca).
Capítulo 03

Tipos primitivos

JavaScript tem oito tipos fundamentais — sete primitivos e o tipo "objeto". Primitivos são imutáveis e passados por valor; objetos são mutáveis e passados por referência. Entender essa distinção é pré-requisito para não apanhar da linguagem.

# Os sete primitivos

primitivos.js
// 1) string — texto
const nome = "Ana";
const saudacao = `Olá, ${nome}`;

// 2) number — inteiros e decimais (mesmo tipo!)
const idade = 30;
const preco = 19.99;
const infinito = Infinity;
const naoNumero = NaN; // "Not a Number"

// 3) bigint — inteiros arbitrariamente grandes
const gigante = 9007199254740993n; // note o 'n' no final

// 4) boolean — true ou false
const ativo = true;

// 5) null — "valor intencionalmente ausente"
const resultado = null;

// 6) undefined — "ainda não foi atribuído"
let indefinido;
console.log(indefinido); // undefined

// 7) symbol — identificadores únicos (cap. dedicado adiante)
const id = Symbol("identificador");

# null vs undefined

Esta é uma distinção frequentemente confundida. Os dois representam "ausência de valor", mas com intenções diferentes:

  • undefined é o valor padrão do JavaScript quando nada foi atribuído (variável sem valor, parâmetro não passado, propriedade inexistente).
  • null é um valor que você, programador, coloca para dizer "aqui não há nada intencionalmente".
null-vs-undefined.js
let x;                     // x é undefined (sem atribuição)
function f(a, b) {
  console.log(b);          // undefined se chamada com f(1)
}

const usuario = { nome: null }; // "nome não existe, e eu sei disso"

// Na comparação rasa, eles são iguais (==), mas não idênticos (===):
null == undefined;  // true  (coerção)
null === undefined; // false (tipos diferentes)

# O operador typeof e suas armadilhas

typeof devolve uma string que descreve o tipo de um valor. É útil, mas tem duas peculiaridades históricas que você precisa conhecer.

typeof.js
typeof "oi";        // "string"
typeof 42;          // "number"
typeof true;        // "boolean"
typeof undefined;   // "undefined"
typeof 10n;         // "bigint"
typeof Symbol();    // "symbol"
typeof function(){};// "function"
typeof {};          // "object"
typeof [];          // "object"  ← arrays são objetos!

// ⚠️ Armadilha #1: null retorna "object" — um bug histórico:
typeof null;        // "object" (deveria ser "null")

// ⚠️ Armadilha #2: NaN é do tipo "number":
typeof NaN;         // "number"

// Para testar array, use Array.isArray:
Array.isArray([1, 2, 3]); // true
Array.isArray("abc");     // false

# Primitivos vs objetos — valor vs referência

Entender isto é essencial. Primitivos são copiados ao serem atribuídos ou passados como argumento. Objetos são compartilhados por referência.

valor-vs-ref.js
// Primitivos — cópia por valor:
let a = 10;
let b = a;      // b recebe uma cópia
b = 20;
console.log(a); // 10 (a não mudou)

// Objetos — cópia por referência:
const obj1 = { x: 10 };
const obj2 = obj1; // obj2 aponta para o MESMO objeto
obj2.x = 20;
console.log(obj1.x); // 20 (o mesmo objeto mudou!)

// Para copiar de verdade:
const copia = { ...obj1 };           // shallow copy
const funda = structuredClone(obj1); // deep clone nativo

# Coerção de tipos e igualdade

JavaScript converte tipos automaticamente em muitas operações — isso é a famosa "coerção". Ela explica comportamentos surpreendentes que viraram memes ([] + [] === ""). A regra prática é: use sempre === e !== e escolha a conversão explicitamente.

coercao.js
// == (igualdade frouxa) — aplica coerção:
"5" == 5;          // true  (string vira number)
null == undefined; // true
[] == false;       // true  (!!)
"0" == false;      // true

// === (igualdade estrita) — não aplica coerção:
"5" === 5;         // false
0 === -0;          // true
NaN === NaN;       // false (!) — NaN nunca é igual a nada, nem a si mesmo!

// Para testar NaN:
Number.isNaN(NaN);   // true
Object.is(NaN, NaN); // true — quase como ===, mas NaN é igual a NaN

// Conversões explícitas (SEMPRE prefira):
Number("42");     // 42
Number("abc");    // NaN
String(42);       // "42"
Boolean(0);       // false
Boolean("");      // false
Boolean("0");     // true (!) — strings não-vazias são truthy

// Falsy values — convertidos para false:
// false, 0, -0, 0n, "", null, undefined, NaN
// Tudo o mais é truthy (incluindo [], {}, "false")
Nunca use ==
A única exceção legítima é x == null, que testa null e undefined ao mesmo tempo. Fora isso, use === sempre. Seu linter provavelmente já impõe isso.
Capítulo 04

Operadores e fluxo

Além dos operadores tradicionais, JavaScript moderno acrescentou um punhado de novos operadores que substituíram com elegância padrões comuns feitos à mão. Dominá-los deixa o código mais curto e mais legível ao mesmo tempo.

# Operadores aritméticos e lógicos

operadores.js
// Aritméticos:
10 + 3;    // 13
10 - 3;    // 7
10 * 3;    // 30
10 / 3;    // 3.333...
10 % 3;    // 1 (resto)
10 ** 3;   // 1000 (potência — equivalente a Math.pow)

// Lógicos (com short-circuit):
true && "oi";     // "oi" — retorna o último valor se tudo truthy
false && "oi";    // false — para no primeiro falsy
null || "padrão"; // "padrão" — retorna o primeiro truthy
"ana" || "padrão";// "ana"

// Negação:
!true;     // false
!"oi";     // false (string não-vazia é truthy)
!!"oi";    // true (truque: converte em boolean)

# Operadores modernos essenciais

modernos.js
// Nullish coalescing (??) — fallback SÓ em null/undefined:
const porta = config.porta ?? 8080;
// Diferente de || que também dispara em 0, "", false:
0 || 8080;   // 8080 (porque 0 é falsy) ← pode ser bug!
0 ?? 8080;   // 0 (porque 0 não é null/undefined) ← correto

// Optional chaining (?.) — acessa com segurança:
const cidade = usuario?.endereco?.cidade;
// Sem erro se usuario ou endereco forem null/undefined

// Chamada opcional:
usuario?.salvar?.();

// Acesso opcional de array:
const primeiro = lista?.[0];

// Atribuições compostas (ES2021):
let x = null;
x ??= 10;    // x = x ?? 10 → x agora é 10
x ||= 5;     // x = x || 5
x &&= 20;   // x = x && 20

// Spread (...) — expande iteráveis:
const a = [1, 2, 3];
const b = [...a, 4, 5];       // [1, 2, 3, 4, 5]
const max = Math.max(...a);   // equivalente a Math.max(1, 2, 3)

// Spread em objetos:
const base = { nome: "Ana" };
const completo = { ...base, idade: 30 }; // { nome: "Ana", idade: 30 }

# Condicionais

condicionais.js
// if / else if / else
const idade = 20;
if (idade < 18) {
  console.log("menor");
} else if (idade < 65) {
  console.log("adulto");
} else {
  console.log("sênior");
}

// Ternário (expressão, pode retornar valor):
const tipo = idade < 18 ? "menor" : "adulto";
const classe = idade < 18 ? "menor" : idade < 65 ? "adulto" : "sênior";

// switch com fall-through:
switch (idade) {
  case 0:
  case 1:
  case 2:
    console.log("bebê");
    break;
  case 18:
    console.log("maioridade");
    break;
  default:
    console.log("outra idade");
}

// Padrão moderno — objeto como "switch":
const mensagens = {
  loading: "Carregando...",
  ok: "Tudo certo!",
  erro: "Falhou.",
};
const msg = mensagens[estado] ?? "desconhecido";

# Loops

loops.js
// for clássico — quando você precisa do índice:
for (let i = 0; i < 5; i++) {
  console.log(i);
}

// while — enquanto condição é verdadeira:
let n = 0;
while (n < 5) {
  console.log(n);
  n++;
}

// do...while — executa pelo menos uma vez:
do {
  n++;
} while (n < 10);

// for...of — itera sobre valores de iteráveis (arrays, strings, Map, Set):
const cores = ["red", "green", "blue"];
for (const cor of cores) {
  console.log(cor);
}

// for...of com índice (usando entries):
for (const [i, cor] of cores.entries()) {
  console.log(i, cor);
}

// for...in — itera sobre CHAVES de um objeto (cuidado!):
const obj = { a: 1, b: 2, c: 3 };
for (const chave in obj) {
  console.log(chave, obj[chave]);
}

// break e continue:
for (let i = 0; i < 10; i++) {
  if (i === 3) continue;   // pula essa iteração
  if (i === 7) break;      // encerra o loop
  console.log(i);
}

// Labels — break de loops aninhados:
externo: for (let i = 0; i < 3; i++) {
  for (let j = 0; j < 3; j++) {
    if (i === 1 && j === 1) break externo;
  }
}
Qual loop usar?
Para arrays, prefira métodos como .forEach(), .map(), .filter(). Quando precisar de break/continue, use for...of. Evite for...in com arrays — ele itera sobre chaves (incluindo herdadas), não valores.
Capítulo 05

Strings

Strings em JavaScript são imutáveis — toda "modificação" retorna uma nova string. A linguagem oferece uma biblioteca rica de métodos, e os template literals de ES2015 transformaram completamente como escrevemos texto dinâmico.

# Template literals

templates.js
const nome = "Ana";
const idade = 30;

// Interpolação com ${}:
const frase = `${nome} tem ${idade} anos`;

// Multi-linhas sem truques:
const html = `
  <div>
    <h1>${nome}</h1>
    <p>Idade: ${idade}</p>
  </div>
`;

// Expressões dentro de ${}:
const maior = `${idade >= 18 ? "adulto" : "menor"}`;
const dobro = `O dobro de ${idade} é ${idade * 2}`;

// Tagged templates — função que processa o template:
function destacar(strings, ...valores) {
  return strings.reduce((acc, str, i) => {
    const v = valores[i] ? `[${valores[i]}]` : "";
    return acc + str + v;
  }, "");
}

const resultado = destacar`Olá ${nome}, você tem ${idade} anos.`;
// "Olá [Ana], você tem [30] anos."

# Métodos essenciais

metodos-string.js
const s = "JavaScript moderno";

// Comprimento:
s.length; // 18

// Acesso por índice:
s[0];           // "J"
s.at(-1);       // "o" (índice negativo suportado)
s.charAt(4);    // "S"

// Busca:
s.includes("Script");     // true
s.startsWith("Java");     // true
s.endsWith("no");         // true
s.indexOf("a");           // 1 (primeira ocorrência)
s.lastIndexOf("a");       // 3 (última)

// Transformação (lembre: strings são imutáveis):
s.toUpperCase();          // "JAVASCRIPT MODERNO"
s.toLowerCase();          // "javascript moderno"
s.trim();                 // remove espaços nas pontas
s.trimStart();            // só no início
s.trimEnd();              // só no fim
s.padStart(25, "-");      // "-------JavaScript moderno"
s.padEnd(25, ".");        // "JavaScript moderno......."

// Substrings:
s.slice(0, 10);           // "JavaScript"
s.slice(-7);              // "moderno" (do fim)
s.substring(0, 10);       // "JavaScript"

// Substituição:
"banana".replace("a", "X");      // "bXnana" (primeira ocorrência)
"banana".replaceAll("a", "X");   // "bXnXnX" (todas)
"banana".replace(/a/g, "X");     // mesma coisa, via regex

// Divisão e junção:
"a,b,c".split(",");        // ["a", "b", "c"]
["a", "b", "c"].join("-"); // "a-b-c"
"abc".split("");           // ["a", "b", "c"]

// Repetir:
"ab".repeat(3);            // "ababab"

// Escape de HTML (não embutido — implemente ou use lib):
function escapar(s) {
  return s.replace(/[&<>"']/g, c => ({
    "&": "&amp;", "<": "&lt;", ">": "&gt;",
    '"': "&quot;", "'": "&#39;",
  })[c]);
}

# Unicode e caracteres especiais

unicode.js
// Escape padrão (4 dígitos hex):
"\u00E9";           // "é"

// Escape de code point (para caracteres fora do BMP):

// Emojis e length — cuidado!

Capítulo 06

Números e matemática

Em JavaScript, "number" é um tipo único que cobre inteiros e decimais — usando internamente IEEE 754 de 64 bits, igual a double em C. Isso traz consequências: 0.1 + 0.2 não é 0.3, e há limites para inteiros seguros. Conhecê-los evita bugs financeiros clássicos.

# O tipo Number

numeros.js
// Literais numéricos:
const decimal = 1000000;
const legivel = 1_000_000;      // underscore para legibilidade
const hex = 0xff;               // 255
const bin = 0b1010;             // 10
const oct = 0o17;               // 15
const cientifico = 1.5e3;       // 1500
const pequeno = 1.5e-4;         // 0.00015

// Valores especiais:
Infinity;
-Infinity;
NaN;
Number.MAX_SAFE_INTEGER;  // 9007199254740991 (2^53 - 1)
Number.MIN_SAFE_INTEGER;  // -9007199254740991
Number.EPSILON;           // menor diferença representável

// Métodos do objeto Number:
Number.isInteger(42);       // true
Number.isFinite(Infinity);  // false
Number.isNaN(NaN);          // true (mais confiável que isNaN global)
Number.isSafeInteger(2n);   // false (não é number)

// Parsing (o Number() vs parseInt/parseFloat):
Number("42");        // 42
Number("42abc");     // NaN (precisa ser string limpa)
parseInt("42abc");   // 42 (lê até achar não-dígito)
parseInt("ff", 16);  // 255 (com base)
parseFloat("3.14x"); // 3.14

// Métodos de instância:
(3.14159).toFixed(2);       // "3.14" (string!)
(1234.5678).toPrecision(6); // "1234.57"
(255).toString(16);         // "ff"
(10).toString(2);           // "1010"

# A armadilha do ponto flutuante

precisao.js
0.1 + 0.2;         // 0.30000000000000004 (!)
0.1 + 0.2 === 0.3; // false

// Motivo: números binários não representam 0.1 exatamente.
// Soluções:

// 1) Comparar com margem de erro:
Math.abs(0.1 + 0.2 - 0.3) < Number.EPSILON; // true

// 2) Arredondar:
+(0.1 + 0.2).toFixed(10) === 0.3; // true

// 3) Para dinheiro, trabalhe em centavos (inteiros):
const precoCent = 1999;     // R$ 19,99
const total = precoCent * 3; // 5997 (correto!)

// 4) Use bibliotecas como decimal.js ou dinero.js
//    para dinheiro em produção.

# O objeto Math

math.js
// Constantes:
Math.PI;         // 3.14159...
Math.E;          // 2.71828...

// Arredondamento:
Math.round(1.5);   // 2
Math.floor(1.9);   // 1 (para baixo)
Math.ceil(1.1);    // 2 (para cima)
Math.trunc(-1.9);  // -1 (remove decimais)

// Mínimo, máximo:

// Potência e raiz:

Math.sqrt(16);     // 4
Math.cbrt(27);     // 3

// Absoluto e sinal:
Math.abs(-5);      // 5
Math.sign(-42);    // -1 (-1, 0 ou 1)

// Logs e exponenciais:

Arrays

Arrays em JavaScript são objetos especializados: chaves numéricas, uma propriedade length mágica, e uma coleção generosa de métodos. Dominar a API de arrays é dominar boa parte do que se faz no dia-a-dia — desde manipular listas até transformar dados de APIs.

# Métodos essenciais — ordem e mutação

// sort — MUTA o array. Sem comparator, ordena como STRING!

[1, 2, 3].reverse(); // [3, 2, 1]

// MÉTODOS IMUTÁVEIS (ES2023) — retornam nova cópia:
[3, 1, 2].toSorted();       // [1, 2, 3] — original intacto
[1, 2, 3].toReversed();     // [3, 2, 1]
[1, 2, 3].toSpliced(1, 1);  // [1, 3]
[1, 2, 3].with(1, 99);      // [1, 99, 3] (substitui índice 1)

// push / pop — no FIM:
const arr = [1, 2];
arr.push(3);       // [1, 2, 3] — retorna novo length
arr.pop();         // 3 — retorna removido

// shift / unshift — no INÍCIO:
arr.unshift(0);    // [0, 1, 2]
arr.shift();       // 0 — removido

// splice — remove/insere no meio (MUTA):
const lista = ["a", "b", "c", "d"];
lista.splice(1, 2);           // remove 2 desde índice 1 → ["a", "d"]
lista.splice(1, 0, "x", "y"); // insere sem remover → ["a", "x", "y", "d"]

["a","b","c","d"].slice(-2);   // ["c", "d"]

// concat — junta (NÃO muta):

Métodos imutáveis vs mutáveis
Em código moderno, prefira os métodos que retornam novo array (map, filter, slice, toSorted, toReversed, with). Eles são mais previsíveis, combinam bem e evitam bugs de estado compartilhado. Use mutadores (push, sort, splice) conscientemente, não por reflexo.

Map, Set e variantes Weak

Objetos comuns servem como dicionários desde sempre — mas têm limitações (chaves só strings/símbolos, prototype herdado, ordem de inserção não-garantida em casos extremos). ES2015 introduziu estruturas dedicadas: Map, Set, WeakMap, WeakSet. Cada uma resolve um problema específico.

# Map — dicionário de qualquer coisa para qualquer coisa

map.js

// Set / get:

m.set(42, "numérico");
m.set(true, "booleano");

m.set(objChave, "usando objeto como chave!");

m.get("chave");       // "valor"
m.get(objChave);      // "usando objeto como chave!"
m.has("chave");       // true
m.size;               // 4
m.delete("chave");
m.clear();

// Inicialização com entries:
const m2 = new Map([
  ["nome", "Ana"],
  ["idade", 30],
]);

// Iteração (mantém ordem de inserção):
for (const [k, v] of m2) console.log(k, v);
for (const k of m2.keys()) console.log(k);
for (const v of m2.values()) console.log(v);
m2.forEach((v, k) => console.log(k, v));

// Conversão:

new Map(Object.entries({a:1, b:2}));

Use Map quando: as chaves não são strings/símbolos; você precisa iterar em ordem garantida; você muda o conjunto de chaves frequentemente; o tamanho importa (.size é O(1)). Use objeto comum para estruturas com chaves fixas conhecidas na compilação — é mais leve e tem sintaxe melhor.

# WeakMap e WeakSet

Versões "fracas" cujos valores (em WeakMap) ou membros (em WeakSet) não impedem o garbage collector de limpá-los quando nenhuma outra referência existe. Chaves precisam ser objetos. Sem .size, sem iteração — o GC pode remover qualquer coisa a qualquer momento.

Funções em profundidade

Funções em JavaScript são cidadãs de primeira classe: podem ser atribuídas a variáveis, passadas como argumento, retornadas de outras funções, e guardam estado através de closures. Elas são o bloco de construção mais fundamental da linguagem — mais do que objetos, mais do que classes.

  return a - b;

// 3) Named function expression (útil para recursão/debug):
const fatorial = function fact(n) {
  return n <= 1 ? 1 : n * fact(n - 1);

# Closures

Uma closure é uma função que "se lembra" do escopo onde foi criada, mesmo quando executada fora dele. Esse é possivelmente o conceito mais poderoso de JavaScript — e o alicerce de padrões como hooks em React, factories, módulos e callbacks.

closures.js
// Exemplo clássico — contador com estado privado:
function criarContador() {
  let count = 0;                 // variável "capturada" pela closure
  return {
    incrementar() { count++; },
    valor() { return count; },
  };
}

const c = criarContador();
c.incrementar();
c.incrementar();
c.valor();        // 2
// count é inacessível de fora — está encapsulado.

// Factory de funções:
function multiplicadorPor(fator) {
  return (n) => n * fator;
}
const dobrar = multiplicadorPor(2);
const triplicar = multiplicadorPor(3);
dobrar(5);      // 10
triplicar(5);   // 15

// Memoization — cache em closure:
function memoize(fn) {
  const cache = new Map();
  return function(...args) {
    const chave = JSON.stringify(args);
    if (!cache.has(chave)) cache.set(chave, fn.apply(this, args));
    return cache.get(chave);
  };
}

// Armadilha clássica — var em loop:
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0); // 3, 3, 3
}
// Com let, cada iteração cria nova closure:
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0); // 0, 1, 2
}

# Funções de ordem superior

Funções que recebem outras funções, ou retornam funções. Essa é a base do paradigma funcional em JS.

higher-order.js
// Composição — encadeia funções:
const compor = (...fns) => x => fns.reduceRight((v, f) => f(v), x);
const pipe = (...fns) => x => fns.reduce((v, f) => f(v), x);

const somarDois = n => n + 2;
const dobrar = n => n * 2;
const quadrado = n => n ** 2;

const processo = pipe(somarDois, dobrar, quadrado);
processo(3); // ((3+2)*2)^2 = 100

// Currying — transforma f(a,b,c) em f(a)(b)(c):
const curry = (fn) => {
  return function curried(...args) {
    if (args.length >= fn.length) return fn(...args);
    return (...next) => curried(...args, ...next);
  };
};

const somar = (a, b, c) => a + b + c;
const somarCurry = curry(somar);
somarCurry(1)(2)(3);     // 6
somarCurry(1, 2)(3);     // 6
somarCurry(1)(2, 3);     // 6

// Debounce — adia execução:
function debounce(fn, ms) {
  let timer;
  return (...args) => {
    clearTimeout(timer);
    timer = setTimeout(() => fn(...args), ms);
  };
}

// Throttle — limita frequência:
function throttle(fn, ms) {
  let ultimo = 0;
  return (...args) => {
    const agora = Date.now();
    if (agora - ultimo >= ms) {
      ultimo = agora;
      fn(...args);
    }
  };
}

O this em profundidade

Nenhum tópico causou mais confusão na história de JavaScript do que o this. Em outras linguagens, this sempre se refere à "instância atual". Em JavaScript, o valor de this depende de como a função foi chamada, não de onde foi definida. Dominar isso é separador de águas.

# As cinco regras do this

this-regras.js
// REGRA 1 — Chamada de método: this é o objeto antes do ponto.
const user = {
  nome: "Ana",
  falar() { return this.nome; },
};
user.falar();              // "Ana" — this = user

// REGRA 2 — Chamada standalone: this é undefined (strict) ou global (não-strict).
const f = user.falar;
f();                       // undefined (ou "" no modo não-strict)

// REGRA 3 — new: this é o objeto novo criado.
function Produto(nome) {
  this.nome = nome;         // this = objeto novo
}
const p = new Produto("Livro");

// REGRA 4 — call/apply/bind: this é o primeiro argumento.
function falar() { return this.nome; }
falar.call({ nome: "Bruno" });  // "Bruno"
falar.apply({ nome: "Carla" }); // "Carla"
const ligado = falar.bind({ nome: "Dora" });
ligado();                       // "Dora"

// REGRA 5 — Arrow functions: this vem do escopo léxico de ONDE foi definida.
const obj = {
  nome: "Ana",
  falarArrow: () => this.nome,   // this NÃO é obj — é o escopo de fora!
  esperar() {
    setTimeout(() => console.log(this.nome), 100); // ✅ this é obj
  },
  esperarRuim() {
    setTimeout(function() { console.log(this.nome); }, 100); // ❌ this não é obj
  },
};

# call, apply e bind

Três métodos que todas as funções herdam, para controlar manualmente o this.

call-apply-bind.js

}

apresentar.call(ana, "Olá", "!");     // "Olá, Ana!"

// apply — argumentos em array:
apresentar.apply(ana, ["Oi", "."]);   // "Oi, Ana."

// bind — retorna NOVA função ligada ao this:
const cumprimentar = apresentar.bind(ana, "Hey");
cumprimentar("?");                    // "Hey, Ana?"

// Aplicações úteis:
// 1) Emprestar método de array:
const aLike = { 0: "a", 1: "b", length: 2 };
const arr = Array.prototype.slice.call(aLike); // ["a", "b"]

// 2) Encontrar máximo em array:
Math.max.apply(null, [1, 2, 3]);      // ou Math.max(...[1,2,3])

// 3) "Congelar" this em event handler:

    this.nome = "Botão";

Event loop e o modelo assíncrono

JavaScript é single-threaded — roda uma coisa de cada vez. Como, então, ele consegue lidar com requisições simultâneas, timers, eventos? A resposta é o event loop, um mecanismo elegante que separa o que roda agora do que roda depois. Este é o capítulo que desmistifica assíncrono.

# Como o event loop funciona

Imagine três estruturas principais:

  • Call stack — onde as funções em execução vivem. Empilham quando chamadas, desempilham quando retornam.
  • Task queue (macrotasks) — fila de coisas agendadas: setTimeout, eventos de I/O, mensagens.
  • Microtask queue — fila prioritária: Promise.then, queueMicrotask, MutationObserver.

O event loop roda num ciclo: (1) executa até esvaziar a call stack, (2) drena toda a microtask queue, (3) pega uma macrotask, volta ao passo 1.

ordem.js
console.log("1 — síncrono");

setTimeout(() => console.log("4 — macrotask"), 0);

Promise.resolve().then(() => console.log("3 — microtask"));

console.log("2 — síncrono");

// Saída:
// 1 — síncrono
// 2 — síncrono
// 3 — microtask   ← antes do setTimeout!
// 4 — macrotask
Microtasks primeiro
Microtasks são executadas antes da próxima macrotask — mesmo que o setTimeout esteja com 0ms. Isso é a razão pela qual await/then "encadeiam rápido" e o UI não fica travado por promises.

# setTimeout, setInterval, queueMicrotask

timers.js
// setTimeout — executa uma vez, após ms:
const id = setTimeout(() => console.log("depois"), 1000);
clearTimeout(id); // cancela

// setInterval — executa repetidamente:
const intId = setInterval(() => console.log("tick"), 1000);
clearInterval(intId);

// Cuidado — "0ms" não é zero. O mínimo prático é 4ms em navegadores,
// e o callback só roda quando a stack esvaziar.
setTimeout(() => console.log("imediato"), 0);
for (let i = 0; i < 1e9; i++) {} // bloqueia
// "imediato" imprime SÓ depois do loop.

// queueMicrotask — agenda microtask:
queueMicrotask(() => console.log("microtask"));

// requestAnimationFrame — sincroniza com o próximo frame de renderização (browser):
requestAnimationFrame(() => {
  // executa antes do próximo paint (~16ms a 60fps)
});

Promises

Uma promise representa um valor que pode ainda não existir — o resultado futuro de uma operação assíncrona. Ela substitui callbacks aninhados ("callback hell") por uma API encadeável e componível, que serviu de base para o async/await.

# Estados e criação

promises.js
// Uma promise está em 1 de 3 estados:
// - pending  (pendente)
// - fulfilled (resolvida)  — tem um valor
// - rejected  (rejeitada)  — tem um motivo

// Criação a partir de um executor:
const p = new Promise((resolve, reject) => {
  setTimeout(() => {
    if (Math.random() > 0.5) resolve("deu certo");
    else reject(new Error("falhou"));
  }, 1000);
});

// Consumo:
p.then(valor => console.log("ok:", valor))
 .catch(erro => console.log("erro:", erro.message))
 .finally(() => console.log("limpa"));

// Promises "já resolvidas" — úteis em código que precisa retornar promise:
Promise.resolve(42);
Promise.reject(new Error("erro"));

# Encadeamento e transformação

encadeamento.js
fetch("/api/user")

  .then(user => fetch(`/api/posts?user=${user.id}`))
  .then(res => res.json())
  .then(posts => console.log(posts))
  .catch(erro => console.error(erro));

// Erros "pulam" para o catch mais próximo:
Promise.resolve(10)
  .then(n => { throw new Error("ops"); })

  .catch(e => console.log(e.message)) // "ops"
  .then(() => console.log("continua depois do catch"));

// Cuidado — esquecer de retornar quebra a cadeia:
p.then(x => {
  fetch("/api").then(() => {}); // promise "perdida"!
})
// Correto:

  return fetch("/api").then(() => {}); // ou use async/await
})

# Promise combinators

combinators.js
const p1 = Promise.resolve(1);

await Promise.all([p1, p2]);           // [1, 2]
await Promise.all([p1, p3]);           // ❌ rejected com Error("x")

// Promise.allSettled — espera todas; retorna status individual:
await Promise.allSettled([p1, p3]);
// [
//   { status: "fulfilled", value: 1 },
//   { status: "rejected", reason: Error("x") }
// ]

// Promise.any — resolve com a primeira a cumprir (ignora rejeições,
// só falha se TODAS falharem, com AggregateError):
await Promise.any([p3, p1]);           // 1

// Uso real — timeout manual:
function comTimeout(promise, ms) {
  const timeout = new Promise((_, reject) =>
    setTimeout(() => reject(new Error("timeout")), ms)
  );
  return Promise.race([promise, timeout]);
}

async / await

Adicionado em ES2017, async/await é açúcar sintático sobre promises — mas um açúcar transformador. Código assíncrono passa a parecer síncrono, o que resolve boa parte dos problemas de legibilidade que os .then em cadeia ainda causavam.

# Escrevendo código assíncrono claro

async-await.js
// Função async SEMPRE retorna uma Promise:
async function carregarUsuario(id) {
  const res = await fetch(`/api/users/${id}`);
  if (!res.ok) throw new Error("404");
  const user = await res.json();
  return user;
}

// Uso:
carregarUsuario(1).then(u => console.log(u));
// ou dentro de outra async:
const u = await carregarUsuario(1);

// Top-level await — funciona em módulos ESM:
// arquivo.mjs
const dados = await fetch("/api").then(r => r.json());

// Tratamento de erros com try/catch:
async function carregarSeguro(id) {
  try {
    const user = await carregarUsuario(id);
    return user;
  } catch (e) {
    console.error("falhou:", e);
    return null;
  } finally {
    console.log("limpa sempre");
  }
}

// Sequencial vs paralelo:
// ❌ sequencial — lento (um depois do outro):
const a = await carregarUsuario(1);
const b = await carregarUsuario(2);

// ✅ paralelo — começam juntos, aguarda os dois:
const [a2, b2] = await Promise.all([
  carregarUsuario(1),
  carregarUsuario(2),
]);

# Async em loops

async-loops.js
const ids = [1, 2, 3, 4];

// Sequencial — uma de cada vez:
for (const id of ids) {
  const user = await carregarUsuario(id);
  console.log(user);
}

// Paralelo — todas de uma vez (CUIDADO: pode sobrecarregar):

async function emBatches(items, tamanho, fn) {
  const resultados = [];
  for (let i = 0; i < items.length; i += tamanho) {
    const batch = items.slice(i, i + tamanho);
    const res = await Promise.all(batch.map(fn));
    resultados.push(...res);
  }
  return resultados;
}

Iterators e Generators

O protocolo de iteração é o contrato secreto que faz for...of, spread e destructuring funcionarem com qualquer coisa. Generators são funções especiais que produzem sequências preguiçosas — valores só são calculados quando pedidos.

# O protocolo iterável

iteradores.js
// Um OBJETO É ITERÁVEL se tem método [Symbol.iterator]() que retorna um ITERATOR.
// Um ITERATOR é um objeto com método next() que retorna { value, done }.

// Arrays, strings, Map, Set, NodeList — todos iteráveis:
for (const c of "abc") console.log(c); // "a", "b", "c"

// Criando um iterável manualmente:
const range = {
  inicio: 1,
  fim: 5,
  [Symbol.iterator]() {
    let atual = this.inicio;
    const fim = this.fim;
    return {
      next() {
        if (atual <= fim) return { value: atual++, done: false };
        return { value: undefined, done: true };
      },
    };
  },
};

for (const n of range) console.log(n);  // 1, 2, 3, 4, 5
[...range];                               // [1, 2, 3, 4, 5]

# Generators

generators.js
// function* define um generator; 'yield' emite valor e pausa.
function* contar() {
  yield 1;
  yield 2;
  yield 3;
}

const gen = contar();
gen.next(); // { value: 1, done: false }
gen.next(); // { value: 2, done: false }
gen.next(); // { value: 3, done: false }
gen.next(); // { value: undefined, done: true }

// Generators são iteráveis:
for (const v of contar()) console.log(v); // 1, 2, 3
[...contar()];                             // [1, 2, 3]

// Sequências infinitas (uma das magias):
function* naturais() {
  let n = 1;
  while (true) yield n++;
}

const n = naturais();
n.next().value; // 1
n.next().value; // 2

// yield* — delega a outro iterável:
function* a() { yield 1; yield 2; }
function* b() { yield* a(); yield 3; } // 1, 2, 3

// Comunicação bidirecional — next(arg) passa valor de volta:
function* conversa() {
  const x = yield "qual seu nome?";
  yield `olá, ${x}`;
}
const c = conversa();
c.next();           // { value: "qual seu nome?", done: false }
c.next("Ana");      // { value: "olá, Ana", done: false }

Proxy e Reflect

Proxy permite interceptar praticamente qualquer operação em um objeto — leitura, escrita, deletar, testar propriedade. É a ferramenta por trás de sistemas de reatividade como Vue e algumas partes do SolidJS, e de ORMs como Prisma e Drizzle.

# Proxy — interceptando operações

proxy.js
const alvo = { nome: "Ana", idade: 30 };

    console.log(`lendo ${String(prop)}`);
    return obj[prop];
  },
  set(obj, prop, valor) {
    console.log(`escrevendo ${String(prop)} = ${valor}`);
    obj[prop] = valor;
    return true;  // indica sucesso
  },
  has(obj, prop) {
    console.log(`testando ${String(prop)} in`);
    return prop in obj;
  },
  deleteProperty(obj, prop) {
    console.log(`deletando ${String(prop)}`);
    delete obj[prop];
    return true;
  },

proxy.nome;         // "lendo nome" → "Ana"
proxy.idade = 31;   // "escrevendo idade = 31"
"nome" in proxy;    // "testando nome in" → true
delete proxy.idade; // "deletando idade"

# Casos de uso práticos

proxy-usos.js
// 1) Defaults automáticos:
function comDefaults(obj, defaultValue) {
  return new Proxy(obj, {
    get: (t, p) => p in t ? t[p] : defaultValue,
  });
}
const cfg = comDefaults({}, "—");
cfg.qualquer; // "—"

// 2) Validação automática:
function validado(schema) {
  return new Proxy({}, {
    set(t, p, v) {
      if (!schema[p]) throw new Error(`${String(p)} inválido`);
      if (typeof v !== schema[p]) throw new TypeError();
      t[p] = v;
      return true;
    },
  });
}
const u = validado({ nome: "string", idade: "number" });
u.nome = "Ana";     // ok
// u.cor = "azul";   // ❌

// 3) Objetos reativos (princípio do Vue/Svelte runes/etc):
function reativo(obj, onChange) {
  return new Proxy(obj, {
    set(t, p, v) {
      const antigo = t[p];
      t[p] = v;
      if (antigo !== v) onChange(p, v, antigo);
      return true;
    },
  });
}

// 4) Acesso profundo seguro:
function seguro(obj) {
  return new Proxy(obj, {
    get: (t, p) => {
      const v = t[p];
      return v && typeof v === "object" ? seguro(v) : (v ?? undefined);
    },
  });
}

# Reflect — API espelhada

Reflect é um objeto com métodos estáticos que correspondem exatamente às operações que Proxy pode interceptar. É mais expressivo do que acessar diretamente e, dentro de handlers de Proxy, é o jeito canônico de "delegar para o comportamento padrão".

reflect.js
const obj = { x: 10 };

// Operações espelhadas:
Reflect.get(obj, "x");              // 10 (equivale a obj.x)
Reflect.set(obj, "y", 20);          // true
Reflect.has(obj, "x");              // true
Reflect.deleteProperty(obj, "x");
Reflect.ownKeys(obj);               // ["y"]
Reflect.defineProperty(obj, "z", { value: 30 });

// Construção dinâmica:
class Foo { constructor(a, b) { this.a = a; this.b = b; } }
const f = Reflect.construct(Foo, [1, 2]);

// Uso canônico dentro de Proxy — delegar ao alvo:
new Proxy(obj, {
  get(t, p, receiver) {
    console.log("log");
    return Reflect.get(t, p, receiver);  // preserva semântica correta
  },
});

Módulos ESM

ES Modules (ESM) é o sistema de módulos nativo de JavaScript — padronizado em ES2015 e hoje universalmente suportado. Cada arquivo é um módulo com seu próprio escopo; você compartilha explicitamente o que quer com export e usa o que precisa com import.

# Exportando de um módulo

matematica.js
// Named exports:
export const PI = 3.14159;
export function somar(a, b) { return a + b; }
export class Calculadora { /* ... */ }

// Exportar em bloco no final:
function multiplicar(a, b) { return a * b; }
const E = 2.71828;
export { multiplicar, E };

// Renomear na exportação:
export { multiplicar as multiply };

// Default export (um por módulo):
export default function principal() { /* ... */ }

// Re-export (atalhos de módulo — "barrel"):
export * from "./outro.js";
export { algo } from "./outro.js";
export { default as Util } from "./util.js";

# Importando em outro módulo

app.js
// Named imports:
import { PI, somar } from "./matematica.js";

// Renomear:
import { somar as soma } from "./matematica.js";

// Namespace — importa tudo como um objeto:
import * as Math from "./matematica.js";
Math.somar(1, 2);

// Default:
import principal from "./matematica.js";

// Mistura:
import principal, { PI } from "./matematica.js";

// Só efeitos colaterais (sem nomes):
import "./reset.css";
import "./polyfill.js";

// Dynamic import — assíncrono, retorna Promise:
const mod = await import("./pesado.js");
mod.usar();

// Útil para code splitting (imports condicionais):
if (usuario.admin) {
  const { painel } = await import("./admin.js");
  painel();
}

// import.meta — metadados do módulo:
console.log(import.meta.url);    // URL deste módulo
CommonJS (require/module.exports)
É o sistema antigo de módulos do Node.js, síncrono. Você encontrará muito em código legado. Node moderno aceita ambos, mas um projeto novo deve usar ESM ("type": "module" no package.json, ou extensão .mjs).

# Características dos módulos ES

  • Strict mode implícito — todo módulo é sempre strict, não precisa declarar.
  • Escopo próprio — variáveis e funções não vazam para o global.
  • Executado uma vez — mesmo importado em vários lugares, o código do módulo roda só uma vez; as exportações são compartilhadas.
  • Top-level await — em ESM, você pode usar await direto no nível do arquivo.
  • Estático por padrão — import/export são analisáveis em tempo de build, permitindo tree-shaking.

Tratamento de erros

Todo programa não-trivial precisa lidar com o que dá errado — falhas de rede, inputs inválidos, estados impossíveis. JavaScript oferece try/catch/finally, uma hierarquia de tipos de erro, e a possibilidade de lançar e criar erros customizados.

# try, catch, finally, throw

try-catch.js
try {
  const dados = JSON.parse(entrada);
  processar(dados);
} catch (e) {
  console.error("falha:", e.message);
} finally {
  limpar(); // roda sempre, com ou sem erro
}

// catch sem binding (ES2019):
try { pode_falhar(); } catch { fallback(); }

// throw — qualquer valor, mas SEMPRE lance Error (ou subclasse):
throw new Error("algo deu errado");
throw new TypeError("esperava número");

// ❌ Evite lançar strings ou outros valores:
// throw "erro"; // funciona, mas perde stack trace e info

// try/catch em torno de assíncrono — NÃO pega rejeições de promises
// a menos que você use await:
try {
  minhaPromise();  // rejeição aqui NÃO é capturada
} catch {}

try {
  await minhaPromise();  // agora sim
} catch {}

# Erros customizados

custom-errors.js
class ValidationError extends Error {
  constructor(campo, mensagem) {
    super(`${campo}: ${mensagem}`);
    this.name = "ValidationError";
    this.campo = campo;
  }
}

class NotFoundError extends Error {
  constructor(recurso) {
    super(`${recurso} não encontrado`);
    this.name = "NotFoundError";
    this.status = 404;
  }
}

try {
  throw new ValidationError("email", "inválido");
} catch (e) {
  if (e instanceof ValidationError) {
    console.log("campo:", e.campo);
  } else if (e instanceof NotFoundError) {
    console.log("status:", e.status);
  } else {
    throw e; // re-throw o que não reconheço
  }
}

// Error cause (ES2022) — preserva a causa original:
try {
  await banco.conectar();
} catch (causa) {
  throw new Error("falha ao iniciar", { cause: causa });
}
// Quando printa, a cadeia inteira de causes aparece.
Onde tratar
Não embrulhe tudo em try/catch defensivamente. Deixe o erro propagar até o lugar onde há informação suficiente para tratar. Em aplicações reais, isso costuma ser: um handler de rota (back-end) ou um error boundary de UI (front-end).

// Hash SHA-256:
async function hash(mensagem) {
  const bytes = new TextEncoder().encode(mensagem);
  const buf = await crypto.subtle.digest("SHA-256", bytes);
  return [...new Uint8Array(buf)]
    .map(b => b.toString(16).padStart(2, "0"))
    .join("");
}

Regex em JavaScript

Expressões regulares em JavaScript têm uma sintaxe própria (literais /.../), flags específicas, e se integram a vários métodos de string. Esta seção é uma referência rápida — para o estudo profundo de regex em geral, existe um manual dedicado mais adiante na coleção.

# Criação e uso

regex.js
// Duas formas:
const r1 = /\d+/g;                       // literal
const r2 = new RegExp("\\d+", "g");      // construtor (strings escapam de novo)

// Flags:
// g  — global (todas as ocorrências)
// i  — case-insensitive
// m  — multi-linha (^ $ casam com início/fim de linha)
// s  — dotAll (. casa com \n também)
// u  — unicode (para \u{xxxx})
// y  — sticky (busca a partir do lastIndex)
// d  — retorna índices dos grupos

// Métodos em STRING:
"abc123".match(/\d+/);              // ["123", index: 3, ...]
"abc123".match(/\d+/g);             // ["123"] — só matches, sem detalhes
"abc123".matchAll(/\d+/g);          // iterator com todos os detalhes
[..."a1b2c3".matchAll(/\d/g)];      // [["1"], ["2"], ["3"]]
"abc".replace(/b/, "X");            // "aXc"
"abc".replaceAll("a", "X");
"ab1cd2".split(/\d/);               // ["ab", "cd", ""]
"abc".search(/b/);                  // 1 (índice)

// Métodos em REGEX:
/\d+/.test("abc123");               // true (mais rápido se só quer bool)
/\d+/.exec("abc123");               // ["123", index: 3, ...]

# Grupos nomeados e referências

grupos.js
// Grupos de captura (numerados):
"2024-03-15".match(/(\d{4})-(\d{2})-(\d{2})/);
// ["2024-03-15", "2024", "03", "15"]

// Grupos nomeados — mais legível:
const m = "2024-03-15".match(/(?<ano>\d{4})-(?<mes>\d{2})-(?<dia>\d{2})/);
m.groups;  // { ano: "2024", mes: "03", dia: "15" }

// Replace usando grupos:
"2024-03-15".replace(
  /(?<ano>\d{4})-(?<mes>\d{2})-(?<dia>\d{2})/,
  "$<dia>/$<mes>/$<ano>"
);
// "15/03/2024"

// Replace com função:
"abc 123 def 456".replace(/\d+/g, (match) => Number(match) * 2);
// "abc 246 def 912"

// Lookahead e lookbehind:
/foo(?=bar)/      // "foo" seguido de "bar", sem consumir bar
/foo(?!bar)/      // "foo" NÃO seguido de "bar"
/(?<=\$)\d+/      // dígitos precedidos de $
/(?<!\$)\d+/      // dígitos NÃO precedidos de $

Boas práticas

A mecânica da linguagem você já conhece. Agora, um conjunto de princípios que a comunidade JavaScript desenvolveu ao longo de décadas — o tipo de coisa que faz o código envelhecer bem.

# Use const como padrão

A maior parte das suas variáveis não precisa mudar. Começar tudo com const e só trocar para let quando necessário elimina uma categoria inteira de bugs — a reatribuição acidental.

# Prefira imutabilidade

Use map, filter, spread e os novos métodos imutáveis de array (toSorted, toReversed, with) em vez de mutar estruturas. Isso torna o código mais previsível e trabalha muito melhor com React e paradigmas funcionais.

# Sempre ===

Ignore ==. A única exceção defensável é x == null para testar "null ou undefined" — e mesmo isso é controverso. Linters modernos impõem === por padrão.

# Nomes comunicam intenção

  • camelCase para variáveis e funções: buscarUsuario, totalCarrinho.
  • PascalCase para classes e construtores: Usuario, HttpClient.
  • CONSTANT_CASE para constantes verdadeiras de módulo: MAX_RETRIES, DEFAULT_TIMEOUT.
  • Booleanos começam com verbo: estaAtivo, temPermissao, podeEditar.
  • Prefira nomes descritivos e pronunciáveis — usrCnt economiza 3 letras e custa semanas de clareza.

# Funções pequenas, com uma responsabilidade

Uma função que faz uma coisa bem é mais reutilizável, testável e legível do que uma que faz cinco coisas razoavelmente. Se uma função está ficando difícil de nomear, é provavelmente porque está fazendo coisas demais.

# Trate erros onde faz sentido

Não embrulhe tudo em try/catch defensivamente. Deixe o erro propagar até o lugar onde há informação suficiente para tratar. Em aplicações reais, isso costuma ser: um handler de rota (back-end) ou um error boundary de UI (front-end).

# Prefira async/await a then chains

São equivalentes tecnicamente, mas async/await resulta em código linear, com estruturas de controle familiares (try/catch, for) — muito mais legível para código com mais de dois passos.

# Valide dados externos

Tudo que vem de fora do seu código — respostas de API, localStorage, variáveis de ambiente, inputs de usuário — precisa ser validado antes de ser usado. Bibliotecas como Zod, Valibot e ArkType facilitam isso criando schemas reutilizáveis.

# Use ferramentas modernas

  • ESLint com regras recomendadas — detecta bugs e impõe estilo.
  • Prettier — formatação automática.
  • TypeScript — a maior evolução que JS recebeu na última década; vale o custo inicial.
  • Vitest ou Node test runner — testes rápidos e modernos.
  • Bun ou tsx — para rodar arquivos direto sem build.

# Leia o código que você usa

Bibliotecas populares — Lodash, date-fns, React, Vue — são algumas das melhores fontes de aprendizado de JavaScript moderno. Abra os fontes delas de vez em quando. Você vai ver padrões, truques e decisões de design que não aparecem em tutorial nenhum.

Palavra final
JavaScript recompensa quem persiste. A linguagem tem esquinas estranhas — fruto de 30 anos de retrocompatibilidade — mas também tem recursos que poucas outras oferecem: closures poderosas, sistema de prototypes flexível, iteradores preguiçosos, proxies para meta-programação. Aprender bem o básico e os recantos é o que transforma um dev em alguém que faz JavaScript trabalhar para si.
