Capítulo 01

Introdução

Node.js é o que aconteceu quando alguém pegou o motor JavaScript do Chrome e perguntou: e se isso rodasse fora do navegador, com acesso a arquivos, sockets e processos? A resposta — runtime single-thread, orientado a eventos, especializado em I/O — virou padrão de fato para back-end JavaScript. Antes de ver código, vale entender o modelo mental: o que Node é, como ele consegue ser rápido sendo single-thread, e onde ele brilha (e onde tropeça).

# O que é Node.js

Node.js é um runtime — um ambiente de execução — para JavaScript fora do navegador. Tecnicamente, é um binário em C++ que combina três coisas: o motor V8 (que executa JavaScript), a biblioteca libuv (que cuida de I/O assíncrono e do event loop), e uma API de bibliotecas nativas (fs, http, net, crypto, etc.) escritas em C++ e expostas ao JavaScript.

O resultado é um interpretador único: você escreve JavaScript, mas em vez de manipular o DOM, manipula arquivos, requisições HTTP, processos filhos, streams binários. Não há window, não há document — em compensação, há process, Buffer, __dirname, e o conjunto inteiro de módulos prefixados com node:.

▸ Nota
Hoje existem outros runtimes JS para servidor — Deno e Bun são os principais. Ambos rodam código Node em boa parte dos casos, mas têm prioridades diferentes (Deno: segurança e TypeScript nativo; Bun: velocidade e tooling integrado). Este manual é sobre Node.js, que continua sendo o runtime mais maduro e amplamente adotado em produção.

# V8 — o motor JavaScript

O V8 é o motor JavaScript do Google Chrome, escrito em C++. Ele é o que transforma o seu código JavaScript em código de máquina executável — e o faz com várias camadas de otimização:

  • Parser → transforma o texto em AST (árvore sintática).
  • Ignition → interpretador que gera bytecode e roda direto.
  • TurboFan / Maglev / Sparkplug → compiladores JIT que recompilam funções "quentes" em código nativo otimizado, usando inferência de tipos baseada no que viu rodando.
  • Garbage collector → gerencia memória automaticamente, com geração jovem (Scavenger) e velha (Mark-Sweep-Compact).

A parte importante para quem programa: o V8 é muito rápido em código onde os tipos são previsíveis. Funções que recebem sempre os mesmos formatos de objetos são otimizadas; funções que recebem objetos com shapes diferentes a cada chamada acabam desotimizadas. Isso quase nunca importa no dia a dia — mas explica por que Node consegue performance próxima a linguagens compiladas em hot paths bem escritos.

# libuv — a outra metade

Se o V8 só executasse JavaScript, Node seria um interpretador qualquer. O que faz Node ser Node é a libuv — uma biblioteca em C que provê:

  • O event loop — o coração assíncrono que orquestra callbacks.
  • Acesso assíncrono a arquivos, sockets e processos, usando as APIs nativas do sistema operacional (epoll no Linux, kqueue no macOS/BSD, IOCP no Windows).
  • Uma thread pool (4 threads por padrão) para operações que não têm versão assíncrona no SO — como fs.readFile, DNS, crypto.pbkdf2, compressão.

O ponto importante: I/O é, na maioria dos casos, delegado ao kernel. Quando você abre um arquivo ou recebe uma conexão TCP, libuv não fica esperando — ela registra interesse no descritor e segue tocando outras tarefas. Quando o kernel avisa que os dados chegaram, o callback correspondente é enfileirado para rodar.

# Single-thread + event loop: o modelo mental

Esta é a frase que define Node: seu código JavaScript roda em uma única thread. Não há threads disputando o mesmo objeto, não há mutexes, não há race conditions sobre variáveis em memória. Em compensação, qualquer trecho síncrono que demore 200 ms congela o servidor inteiro por 200 ms.

Como então um servidor Node atende milhares de requisições simultâneas com uma thread só? A resposta é o event loop:

conceitual
// Pseudocódigo do que Node faz, conceitualmente
while (haTarefasPendentes()) {
  // 1. Roda callbacks que já estão prontos
  rodarCallbacksProntos();

  // 2. Pede ao SO os eventos de I/O que terminaram
  const eventos = esperarKernel();

  // 3. Para cada evento, enfileira o callback associado
  for (const ev of eventos) enfileirarCallback(ev);

  // 4. Repete — eternamente, até não haver mais nada
}

Quando seu código pede fs.readFile('a.txt', cb), Node delega a leitura à thread pool do libuv (porque fs não tem API assíncrona universal no SO), volta para o event loop, e quando a leitura termina, cb entra na fila para rodar na próxima volta. Enquanto isso, sua função pode ter aceito 50 conexões HTTP, respondido 30 e ficado esperando o banco em outras 20 — tudo na mesma thread.

▸ Cuidado
Single-thread não significa um core só. Você pode (e deve) escalar Node em produção rodando uma instância por core via cluster, PM2, ou containers replicados. O que é single-thread é o event loop de cada processo — não o limite do servidor.

O campo "scripts" define comandos acionados por npm run <nome>. Quatro comportamentos especiais valem a pena conhecer:

{

# Lockfiles

O lockfile registra a versão exata de cada dependência instalada — incluindo transitivas. Sem ele, npm install em máquinas diferentes pode instalar versões diferentes, quebrando de formas sutis.

  • Sempre comite o lockfile no repositório — é o que garante builds reproduzíveis.
  • Use npm ci (em vez de npm install) em CI/CD: ele instala exatamente o lockfile, sem modificá-lo, e falha se houver divergência.
  • Nunca misture gerenciadores de pacotes no mesmo projeto: npm + pnpm geram dois lockfiles e criam inconsistências.

Capítulo 03

ESM vs CommonJS

Node nasceu em 2009 com seu próprio sistema de módulos — CommonJS — porque na época o JavaScript não tinha um. Em 2015, o ECMAScript especificou um sistema oficial: ESM (ECMAScript Modules). Por anos os dois conviveram em tensão; hoje, em 2026, ESM é o padrão recomendado para código novo, e CommonJS é o legado que ainda sustenta metade do npm. Este capítulo explica como Node decide qual carregar, o que cada um faz de diferente, e como interoperar quando você inevitavelmente precisar.

# Dois sistemas de módulos, lado a lado

A diferença mais profunda entre os dois não é sintática — é semântica. CommonJS resolve módulos dinamicamente, em tempo de execução: require() é uma função normal que pode ser chamada condicionalmente, dentro de if, em qualquer ponto do código. ESM resolve estaticamente, antes de executar: import é uma diretiva sintática, é içada para o topo do arquivo, e o grafo de dependências é montado pelo runtime antes de qualquer linha rodar.

Essa distinção tem consequências em cascata: ESM pode carregar módulos em paralelo, suporta top-level await, permite tree-shaking agressivo em bundlers, e tem live bindings entre módulos. CommonJS é mais simples mentalmente, totalmente síncrono, e funciona em qualquer versão de Node — mas paga o preço em flexibilidade.

# Como o Node decide qual sistema usar

A regra é uma combinação da extensão do arquivo e do campo "type" no package.json mais próximo:

Extensão"type" no package.jsonCarregado como
.mjs(irrelevante)ESM
.cjs(irrelevante)CommonJS
.jsausente ou "commonjs"CommonJS (default histórico)
.js"module"ESM

Na prática, projetos novos colocam "type": "module" no package.json e usam .js para todos os arquivos. Quando você precisa pontualmente do outro sistema (raro), usa a extensão explícita: .cjs para um arquivo CommonJS dentro de um projeto ESM, ou .mjs para um arquivo ESM dentro de um projeto CommonJS.

# CommonJS — o antigo, ainda vivo

math.cjs
// Exporta uma função usando module.exports
function soma(a, b) { return a + b; }
function subtrai(a, b) { return a - b; }

module.exports = { soma, subtrai };

// Variantes equivalentes:
// exports.soma = soma;
// module.exports.soma = soma;
app.cjs
const { soma, subtrai } = require('./math.cjs');

if (process.env.DEBUG) {
  const debug = require('debug');

// Variáveis especiais que só existem em CJS:

# ESM — o moderno, agora padrão

math.js (com "type": "module")
// Named exports: várias formas equivalentes
export function soma(a, b) { return a + b; }
export function subtrai(a, b) { return a - b; }

// Ou tudo de uma vez no final:
// export { soma, subtrai };

// Default export: zero ou um por arquivo
export default function multiplica(a, b) { return a * b; }
app.js
// import deve ficar no topo do arquivo (estaticamente analisável)
import { soma, subtrai } from './math.js';     // named
import multiplica from './math.js';            // default
import * as math from './math.js';          // namespace
import { soma as add } from './math.js';       // renomeando

// node: protocol é OBRIGATÓRIO em ESM para built-ins
import { readFile } from 'node:fs/promises';

// import dinâmico — único caso "function-like" — retorna Promise
if (process.env.DEBUG) {
  const { default: debug } = await import('debug');
  debug('modo debug ativo');
}

Live bindings — uma diferença sutil mas real

Em CommonJS, require retorna um snapshot do objeto exportado no momento da chamada. Em ESM, o que você importa é uma referência ao vivo: se o módulo de origem reatribuir um let exportado, o consumidor vê o novo valor automaticamente.

contador.js (ESM)
export let contador = 0;
export function incrementar() { contador++; }
app.js (ESM)
import { contador, incrementar } from './contador.js';

console.log(contador);  // 0
incrementar();
console.log(contador);  // 1 — em ESM, vê o valor atualizado
                          // em CJS o equivalente continuaria 0
▸ Nota
Apesar de live bindings serem possíveis, raramente é boa ideia exportar variáveis mutáveis. O padrão recomendado continua sendo exportar funções e objetos imutáveis. Live bindings importam mais como garantia de que importar e reimportar o mesmo módulo retorna a mesma instância.

# O prefixo node:

Built-ins do Node como fs, http, path, crypto têm um prefixo opcional em CJS e fortemente recomendado em ESM: node:fs, node:http, etc. Por que existe?

  • Desambiguação: deixa explícito que é um built-in, não um pacote npm. Se alguém publicar um pacote chamado fs no npm, import 'fs' ficaria ambíguo; import 'node:fs' não.
  • Segurança: previne ataques de typosquatting e dependency confusion.
  • Performance: o resolver pula a busca em node_modules direto.

import { readFile } from 'node:fs/promises';

# Top-level await

Em ESM, você pode usar await diretamente no nível do módulo, sem envolver em uma função async. Isso muda padrões inteiros de inicialização — substitui o velho async IIFE por algo direto e legível:

init.js — antes (CommonJS)
const fs = require('node:fs/promises');

// Padrão IIFE async — feio mas necessário em CJS
(async () => {
  const config = JSON.parse(await fs.readFile('./config.json', 'utf8'));
  // ... resto da inicialização aninhado aqui dentro
})();
init.js — agora (ESM)
import { readFile } from 'node:fs/promises';

// Top-level await — código linear, sem IIFE
const config = JSON.parse(await readFile('./config.json', 'utf8'));

export const port = config.port;
export const dbUrl = config.dbUrl;
▸ Cuidado
Top-level await bloqueia a finalização do carregamento do módulo até a Promise resolver — qualquer importador espera. Use para inicialização legítima (config, conexões), não para trabalho que poderia ficar para depois. E nunca para algo que pode demorar indefinidamente: o módulo nunca terminaria de carregar.

As variáveis especiais __dirname e __filename de CommonJS não existem em ESM. Em troca, há import.meta.url — uma URL file:// apontando para o módulo atual:

// Uso típico: caminho absoluto para arquivo vizinho

▸ Node 22

# Interop entre ESM e CJS

Você consegue importar CJS de ESM, mas não consegue fazer require() de ESM em CJS — porque require é síncrono e ESM tem top-level await (potencialmente assíncrono). A saída é o import() dinâmico, que retorna Promise.

esm-importando-cjs.js
// Pacote CommonJS: o module.exports inteiro vira o "default"
import lodash from 'lodash';
lodash.chunk([1, 2, 3, 4], 2);

// Para alguns pacotes CJS, Node detecta named exports automaticamente:
import { chunk } from 'lodash';  // ✅ funciona em muitos casos

// Quando a detecção falha (raro), use a forma com default:
import pkg from 'pacote-cjs-problemático';
const { funcaoX } = pkg;
cjs-importando-esm.cjs
// require() falha em pacotes só-ESM:
// const fetch = require('node-fetch');  // ❌ ERR_REQUIRE_ESM

// Use import() dinâmico — retorna Promise

  const { default: fetch } = await import('node-fetch');
  const res = await fetch('https://api.example.com');
  console.log(await res.json());

main();

# Importando JSON

Em ESM, importar JSON exige import attributes — sintaxe que diz ao runtime como interpretar o arquivo. Foi padronizada como estável no Node 22 com a palavra-chave with:

# Qual escolher hoje

Para código novo, em 2026, a recomendação é direta: ESM. Coloque "type": "module" no package.json, use .js em todos os arquivos, adote o prefixo node: e abrace o top-level await. As únicas razões para escolher CommonJS hoje são:

  • Você está mantendo uma base CommonJS grande e estável — migração não compensa o risco.
  • Você publica uma biblioteca que precisa ser usável por consumidores antigos. Nesse caso, exporte dual (campo "exports" com "import" e "require", como visto no capítulo 2).
  • Você depende de uma ferramenta legada que ainda não suporta ESM bem (raro hoje).

Capítulo 04

Event loop

Você já entende o que o event loop faz — orquestrar callbacks. Falta entender como: quais filas existem, em que ordem rodam, por que setTimeout(fn, 0) e setImmediate(fn) alternam ordem em situações diferentes, e como uma chamada inocente a JSON.parse de um arquivo grande pode travar 10 mil requisições em paralelo. Este é o capítulo mais denso do volume — guarde-o; é o que distingue quem usa Node de quem entende Node.

# Por que existe um loop

Imagine seu programa como uma lista de coisas a fazer: rodar tal código quando uma requisição HTTP chegar, rodar tal outro quando o arquivo terminar de ler, rodar este quando o timer expirar. Em vez de criar uma thread para cada uma dessas espera-execuções, libuv mantém um loop infinito que pergunta ao kernel: "o que aconteceu desde a última volta?", executa os callbacks correspondentes, e dá outra volta.

O loop só termina quando não há mais nada para fazer — nenhum timer ativo, nenhum socket aberto, nenhum callback pendente. Aí o processo encerra. É por isso que um servidor HTTP simples roda "para sempre": tem sempre um socket escutando, então há sempre algo no loop.

# As seis fases

O event loop não é uma fila única — é uma sequência ordenada de seis filas, cada uma com seu propósito. Toda volta percorre as fases na mesma ordem:

as fases do event loop
      ┌────────────────────────────────────────────────────┐
      │                                                    │
      │   ┌─────────────┐    setTimeout, setInterval       │
   ┌─→│   │   TIMERS    │    cujo tempo já expirou         │
   │  │   └──────┬──────┘                                   │
   │  │          ↓                                          │
   │  │   ┌─────────────┐    callbacks de I/O adiados      │
   │  │   │   PENDING   │    (alguns erros TCP, etc.)      │
   │  │   └──────┬──────┘                                   │
   │  │          ↓                                          │
   │  │   ┌─────────────┐    interno do libuv               │
   │  │   │ IDLE / PREP │    (você não interage com isso)  │
   │  │   └──────┬──────┘                                   │
   │  │          ↓                                          │
   │  │   ┌─────────────┐    NOVOS eventos de I/O          │
   │  │   │    POLL     │    onde o loop passa MAIS tempo  │
   │  │   └──────┬──────┘    (HTTP, fs, net, ...)           │
   │  │          ↓                                          │
   │  │   ┌─────────────┐    setImmediate                  │
   │  │   │    CHECK    │                                   │
   │  │   └──────┬──────┘                                   │
   │  │          ↓                                          │
   │  │   ┌─────────────┐    socket.on('close'), etc.      │
   │  │   │    CLOSE    │                                   │
   │  │   └──────┬──────┘                                   │
   │  │          │                                          │
   └──┘          │                                          │
      │   loop volta ao topo se há trabalho                 │
      └────────────────────────────────────────────────────┘

   ★ ENTRE CADA callback de QUALQUER fase, Node:
       1. drena a fila de process.nextTick
       2. drena a fila de microtasks (Promises, queueMicrotask)

Vale destacar duas fases:

  • poll — onde quase tudo acontece. Aqui o loop pede ao kernel os eventos de I/O que terminaram (uma requisição chegou, um arquivo foi lido, um socket recebeu dados). Se não houver nada pendente em outras fases, libuv pode bloquear aqui esperando o kernel — é como um servidor "dorme" até a próxima requisição.
  • check — fase exclusiva do setImmediate. Sempre vem logo após poll, sem competir com timers — esta é a chave para a previsibilidade que vamos explorar adiante.

# Microtasks: process.nextTick e Promises

Além das seis fases, Node mantém duas filas especiais que rodam entre qualquer dois callbacks de qualquer fase:

  • process.nextTick — exclusivo do Node, prioridade máxima. Roda antes de qualquer microtask de Promise.
  • Microtasks — Promises (.then, await) e queueMicrotask. Padrão da linguagem.

A ordem de drenagem é:

o que acontece após cada callback
callback de uma fase termina
  ↓
drenar process.nextTick — roda TUDO que estiver na fila
  └→ se um nextTick agendar outro nextTick, roda também
  ↓
drenar microtasks (Promises) — roda TUDO
  └→ se uma Promise agendar outra microtask, roda também
  └→ entre microtasks NÃO drena nextTick de novo no padrão atual
  ↓
próximo callback (mesma fase ou próxima)
▸ Nota
A ordem entre process.nextTick e microtasks já variou entre versões de Node. No Node 22 atual: nextTick sempre antes de microtasks. Mas a regra prática é a mesma há anos — não dependa de detalhes finos da ordem: se sua lógica precisa de process.nextTick rodando em ordem específica em relação a Promises, há provavelmente um bug de design.

# A ordem completa: um exemplo guiado

Veja se consegue prever a saída antes de ler:

ordem.js
console.log('1. sync');

setTimeout(() => console.log('5. timeout'), 0);
setImmediate(() => console.log('6. immediate'));

Promise.resolve().then(() => console.log('4a. promise'));
queueMicrotask(() => console.log('4b. microtask'));

process.nextTick(() => console.log('3. nextTick'));

console.log('2. sync');

// Saída:
// 1. sync
// 2. sync
// 3. nextTick
// 4a. promise
// 4b. microtask
// 5. timeout    (ou 6, depende do timing — veja próxima seção)
// 6. immediate  (ou 5)

O raciocínio:

  1. Tudo que é síncrono roda primeiro, na ordem do código (1. sync, 2. sync).
  2. Quando o código síncrono termina, Node drena process.nextTick antes de qualquer outra coisa (3. nextTick).
  3. Em seguida, drena microtasks de Promise e queueMicrotask em ordem de inserção (4a, 4b).
  4. Só então o event loop entra na primeira volta — fase timers ou check dependendo de qual estiver "pronto".

# setTimeout(fn, 0) vs setImmediate(fn)

Esta é a pegadinha clássica de Node. A ordem entre os dois depende do contexto:

caso-1-fora-de-io.js
// Fora de qualquer callback de I/O
setTimeout(() => console.log('timeout'), 0);
setImmediate(() => console.log('immediate'));

// Saída: NÃO DETERMINÍSTICA. Pode ser:
//   timeout → immediate    (se <1ms passou na primeira volta)
//   immediate → timeout    (se o timer ainda não atingiu o threshold)
caso-2-dentro-de-io.js
import { readFile } from 'node:fs';

readFile(import.meta.filename, () => {
  // Estamos DENTRO de um callback de I/O — fase poll já passou
  setTimeout(() => console.log('timeout'), 0);
  setImmediate(() => console.log('immediate'));
});

// Saída: SEMPRE
//   immediate
//   timeout

Por quê? Quando você está dentro de um callback de I/O, o loop acabou de sair da fase poll. A próxima fase é check — e é justamente lá que setImmediate roda. Só na volta seguinte do loop é que voltamos a timers. Resultado: setImmediate sempre vence dentro de I/O.

Fora de I/O, o resultado depende de qual fase Node entrou primeiro e se 1ms já passou desde o agendamento do setTimeout(fn, 0). Não confie nessa ordem.

▸ Dica
Se você quer "rodar isso assim que possível, mas depois do código atual": prefira queueMicrotask(fn) (mais rápido, garante que roda na mesma volta) ou setImmediate(fn) (próxima volta, mas determinístico). Evite setTimeout(fn, 0) — comportamento dependente do contexto.

# Starvation: quando uma fila bloqueia outras

Como process.nextTick e microtasks rodam antes de o loop avançar, é possível agendá-los recursivamente e nunca deixar o loop entrar em poll. Resultado: o servidor não atende mais nada.

starvation.js
// ❌ ANTI-PADRÃO: nunca devolve o controle ao event loop
function tickInfinito() {
  process.nextTick(tickInfinito);  // nextTick sempre roda antes de I/O
}
tickInfinito();

// O servidor HTTP abaixo nunca recebe uma requisição — nenhuma!
import { createServer } from 'node:http';
createServer((req, res) => res.end('oi')).listen(3000);

O mesmo padrão com Promise.resolve().then(...) recursivo causa starvation parecida — microtasks também rodam antes do loop avançar. A regra prática é simples: se você precisa repetir trabalho continuamente, use setImmediate (entrega o controle ao loop) ou setTimeout, nunca process.nextTick ou Promises encadeadas em loop fechado.

# Bloquear o loop = travar o servidor

O event loop é single-thread. Qualquer código síncrono que demore é tempo em que nada mais acontece — nenhuma requisição é aceita, nenhum timer dispara, nenhum callback de I/O é processado.

vilao.js
import { createServer } from 'node:http';
import { readFileSync } from 'node:fs';

createServer((req, res) => {
  // ❌ readFileSync trava o event loop pelo tempo da leitura
  const dados = readFileSync('/tmp/grande.json', 'utf8');

  // ❌ JSON.parse é síncrono e proporcional ao tamanho
  const obj = JSON.parse(dados);

  // ❌ for-loop síncrono pesado bloqueia tudo
  let total = 0;
  for (let i = 0; i < obj.itens.length; i++) {
    total += calcularPesado(obj.itens[i]);
  }

  res.end(JSON.stringify({ total }));
}).listen(3000);

As três armadilhas comuns:

  1. APIs com sufixo Sync — readFileSync, writeFileSync, execSync. Aceitáveis em scripts de CLI ou inicialização (uma única vez); proibidos em hot path de servidor.
  2. JSON.parse de payloads grandes — é síncrono e CPU-bound. Para arquivos > alguns MB, considere streaming JSON parsers.
  3. Loops longos sobre arrays grandes — qualquer iteração que demore > 10ms precisa ser quebrada (em chunks com setImmediate entre eles) ou movida para Worker Threads.

# Detectando bloqueios em produção

Node oferece um histograma do delay do event loop — uma métrica direta de quão "ocupado" o loop está. Se o p99 sobe, há trabalho síncrono comendo tempo.

monitor-loop.js
import { monitorEventLoopDelay } from 'node:perf_hooks';

// Sample a cada 20ms — overhead negligível
const histogram = monitorEventLoopDelay({ resolution: 20 });
histogram.enable();

setInterval(() => {
  const ms = (n) => (n / 1e6).toFixed(2);  // ns → ms
  console.log({
    p50: ms(histogram.percentile(50)),
    p99: ms(histogram.percentile(99)),
    max: ms(histogram.max),
  });
  histogram.reset();
}, 5000);

Em servidor saudável, p99 deve ficar < 10ms; max ocasional acima disso é tolerável (GC, I/O em surto). Picos sustentados acima de 50ms indicam bloqueio significativo — vale investigar com --prof ou clinic.js (capítulo 18).

▸ Performance
A regra prática que vale a pena memorizar: uma task só (callback inteiro mais drenagem de microtasks) idealmente fica abaixo de 10ms. Acima disso, você está roubando latência de outras requisições. Se uma operação genuinamente demora mais — criptografia pesada, parsing massivo, cálculo numérico —, ela pertence a um Worker Thread (capítulo 11), não ao event loop principal.

# Quando você precisa sair do loop

Há três escapes legítimos para CPU-bound em Node, em ordem crescente de complexidade:

  1. Quebrar em chunks com setImmediate — para loops grandes que cabem na mesma thread mas precisam ceder o controle periodicamente.
  2. Worker Threads (capítulo 11) — thread real, com seu próprio V8 e event loop, comunicando-se por mensagens.
  3. Processo filho ou serviço separado — quando o trabalho é pesado o suficiente para justificar isolamento de processo.
chunk-com-setImmediate.js
// Em vez de processar 1 milhão de itens de uma vez bloqueando o loop,
// processa 1000 por vez e cede o controle entre os chunks
async function processarEmChunks(itens, processar, tamanho = 1000) {
  for (let i = 0; i < itens.length; i += tamanho) {
    const chunk = itens.slice(i, i + tamanho);
    chunk.forEach(processar);

    // Cede o controle: outras requisições e callbacks podem rodar
    await new Promise(resolve => setImmediate(resolve));
  }
}

Promises & async/await

Promises e async/await são as duas faces da mesma moeda: async/await é apenas açúcar sintático sobre Promises. Este capítulo vai além do básico e cobre o que projetos reais exigem: combinar múltiplas operações assíncronas, cancelar trabalho em andamento, implementar timeouts e retries robustos, e tratar erros de forma que nenhuma rejeição passe em silêncio.

# Promise vs async/await: a relação

Toda função async retorna uma Promise. await desempacota a Promise e pausa a execução da função (não do event loop — que continua rodando). Isso significa que as duas formas abaixo são semanticamente idênticas:

equivalencia.js
// Com .then — encadeamento explícito
function buscarUsuario(id) {
  return fetch(`/usuarios/${id}`)
    .then(res => {
      if (!res.ok) throw new Error(`HTTP ${res.status}`);
      return res.json();
    });
}

// Com async/await — mesma coisa, mais legível
async function buscarUsuario(id) {
  const res = await fetch(`/usuarios/${id}`);
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  return res.json();  // return de async função wrapa em Promise automaticamente
}

# Os combinadores

Quatro métodos estáticos de Promise para trabalhar com múltiplas Promises em paralelo — escolher o certo é questão de qual comportamento você quer em caso de falha:

MétodoResolve quandoRejeita quandoUse para…
Promise.allTodas resolvemQualquer uma falhaTodas devem ter sucesso (ex: buscar user + roles)
Promise.allSettledSempre (após todas terminarem)NuncaDisparar N tarefas independentes e inspecionar resultados

combinadores.js
import { setTimeout as sleep } from 'node:timers/promises';

// Promise.all — falha rápida, tudo ou nada
const [usuario, posts, roles] = await Promise.all([
  buscarUsuario(userId),
  buscarPosts(userId),
  buscarRoles(userId),
]); // as três correm em paralelo — sem await sequencial!

// Promise.allSettled — inspecionar individualmente
const resultados = await Promise.allSettled([
  enviarEmail(user1),
  enviarEmail(user2),
  enviarEmail(user3),
]);
const falhas = resultados
  .filter(r => r.status === 'rejected')
  .map(r => r.reason);

// Promise.any — corrida, usar o primeiro sucesso
const resposta = await Promise.any([
  fetch('https://api-us.example.com/data'),
  fetch('https://api-eu.example.com/data'),
  fetch('https://api-br.example.com/data'),
]); // usa o endpoint que responder primeiro
▸ Cuidado
Await em loop sequencial é um dos anti-padrões mais comuns em Node. for (const id of ids) { await buscar(id) } faz uma requisição de cada vez. Se as operações são independentes, use Promise.all(ids.map(buscar)) para disparar todas em paralelo.

# node:timers/promises

Node 22 inclui versões baseadas em Promise dos timers clássicos, acessíveis em node:timers/promises. Eliminam o anti-padrão de new Promise(resolve => setTimeout(resolve, ms)):

timers-promises.js
import {
  setTimeout as sleep,
  setInterval as interval,
  setImmediate as immediate,
} from 'node:timers/promises';

// Aguardar N milissegundos — limpo, sem wrapper manual
await sleep(1000);

// setTimeout com valor de retorno
const resultado = await sleep(500, 'pronto');  // resolve com 'pronto'

// setImmediate como Promise — cede o loop e continua
await immediate();

// Iterador assíncrono de setInterval — ticker
const tick = interval(1000, 'ping');

for await (const msg of tick) {
  console.log(msg);  // 'ping' a cada segundo
  if (deveParar()) tick.return();  // cancela o iterador
}

# AbortController e cancelamento

O padrão de cancelamento moderno em JavaScript é baseado em AbortController: você cria um controller, passa o signal para operações que o suportam, e chama controller.abort() quando quer cancelar. Em Node 22, fetch, streams, setTimeout/setInterval de node:timers/promises, e child_process todos aceitam signal.

abort-controller.js
import { setTimeout as sleep } from 'node:timers/promises';

// Cancelar uma operação em andamento
const controller = new AbortController();
const { signal } = controller;

// Cancela após 5 segundos
setTimeout(() => controller.abort(new Error('timeout')), 5000);

try {
  const res = await fetch('https://api.example.com/dados', { signal });
  const dados = await res.json();
  return dados;
} catch (err) {
  if (err.name === 'AbortError') {
    console.log('Requisição cancelada');
    return null;
  }
  throw err;  // relança outros erros
}

// Cancelar timers também
await sleep(10_000, undefined, { signal }); // cancela se signal disparar

AbortSignal.timeout — atalho para timeout de uma operação

signal-timeout.js
// AbortSignal.timeout: cria um signal que dispara após N ms
// Sem precisar de AbortController manual — elegante e menos verboso
const res = await fetch('https://api.example.com', {
  signal: AbortSignal.timeout(3000),  // cancela após 3s
});

// Combinar múltiplos signals: AbortSignal.any (Node 20+)
const perReqController = new AbortController();
const signal = AbortSignal.any([
  perReqController.signal,       // cancelamento manual
  AbortSignal.timeout(5000),   // timeout automático
]);

# Tratando erros em código assíncrono

Erros em código assíncrono têm uma regra simples: toda Promise que pode rejeitar precisa de um .catch ou de um try/catch em volta do await. Promises rejeitadas sem handler são silenciosas por padrão — e em Node 22 causam um warning grave:

tratamento-erros.js
// ✅ try/catch com await — padrão mais comum
async function buscar(id) {
  try {
    const res = await fetch(`/api/${id}`);
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
    return await res.json();
  } catch (err) {
    console.error('Falha ao buscar', err);
    throw err;  // relança para o chamador decidir
  }
}

// ❌ Promise flutuante — silenciosa, sem tratamento

# Retry com backoff exponencial

Operações de rede falham. Um bom retry implementa três coisas: limite de tentativas, espera crescente entre elas (backoff), e um toque de aleatoriedade (jitter) para evitar que muitos clientes tentem exatamente ao mesmo tempo:

retry.js
import { setTimeout as sleep } from 'node:timers/promises';

async function comRetry(fn, {
  tentativas = 3,
  baseMs = 200,
  maxMs = 10_000,
  signal,
} = {}) {
  for (let tentativa = 0; tentativa <= tentativas; tentativa++) {
    try {
      return await fn({ tentativa, signal });
    } catch (err) {
      const ultima = tentativa === tentativas;
      const cancelado = err.name === 'AbortError';

      if (ultima || cancelado) throw err;

      // Backoff exponencial + jitter: 200ms, 400ms, 800ms… com ruído
      const jitter = Math.random() * baseMs;
      const delay = Math.min(baseMs * 2 ** tentativa + jitter, maxMs);

      console.warn(`Tentativa ${tentativa + 1} falhou, aguardando ${delay.toFixed(0)}ms`);
      await sleep(delay, undefined, { signal });
    }
  }
}

// Uso
const dados = await comRetry(
  ({ signal }) => fetch('https://api.example.com/dados', { signal }),
  { tentativas: 4, baseMs: 300, signal: AbortSignal.timeout(30_000) }
);

# Padrões de execução paralela com limite

Promise.all com um array grande dispara todas as operações de uma vez — pode sobrecarregar APIs externas ou esgotar conexões do banco. Para disparar N operações com no máximo X em paralelo, use um pool simples:

pool.js
// Processa itens com no máximo `concorrencia` Promises ativas
async function mapPool(itens, fn, concorrencia = 5) {
  const resultados = [];
  const ativas = new Set();

  for (const [i, item] of itens.entries()) {
    const p = fn(item, i).then(v => { resultados[i] = v; ativas.delete(p); });
    ativas.add(p);
    if (ativas.size >= concorrencia) await Promise.race(ativas);
  }
  await Promise.all(ativas);
  return resultados;
}

// Exemplo: processar 500 usuários, 10 por vez
const resultados = await mapPool(usuarios, usuario =>
  enviarEmail(usuario), 10
);
▸ Node 22
A partir do Node 22, fetch global é considerado estável — sem flag experimental. Suporta AbortSignal, streaming de corpo via response.body (ReadableStream), e keepalive. Para produção com muitas requisições, considere configurar um dispatcher customizado via undici (a engine por trás do fetch nativo) para controlar pool de conexões.

# Anti-padrões comuns

// ❌ await desnecessário no return — cria stack frame extra

// ✅ return direto (exceto quando dentro de try/catch — aí await é necessário!)

// ✅ Dentro de try/catch, DEVE ter await para capturar rejeição
async function correto() {
  try { return await buscar(); } catch(e) { return null; }
}

// ❌ new Promise desnecessário — wrapping de função já async
const dados = await new Promise((resolve, reject) => {
  fetch('/api').then(resolve).catch(reject);  // desnecessário
});
// ✅ Simples e direto
const dados = await fetch('/api');

// ❌ Sequential quando podia ser paralelo
const a = await buscar('a');  // espera A terminar...
const b = await buscar('b');  // ...para SÓ ENTÃO começar B
// ✅ Paralelo
const [a, b] = await Promise.all([buscar('a'), buscar('b')]);

Streams

Streams são a abstração que Node usa para mover dados que não cabem na memória — ou que ainda nem chegaram inteiros. Ler um arquivo de 4GB byte por byte, transmitir uma resposta HTTP enquanto ela é gerada, comprimir em tempo real entre o disco e a rede: tudo passa por streams. Este capítulo explica os quatro tipos, o problema do backpressure que motiva tudo, e os padrões modernos que substituíram o .pipe() clássico.

# Por que streams existem

O caminho ingênuo para copiar um arquivo é ler tudo na memória e escrever tudo de uma vez:

copia-ingenua.js
import { readFile, writeFile } from 'node:fs/promises';

// ❌ Funciona para 1MB, falha para 4GB — RAM esgotada
const conteudo = await readFile('enorme.bin');
await writeFile('copia.bin', conteudo);

O caminho com streams move os dados em chunks — pedaços de 16KB-64KB que entram, são processados e saem, sem nunca ficar todos juntos na memória:

copia-com-stream.js
import { createReadStream, createWriteStream } from 'node:fs';
import { pipeline } from 'node:stream/promises';

// ✅ Funciona para qualquer tamanho — memória constante
await pipeline(
  createReadStream('enorme.bin'),
  createWriteStream('copia.bin')
);

A diferença entre os dois exemplos não é estética — é a diferença entre um servidor que cai sob carga e um que não cai. Streams são memória constante independente do tamanho dos dados.

A maior parte do código de aplicação interage com streams já existentes — você não cria sockets do zero, recebe-os do http. Mas Transform é o tipo que mais aparece em código novo: parsers, formatadores, validadores, qualquer coisa que processe dados em fluxo.

# O problema central: backpressure

Imagine um leitor (do disco) que produz a 100 MB/s e um escritor (na rede) que consome a 10 MB/s. Se conectar os dois ingenuamente, o leitor vai produzir 90 MB a mais do que o escritor consegue absorver a cada segundo. Esses 90 MB têm que ir para algum lugar — e o lugar é a memória do processo. Em poucos segundos, o RAM acabou.

Esse é o problema do backpressure: quando produtor é mais rápido que consumidor, o produtor precisa desacelerar. Streams resolvem isso com um buffer interno limitado (o highWaterMark, default 16KB) e um sinal: quando o buffer enche, writable.write(chunk) retorna false, dizendo "pare de me alimentar até eu emitir 'drain'".

backpressure-manual.js
// O que pipeline() faz por você, conceitualmente
readable.on('data', (chunk) => {
  const ok = writable.write(chunk);
  if (!ok) {
    readable.pause();                        // pare de produzir
    writable.once('drain', () => readable.resume()); // retome quando pronto
  }
});
readable.on('end', () => writable.end());
▸ Cuidado
Esse código parece certo mas tem três bugs sutis que você não quer escrever: erros não são propagados entre os streams, o destino não é fechado se a origem falhar, e 'data'/'end' em vez de 'readable'/'end' têm pegadinhas com sequência de eventos. Use pipeline. Sempre.

# pipeline — a forma correta

pipeline conecta múltiplos streams, propaga backpressure automaticamente, encerra todos se qualquer um falhar, e retorna uma Promise. Use sempre que conectar dois ou mais streams:

pipeline-completa.js
import { createReadStream, createWriteStream } from 'node:fs';
import { pipeline } from 'node:stream/promises';
import { createGzip } from 'node:zlib';

// Compactar um arquivo grande — leitura → gzip → escrita
try {
  await pipeline(
    createReadStream('log.txt'),
    createGzip(),
    createWriteStream('log.txt.gz')
  );
} catch (err) {
  console.error('Falha em algum estágio:', err);
}

// Cancelamento via AbortSignal — encerra todos os streams limpos
const signal = AbortSignal.timeout(30_000);
await pipeline(
  createReadStream('lento.bin'),
  createWriteStream('destino.bin'),
  { signal }
);

# Async iterators: streams como for-of

Todo Readable é async iterable. Isso significa que você pode percorrê-lo com for await...of e a linguagem cuida do backpressure por você — cada await da iteração só pede o próximo chunk quando o handler termina:

for-await-stream.js
import { createReadStream } from 'node:fs';
import { createInterface } from 'node:readline';

// Processar arquivo linha por linha — backpressure automático
const rl = createInterface({
  input: createReadStream('enorme.log'),
  crlfDelay: Infinity,  // trata \r\n como uma única quebra
});

let contagem = 0;
for await (const linha of rl) {
  if (linha.includes('ERROR')) {
    await salvarErro(linha);  // stream pausa naturalmente aqui
    contagem++;
  }
}
console.log(`${contagem} erros encontrados`);

Para streams que produzem objetos (não só bytes), for await...of é frequentemente a forma mais limpa de consumir. Compare com a forma "tradicional" baseada em eventos:

eventos-vs-async-iter.js
// ❌ Tradicional — baseado em eventos, controle de fluxo manual
function processar(stream) {
  return new Promise((resolve, reject) => {
    let total = 0;
    stream.on('data', (chunk) => { total += chunk.length; });
    stream.on('end',  () => resolve(total));
    stream.on('error', reject);
  });
}

// ✅ Moderno — async iterator, controle de fluxo automático
async function processar(stream) {
  let total = 0;
  for await (const chunk of stream) total += chunk.length;
  return total;
}

# Modo objeto vs modo binário

Streams têm dois modos. No padrão (modo binário), chunks são Buffer ou string; tamanhos são contados em bytes. No modo objeto (objectMode: true), chunks são qualquer valor JS, e o highWaterMark é contado em número de itens, não bytes.

object-mode.js
import { Readable, Writable } from 'node:stream';

// Stream que produz objetos a partir de um array
const origem = Readable.from([
  { id: 1, nome: 'Ana' },
  { id: 2, nome: 'Beto' },
  { id: 3, nome: 'Carla' },
]); // objectMode: true automático em Readable.from(array)

// Writable em modo objeto — recebe um objeto por write
const destino = new Writable({
  objectMode: true,
  write(obj, _enc, cb) {
    console.log('Recebido:', obj);
    cb();  // sinaliza que processou — libera o próximo
  },
});

await pipeline(origem, destino);
▸ Nota
Readable.from(iterable) é o atalho moderno para criar Readables a partir de arrays, geradores, async iterators ou qualquer iterable. Em quase todos os casos é mais limpo que herdar de Readable e implementar _read().

# Transform — pipeline de processamento

Transform é o tipo que você mais escreve. Recebe chunks pela entrada, produz chunks na saída. Use a forma funcional quando puder — herdar de Transform só quando precisar manter estado ou implementar lógica complexa.

transform-funcional.js
import { Transform } from 'node:stream';
import { pipeline } from 'node:stream/promises';

// Transform com generator: forma mais limpa para casos comuns
await pipeline(
  origem,

  // Cada chunk produz zero ou mais chunks de saída via yield
  async function* filtraAtivos(source) {
    for await (const usuario of source) {
      if (usuario.ativo) yield usuario;
    }
  },

  async function* enriquece(source) {
    for await (const usuario of source) {
      const roles = await buscarRoles(usuario.id);
      yield { ...usuario, roles };
    }
  },

  destino
);

// Equivalente clássico — Transform com objectMode
const filtraAtivosClassic = new Transform({
  objectMode: true,
  transform(usuario, _enc, cb) {
    if (usuario.ativo) this.push(usuario);
    cb();
  },
});

Generators async como estágios de pipeline são uma das formas mais elegantes de processar dados em Node moderno. Ganham em legibilidade, herdam controle de fluxo do for await, e se compõem naturalmente.

# Erros e cleanup

O comportamento de streams diante de erros é a fonte mais comum de leaks em Node. As regras:

  • Streams emitem 'error'. Se ninguém escuta, o processo crasha — e geralmente isso é o que você quer.
  • Erro em um estágio não fecha automaticamente os outros — exceto se você usou pipeline.
  • Sempre que criar streams manualmente fora de pipeline, plugue um handler de 'error' em cada um — ou (preferível) reestruture para usar pipeline.
cleanup.js
import { finished } from 'node:stream/promises';

// finished: aguarda stream terminar (com sucesso ou erro)
// útil quando NÃO há pipeline — ex: você só consome um stream
const stream = criarStream();

try {
  for await (const chunk of stream) {
    await processar(chunk);
  }
  await finished(stream);  // confirma término limpo
} catch (err) {
  stream.destroy(err);  // fecha recursos do stream em caso de erro
  throw err;
}

# Padrões de stream em servidores HTTP

Em handlers HTTP do Node, req é um Readable e res é um Writable. Streamar a resposta em vez de bufferizar reduz drasticamente o uso de memória sob carga.

servidor-streaming.js
import { createServer } from 'node:http';
import { createReadStream } from 'node:fs';
import { pipeline } from 'node:stream/promises';
import { createGzip } from 'node:zlib';

createServer(async (req, res) => {
  if (req.url === '/download') {
    res.setHeader('Content-Encoding', 'gzip');
    res.setHeader('Content-Type', 'application/octet-stream');

    // Memória constante mesmo para arquivos enormes
    try {
      await pipeline(
        createReadStream('./grandao.bin'),
        createGzip(),
        res
      );
    } catch (err) {
      // Cliente desconectou ou erro de leitura — pipeline já fechou tudo
      console.error('Falha no download:', err.message);
    }
  }
}).listen(3000);
▸ Performance
Em servidores HTTP, sempre que a resposta puder ser maior que ~1MB, considere streaming. A diferença prática: 1000 clientes baixando um arquivo de 100MB com bufferização → 100GB de RAM (impossível); com streaming → ~16MB total. Streams são o que permite Node ser viável para CDNs, proxies e file servers.

Buffers & Binário

JavaScript nasceu sem tipo nativo para dados binários — strings eram suficientes para o DOM. Node precisava lidar com arquivos, sockets e protocolos de rede, então criou o Buffer: um bloco de memória de tamanho fixo, fora do heap do V8, para dados brutos. Hoje Buffer e TypedArray convivem; entender quando usar cada um, como converter entre formatos, e os cuidados de segurança evita bugs sutis e vazamentos.

# Buffer: o básico

Buffer é uma subclasse de Uint8Array — cada elemento é um inteiro de 0 a 255. A diferença principal é que Buffer tem métodos adicionais de encoding e vem do runtime Node, não da linguagem. Três formas de criar:

criar-buffers.js
// 1. A partir de uma string — especifique sempre o encoding
const b1 = Buffer.from('Olá, mundo', 'utf8');
const b2 = Buffer.from('SGVsbG8=', 'base64');   // "Hello" em base64
const b3 = Buffer.from('deadbeef', 'hex');       // 4 bytes: 0xde 0xad 0xbe 0xef

// 2. A partir de array de bytes
const b4 = Buffer.from([0x48, 0x65, 0x6c, 0x6c, 0x6f]); // "Hello"

// 3. Buffer zerado de tamanho fixo — para alocar e preencher depois
const b5 = Buffer.alloc(16);          // 16 bytes, todos 0x00
const b6 = Buffer.alloc(16, 0xff);     // 16 bytes, todos 0xff

// ❌ Buffer.allocUnsafe — MAIS rápido, mas contém lixo de memória anterior
// Use apenas quando vai preencher TUDO imediatamente — ex: geração de IDs
const b7 = Buffer.allocUnsafe(32);   // rápido, mas perigoso se não preencher
▸ Cuidado
Buffer.allocUnsafe é mais rápido porque não zera a memória — ela pode conter dados de alocações anteriores do processo. Nunca exponha esse Buffer para clientes ou logs sem preencher completamente. Para tamanhos abaixo de ~4KB, a diferença de performance é negligível — prefira Buffer.alloc.

# Buffer e TypedArrays

Como Buffer é subclasse de Uint8Array, ele é também um ArrayBuffer view. Isso significa que APIs Web (WebCrypto, WebAssembly, fetch) aceitam Buffer diretamente — mas há uma pegadinha com o offset:

buffer-typed-array.js
const buf = Buffer.from([1, 2, 3, 4]);

// Buffer é Uint8Array — acessa bytes por índice
console.log(buf[0]);               // 1
console.log(buf instanceof Uint8Array); // true

// CUIDADO: buf.buffer (ArrayBuffer subjacente) pode ser maior que buf!
// Buffer usa pool interno — byteOffset pode ser > 0
console.log(buf.byteOffset);  // pode não ser 0 para buffers pequenos
console.log(buf.buffer.byteLength);  // pode ser 8192 (tamanho do pool)

// ✅ Para passar como ArrayBuffer para WebCrypto e similares, extraia uma cópia:
const ab = buf.buffer.slice(buf.byteOffset, buf.byteOffset + buf.byteLength);

// Ou use o atalho:
const ab2 = buf.buffer.slice(buf.byteOffset, buf.byteOffset + buf.byteLength);
const u32 = new Uint32Array(ab2);  // view de 32-bit sobre os mesmos bytes

DataView — leitura de inteiros multi-byte

# Bytes e strings: tamanho não é o que parece

Um erro comum é usar string.length para medir quantos bytes uma string ocupa. Em UTF-8, caracteres fora do ASCII básico ocupam 2, 3 ou 4 bytes — não 1:

tamanho-bytes.js
const s1 = 'hello';
const s2 = 'olá';
const s3 = '你好';
const s4 = '🚀';

console.log(s1.length, Buffer.byteLength(s1));  // 5, 5   (ASCII puro)
console.log(s2.length, Buffer.byteLength(s2));  // 3, 4   (á = 2 bytes)
console.log(s3.length, Buffer.byteLength(s3));  // 2, 6   (CJK = 3 bytes cada)
console.log(s4.length, Buffer.byteLength(s4));  // 2, 4   (emoji = 4 bytes, par substituto)

// Use Buffer.byteLength quando precisar do tamanho em bytes (headers HTTP, etc.)
res.setHeader('Content-Length', Buffer.byteLength(body, 'utf8'));
▸ Dica
Sempre que precisar medir o tamanho de uma string para um header HTTP, banco de dados com limite de bytes, ou protocolo de rede: use Buffer.byteLength(str, encoding) — nunca str.length. Para truncar texto a N bytes sem cortar caracteres multibyte pela metade, escreva no Buffer e converta de volta com a string resultante de buf.subarray(0, N).toString('utf8').

EventEmitter

O padrão pub/sub é a cola que une Node internamente: streams, processos filhos, sockets, o próprio process — todos são EventEmitters. Entender a API, os riscos de acumular listeners e as formas modernas de consumir eventos como async iterables é fundamental para trabalhar com qualquer API nativa do Node.

# Conceito: pub/sub

EventEmitter implementa o padrão publicador/assinante: quem emite um evento não sabe quem está escutando, e quem escuta não precisa saber de onde vem. A classe está em node:events e é a base de quase todo o runtime:

basico.js
import { EventEmitter } from 'node:events';

const emitter = new EventEmitter();

// Registrar listener
emitter.on('dados', (payload) => {
  console.log('Recebido:', payload);
});

// Listener de uma só vez — remove a si mesmo após disparar
emitter.once('conectado', () => {
  console.log('Conexão estabelecida');
});

// Emitir — passa argumentos diretamente para os listeners
emitter.emit('dados', { id: 1, valor: 'abc' });
emitter.emit('conectado');
emitter.emit('conectado'); // listener once não dispara mais

// Remover listener específico
function handler(d) { console.log(d); }
emitter.on('dados', handler);
emitter.off('dados', handler);   // remove — precisa da mesma referência!

// Listar e remover todos os listeners de um evento
console.log(emitter.listenerCount('dados'));
emitter.removeAllListeners('dados');  // limpa evento específico
emitter.removeAllListeners();         // limpa TUDO — use com cuidado

# Herdando de EventEmitter

O padrão idiomático é criar classes que estendem EventEmitter — não instanciar a classe diretamente. Assim você encapsula estado e expõe uma API de eventos tipada:

fila.js
import { EventEmitter } from 'node:events';

class FilaDeTrabalho extends EventEmitter {
  constructor() {
    super();
    this.itens = [];
    this.processando = false;
  }

  adicionar(item) {
    this.itens.push(item);
    this.emit('item:adicionado', item);
    if (!this.processando) this._processar();
  }

  async _processar() {
    this.processando = true;
    while (this.itens.length > 0) {
      const item = this.itens.shift();
      try {
        await executarTrabalho(item);
        this.emit('item:concluido', item);
      } catch (err) {
        this.emit('item:erro', err, item);
      }
    }
    this.processando = false;
    this.emit('fila:vazia');
  }
}

// Uso
const fila = new FilaDeTrabalho();
fila.on('item:concluido', (item) => console.log('Feito:', item));
fila.on('item:erro', (err) => console.error('Erro:', err));
fila.on('fila:vazia', () => console.log('Tudo processado'));

# Memory leaks de listeners

O bug de EventEmitter mais comum em produção: registrar listeners dentro de um loop ou handler — sem nunca removê-los — até o processo travar com MaxListenersExceededWarning. Node avisa quando um evento tem mais de 10 listeners por padrão:

leak.js
// ❌ LEAK: registra novo listener a cada requisição
function handleRequest(req, res) {
  // cada chamada adiciona um listener novo — nunca removido
  process.on('SIGTERM', () => res.end('encerrando'));
}

// ✅ Registra UMA vez, fora do handler
const respostasAtivas = new Set();
process.on('SIGTERM', () => {
  for (const res of respostasAtivas) res.end('encerrando');
});

function handleRequest(req, res) {
  respostasAtivas.add(res);
  res.on('finish', () => respostasAtivas.delete(res));
}

// ❌ LEAK: arrow function anônima — não tem referência para off()
emitter.on('evento', () => { /* ... */ });
// emitter.off('evento', ???)  — impossível remover depois!

// ✅ Guarde a referência
const onEvento = () => { /* ... */ };
emitter.on('evento', onEvento);
// Quando não precisar mais:
emitter.off('evento', onEvento);
▸ Cuidado
MaxListenersExceededWarning é um aviso, não um erro — o processo continua. Por isso ele é silencioso em produção e causa lentidão crescente até o processo ficar inutilizável. Trate-o como bug crítico: ative monitoramento de process.on('warning', ...) e alerte imediatamente.

# O evento 'error'

O evento 'error' tem tratamento especial em todos os EventEmitters: se for emitido e não houver listener registrado, o Node lança uma exceção não capturada e o processo crasha. Sempre registre um handler de erro:

error-event.js
import { createReadStream } from 'node:fs';

const stream = createReadStream('nao-existe.txt');

// ❌ Sem handler de 'error' — crash garantido se o arquivo não existir
stream.on('data', chunk => console.log(chunk));

// ✅ Sempre adicione handler de 'error'
stream.on('data', chunk => console.log(chunk));
stream.on('error', err => console.error('Erro no stream:', err.message));

// Emitir um erro com contexto
class MinhaClasse extends EventEmitter {
  processar() {
    try {
      /* ... */
    } catch (err) {
      this.emit('error', err);  // idioma correto para propagar erros
    }
  }
}

# events.on — async iterable de eventos

A função events.on transforma qualquer evento de um EventEmitter em um async iterable — eliminando callbacks para consumo sequencial de eventos:

events-on.js
import { on, once } from 'node:events';
import { createInterface } from 'node:readline';

// once() — promisifica um evento de uma só vez
const server = createServer(handler);
server.listen(3000);
await once(server, 'listening');  // aguarda o servidor estar pronto
console.log('Servidor pronto');

// on() — async iterable para eventos recorrentes
const rl = createInterface({ input: process.stdin });
const linhas = on(rl, 'line');  // async iterable de linhas

for await (const [linha] of linhas) {
  // cada iteração espera o próximo evento 'line'
  console.log('Lido:', linha);
  if (linha === 'sair') break;
}
rl.close();
▸ Nota
events.on recebe os argumentos do evento como array — por isso o destructuring [linha] em vez de linha direto. Se o evento emite múltiplos argumentos (emit('msg', id, body)), você recebe [id, body] a cada iteração. Passe { signal } como segundo argumento para interromper o loop externamente.

async function lerJson(caminho) {
  try {
    const txt = await readFile(caminho, 'utf8');
    return JSON.parse(txt);
  } catch (err) {
    if (err.code === 'ENOENT') return null;     // arquivo não existe
    if (err instanceof SyntaxError) throw new Error(`JSON inválido em ${caminho}`);
    throw err;
  }
}

# Watching de arquivos

Node 22 estabilizou fs.watch com suporte a async iterator — a forma mais limpa de observar mudanças em arquivos e diretórios:

watch.js
import { watch } from 'node:fs/promises';

// Observar um arquivo ou diretório
const watcher = watch('./src', { recursive: true });

const controller = new AbortController();

try {
  for await (const evento of watcher) {
    console.log(evento.eventType);  // 'rename' | 'change'
    console.log(evento.filename);   // nome do arquivo alterado

    if (evento.filename === 'parar') controller.signal.abort();
  }
} catch (err) {
  if (err.name !== 'AbortError') throw err;
}

// Padrão comum: rebuildar quando arquivos mudam (tipo --watch do tsc)
async function devWatch(dir, callback) {
  for await (const { filename } of watch(dir, { recursive: true })) {
    console.log(`Mudança: ${filename}`);
    await callback(filename);
  }
}
▸ Nota

# Caminhos absolutos e portabilidade

O erro mais comum de portabilidade em Node é usar caminhos relativos que dependem do diretório de trabalho atual (process.cwd()) — que pode ser qualquer lugar, dependendo de como o processo foi iniciado. A regra: sempre construa caminhos absolutos a partir de uma âncora conhecida.

const config2 = await readFile(configPath, 'utf8');

// ✅ Variante para Node < 22 (sem import.meta.dirname)

const __dirname = dirname(fileURLToPath(import.meta.url));
const configPath2 = join(__dirname, 'config.json');

// path.join vs path.resolve
join('/a', 'b', 'c');          // '/a/b/c'  — concatena com separador do SO
resolve('b', 'c');             // '/cwd/b/c' — absoluto a partir de cwd()
resolve('/a', 'b');            // '/a/b'     — absoluto sobrescreve

// Utilitários de path
basename('/pasta/arquivo.js');  // 'arquivo.js'
basename('/pasta/arquivo.js', '.js'); // 'arquivo'
dirname('/pasta/arquivo.js');   // '/pasta'

Child Processes & Workers

Node é single-thread, mas o servidor real raramente é. Quando você precisa rodar um binário externo, isolar trabalho CPU-bound, ou usar todos os cores da máquina, o runtime oferece quatro caminhos crescentes em isolamento — de threads compartilhando memória até processos completamente independentes. Escolher o errado é fonte fácil de bugs sutis e perda de performance. Este capítulo mapeia os quatro, com critérios para escolher e exemplos idiomáticos.

# O espectro de isolamento

O Node oferece, em ordem crescente de isolamento e custo:

A regra de bolso: workers para JS, child processes para tudo o mais. Cluster é um caso especial de child processes para HTTP. Vamos do mais leve ao mais isolado.

# Worker Threads — JavaScript em outra thread

Worker Threads são threads reais — cada uma com seu próprio interpretador V8, seu próprio event loop, sua própria heap. Não compartilham variáveis JavaScript com a thread principal: a comunicação é por mensagens estruturadamente clonadas, exceto por dois mecanismos especiais que vamos ver adiante.

O caso típico: você tem uma função CPU-bound (parsing de um JSON gigante, hash de um arquivo, render de imagem) que travaria o event loop principal. Um worker resolve isso sem mexer na arquitetura do servidor:

main.js
import { Worker } from 'node:worker_threads';

function processarPesado(dados) {
  return new Promise((resolve, reject) => {
    const worker = new Worker(new URL('./worker.js', import.meta.url), {
      workerData: dados,  // payload inicial — clonado, não compartilhado
    });

    worker.on('message', resolve);
    worker.on('error',   reject);
    worker.on('exit', code => {
      if (code !== 0) reject(new Error(`Worker saiu com código ${code}`));
    });
  });
}

// Servidor HTTP segue respondendo enquanto o worker calcula
const resultado = await processarPesado(payloadGigante);
worker.js
import { workerData, parentPort } from 'node:worker_threads';

// Trabalho CPU-bound — cálculo síncrono pesado
function processar(dados) {
  let total = 0;
  for (const item of dados.itens) {
    total += algoritmoComplexo(item);
  }
  return total;
}

const resultado = processar(workerData);
parentPort.postMessage(resultado);  // envia de volta — também clonado

Como dados cruzam a fronteira

Quando você chama postMessage(obj), o objeto passa pelo structured clone — uma cópia profunda que suporta a maioria dos tipos JS (Date, Map, Set, RegExp, TypedArrays, Buffers). Funções, instâncias de classes customizadas e closures não atravessam — elas perdem o protótipo e viram POJOs.

Para grandes volumes de dados, clonar é caro. Há duas saídas:

  1. Transferência — mover a posse de um ArrayBuffer ou MessagePort para o outro lado. Sem cópia. Após transferir, o lado original perde acesso aos dados.
  2. SharedArrayBuffer — buffer realmente compartilhado entre threads. Acesso direto aos mesmos bytes. Requer cuidado com sincronização (via Atomics).
transferencia.js
// Transferência: zero-copy, mas a referência vira inválida no remetente
const buf = new ArrayBuffer(10 * 1024 * 1024);  // 10 MB
preencherDados(buf);

worker.postMessage({ buf }, [buf]);  // segundo arg = lista de transferíveis

console.log(buf.byteLength);  // 0 — buffer foi neutralizado neste lado

// SharedArrayBuffer: compartilhamento real, requer Atomics para coordenar
const shared = new SharedArrayBuffer(1024);
const view = new Int32Array(shared);

worker.postMessage({ shared });  // passa referência, não cópia

// Ambos os lados podem ler/escrever — mas precisam coordenar:
Atomics.store(view, 0, 42);
Atomics.notify(view, 0);  // acorda outras threads esperando neste índice
▸ Cuidado
Compartilhar memória entre threads traz de volta todos os problemas que Node evitou ao ser single-thread: race conditions, deadlocks, ordem de visibilidade. Use SharedArrayBuffer apenas quando profilou e mostrou que cópia é o gargalo. Para 99% dos casos, postMessage com clone resolve — e nunca trava.

Pool de workers

Criar um worker novo a cada tarefa custa caro — boot do V8, inicialização do módulo, ~30-100 ms. Para tarefas frequentes, mantenha um pool de workers vivos e mande mensagens:

pool.js (esqueleto)
import { Worker } from 'node:worker_threads';
import { availableParallelism } from 'node:os';

class WorkerPool {
  constructor(scriptUrl, tamanho = availableParallelism()) {
    this.workers = Array.from({ length: tamanho }, () => new Worker(scriptUrl));
    this.fila = [];
    this.disponiveis = [...this.workers];
  }

  executar(tarefa) {
    return new Promise((resolve, reject) => {
      const trabalho = { tarefa, resolve, reject };
      if (this.disponiveis.length) this._dispatch(trabalho);
      else this.fila.push(trabalho);
    });
  }

  _dispatch({ tarefa, resolve, reject }) {
    const worker = this.disponiveis.pop();
    const onMessage = (resp) => { cleanup(); resolve(resp); };
    const onError   = (err)  => { cleanup(); reject(err); };
    const cleanup   = () => {
      worker.off('message', onMessage);
      worker.off('error', onError);
      this.disponiveis.push(worker);
      if (this.fila.length) this._dispatch(this.fila.shift());
    };
    worker.once('message', onMessage);
    worker.once('error', onError);
    worker.postMessage(tarefa);
  }
}
▸ Performance
Bibliotecas como piscina (do mesmo autor do Fastify) fornecem implementações de pool prontas, com filas, timeouts, abort, e idle-cleanup. Em produção, prefira piscina a reimplementar pool. A versão acima é didática — entenda o conceito; use a biblioteca.

# child_process.fork — outro processo Node

fork cria um processo Node novo (não thread) já com canal IPC bidirecional pronto. É o jeito tradicional de dividir trabalho Node entre processos — útil quando o isolamento de processo é desejável (crash não derruba o pai) ou quando você precisa de várias instâncias com seu próprio espaço de memória.

main-fork.js
import { fork } from 'node:child_process';

const filho = fork(new URL('./filho.js', import.meta.url));

// IPC pré-configurado: send/on('message') dos dois lados
filho.send({ tipo: 'tarefa', payload: 'olá' });

filho.on('message', (msg) => {
  console.log('Filho respondeu:', msg);
});

filho.on('exit', (codigo, sinal) => {
  console.log(`Filho saiu — código: ${codigo}, sinal: ${sinal}`);
});
filho.js
process.on('message', (msg) => {
  if (msg.tipo === 'tarefa') {
    const resultado = processar(msg.payload);
    process.send({ tipo: 'resposta', resultado });

O custo: cada fork é um processo Node novo — boot inteiro do V8, ~50-150 MB de RSS. Para tarefas frequentes e pequenas, worker threads é mais eficiente. Use fork quando quer isolamento de processo (ex: cada filho carrega configuração diferente, ou pode crashar sem afetar o pai).

# cluster — vários cores em um servidor

O módulo cluster é uma camada sobre fork que automatiza um padrão comum: um processo "master" gera N processos "worker" iguais, todos escutando na mesma porta. O kernel distribui conexões entre eles via balanceamento round-robin (ou similar).

cluster.js
import cluster from 'node:cluster';
import { availableParallelism } from 'node:os';
import { createServer } from 'node:http';

if (cluster.isPrimary) {
  const n = availableParallelism();
  console.log(`Master ${process.pid} — gerando ${n} workers`);
  for (let i = 0; i < n; i++) cluster.fork();

  // Resiliência: respawn se um worker morrer inesperadamente
  cluster.on('exit', (worker, code, signal) => {
    console.warn(`Worker ${worker.process.pid} morreu, recriando...`);
    cluster.fork();
  });
} else {
  // Cada worker abre o servidor; o kernel balanceia conexões
  createServer((req, res) => {
    res.end(`PID ${process.pid}\n`);
  }).listen(3000);
  console.log(`Worker ${process.pid} escutando`);
}
▸ Nota
Em deployments modernos com containers (Docker, Kubernetes), prefira rodar uma instância Node por container e replicar containers — o orquestrador faz o balanceamento. cluster faz mais sentido para deploys em VM única, ou quando você quer aproveitar todos os cores de uma máquina sem orquestrador. PM2 e similares também usam cluster por baixo.

# MessageChannel: comunicação direta entre workers

Por padrão, workers só conversam com a thread principal. Para comunicação direta entre dois workers (sem intermediário), use MessageChannel: um par de portas port1/port2 que você transfere para os workers:

const w1 = new Worker('./w.js');
const w2 = new Worker('./w.js');

const { port1, port2 } = new MessageChannel();

# http.createServer — servidor HTTP nativo

Em produção quase sempre se usa um framework (Hono, Fastify, Express), mas entender o servidor HTTP nativo é fundamental — frameworks são wrappers finos sobre ele:

servidor.js
import { createServer } from 'node:http';

const server = createServer(async (req, res) => {
  // req é IncomingMessage (Readable) — res é ServerResponse (Writable)

  // Roteamento básico
  const url = new URL(req.url, `http://${req.headers.host}`);

  if (req.method === 'GET' && url.pathname === '/health') {
    res.writeHead(200, { 'Content-Type': 'application/json' });
    res.end(JSON.stringify({ ok: true, uptime: process.uptime() }));
    return;
  }

  if (req.method === 'POST' && url.pathname === '/echo') {
    // Ler body: coletar chunks e concatenar
    const chunks = [];
    for await (const chunk of req) chunks.push(chunk);
    const body = Buffer.concat(chunks).toString('utf8');

    res.writeHead(200, { 'Content-Type': 'application/json' });
    res.end(JSON.stringify({ recebido: body }));
    return;
  }

  // 404 genérico
  res.writeHead(404);
  res.end('Not found');
});

// Graceful shutdown
server.keepAliveTimeout = 65_000;  // maior que o LB (ex: 60s do ALB da AWS)
server.headersTimeout  = 66_000;  // maior que keepAliveTimeout

server.listen(3000, () => console.log('http://localhost:3000'));
▸ Nota
O handler de createServer pode ser async, mas o Node não captura rejeições automaticamente nele. Se a função async lançar, a Promise rejeitada some silenciosamente e a conexão fica pendurada até o timeout. Sempre envolva a lógica em try/catch e responda com res.writeHead(500) no catch.

Lendo o body com limite de tamanho

body-parser.js
// Utilitário reutilizável — lê body com limite máximo
async function lerBody(req, limiteBytes = 1024 * 1024) {
  let tamanho = 0;
  const chunks = [];
  for await (const chunk of req) {
    tamanho += chunk.length;
    if (tamanho > limiteBytes) {
      req.destroy();
      throw new Error('Payload muito grande');
    }
    chunks.push(chunk);
  }
  return Buffer.concat(chunks);
}

// Parsear JSON do body
async function lerJson(req) {
  const raw = await lerBody(req);
  try {
    return JSON.parse(raw.toString('utf8'));
  } catch {
    throw new Error('JSON inválido');
  }
}

# WebSocket com ws

Node não tem WebSocket server nativo — use o pacote ws, que é o padrão da comunidade (leve, sem dependências, alta performance):

websocket-server.js
import { WebSocketServer } from 'ws';
import { createServer } from 'node:http';

const httpServer = createServer(); // HTTP e WS no mesmo servidor
const wss = new WebSocketServer({ server: httpServer });

// Map de clientes para broadcast
const clientes = new Map();

wss.on('connection', (ws, req) => {
  const id = crypto.randomUUID();
  clientes.set(id, ws);
  console.log(`Cliente ${id} conectou (total: ${clientes.size})`);

  ws.on('message', (data, isBinary) => {
    const msg = isBinary ? data : data.toString('utf8');
    // Broadcast para todos exceto o remetente
    for (const [cid, client] of clientes) {
      if (cid !== id && client.readyState === ws.OPEN) {
        client.send(msg);
      }
    }
  });

  ws.on('close', () => {
    clientes.delete(id);
    console.log(`Cliente ${id} desconectou`);
  });

  ws.on('error', (err) => console.error(`WS ${id} erro:`, err.message));

  ws.send(JSON.stringify({ tipo: 'boas-vindas', id }));
});

httpServer.listen(3000);

# Server-Sent Events (SSE)

SSE é uma alternativa simples a WebSocket para fluxo unidirecional (servidor → cliente) — chat de IA em streaming, notificações, dashboards em tempo real. Funciona sobre HTTP puro, sem biblioteca extra:

sse.js
import { createServer } from 'node:http';

createServer(async (req, res) => {
  if (req.url !== '/eventos') { res.writeHead(404); res.end(); return; }

  // Headers obrigatórios para SSE
  res.writeHead(200, {
    'Content-Type':  'text/event-stream',
    'Cache-Control': 'no-cache',
    'Connection':    'keep-alive',
    'X-Accel-Buffering': 'no',  // desabilita buffer do nginx
  });

  // Helper: envia um evento SSE
  const enviar = (evento, dados) => {
    res.write(`event: ${evento}\ndata: ${JSON.stringify(dados)}\n\n`);
  };

  // Tick a cada segundo
  const timer = setInterval(() => enviar('tick', { ts: Date.now() }), 1000);

  // Limpa quando cliente desconecta
  req.on('close', () => clearInterval(timer));
}).listen(3000);

TypeScript em Node.js

TypeScript e Node.js se complementam bem — mas a configuração inicial tem armadilhas reais: resolução de módulos, extensões de import, decorators, geração de source maps, e a escolha entre runners de desenvolvimento. # Por que a configuração importa

O TypeScript tem configurações defaults pensadas para o navegador e para CommonJS — não para ESM com Node moderno. Usar o tsconfig.json errado gera erros sutis: import de arquivo sem extensão que funciona no TypeScript mas quebra no Node, await no top-level que não compila, paths que não resolvem em runtime. A configuração abaixo foi ajustada para Node 22 com ESM:

    "lib": ["ES2023"],

Erros & Debugging

Erros em código assíncrono são uma das partes que mais separam o Node de outras runtimes — e onde mais tempo se perde quando a base teórica está vaga. Este capítulo constrói um modelo mental do que é uma stack trace, por que ela é frequentemente "mentirosa" em código async, como criar hierarquias de erros que carregam contexto, e como usar o Inspector e source maps para depurar TypeScript em produção sem cair em becos sem saída.

# O que é uma stack trace, mecanicamente

Uma stack trace não é uma "história do que aconteceu" — é uma fotografia da pilha de chamadas no momento em que o objeto Error foi criado. Cada quadro corresponde a uma função que ainda não retornou. Quando o controle flui para uma callback assíncrona, a pilha que existia antes do await já desapareceu: o JavaScript voltou ao event loop, despachou outras coisas, e só agora retomou.

Isso significa que, sem ajuda, o erro lançado dentro de uma callback de I/O carrega um stack trace que começa em algum lugar arbitrário do event loop — sem nenhuma menção à função que originalmente enfileirou aquela operação. É confuso, e foi por décadas o pesadelo de quem depurava Node.

stack-truncada.js
// Estilo callback antigo — stack trace é truncada onde ocorre o erro,
// SEM mencionar quem chamou lerArquivo
import { readFile } from 'node:fs';

function lerArquivo(caminho) {
  readFile(caminho, (err, dados) => {
    if (err) throw err;  // stack começa aqui, não em lerArquivo!
  });
}

lerArquivo('./inexistente.txt');
// Saída: ENOENT em onread (anônimo) — <event loop interno> — fim

# Async stack traces

O V8 moderno (e portanto Node) implementa async stack traces: para Promises e async/await, o motor mantém uma referência ao quadro que iniciou a operação assíncrona e o anexa à pilha do erro. O resultado é uma trace que atravessa o limite assíncrono naturalmente:

stack-async.js
import { readFile } from 'node:fs/promises';

async function lerArquivo(caminho) {
  return await readFile(caminho, 'utf8');
}

async function main() {
  await lerArquivo('./inexistente.txt');
}

main();
// Error: ENOENT: no such file or directory ...
//     at async open (node:internal/fs/promises:639)
//     at async readFile (node:internal/fs/promises:1227)
//     at async lerArquivo (file:///...:5)   ← preservado!
//     at async main (file:///...:9)         ← também!

Duas regras práticas decorrem disso:

  1. Sempre prefira fs/promises ao fs de callback — você ganha stack traces utilizáveis de graça.
  2. Use return await quando a função puder lançar dentro de um try/catch. Sem o await, a função retorna a Promise e seu próprio quadro some antes do erro ser observado, deixando a trace mais curta.
return-await.js
// ❌ stack vai pular o quadro de buscarUsuario — return sem await
async function buscarUsuario(id) {
  try {
    return db.query('SELECT * FROM users WHERE id = ?', id);
    // ↑ se não há await, o catch também não captura a rejeição
  } catch (err) { /* não chega aqui! */ }
}

// ✅ stack mantém quadro de buscarUsuario; catch funciona
async function buscarUsuario(id) {
  try {
    return await db.query('SELECT * FROM users WHERE id = ?', id);
  } catch (err) {
    throw new DatabaseError('falha ao buscar usuário', { cause: err });
  }
}

# Hierarquias de Error customizadas

Em produção, lançar throw new Error('algo deu errado') joga fora informação valiosa. Erros bem desenhados carregam tipo (para que o chamador escolha como reagir), contexto (dados do que falhou) e causa (o erro original que provocou):

errors.ts
// Classe base — todo erro de domínio herda dela
export class AppError extends Error {
  code: string;
  status: number;
  context: Record<string, unknown>;

  constructor(
    message: string,
    options: { code: string; status?: number; context?: Record<string, unknown>; cause?: unknown }
  ) {
    super(message, { cause: options.cause });
    this.name = this.constructor.name;  // 'NotFoundError', etc.
    this.code = options.code;
    this.status = options.status ?? 500;
    this.context = options.context ?? {};
  }
}

// Subclasses — comunicam intenção sem usar strings mágicas
export class NotFoundError extends AppError {
  constructor(recurso: string, id: string) {
    super(`${recurso} não encontrado: ${id}`, {
      code: 'NOT_FOUND',
      status: 404,
      context: { recurso, id },
    });
  }
}

export class ValidationError extends AppError {
  constructor(campo: string, motivo: string) {
    super(`Validação falhou em ${campo}: ${motivo}`, {
      code: 'VALIDATION',
      status: 400,
      context: { campo, motivo },
    });
  }
}

export class DatabaseError extends AppError {
  constructor(message: string, opts: { cause?: unknown } = {}) {
    super(message, { code: 'DATABASE', status: 503, cause: opts.cause });
  }
}
uso.ts
async function buscarUsuario(id: string) {
  let rows;
  try {
    rows = await db.query('SELECT * FROM users WHERE id = ?', id);
  } catch (err) {
    // Embrulha erro de baixo nível com cause — preserva a stack original
    throw new DatabaseError('falha consultando users', { cause: err });
  }
  if (rows.length === 0) throw new NotFoundError('Usuário', id);
  return rows[0];
}

// No handler HTTP, decida a resposta com base no tipo
try {
  const u = await buscarUsuario(req.params.id);
  res.json(u);
} catch (err) {
  if (err instanceof AppError) {
    res.status(err.status).json({ code: err.code, message: err.message });
  } else {
    res.status(500).json({ code: 'INTERNAL' });
    log('erro inesperado', { err });
  }
}
▸ Nota
A opção { cause } do construtor Error existe desde Node 16 e é ignorada com frequência. Ela é o jeito padrão de encadear erros — preserva a stack trace original e é serializada por loggers como o Pino. Sempre que reembrulhar um erro, passe o original como cause em vez de stringificar a mensagem.

# Erros operacionais vs erros de programação

Esta distinção, popularizada por Joyent há mais de uma década, ainda é o melhor critério para decidir o que fazer com um erro:

  • Operacionais — falhas esperadas de I/O ou input: timeout de rede, arquivo não encontrado, validação inválida. Devem ser tratadas — propagadas até o handler que sabe traduzir em resposta HTTP, retentar, ou degradar graciosamente.
  • De programação — bugs: undefined usado como objeto, asserções violadas, contratos quebrados. Não tente "tratar" — encerre o processo. Continuar com estado potencialmente corrompido é pior do que reiniciar.
handlers-globais.js
// Último recurso: erros que escaparam de todos os handlers locais
process.on('uncaughtException', (err, origin) => {
  log.fatal({ err, origin }, 'exception não capturada');
  // Bug — não tente continuar. Deixe o orquestrador (PM2, K8s) reiniciar.
  process.exit(1);
});

process.on('unhandledRejection', (reason, promise) => {
  log.fatal({ reason, promise }, 'Promise rejeitada sem handler');
  process.exit(1);
});

// process.on('warning') — capture os "soft errors" do Node, como
// MaxListenersExceededWarning, deprecation warnings, etc.
process.on('warning', (warning) => {
  log.warn({ name: warning.name, message: warning.message, stack: warning.stack });
});

Logging Estruturado

Em produção, logs são sua janela para o que está acontecendo. console.log não escala — mistura dados com texto, impossibilita buscas eficientes, e não carrega contexto. Logs estruturados em JSON são indexáveis, pesquisáveis e processáveis por qualquer ferramenta de observabilidade. Este capítulo cobre Pino (o padrão de alta performance), níveis, contexto, correlation IDs e os patterns que fazem logs úteis de verdade.

# Por que não console.log

Três limitações tornam console.log inadequado para produção:

  • Sem estrutura — texto livre é difícil de parsear, impossível de agrupar por campo.
  • Sem níveis — não há como filtrar só erros, ou ligar logs de debug temporariamente.
  • Sem contexto automático — cada log é uma ilha; correlacionar logs de uma mesma requisição exige trabalho manual.

Logs em JSON resolvem os três: ferramentas como Datadog, Grafana Loki, Elasticsearch e CloudWatch indexam campos diretamente, e você filtra por level=error, requestId=abc-123 ou userId=42 em segundos.

# Pino — alta performance por padrão

Pino é o logger mais rápido do ecossistema Node — mais rápido que Winston em 5-8x nos benchmarks — porque serializa JSON de forma extremamente eficiente e, por padrão, delega a formatação "bonita" a um processo separado. Em produção, escreve JSON puro para stdout:

terminal
$ pnpm add pino
$ pnpm add -D pino-pretty  # apenas para dev
src/logger.ts
import pino from 'pino';
import { env } from './env.js';

export const logger = pino({
  // Nível mínimo — abaixo disso, ignorado completamente
  level: env.LOG_LEVEL,  // 'trace' | 'debug' | 'info' | 'warn' | 'error' | 'fatal'

  // Em produção: JSON puro e eficiente
  // Em dev: pretty-print colorido via pino-pretty
  transport: env.NODE_ENV !== 'production'
    ? { target: 'pino-pretty', options: { colorize: true, translateTime: 'SYS:standard' } }
    : undefined,

  // Campos base que aparecem em TODOS os logs
  base: {
    pid: process.pid,
    app: 'minha-api',
    version: process.env.npm_package_version,
  },

  // Serializers: controla como objetos específicos são convertidos
  serializers: {
    err: pino.stdSerializers.err,     // serializa Error com stack, message, etc.
    req: pino.stdSerializers.req,     // serializa req HTTP (sem headers sensíveis)
    res: pino.stdSerializers.res,     // serializa res HTTP
  },

  // Redactar campos sensíveis — nunca logar senhas ou tokens
  redact: {
    paths: ['req.headers.authorization', '*.password', '*.token', '*.secret'],
    censor: '[REDACTED]',

# Child loggers e contexto

Child loggers herdam configuração e campos do pai — sem repetir contexto em cada chamada. O pattern clássico: um child por requisição HTTP com o requestId incluído automaticamente em todos os logs daquela requisição:

child-logger.ts
import { logger } from './logger.js';

// Child com contexto fixo — todos os logs herdados incluem esses campos
const dbLogger = logger.child({ component: 'database' });
dbLogger.info('conectado');
// → { "level":30, "component":"database", "msg":"conectado", ... }

// Child por requisição — cria na entrada do handler
function criarLoggerDeReq(req: IncomingMessage) {
  const requestId =
    (req.headers['x-request-id'] as string) ?? crypto.randomUUID();
  return logger.child({
    requestId,
    method: req.method,
    url: req.url,
  });
}

// Todos os logs do handler carregam requestId automaticamente
function handler(req, res) {
  const log = criarLoggerDeReq(req);
  log.info('requisição recebida');

  const usuario = await buscarUsuario(req, log);
  log.info({ userId: usuario.id }, 'usuário autenticado');
}

# Correlation IDs

Em arquitetura de microsserviços, uma operação cruza vários serviços. Para rastrear o caminho completo, propague um correlation ID em todos os headers e logs. A convenção mais comum é X-Request-ID ou X-Trace-ID:

correlation.ts
import { AsyncLocalStorage } from 'node:async_hooks';
import { logger } from './logger.js';

// AsyncLocalStorage: contexto que flui por toda a cadeia async de uma requisição
// sem precisar passar o log por parâmetro em cada função
const als = new AsyncLocalStorage<{ log: typeof logger; requestId: string }>();

export function getLogger() {
  return als.getStore()?.log ?? logger;
}

export function getRequestId() {
  return als.getStore()?.requestId;
}

// Middleware: inicializa o contexto no começo da requisição
export function correlationMiddleware(req, res, next) {
  const requestId =
    req.headers['x-request-id'] ?? crypto.randomUUID();

  res.setHeader('X-Request-ID', requestId);  // devolve para o cliente

  const log = logger.child({ requestId });
  als.run({ log, requestId }, next);           // tudo chamado dentro de next() vê o contexto
}

// Em qualquer função, sem precisar de parâmetro:
async function buscarUsuario(id: string) {
  const log = getLogger();                    // pega o child da requisição atual
  log.debug({ userId: id }, 'buscando usuário'); // requestId incluído automaticamente
  return await db.query('SELECT * FROM users WHERE id = ?', id);
}

// Propagar em chamadas a outros serviços
async function chamarServico(url: string, body: unknown) {
  return fetch(url, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'X-Request-ID': getRequestId() ?? '',  // propaga para o próximo serviço
    },
    body: JSON.stringify(body),
  });
}
▸ Nota
AsyncLocalStorage é a API Node para contexto de execução assíncrona — funciona como um "thread-local" para código async. Cada chamada a als.run(store, fn) cria um contexto isolado visível por toda a cadeia async descendente, sem interferir com outras requisições. Frameworks como Fastify, NestJS e Hono expõem mecanismos semelhantes por baixo.

# Validação com Zod

Carregar process.env diretamente espalha string | undefined por todo o código. A solução é um módulo central que valida e exporta a configuração com tipos corretos — falha rápido na inicialização se algo estiver errado:

src/env.ts
import { z } from 'zod';

const schema = z.object({
  // Servidor
  NODE_ENV:  z.enum(['development', 'test', 'production']).default('development'),
  PORT:      z.coerce.number().int().min(1).max(65535).default(3000),
  HOST:      z.string().default('0.0.0.0'),

  // Banco de dados
  DATABASE_URL: z.string().url(),
  DB_POOL_MIN:  z.coerce.number().default(2),
  DB_POOL_MAX:  z.coerce.number().default(10),

  // Auth
  JWT_SECRET:  z.string().min(32, 'JWT_SECRET precisa ter ao menos 32 chars'),
  JWT_EXPIRES: z.string().default('7d'),

  // Logging
  LOG_LEVEL: z.enum(['trace', 'debug', 'info', 'warn', 'error', 'fatal']).default('info'),

  // Opcional — ausente é ok
  SENTRY_DSN:  z.string().url().optional(),
  REDIS_URL:   z.string().optional(),
});

const resultado = schema.safeParse(process.env);

if (!resultado.success) {
  const erros = resultado.error.flatten().fieldErrors;
  console.error('\n❌ Variáveis de ambiente inválidas ou ausentes:\n');
  for (const [campo, msgs] of Object.entries(erros)) {
    console.error(`  ${campo}: ${msgs?.join(', ')}`);
  }
  console.error('\nVerifique seu arquivo .env\n');
  process.exit(1);
}

export const env = resultado.data;
export type Env = typeof env;
uso em qualquer módulo

  const v = process.env[nome];

function requerInt(nome: string, padrao?: number): number {
  const v = process.env[nome];

  const n = Number(v);
  if (!v || Number.isNaN(n)) throw new Error(`${nome} deve ser um inteiro`);
  return n;
}

Performance & Profiling

Otimizar Node sem profiling é supersticioso — você ajusta coisas que parecem lentas e ignora os 80% reais do tempo. Este capítulo apresenta uma metodologia de diagnóstico que começa identificando o tipo de gargalo (CPU, event loop, memória, I/O) e só então escolhe a ferramenta certa: perf_hooks para medições in-process, --prof e --cpu-prof do V8 para flamegraphs, clinic.js para diagnóstico amplo. Ao final, você sabe ler um flamegraph sem se enganar.

# Os quatro tipos de gargalo

Antes de medir, decida qual problema você está caçando. Cada tipo tem um sintoma característico e uma ferramenta diferente:

GargaloSintomaFerramenta
CPULatência alta sob carga; %CPU próximo a 100% em um coreCPU profiler (--cpu-prof, clinic flame)
Event loopLatência alta sem CPU saturado; p99 do loop > 50msmonitorEventLoopDelay, clinic doctor
MemóriaRSS cresce sem voltar; OOM kills periódicosHeap snapshots, --heap-prof

A metodologia é sempre a mesma: medir antes de otimizar. Sem dados, otimizações são chutes — frequentemente em pontos errados.

# perf_hooks — medindo in-process

O módulo node:perf_hooks é o ponto de entrada para medições nativas: tempo entre marcos, histograma do event loop, observação do GC. Funciona dentro do próprio processo, sem ferramentas externas:

marks.js
import { performance } from 'node:perf_hooks';

// Medição simples — diferença entre dois momentos
const inicio = performance.now();
await trabalho();
const ms = performance.now() - inicio;
console.log(`Trabalho: ${ms.toFixed(2)}ms`);

// Marks + measures — para múltiplos pontos com nomes
performance.mark('parse-start');
const obj = JSON.parse(textoEnorme);
performance.mark('parse-end');

performance.mark('transform-start');
const resultado = transformar(obj);
performance.mark('transform-end');

const parse = performance.measure('parse', 'parse-start', 'parse-end');
const transform = performance.measure('transform', 'transform-start', 'transform-end');
console.log({ parse: parse.duration, transform: transform.duration });

# CPU profiling — flamegraphs

Quando a CPU está saturada e você não sabe onde, um CPU profiler responde a pergunta: durante 30 segundos de execução, em qual função o processo passou mais tempo? O resultado é normalmente visualizado como flamegraph:

terminal — V8 CPU profiler nativo
# Gera arquivo .cpuprofile compatível com Chrome DevTools
$ node --cpu-prof src/index.js
# Após encerrar, gera CPU.YYYY...PID.cpuprofile

# Diretório customizado e nome controlado
$ node --cpu-prof --cpu-prof-dir=./profiles --cpu-prof-name=run.cpuprofile src/index.js

# Profile detalhado (intervalo menor entre samples = mais resolução, mais overhead)
$ node --cpu-prof --cpu-prof-interval=100 src/index.js

Para abrir: Chrome DevTools → Performance → "Load profile" → arquivo .cpuprofile. Você vê uma visão "bottom-up" (quais funções consumiram mais self-time) e um flamegraph (a árvore de chamadas).

Como ler um flamegraph

Um flamegraph mostra a pilha de chamadas durante o profiling:

  • Eixo Y — profundidade da pilha. Cada função fica acima da que a chamou. Não há ordem temporal aqui.
  • Eixo X — tempo total acumulado em cada função (não tempo absoluto). Larguras representam % do tempo amostrado.
  • Cores — convenção visual; geralmente vermelho/quente para JS, azul para C++, amarelo para internos do Node.
  • Plateaus largos no topo são onde a CPU passou tempo de fato. Procure-os primeiro — são os candidatos a otimização.
  • Pirâmides altas e estreitas indicam recursão profunda mas barata; geralmente não são o problema.
▸ Cuidado
Largura no flamegraph é tempo acumulado de CPU, não tempo total. Uma função que aguarda I/O pode ter latência alta e largura zero — o profile do V8 não vê I/O, vê só CPU. Para investigar latência de chamadas a banco ou rede, use tracing (OpenTelemetry) — não CPU profile.

# Memory profiling: heap snapshots

Para detectar vazamentos de memória, snapshots do heap são a ferramenta principal. O fluxo: tirar dois snapshots em momentos distintos e comparar — objetos que cresceram entre eles são candidatos:

import { writeHeapSnapshot } from 'node:v8';
import { resolve } from 'node:path';

// Tira snapshot programaticamente — útil em rota administrativa
function tirarSnapshot() {
  const arquivo = resolve(`./heap-${Date.now()}.heapsnapshot`);
  writeHeapSnapshot(arquivo);
  return arquivo;
}

# Otimizações que costumam valer a pena

Em ordem de impacto típico em servidores Node de produção:

  1. Tirar trabalho síncrono do hot path — JSON.parse de payloads grandes, readFileSync, criptografia síncrona em handler. Mover para Worker Threads ou caching.
  2. Evitar Promise.all com arrays gigantes — disparar 10 mil queries simultâneas sobrecarrega o pool. Use pool com limite de concorrência (cap. 5).
  3. Streaming em vez de buffer — respostas HTTP grandes, leituras de arquivo. Memória constante, latência percebida menor.
  4. Pool de conexões adequado — undici Agent para fetch, pool do Postgres dimensionado conforme cargas. Sem isso, gargalo invisível.
  5. Caching — em memória (LRU) para hot data, Redis para cache compartilhado. Frequentemente é a maior ganhadora.
  6. Trocar JSON parse pesado por streaming parser — stream-json ou similar — quando payloads passam de poucos MB.
  7. Pre-compilar regex e templates — fora do hot path; caro montar a cada chamada.
  8. Reduzir alocações em hot path — reutilizar Buffers, evitar object spread em loops apertados.
▸ Performance
A heurística que mais economiza tempo: o p99 ruim quase nunca é "JavaScript lento" — é I/O sem timeout (banco, fetch externo) ou trabalho síncrono no event loop. Procure essas duas coisas antes de otimizar algoritmos. Tracing (OpenTelemetry) vai mostrar o I/O; monitorEventLoopDelay vai mostrar o segundo. Otimizar JS puro é quase sempre o último degrau, não o primeiro.

Memória & CPU

Em Node, memória é uma das poucas coisas que você precisa entender mecanicamente para não se enganar. O GC do V8 não é uma caixa preta: ele é geracional, tem tradeoffs claros entre throughput e latência, e responde a padrões de alocação que seu código produz. Este capítulo explica o modelo, identifica os quatro tipos de memory leak mais comuns em Node, mostra como diagnosticá-los, e cobre as flags de produção que importam de verdade.

# O modelo de memória do V8

Quando seu processo Node aloca um objeto JavaScript, ele vai para o heap do V8 — uma região de memória gerenciada pelo motor. O heap não é um espaço único; é dividido em gerações baseadas em uma observação empírica: a maioria dos objetos morre jovem. Coletar o lixo separadamente para cada geração é dramaticamente mais barato do que escanear o heap inteiro toda vez.

a hierarquia do heap V8
┌────────────────────────────────────────────────────────────────────┐
│                          Heap V8                                   │
│                                                                    │
│  ┌──────────────────────┐   ┌────────────────────────────────────┐ │
│  │ Young generation     │   │ Old generation                     │ │
│  │   (~8-32 MB)         │   │   (centenas de MB a vários GB)     │ │
│  │                      │   │                                    │ │
│  │  Eden + 2 semispaces │   │  Old object space                  │ │
│  │  → Scavenger         │   │  Large object space (> 512KB)      │ │
│  │  Coleta a cada       │   │  Code space (JIT)                  │ │
│  │  ~100ms-1s, <1ms     │   │  → Mark-Sweep-Compact              │ │
│  │                      │   │  Coleta esporádica, dezenas-       │ │
│  │                      │   │  centenas de ms                    │ │
│  └──────────────────────┘   └────────────────────────────────────┘ │
│                                                                    │
│  Promovido: objeto sobreviveu a 2 GCs jovens → migra p/ old        │
└────────────────────────────────────────────────────────────────────┘

Fora do heap V8 (mas conta para RSS):
  • Buffers e ArrayBuffers grandes (alocação fora do heap)
  • Pilha de cada thread
  • Memória de bibliotecas nativas

Os dois algoritmos do GC têm perfis muito diferentes — entender quando cada um roda é a chave para diagnosticar problemas:

  • Scavenger (young generation) — copia objetos vivos entre dois semispaces. Rápido (sub-milissegundo), mas roda com frequência. A grande maioria dos objetos morre aqui sem nunca ser visitada de novo.
  • Mark-Sweep-Compact (old generation) — varre o heap velho, marca objetos alcançáveis a partir das raízes (variáveis globais, pilha, registradores), libera o resto. Eventualmente compacta para reduzir fragmentação. Custa dezenas a centenas de ms — é o "GC pause" que você sente.

O V8 moderno faz parte do mark-sweep incrementalmente e concorrente com o JS, justamente para reduzir pausas. Mesmo assim, em servidores com muita memória ativa, GCs do old podem causar latência visível — e é isso que você vai querer monitorar.

# Limites de heap

Por padrão, o V8 limita o heap (old generation) — historicamente em 1.5-4GB dependendo da versão e arquitetura. Ao processar dados grandes ou rodar workloads pesados, você precisa explicitar:

terminal — flags de heap
# Aumentar heap velho para 4GB
$ node --max-old-space-size=4096 src/index.js

# Idem para young generation (raramente necessário)
$ node --max-semi-space-size=64 src/index.js

# Em containers: ajuste com base na memória disponível
# NUNCA configure mais do que o container tem — provoca OOM kill do orquestrador
# Recomendação: max-old-space-size = 75% da memória do container
$ NODE_OPTIONS='--max-old-space-size=3072' node dist/index.js  # para container 4GB
▸ Cuidado
No Node 22, o V8 detecta automaticamente o limite do container (cgroup) e ajusta o heap default — mas só para containers com limite explícito. Em hospedagem que não impõe limite, o default ainda é o histórico (~2GB em x64). Sempre defina --max-old-space-size em produção: previne OOMs do orquestrador e deixa o comportamento de memória previsível.

# RSS, heap e o que cada um significa

process.memoryUsage() retorna várias métricas — entender as diferenças evita falsos alarmes:

memoria.js
const mb = process.memoryUsage();

console.log({
  rss:        mb.rss,         // total de RAM física do processo
  heapUsed:   mb.heapUsed,    // objetos JS vivos no heap V8
  heapTotal:  mb.heapTotal,   // heap V8 alocado (≥ heapUsed)
  external:   mb.external,    // memória de C++ vinculada a JS (Buffers)
  arrayBuffers: mb.arrayBuffers, // subset de external — só ArrayBuffers
});
MétricaO que éQuando preocupa
rssResident Set Size — total na RAM físicaCresce sem teto → leak ou config insuficiente
heapUsedObjetos JS realmente vivosCresce e nunca cai após GC → leak de objetos JS
heapTotalHeap V8 reservadoCresce e fica próximo do --max-old-space-size → vai dar OOM
externalBuffers, certificados, dados nativosCresce sem cair → leak de Buffers (não vê em snapshot do V8)

Diferença prática: heapUsed alto mas estável é normal — significa que sua aplicação tem cache vivo, não que há leak. Leak é quando heapUsed cresce após cada GC do old. Para reconhecer isso, observe a métrica ao longo do tempo, não em pontos isolados.

# Os quatro padrões de memory leak em Node

1. Cache sem limite

leak-cache.js
// ❌ Map cresce para sempre — primeira causa de leak em Node
const cache = new Map();
function buscar(id) {
  if (cache.has(id)) return cache.get(id);
  const dados = consultarBanco(id);
  cache.set(id, dados);  // nunca evicta — leak garantido
  return dados;
}

// ✅ LRU com limite — nunca passa de N entradas
import { LRUCache } from 'lru-cache';
const cache2 = new LRUCache({ max: 10_000, ttl: 1000 * 60 * 5 });

// ✅ Para chave-valor onde você não sabe quando dispensar:
// WeakMap — chave é referência fraca, GC pode coletar quando não há outras refs
const meta = new WeakMap();
meta.set(usuario, { ultimoLogin: Date.now() });
// quando o usuario for coletado, sua entrada na meta some automaticamente

2. Listeners acumulados

leak-listeners.js
// ❌ Listener adicionado dentro de handler — cresce a cada requisição
function handler(req, res) {
  process.on('SIGTERM', () => res.end('tchau'));  // retém res, gera warning
}

// ✅ Registra UMA vez no escopo correto, mantém Set de respostas ativas
const ativas = new Set();
process.on('SIGTERM', () => { for (const r of ativas) r.end('tchau'); });

function handler2(req, res) {
  ativas.add(res);
  res.on('finish', () => ativas.delete(res));  // LIBERA ao concluir
}

3. Closures retendo grandes objetos

leak-closure.js
// ❌ Closure retém o "dadosEnormes" inteiro — mesmo só usando uma propriedade
function criarHandler(dadosEnormes) {
  return () => dadosEnormes.id;  // 50MB do dadosEnormes ficam vivos!
}

// ✅ Extrai apenas o que precisa antes de fechar a closure
function criarHandler(dadosEnormes) {
  const id = dadosEnormes.id;
  return () => id;
}

// ❌ Variante: setTimeout com closure que fica viva por horas
function processar(payload) {
  // payload de 100MB fica retido pelas próximas 24h
  setTimeout(() => log(payload.id), 86_400_000);
}

4. Streams não consumidos / Buffers retidos

leak-streams.js
// ❌ Cliente desconectou no meio do upload, mas req continua bufferizando
async function handler(req, res) {
  const chunks = [];
  for await (const chunk of req) chunks.push(chunk);  // loop nunca termina
  // req.aborted nunca é checado, chunks acumulam, leak de Buffer
}

// ✅ Detecta abort/close e limpa
async function handler2(req, res) {
  const chunks = [];
  req.on('aborted', () => chunks.length = 0);

  try {
    for await (const chunk of req) chunks.push(chunk);
  } catch (err) {
    // erro ou abort do cliente — chunks são liberados ao sair do escopo
    return res.writeHead(400).end();
  }
}
▸ Nota
Buffers grandes não aparecem direito em heap snapshots do V8 — eles são alocados fora do heap e contam só em external/arrayBuffers do memoryUsage. Para diagnosticar leaks de Buffer, observe process.memoryUsage().external ao longo do tempo, não só heapUsed.

# Diagnóstico: o método dos três snapshots

Quando o gráfico de memória sobe sem voltar, o método mais confiável para encontrar a fonte é o "three snapshot pattern":

  1. Snapshot 1 — após warmup (servidor pronto, caches inicializados, em estado estável).
  2. Rode carga de tráfego representativa por alguns minutos.
  3. Snapshot 2 — depois da carga.
  4. Rode mais carga idêntica por outros minutos.
  5. Snapshot 3 — depois da segunda carga.
  6. Compare os três no Chrome DevTools — Memory → "Comparison" entre 1 e 2, depois 2 e 3.

O que importa: objetos que cresceram do snapshot 1 para o 2 e do 2 para o 3. Esses são os candidatos a leak — o que cresceu só uma vez pode ser cache válido. Use a coluna Retained Size (não Shallow Size) para encontrar o objeto que está prendendo o conjunto, e siga "Retainers" para descobrir quem mantém referência.

# CPU — saturação e o que fazer

Diferente de memória, CPU em Node tem um limite duro: um único core para JavaScript. Quando você vê top mostrar 100% num core e os outros ociosos, isso é sinal claro de gargalo de event loop. As respostas:

  1. Mover trabalho CPU-bound para Worker Threads (cap. 11). Hash, parsing pesado, compressão — todos cabem.
  2. Quebrar loops longos em chunks com setImmediate (cap. 4) — para não trancar o loop por > 10ms.
  3. Cluster ou múltiplos containers — se o trabalho é inerentemente paralelizável e independente.
  4. Substituir bibliotecas puras-JS por bindings nativos — argon2 em vez de bcryptjs, parsers em Rust via NAPI quando o gargalo é confirmado.

# Padrões úteis para reduzir pressão no GC

  scratch.valor = item.valor;

# Prototype pollution

Prototype pollution é uma classe de vulnerabilidade exclusiva de JavaScript: se um atacante conseguir definir propriedades em Object.prototype, todos os objetos do processo herdam essas propriedades — podendo alterar comportamento de bibliotecas, bypass de autorizações, e até RCE em casos extremos:

prototype-pollution.js
// Vetor típico: merge de objeto controlado pelo atacante
function mergeInseguro(alvo, fonte) {
  for (const chave in fonte) {
    if (typeof fonte[chave] === 'object') {
      alvo[chave] = alvo[chave] || {};
      mergeInseguro(alvo[chave], fonte[chave]); // ❌ sem checar __proto__
    } else {
      alvo[chave] = fonte[chave];
    }
  }
}

// Payload do atacante:
const payload = JSON.parse('{"__proto__": {"admin": true}}');
mergeInseguro({}, payload);
console.log(({}).admin);  // true — prototype contaminado!

// ✅ Mitigações:

// 1. Criar objetos sem prototype para dados externos
const seguro = Object.create(null);  // sem prototype, sem perigo

// 2. Checagem explícita na função de merge
function mergeSeguro(alvo, fonte) {
  for (const chave of Object.keys(fonte)) {  // Object.keys não retorna __proto__
    if (chave === '__proto__' || chave === 'constructor' || chave === 'prototype') continue;
    alvo[chave] = fonte[chave];
  }
  return alvo;
}

// 3. structuredClone — nunca propaga __proto__
const copia = structuredClone(dadosExternos);

// 4. Usar Map em vez de objetos para dados de chave arbitrária
const mapa = new Map(dados);  // __proto__ é só uma chave normal

# Rate limiting

Sem rate limiting, endpoints de login, cadastro e API pública ficam abertos a brute force, credential stuffing e DoS por custo. Implemente em múltiplas camadas:

import rateLimit from 'express-rate-limit';
import { RedisStore } from 'rate-limit-redis';

// Limite global — proteção básica
const global = rateLimit({
  windowMs: 15 * 60 * 1000,  // 15 minutos
  max: 300,                   // 300 reqs por IP por janela
  standardHeaders: true,     // RateLimit-* headers no response
  legacyHeaders: false,
});

// Limite estrito para auth — brute force
const authLimit = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 10,
  skipSuccessfulRequests: true,  // só conta falhas
  store: new RedisStore({ ... })  // distribuído entre instâncias
});

app.use(global);
app.use('/auth', authLimit);
▸ Nota
Em múltiplas instâncias Node (cluster, containers), rate limit em memória é por processo — cada instância tem seu próprio contador. Use Redis como store (rate-limit-redis) para compartilhar estado entre instâncias. Se você tem Nginx ou um load balancer na frente, considere também rate limiting na camada de proxy.

    res.writeHead(400).end(JSON.stringify({
      code: 'VALIDATION',
      errors: resultado.error.flatten().fieldErrors,
    }));
    return;
  }
  // resultado.data é tipado e sanitizado
  const { nome, email, senha } = resultado.data;
  await cadastrar({ nome, email, senha });  // sem campos extras injetados
}

# Controle de uploads de arquivos

upload-seguro.js
import { createWriteStream } from 'node:fs';
import { pipeline } from 'node:stream/promises';
import { basename, extname, join } from 'node:path';
import { randomUUID } from 'node:crypto';

const EXTS_PERMITIDAS = new Set(['.jpg', '.jpeg', '.png', '.webp', '.pdf']);
const LIMITE_BYTES = 10 * 1024 * 1024;  // 10MB

async function salvarUpload(stream, nomeOriginal) {
  // 1. Validar extensão
  const ext = extname(nomeOriginal).toLowerCase();
  if (!EXTS_PERMITIDAS.has(ext)) {
    throw new Error(`Extensão não permitida: ${ext}`);
  }

  // 2. Nome aleatório — nunca use o nome original
  const nomeSeguro = randomUUID() + ext;
  const caminho = join('/var/uploads', nomeSeguro);

  // 3. Limite de tamanho — encerra se ultrapassar
  let bytes = 0;
  async function* comLimite(src) {
    for await (const chunk of src) {
      bytes += chunk.length;
      if (bytes > LIMITE_BYTES) throw new Error('Arquivo muito grande');
      yield chunk;
    }
  }

  await pipeline(comLimite(stream), createWriteStream(caminho));
  return nomeSeguro;  // retorna apenas o nome, não o path completo
}
▸ Cuidado
Extensão e Content-Type são facilmente falsificados. Para uploads críticos (imagens que serão servidas), valide o magic bytes do arquivo (primeiros bytes que identificam o formato) e reprocesse com uma biblioteca confiável. Um arquivo .jpg com magic bytes de SVG pode conter XSS. Nunca sirva uploads de usuários do mesmo domínio da aplicação — use um subdomínio separado ou CDN.

# Secrets e criptografia

crypto-seguro.ts
import { randomBytes, timingSafeEqual, createHmac, scrypt } from 'node:crypto';
import { promisify } from 'node:util';

const scryptAsync = promisify(scrypt);

// Gerar token seguro (para reset de senha, API keys)
const token = randomBytes(48).toString('base64url');

// Hash de senha — scrypt (robusto) ou argon2 (recomendado quando disponível)
async function hashSenha(senha: string) {
  const salt = randomBytes(16);
  const derivado = await scryptAsync(senha, salt, 64) as Buffer;
  return salt.toString('hex') + ':' + derivado.toString('hex');
}

async function verificarSenha(senha: string, hash: string) {
  const [saltHex, hashHex] = hash.split(':');
  const salt = Buffer.from(saltHex, 'hex');
  const derivado = await scryptAsync(senha, salt, 64) as Buffer;
  const esperado = Buffer.from(hashHex, 'hex');
  // Comparação em tempo constante — previne timing attack
  return derivado.length === esperado.length && timingSafeEqual(derivado, esperado);
}

// HMAC — assinar payloads, webhooks
function assinar(payload: string, secret: string) {
  return createHmac('sha256', secret).update(payload).digest('hex');
}

function verificarHmac(payload: string, assinatura: string, secret: string) {
  const esperada = assinar(payload, secret);
  return timingSafeEqual(Buffer.from(assinatura), Buffer.from(esperada));
}

# Estrutura de projeto que escala

Não existe estrutura universal — existe estrutura que torna o código encontrável, testável e substituível. O critério: qualquer dev que entrou no time hoje consegue achar onde mudança X deve ir sem perguntar:

estrutura recomendada — API Node + TypeScript
src/
├── index.ts           # bootstrap: cria server, registra handlers, listen()
├── env.ts             # validação de process.env — importado ANTES de tudo
│
├── routes/            # handlers HTTP agrupados por domínio de negócio
│   ├── users.ts       # GET /users, POST /users, GET /users/:id
│   └── orders.ts
│
├── services/          # lógica de negócio pura, sem dependência de HTTP
│   ├── UserService.ts # createUser, findById, updateEmail...
│   └── OrderService.ts
│
├── db/                # acesso ao banco — queries, migrations, pool
│   ├── index.ts       # exporta o pool/cliente configurado
│   └── queries/       # ou schema Drizzle/Prisma
│
├── middleware/        # auth, logging de request, rate limit, cors
│
├── lib/               # utilitários compartilhados (crypto, date, string)
│
└── types/             # tipos compartilhados, DTOs, interfaces de domínio

tests/
├── unit/              # serviços, utilitários — sem I/O real
├── integration/       # banco, APIs externas — com containers ou mocks
└── e2e/               # fluxos completos via HTTP

A separação routes → services → db força uma separação de responsabilidades que paga dividendos na hora de testar: services são testáveis sem HTTP; queries são testáveis sem servidor.

# Anti-padrões que destroem servidores Node

Estes erros são comuns, às vezes passam em code review, e só aparecem sob carga real:

anti-padroes.ts
// ❌ 1. Trabalho síncrono em handler de request
app.get('/export', (req, res) => {
  const csv = gerarCsvDe100MbSync(dados);  // trava todos os outros requests!
  res.send(csv);
});
// ✅ Worker Thread ou streaming incremental

// ❌ 2. await em loop sequencial quando operações são independentes
for (const userId of ids) {
  users.push(await buscar(userId));  // 100 ids = 100 roundtrips sequenciais
}
// ✅ Promise.all ou mapPool com limite de concorrência

// ❌ 3. Handler async sem try/catch — Promise rejeita silenciosamente
app.get('/users/:id', async (req, res) => {
  const u = await buscar(req.params.id);  // rejeição não capturada → timeout
  res.json(u);
});
// ✅ try/catch + resposta de erro; ou middleware de erro global

// ❌ 4. Registrar listener dentro de handler
app.use((req, res) => {
  process.on('SIGTERM', () => res.end());  // vazamento de listener por request
});
// ✅ Registrar uma vez no escopo do módulo

// ❌ 5. readFileSync/writeFileSync em servidor HTTP
app.get('/config', (req, res) => {
  const cfg = readFileSync('./config.json');  // bloqueia o event loop
  res.json(JSON.parse(cfg));
});
// ✅ readFile de fs/promises, ou cache na inicialização

// ❌ 6. Construir query SQL por concatenação de strings
db.query(`SELECT * FROM users WHERE id = ${req.params.id}`);  // SQL injection
// ✅ Parâmetros posicionais sempre
db.query('SELECT * FROM users WHERE id = $1', [req.params.id]);

// ❌ 7. JSON.parse sem try/catch em input externo
const body = JSON.parse(rawBody);  // SyntaxError derruba o handler
// ✅ Sempre envolver em try/catch ou usar z.safeParse

// ❌ 8. process.exit() sem flush de streams
server.close(() => process.exit(0));  // logs podem não ter sido gravados
// ✅ Usar process.exitCode + aguardar streams
server.close(async () => {
  await logger.flush();
  process.exitCode = 0;
});

# Inicialização segura

A sequência de bootstrap importa — erros de configuração devem derrubar o processo antes do servidor aceitar conexões:

src/index.ts — bootstrap ordenado
// 1. Variáveis de ambiente — falha aqui se config estiver errada
import { env } from './env.js';

// 2. Logger — pronto para logar erros de inicialização
import { logger } from './lib/logger.js';

// 3. Handlers globais de erro — antes de qualquer I/O
process.on('uncaughtException', (err) => { logger.fatal({ err }, 'uncaughtException'); process.exit(1); });
process.on('unhandledRejection', (r) => { logger.fatal({ reason: r }, 'unhandledRejection'); process.exit(1); });

async function main() {
  // 4. Conexões de infraestrutura — banco, Redis, etc.
  const db = await conectarBanco(env.DATABASE_URL);
  logger.info('banco conectado');

  // 5. Servidor HTTP — só listen() depois que tudo está pronto
  const app = criarApp({ db, logger });
  await new Promise<void>(resolve => app.listen(env.PORT, env.HOST, resolve));
  logger.info({ port: env.PORT }, 'servidor pronto');

  // 6. Graceful shutdown
  const shutdown = async (sinal: string) => {
    logger.info({ sinal }, 'encerrando...');
    await new Promise(r => app.close(r));
    await db.end();
    process.exit(0);
  };
  process.on('SIGTERM', () => shutdown('SIGTERM'));
  process.on('SIGINT',  () => shutdown('SIGINT'));
}

main().catch(err => { logger.fatal({ err }, 'falha no bootstrap'); process.exit(1); });

# O que Node não é — limites reais

Saber os limites com clareza é tão importante quanto saber as forças. Node não é a melhor escolha para:

  • Cálculo científico ou ML pesado — use Python, Julia, ou exporte para WASM.
  • Jobs com paralelismo massivo de CPU — Elixir, Go ou Rust fazem melhor com menos configuração.
  • Sistemas de baixíssima latência (< 100µs) — GC pauses tornam o tail latency imprevisível; prefira C++ ou Rust.
  • Processamento de streams de vídeo pesado — delegue ao ffmpeg ou serviço dedicado.

Para APIs, BFFs, proxies, real-time, CLIs, build tools e microsserviços que fazem I/O: Node em 2025 é uma escolha excelente — madura, rápida, com ecossistema vasto e modelo de concorrência comprovado em escala.
