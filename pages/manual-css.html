Capítulo 01

Como o CSS funciona

CSS é declarativo: você descreve como deve ser, não como chegar lá. Essa única característica define tudo o que vem depois — cascata, especificidade, herança, e por que às vezes uma regra "não pega". Este capítulo é o modelo mental que transforma CSS de misterioso em previsível.

# Anatomia de uma regra CSS

anatomia.css
/* Uma regra CSS é feita de seletor + bloco de declarações */

.card {              /* seletor  — "quais elementos afetar" */
  padding: 16px;     /* declaração: propriedade + valor */
  color: #333;       /* cada declaração termina com ; */
  background: white; /* pode ter quantas quiser */
}

/* Uma declaração = propriedade: valor;
   - Propriedade: o QUE estilizar (cor, tamanho, margem...)
   - Valor: COMO estilizar (vermelho, 20px, 50%) */

# Como o navegador lê o CSS

Quando o navegador recebe o HTML e o CSS, ele executa um pipeline que acontece em milissegundos, mas vale entender:

  1. Parse do HTML — constrói o DOM (árvore de elementos).
  2. Parse do CSS — constrói o CSSOM (árvore de regras).
  3. Render tree — combina DOM e CSSOM para saber o que precisa ser pintado.
  4. Layout — calcula posição e tamanho de cada elemento.
  5. Paint — pinta os pixels na tela.
  6. Composite — combina camadas, incluindo aceleração por GPU.

Saber disso ajuda em performance (cap. 28): mudar transform só recompõe camadas; mudar width força layout + paint + composite — muito mais caro.

# Três formas de aplicar CSS

tres-formas.html
<!-- 1) External — arquivo separado (o padrão profissional) -->
<link rel="stylesheet" href="/estilo.css">

<!-- 2) Internal — dentro de <style> no head
        (bom para CSS crítico de primeira renderização) -->
<style>
  body { font-family: system-ui; }
</style>

<!-- 3) Inline — atributo 'style' direto no elemento
        (use apenas em casos muito pontuais) -->
<p style="color: red;">Texto vermelho</p>
Evite inline sem motivo
CSS inline tem especificidade altíssima, é difícil de sobrescrever, duplica em cada elemento, e não pode ser cacheado. Use-o apenas para valores dinâmicos gerados por JavaScript (posição de tooltip, cor tema do usuário) — nunca para estilos estáticos que caberiam em um arquivo CSS.

# Herança — propriedades que descem a árvore

Algumas propriedades herdam automaticamente do pai para os filhos. Isso é uma das mecânicas mais subestimadas do CSS.

heranca.css
/* Propriedades que HERDAM por padrão:
   - Tipografia: font-family, font-size, color, line-height, text-align
   - Listagem: list-style
   - Cursor, visibility
   - Algumas de tabela */

body {
  font-family: "Inter", sans-serif;  /* todo filho herda */
  color: #333;                        /* todo filho herda */
}

/* Propriedades que NÃO herdam por padrão:
   - Box model: margin, padding, border, width, height
   - Background, position, display, z-index */

/* Forçar herança explicitamente */
a {
  color: inherit;  /* usa a cor do pai */
}

/* Três palavras-chave universais */
* {
  color: inherit;   /* copia do pai */
  margin: initial;  /* volta ao padrão do navegador */
  padding: unset;   /* inherit se herdável, senão initial */
  border: revert;   /* volta ao padrão do browser (user-agent stylesheet) */
}

# O reset — começar do zero

Cada navegador vem com uma "folha de estilo do usuário" com valores padrão (margens em <body>, bullets em <ul>, etc). Eles variam entre navegadores, criando inconsistências. A solução é um reset: um CSS inicial que normaliza o começo.

reset-moderno.css
/* Reset moderno e enxuto — essencial para projetos novos */

*, *::before, *::after {
  box-sizing: border-box;   /* box model previsível (cap. 5) */
}

html {
  -webkit-text-size-adjust: 100%;  /* iOS: não amplia texto no rotate */
  text-size-adjust: 100%;
  scroll-behavior: smooth;
}

body {
  margin: 0;
  min-height: 100dvh;        /* usa altura real do viewport em mobile */
  line-height: 1.5;
  font-family: system-ui, sans-serif;
}

/* Imagens se comportam como blocos, não vazam */
img, picture, video, canvas, svg {
  display: block;
  max-width: 100%;
}

/* Inputs herdam fonte do body (por padrão NÃO herdam!) */
input, button, textarea, select {
  font: inherit;
}

/* Texto quebra corretamente em todas as situações */
p, h1, h2, h3, h4, h5, h6 {
  overflow-wrap: break-word;
}

/* Headings começam no mesmo tamanho do parágrafo — você define depois */
h1, h2, h3, h4, h5, h6 {
  font-size: inherit;
  font-weight: inherit;
}
Resets prontos
Para projetos novos, vale usar um reset consolidado: modern-normalize (pragmático, mantém defaults sensatos) ou o reset de Andy Bell (mais opinativo, ideal para design systems). Frameworks como Tailwind já incluem preflight, que é o próprio reset deles.

Capítulo 02

Seletores

Seletores são como o CSS encontra os elementos que ele quer estilizar. Eles formam uma linguagem própria dentro do CSS — e dominá-la significa escrever muito menos código, mais direto e mais manutenível. O que antes exigia classes extras ou JavaScript, hoje resolve em um seletor bem escolhido.

# Seletores básicos

basicos.css
/* Universal — todo elemento */
* { box-sizing: border-box; }

/* Por tag */
h1 { font-size: 2rem; }
button { cursor: pointer; }

/* Por classe (o mais comum) */
.card { padding: 16px; }

/* Por ID (especificidade alta — use com parcimônia) */
#header { position: sticky; }

/* Por atributo */
[disabled] { opacity: 0.5; }
[type="email"] { text-transform: lowercase; }
[href^="https://"] { /* começa com */ }
[href$=".pdf"] { /* termina com */ }
[class*="btn"] { /* contém */ }
[lang|="en"] { /* lang="en" ou lang="en-*" */ }
[title~="important"] { /* palavra exata em lista */ }

/* Atributo case-insensitive (flag 'i') */
[type="EMAIL" i] { /* casa com EMAIL, email, Email... */ }

# Combinadores — relações entre elementos

combinadores.css
/* Descendente (espaço) — qualquer nível abaixo */
article p { line-height: 1.6; }
/* Pega <p> dentro de <article>, em qualquer profundidade */

/* Filho direto (>) — apenas filhos imediatos */
nav > ul { display: flex; }
/* <ul> que é filho DIRETO de <nav>, não neto */

/* Irmão adjacente (+) — IMEDIATAMENTE depois do mesmo pai */
h2 + p { margin-top: 0; }
/* <p> logo após um <h2> */

/* Irmão geral (~) — qualquer irmão posterior */
h2 ~ p { color: gray; }
/* Todos os <p> que vêm depois de um <h2> (mesmos pais) */

/* Combinar várias classes */
.btn.primary { background: blue; }
/* Elemento com ambas as classes */

/* Combinar classe + tag */
button.danger { background: red; }

/* Agrupar seletores (vírgula) */
h1, h2, h3 { font-weight: 700; }

# Pseudoclasses — estados e relações

Pseudoclasses (prefixo :) selecionam elementos em estados específicos — hover, foco, marcado — ou com relações estruturais.

Estados de interação

estados.css
a:hover { text-decoration: underline; }
button:active { transform: scale(0.98); }
input:focus { outline: 2px solid blue; }
input:focus-visible { outline: 2px solid blue; } /* só teclado — cap. 18 do manual HTML */
button:disabled { opacity: 0.5; }
input:checked { accent-color: green; }
input:placeholder-shown { font-style: italic; }
input:valid { border-color: green; }
input:invalid { border-color: red; }
input:user-invalid { border-color: red; } /* só depois do usuário interagir */
input:required { border-left: 3px solid red; }
input:optional { border-left: 3px solid gray; }
input:read-only { background: #f5f5f5; }
input:in-range { color: green; }
input:out-of-range { color: red; }

Estrutura — posição entre irmãos

estrutura.css
li:first-child { font-weight: bold; }
li:last-child { border-bottom: none; }
li:only-child { text-align: center; }
li:nth-child(2) { background: #eee; }        /* segundo filho */
li:nth-child(odd) { background: #f9f9f9; }   /* ímpares */
li:nth-child(even) { background: white; }    /* pares */
li:nth-child(3n) { color: red; }              /* 3, 6, 9... */
li:nth-child(3n + 1) { color: blue; }         /* 1, 4, 7... */
li:nth-child(-n + 3) { font-weight: bold; }   /* primeiros 3 */
li:nth-last-child(2) { /* penúltimo */ }

/* "of type" — mesma lógica mas conta só o MESMO tipo */
p:first-of-type { font-size: 1.2em; }  /* primeiro <p> (ignora outros tags) */
img:nth-of-type(2) { /* segundo <img> */ }

/* :nth-child com filtro (CSS moderno) */
li:nth-child(2 of .ativo) { /* segundo li COM classe .ativo */ }

/* Vazio / apenas espaços */
p:empty { display: none; }
p:blank { /* vazio ou só whitespace — proposta mais nova */ }

Negação e lógica — :not(), :is(), :where()

logica.css
/* :not() — exclui elementos */
button:not(.primary) { background: gray; }
li:not(:last-child) { border-bottom: 1px solid; }

/* Múltiplos argumentos (moderno) */
input:not([type="hidden"], [disabled]) { padding: 8px; }

/* :is() — aceita lista de seletores, usa a maior especificidade */
:is(h1, h2, h3) a { color: inherit; }
/* Equivalente a: h1 a, h2 a, h3 a — mas mais curto */

/* :where() — igual a :is() MAS com especificidade ZERO */
:where(h1, h2, h3) a { color: inherit; }
/* Muito útil em resets e defaults — não compete na cascata */

/* Diferença prática */
:is(section) p { color: red; }       /* especificidade: (0,0,1) */
:where(section) p { color: red; }    /* especificidade: (0,0,0) — qualquer regra sobrescreve */
:where() é subestimado
Use :where() em resets, bases e design systems. Como ele tem especificidade zero, você evita guerras de especificidade — qualquer classe do componente sobrescreve sem esforço. É uma das adições mais elegantes do CSS moderno.

# Pseudoelementos — partes de um elemento

Pseudoelementos (prefixo ::) selecionam partes de um elemento, ou criam elementos virtuais.

pseudo-elementos.css
/* ::before e ::after — elementos virtuais antes/depois do conteúdo */
.card::before {
  content: "";              /* OBRIGATÓRIO — sem content, não aparece */
  display: block;
  width: 40px;
  height: 4px;
  background: var(--accent);
}

.required::after {
  content: " *";
  color: red;
}

/* ::first-line e ::first-letter — tipografia refinada */
p::first-line { font-weight: bold; }
p::first-letter { font-size: 3em; float: left; }

/* ::marker — o bullet ou número da lista */
li::marker {
  color: var(--accent);
  font-weight: bold;
}

/* ::selection — quando o usuário seleciona com o mouse */
::selection {
  background: var(--accent);
  color: white;
}

/* ::placeholder — texto do placeholder */
input::placeholder {
  color: #999;
  opacity: 1;  /* Firefox aplica opacity padrão */
}

/* ::backdrop — o fundo de <dialog> e fullscreen */
dialog::backdrop {
  background: rgba(0, 0, 0, 0.5);
}

/* ::details-content — conteúdo expansível de <details> (moderno) */
details::details-content {
  overflow: hidden;
}

/* ::file-selector-button — botão do input type="file" */
input[type="file"]::file-selector-button {
  padding: 8px 16px;
  border-radius: 6px;
}

# Seletores de ambiente — :root, :target, :lang

ambiente.css
/* :root — o elemento raiz (<html>), mas com especificidade maior que 'html'.
   É onde você declara variáveis globais. */
:root {
  --cor-primaria: #f0a830;
  --espaco: 8px;
}

/* :target — elemento cujo id corresponde ao hash da URL */
/* Ex: URL tem #capitulo-3 → :target ativa em #capitulo-3 */
section:target {
  outline: 2px solid var(--accent);
  scroll-margin-top: 80px;
}

/* :lang(pt) — elementos com idioma específico */
:lang(pt) q { quotes: "“" "”"; }
:lang(fr) q { quotes: "« " " »"; }

/* :dir(ltr) / :dir(rtl) — direção de escrita */
:dir(rtl) .icon { transform: scaleX(-1); }

/* :scope — usado em contexto específico (ex: querySelector) */
:scope > li { /* filhos diretos do elemento de escopo */ }

/* Padrão :root universal para design tokens */
:root {
  color-scheme: light dark;  /* aceita ambos os temas */
  --text: light-dark(#111, #eee);
  --bg: light-dark(#fff, #111);
}

# O poder do & — parent selector moderno

CSS nativo agora tem nesting e o símbolo & referencia o pai — como em Sass. Voltaremos a isso no capítulo 4, mas aqui fica o primeiro contato:

nesting-preview.css
.btn {
  padding: 10px 20px;

  &:hover {
    background: blue;
  }

  &.primary {
    background: var(--accent);
  }

  .icon + & {
    /* '&' dentro de contexto — aqui quer dizer
       ".btn que vem depois de .icon no mesmo pai" */
  }
}
Regra prática
Seletores simples (uma classe, um atributo) são mais rápidos, mais previsíveis e mais fáceis de manter. Prefira escrever .btn-primary a nav > ul > li:first-child a.active. Reserve seletores complexos para casos em que mudar o HTML não é opção.
Capítulo 03

Especificidade e cascata

"Por que essa regra não está sendo aplicada?" Esta é provavelmente a pergunta mais feita por quem está começando com CSS. A resposta quase sempre está na cascata e na especificidade — as duas engrenagens que decidem qual regra vence quando várias tentam estilizar o mesmo elemento.

# A cascata — como o CSS resolve conflitos

Quando duas ou mais regras aplicam-se ao mesmo elemento e tentam mudar a mesma propriedade, o CSS segue uma ordem precisa para decidir qual vence. A palavra "cascata" no nome da linguagem vem daí.

A ordem de decisão é:

  1. Origem e importância — de onde vem a regra e se tem !important.
  2. Cascade layers (@layer) — visto no próximo capítulo.
  3. Especificidade — quanto o seletor é "específico".
  4. Ordem do código — a última regra declarada vence (em empate).

# Especificidade — a métrica

Cada seletor tem um "peso" calculado em três números: (a, b, c).

ComponenteO que conta
a (IDs)Quantos IDs (#x) o seletor tem
b (classes)Classes (.x), atributos ([x]), pseudoclasses (:hover)
c (tipos)Tags (div) e pseudoelementos (::before)
especificidade.css
/* Calcule: conte IDs, classes+atributos+pseudoclasses, tags+pseudoelementos */

*                             /* (0,0,0) — universal não conta */
p                             /* (0,0,1) */
p::before                     /* (0,0,2) — tag + pseudoelemento */
.card                         /* (0,1,0) */
.card:hover                   /* (0,2,0) */
.card p                       /* (0,1,1) */
article .card p               /* (0,1,2) */
[type="email"]                /* (0,1,0) */
:is(h1, h2, h3)               /* (0,0,1) — usa a MAIOR do grupo */
:where(h1, h2, h3)            /* (0,0,0) — SEMPRE zero */
:not(.foo)                    /* (0,1,0) — usa a do argumento */
#header                       /* (1,0,0) */
#header .nav li               /* (1,1,2) */
style=""                      /* (1,0,0,0) — "inline" é uma 4ª coluna */
!important                    /* trunfo — ignora tudo acima */

# Comparando especificidades

Comparação é posição-por-posição, da esquerda para a direita. Um único ID vence qualquer quantidade de classes.

comparacao.css
/* Qual vence? */

#btn { color: red; }                  /* (1,0,0) */
.btn.primary.large { color: blue; }   /* (0,3,0) */
/* Vence: vermelho. UM id vence TRÊS classes. */

.card p { color: gray; }              /* (0,1,1) */
article .text { color: black; }       /* (0,1,1) */
/* Empate em especificidade → vence a ÚLTIMA declarada (preto) */

a.link { color: blue; }               /* (0,1,1) */
ul li a { color: red; }               /* (0,0,3) */
/* Vence: azul. Uma classe > três tags. */
Notação rápida no DevTools
No inspetor do Chrome, ao passar o mouse sobre um seletor, ele mostra a especificidade calculada. Quando você está debugando por que uma regra não aplica, é o caminho mais rápido — compare os números dos seletores que estão brigando.

# Ordem do código como desempate

ordem.css
/* Quando a especificidade é igual, a ÚLTIMA regra vence */

.btn { color: red; }
.btn { color: blue; }
/* Resultado: azul */

/* Por isso a ordem dos arquivos importa */
<link rel="stylesheet" href="/base.css">       <!-- primeiro -->
<link rel="stylesheet" href="/componentes.css"> <!-- sobrescreve base -->
<link rel="stylesheet" href="/temas.css">       <!-- sobrescreve tudo -->

/* Dentro do mesmo arquivo, leia de cima para baixo */
.btn { padding: 10px; }         /* valor inicial */
.btn.large { padding: 20px; }   /* só quando tem classe .large */

# !important — o martelo

important.css
/* !important sobrescreve QUALQUER coisa (quase) */
.alerta {
  color: red !important;
}

/* Só outro !important com MAIOR especificidade consegue vencer */
#admin .alerta {
  color: blue !important;  /* este vence */
}

/* A ordem real de importância completa:
   1. user-agent (navegador) normal
   2. usuário normal
   3. autor (você) normal          ← CSS que você escreve
   4. autor !important
   5. usuário !important
   6. user-agent !important         ← ninguém vence isso
   7. animações em execução */
Evite !important
Cada !important é uma pequena dívida técnica. Uma vez que você usa, a próxima regra que quiser sobrescrever precisa também ser !important, e virou uma espiral. Use apenas quando: (1) está sobrescrevendo CSS de terceiros que você não controla, ou (2) em utility classes (Tailwind usa para garantir que .mt-4 sempre funciona). Em código próprio comum, quase sempre é melhor aumentar a especificidade do seletor de forma controlada.

# Armadilhas comuns

Armadilha 1 — especificidade acidental

armadilha-1.css
/* Você usa IDs no HTML por conveniência (link interno, JS) */
<nav id="principal">
  <a class="active">Home</a>
</nav>

/* Você estiliza assim, acha elegante */
#principal .active { color: red; }  /* (1,1,0) */

/* Agora tentam sobrescrever e falham silenciosamente */
.active { color: blue; }  /* (0,1,0) — NÃO vence */

/* Solução: não estilize via ID. Estilize só por classe. */
.nav-principal .active { color: red; }  /* (0,2,0) */
.active { color: blue; }                /* (0,1,0) — ainda perde, mas é mais controlável */

Armadilha 2 — reset tarde demais

armadilha-2.css
/* ❌ Arquivo botao.css */
.btn { padding: 10px; }

/* ❌ Arquivo reset.css importado DEPOIS */
button { padding: 0; }  /* sobrescreve .btn por ordem */

/* ✅ Correto: importe o reset PRIMEIRO */
<link rel="stylesheet" href="/reset.css">
<link rel="stylesheet" href="/componentes.css">

Armadilha 3 — o estilo inline que ninguém vê

armadilha-3.html
<!-- Um dev coloca inline por "praticidade" -->
<button style="color: red">Click</button>

<!-- Outro dev tenta sobrescrever via CSS externo -->
/* button { color: blue; } */     ← não vai funcionar

<!-- Único jeito de sobrescrever inline sem !important -->
/* Não tem. Inline tem especificidade (1,0,0,0) — acima de IDs. */
/* Para vencer, só !important ou remover o inline. */

# Estratégias para controlar especificidade

estrategias.css
/* 1. Mantenha especificidade baixa — use classes simples */ .card { } .card-title { } .card-body { } /* Melhor que: .card > header > h2 */ /* 2. Use :where() para zerar a especificidade em defaults */ :where(h1, h2, h3) { margin: 0; } /* Qualquer regra com classe sobrescreve sem luta */ /* 3. Use :is() para agrupar sem duplicar */ :is(.btn-primary, .btn-secondary):hover { transform: translateY(-1px); } /* 4. Aumente especificidade deliberadamente (quando necessário) */ .btn.btn { /* Duplicar a classe aumenta especificidade SEM !important */ /* (0,2,0) em vez de (0,1,0) */ } /* 5. Use cascade layers (próximo capítulo) — a forma MODERNA */ @layer reset, base, componentes, utilitarios; @layer componentes { .btn { padding: 10px; } } @layer utilitarios { .p-20 { padding: 20px; } /* layer posterior vence, independente de especificidade */ }
O futuro é com layers
@layer (cascade layers) é a forma moderna e limpa de controlar cascata — torna obsoletos vários truques antigos de especificidade. Veremos a fundo no próximo capítulo. Se você está começando um projeto novo em 2025+, vale aprender layers antes de memorizar hacks de especificidade.

# Debugando conflitos na prática

Quando algo não aplica, o roteiro mental é sempre o mesmo:

  1. Inspecione o elemento no DevTools. Veja quais regras estão ativas e quais foram sobrescritas (aparecem riscadas).
  2. Compare especificidades dos seletores que brigam.
  3. Verifique a ordem — se a especificidade empata, o último vence.
  4. Procure !important (às vezes vem de libraries).
  5. Verifique inline styles — eles vêm antes de regras externas no DevTools.

Layers, @scope e nesting

Três recursos que chegaram juntos no CSS moderno e transformaram como arquitetamos folhas de estilo. Cascade layers organizam a cascata de forma explícita. @scope isola estilos sem precisar de classes namespace. Nesting nativo traz o que antes exigia Sass. Juntos, tornam obsoletas várias convenções antigas.

# Cascade layers (@layer)

Cascade layers permitem agrupar regras em "camadas" com prioridade explícita. A ordem das camadas vence sobre especificidade — uma regra em uma camada posterior sempre vence, mesmo que tenha menos especificidade.

layers-basico.css
/* Declare a ORDEM das layers primeiro — a última tem mais prioridade */
@layer reset, base, componentes, utilitarios;

@layer reset {
  * { box-sizing: border-box; margin: 0; }
}

@layer base {
  body { font-family: sans-serif; }
  a { color: blue; }
}

@layer componentes {
  .btn {
    padding: 10px 20px;
    background: gray;
  }
}

@layer utilitarios {
  .bg-red { background: red; }
}

/* Agora, esta linha... */
.bg-red { background: red; }
/* ...no layer "utilitarios" VENCE esta... */
.btn { background: gray; }
/* ...no layer "componentes", mesmo com especificidade igual! */
A ordem é tudo
A regra fundamental das layers: a camada posterior sempre vence. Especificidade só é considerada dentro de cada camada. Se você colocou reset na primeira camada, tudo o resto sobrescreve sem luta. Se utilities vêm por último, vencem qualquer componente.

# Layers para integrar bibliotecas de terceiros

Um dos casos mais úteis: você usa uma biblioteca (Bootstrap, um component kit) e quer que seu CSS sempre vença, sem usar !important.

com-biblioteca.css
/* Importar biblioteca inteira em uma layer "baixa" */
@import url("https://cdn.bootstrap.css") layer(bootstrap);

/* Declarar ordem — seu CSS vem depois */
@layer bootstrap, meu-app;

@layer meu-app {
  .btn {
    background: var(--minha-cor);  /* vence QUALQUER regra do Bootstrap */
  }
}

/* Sem precisar de !important, sem precisar saber
   a especificidade do CSS original do Bootstrap. */

# Layers aninhadas e anônimas

layers-avancado.css
/* Layers aninhadas — use '.' para especificar sub-layers */
@layer componentes {
  @layer atomos, moleculas, organismos;

  @layer atomos {
    .btn { padding: 10px; }
  }

  @layer moleculas {
    .card .btn { padding: 12px; }  /* vence atomos */
  }
}

/* Ou usar notação com ponto */
@layer componentes.atomos {
  .btn { padding: 10px; }
}

/* Layer anônima — sem nome */
@layer {
  /* útil para regras que você não quer que ninguém sobrescreva via layer */
}

/* Regras SEM layer vencem todas as layers! */
.important-default { color: red; }  /* vence qualquer @layer */

/* Ordem real completa (menor para maior prioridade):
   1. Regras em @layer declaradas
   2. Regras SEM layer (implicitamente "normal layer")
   3. Regras !important de layers (em ordem INVERSA!)
   4. Regras !important sem layer */

# @scope — isolamento real

@scope resolve um problema antigo: como estilizar "tudo dentro de X, mas parar quando chegar em Y"? Antes, a resposta era nomeação BEM, CSS-in-JS, ou hacks com :not(). Agora é nativo.

scope-basico.css
/* Sintaxe simples — escopa regras a um seletor */
@scope (.card) {
  p { color: var(--muted); }         /* p DENTRO de .card */
  h2 { font-size: 1.5rem; }
  a { color: var(--accent); }
}

/* Esses p, h2, a dentro do scope só se aplicam quando
   estiverem dentro de um elemento .card */

/* Scope com limite — "pare quando encontrar" */
@scope (.menu) to (.submenu) {
  a { color: blue; }
  /* Só pega <a> que está dentro de .menu, MAS para de
     aplicar quando encontra .submenu descendo a árvore. */
}

/* Uso prático — conteúdo markdown dentro de artigo */
@scope (article.prose) to (.no-prose) {
  p  { line-height: 1.7; }
  h2 { font-size: 2rem; margin-top: 2em; }
  ul { padding-left: 1.5em; }
}
/* Tudo acima aplica dentro de <article class="prose">,
   mas não em elementos dentro de <div class="no-prose"> */

# :scope e proximity dentro de @scope

scope-avancado.css
@scope (.card) {
  /* :scope se refere AO PRÓPRIO .card */
  :scope {
    background: white;
    padding: 16px;
  }

  /* & também funciona dentro de @scope */
  & p { font-size: 14px; }
}

/* Proximity — quando dois scopes se sobrepõem, vence O MAIS PRÓXIMO */
@scope (.tema-escuro) {
  p { color: #eee; }
}
@scope (.tema-claro) {
  p { color: #111; }
}

<div class="tema-escuro">
  <div class="tema-claro">
    <p>Este p usa a cor escura (tema-claro é mais próximo)</p>
  </div>
</div>
Suporte em 2025
@scope tem suporte completo em Chrome, Edge, Safari e Firefox desde 2024. Para aplicações que precisam suportar navegadores muito antigos, use como progressive enhancement — se falhar, os estilos ainda aplicam, só sem o isolamento.

# Nesting nativo — o & do CSS

Antes o nesting era o maior motivo para usar Sass. Agora está no CSS padrão — e funciona em todo lugar.

# O uso do & — casos importantes

ampersand.css
.btn {

Cuidados com nesting
Nesting torna o código mais curto, mas também pode torná-lo mais escondido. Três armadilhas comuns: (1) aninhamento profundo demais — se passar de 3 níveis, a especificidade sobe demais e fica difícil de debugar; (2) perder de vista o seletor final — sempre pense "o que este selector resolve expandido"; (3) duplicação — se você está repetindo &:hover em 20 componentes, talvez valha uma utility class.

# Combinando tudo — arquitetura moderna

Box model

Todo elemento HTML é uma caixa. O box model é a lei de como essa caixa é construída — content, padding, border, margin — e como o tamanho final é calculado. Aparentemente simples, mas com armadilhas que confundem até desenvolvedores experientes.

# As quatro camadas da caixa

Da camada interna para a externa:

CamadaO que é

paddingEspaço interno, entre o conteúdo e a borda
borderA borda em si — pode ser visível ou não
marginEspaço externo, entre a borda e outros elementos

# A armadilha do padrão — content-box

Por padrão, a propriedade width define a largura do conteúdo, não da caixa inteira. Isso é box-sizing: content-box — e é a origem de muita confusão.

content-box.css
.caixa-default {
  width: 200px;      /* conteúdo = 200px */
  padding: 20px;     /* +40px (20 de cada lado) */
  border: 4px solid; /* +8px */
  /* Largura REAL: 200 + 40 + 8 = 248px */
}

/* Isso causa quebras — você calculou 200px, mas o elemento ocupa 248px */

# A solução — border-box

A propriedade box-sizing: border-box faz width incluir padding e border. É o comportamento intuitivo, e o padrão universal em qualquer projeto moderno.

border-box.css
/* O reset que todo projeto moderno começa com */
*, *::before, *::after {
  box-sizing: border-box;
}

.caixa {
  width: 200px;      /* agora é a largura TOTAL */
  padding: 20px;     /* "come" do conteúdo, não adiciona */
  border: 4px solid; /* idem */
  /* Largura real: 200px. Conteúdo interno: 152px. */
}
Regra universal
Todo projeto profissional em 2025 começa com *, *::before, *::after { box-sizing: border-box; }. Tailwind, Bootstrap, modern-normalize — todos incluem. Se você for criar um projeto novo, essa é a primeira linha do CSS.

# Margin e o famoso colapso

Margin tem uma particularidade que não existe em padding: colapso de margens. Quando duas margens verticais se encontram, elas não somam — colapsam na maior.

margin-collapse.css
/* Colapso em ação */
.primeiro { margin-bottom: 40px; }
.segundo  { margin-top: 20px; }
/* Espaço REAL entre os dois: 40px (não 60px!). A menor foi absorvida. */

/* Colapso também acontece com o pai */
<div class="pai">
  <p class="filho">...</p>  /* margin-top do filho "vaza" para cima do pai */
</div>

.pai { margin-top: 10px; }
.filho { margin-top: 20px; }
/* O pai efetivamente recebe margin-top de 20px */

/* Como PARAR o colapso */
.pai {
  padding-top: 1px;           /* qualquer coisa no meio quebra o colapso */
  /* ou */
  border-top: 1px solid;
  /* ou */
  overflow: hidden;           /* cria formatting context novo */
  /* ou */
  display: flex;              /* flex/grid não colapsa */
}

/* ⚠️ Colapso SÓ acontece na vertical (block axis).
   Horizontal (inline) NÃO colapsa — margens somam normalmente. */
Margin negativo
Diferente de padding, margin aceita valores negativos. É útil para sobrepor elementos ou compensar espaçamentos — mas use com intenção. Margin negativo gera layouts frágeis se não for bem pensado.

# margin: auto — o truque universal

auto-margin.css
/* Centralizar horizontalmente um bloco com largura fixa */
.container {

  max-width: 100%;      /* não ultrapassa o pai */
  margin: 0 auto;       /* top/bottom: 0 | left/right: auto */
  /* 'auto' distribui o espaço disponível igualmente */
}

/* Empurrar um item para a direita em Flexbox */

  display: flex;

.nav .sair {

# Outline — a borda que não conta

outline parece com border, mas não ocupa espaço no layout. É pintado por cima do elemento. É a propriedade que desenha o anel de foco do teclado.

outline.css

  outline-offset: 2px;   /* distância entre elemento e outline */
}

/* Diferenças de border:
   - Não afeta dimensões do elemento (não empurra nada)
   - Pode ser retangular mesmo com border-radius
   - Usa outline-offset, border usa borda mesmo */
Debug visual

A propriedade display é talvez a mais fundamental do CSS. Ela decide como o elemento se comporta no fluxo da página — se ocupa a linha inteira, se flui com o texto, se forma uma grade, se some. Antes de aprender Flexbox e Grid, é preciso entender display nos seus níveis clássicos.

# Block vs inline na prática

.block {
  display: block;
  width: 300px;         /* funciona */
  height: 100px;        /* funciona */
  margin: 10px;         /* funciona em todos os lados */
  padding: 20px;

  display: inline;
  width: 300px;         /* IGNORADO */
  height: 100px;        /* IGNORADO */

  padding: 20px;        /* vaza fora da linha verticalmente */

# display: none vs visibility vs opacity

  display: none;

}

.invisivel {
  visibility: hidden;
  /* Reserva o espaço. Não é clicável nem focável.
     Screen readers pulam. */

opacity: 0 é traiçoeiro

# display: contents — o elemento invisível

Um valor peculiar e moderno: o elemento "desaparece" do layout, mas seus filhos permanecem, como se fossem filhos do avô.

/* Útil quando o HTML exige um wrapper, mas o layout não deveria */

  <div class="wrapper">  /* display: contents */
    <div>Item 1</div>    /* tratados como filhos diretos do .grid */
    <div>Item 2</div>
  </div>
</div>

  display: contents;
  /* O .wrapper some do layout; Item 1 e Item 2 participam
     do grid diretamente. */
}

/* ⚠️ Cuidado: backgrounds, borders e padding do elemento
   com display: contents não são renderizados. */

# Float — o método legado

float foi por anos a principal ferramenta de layout. Hoje é usado quase exclusivamente para texto fluindo ao redor de imagens — o uso original para o qual foi desenhado.

.d { overflow: auto; }     /* barras aparecem só se necessário */
.e { overflow: clip; }     /* como hidden mas sem criar novo scroll context */

  overflow-y: hidden; /* vertical independente */

# Formatting contexts — o conceito por trás

Cada elemento participa de um "contexto de formatação" — o conjunto de regras que rege como ele se relaciona com vizinhos. Isso raramente aparece em tutoriais, mas explica comportamentos que parecem mágica.

  • Block Formatting Context (BFC) — o clássico. Elementos block empilham verticalmente, margens colapsam.
  • Inline Formatting Context — elementos inline fluem horizontalmente como texto.
  • Flex Formatting Context — criado por display: flex. Comportamento totalmente diferente dentro.
  • Grid Formatting Context — criado por display: grid.

Algumas propriedades criam um novo BFC no elemento, o que: contém floats internos, impede colapso de margens com filhos, isola o elemento do fluxo externo. Propriedades que fazem isso:

novo-bfc.css
/* Qualquer uma dessas cria um novo BFC */
.bfc {
  overflow: hidden;       /* ou auto, scroll — qualquer valor != visible */
  display: flow-root;     /* moderno — criado especificamente para isso */
  display: flex;
  display: grid;
  position: absolute;
  position: fixed;
  float: left;            /* floats em si criam BFC */
}

/* display: flow-root é o mais limpo — sem efeitos colaterais */
.wrapper {
  display: flow-root;
  /* Contém floats, impede colapso com pai/filhos.
     Substitui o velho "clearfix" totalmente. */
}
display: flow-root
Antes de flow-root, para "conter" floats ou impedir colapso de margens, usava-se overflow: hidden — que tinha efeito colateral de recortar conteúdo. flow-root existe exatamente para isso, sem efeitos colaterais. Se você precisar criar um BFC limpo, é a escolha certa.

# Valores de display com dois componentes

Na especificação moderna, display tem dois componentes: o outer display (como o elemento se comporta por fora) e o inner display (como organiza filhos por dentro). É possível declarar os dois separados.

display-moderno.css
/* Sintaxe clássica */
.a { display: flex; }
/* Equivalente moderno: */
.a { display: block flex; }

/* Um elemento que aparece inline mas organiza filhos como grid */
.b { display: inline grid; }

/* Block com fluxo normal de filhos */
.c { display: block flow; }

/* Block que contém floats internos e não colapsa */
.d { display: block flow-root; }

/* Nota: navegadores aceitam a sintaxe antiga;
   a nova é mais explícita mas raramente usada ainda. */

Posicionamento

A propriedade position decide como um elemento se relaciona com o fluxo normal da página. Ela controla cinco comportamentos radicalmente diferentes — do elemento preso à tela, ao elemento que "gruda" ao rolar, ao elemento que sai do fluxo e flutua livremente. É aqui que o CSS começa a parecer mágico.

# Os cinco valores

ValorComportamento
staticPadrão — no fluxo normal, ignora top/left/etc
relativeNo fluxo, mas aceita offset (top/left/etc)
absoluteSai do fluxo, posicionado vs. ancestral "positioned"
fixedSai do fluxo, preso ao viewport
stickyHíbrido — relative até atingir um limite, depois fixed

# static e relative

static-relative.css
/* static — o padrão.
   O elemento respeita o fluxo normal. top/left/etc não funcionam. */
.a {
  position: static;
  top: 20px;  /* ignorado */
}

/* relative — no mesmo lugar, mas aceita deslocamento.
   O espaço original É PRESERVADO. */
.b {
  position: relative;
  top: 20px;    /* desloca 20px para baixo */
  left: 10px;   /* desloca 10px para direita */
  /* Nota: o espaço onde ele estaria ainda existe no layout,
     ficando visualmente um "buraco" onde estava. */
}

/* O uso REAL de relative em 2025: ser âncora para absolute filhos */
.card {
  position: relative;
}
.card .badge {
  position: absolute;
  top: 8px;
  right: 8px;
  /* absolute usa o relative mais próximo como referência */
}

# absolute — fora do fluxo

absolute.css
/* absolute — sai completamente do fluxo.
   O espaço não é reservado; outros elementos fluem como se não existisse. */
.badge {
  position: absolute;
  top: 0;
  right: 0;
}

/* ↑ Referência: o ancestral mais próximo que TEM position
   diferente de static. Se não houver, referência é o <html>. */

/* Padrão clássico — contêiner relative + filho absolute */

}

  left: 0;

}

/* Ocupar o ancestral inteiro */
.cobrir-tudo {
  position: absolute;
  top: 0;
  right: 0;
  bottom: 0;
  left: 0;

  inset: 0;             /* shorthand para top/right/bottom/left */
}

fixed dentro de transform
Quando um ancestral tem transform, filter, perspective ou will-change, ele vira referência para elementos position: fixed filhos — em vez do viewport. Isso pega muita gente de surpresa: "meu modal fixed parou de funcionar" é quase sempre porque algum pai ganhou um transform.

# sticky — o melhor dos mundos

sticky é o valor mais novo e provavelmente o mais útil no dia-a-dia. Ele age como relative até o elemento atingir uma posição de scroll, quando passa a agir como fixed.

sticky.css
/* Header que "gruda" ao topo quando passa */
.nav {
  position: sticky;
  top: 0;
  background: white;
  z-index: 10;
}

/* Títulos de seção que grudam enquanto você lê a seção */
h2 {
  position: sticky;
  top: 60px;   /* respeita um nav fixo de 60px acima */
  background: white;
  padding: 8px 0;
}

/* Sidebar "flutuante" que para ao fim do conteúdo */
.layout {
  display: grid;
  grid-template-columns: 1fr 280px;
  gap: 32px;
}
.sidebar {
  position: sticky;
  top: 20px;
  align-self: start;     /* importante em grid! */

}

/* ⚠️ Armadilhas comuns do sticky:
   1. O PAI precisa ter altura maior que o sticky (senão fica onde está).
   2. Nenhum ancestral pode ter overflow: hidden ou auto — quebra sticky.
   3. Em grid, precisa de align-self: start; senão o item estica. */

Capítulo 08

Z-index e stacking contexts

Z-index parece simples: número maior fica por cima. Na prática, é uma das áreas mais confusas do CSS — porque envolve o conceito silencioso dos stacking contexts. Entender isso resolve de uma vez todas aquelas vezes que seu modal com z-index: 9999 mesmo assim ficou atrás de algo.

# A regra básica

basico.css
/* z-index só funciona em elementos POSICIONADOS
   (position: relative, absolute, fixed, sticky) — ou em flex/grid items */

.normal {
  z-index: 10;       /* ignorado — não está posicionado */
}

.posicionado {
  position: relative; /* ou absolute/fixed/sticky */
  z-index: 10;        /* agora funciona */
}

/* Valores aceitos */
.a { z-index: auto; }    /* padrão — herda stacking do pai */
.b { z-index: 0; }       /* nível base do contexto */
.c { z-index: 10; }      /* acima */
.d { z-index: -1; }      /* abaixo (atrás de irmãos) */

/* z-index SEM unidade — nunca use px, %, em. É um número puro. */

# O que é um stacking context

Um stacking context é um "grupo isolado" de empilhamento. Dentro dele, os z-index dos filhos competem entre si — mas o conjunto todo compete no mundo externo como um único bloco.

stacking-context.css
/* Exemplo clássico que confunde todo mundo */

<div class="pai-A">          /* z-index: 1 */
  <div class="modal"></div>  /* z-index: 9999 */
</div>

<div class="pai-B">          /* z-index: 2 */
  <div class="overlay"></div> /* z-index: 1 */
</div>

.pai-A, .pai-B { position: relative; }

/* Quem fica por cima?
   NÃO é o modal com 9999 — é a overlay com 1!

   Por quê: .pai-A e .pai-B são stacking contexts separados.
   Primeiro, o navegador compara pai-A (1) vs pai-B (2) — B vence.
   O z-index 9999 do modal só compete DENTRO de pai-A. */
A regra que todo mundo aprende difícil
Um z-index alto dentro de um stacking context "baixo" nunca vence um z-index baixo em um context "mais alto". Para o modal do exemplo acima aparecer sobre a overlay, ou ele precisa sair do .pai-A (mover no DOM, usar portal), ou o .pai-A precisa ter z-index maior que .pai-B.

# O que cria um stacking context

Várias propriedades criam um novo stacking context — e esse é o pulo do gato. Muitas não são óbvias:

PropriedadeQuando cria stacking context
position + z-indexQualquer valor de z-index exceto auto
position: fixed / stickySempre, mesmo sem z-index
opacityQualquer valor menor que 1
transformQualquer valor != none
filterQualquer valor != none
backdrop-filterQualquer valor != none
perspectiveQualquer valor != none
will-changeQualquer valor que o browser considere "composito"

# A armadilha silenciosa — opacity e transform

armadilha-silenciosa.css
/* Bug clássico: tooltip com z-index sumiu */

.card {
  opacity: 0.95;   /* ← isto já criou um stacking context! */
}
.card .tooltip {
  position: absolute;
  z-index: 9999;   /* agora está preso dentro do context de .card */
}

/* Com outro card ao lado sem opacity, os dois stacking contexts
   competem — o z-index 9999 do tooltip não vale nada "lá fora". */

/* Solução: elevar o .card também */
.card-ativo {
  opacity: 0.95;
  z-index: 10;
  position: relative;
}

# isolation: isolate — contexto de propósito

A propriedade isolation foi criada exatamente para você escolher criar um stacking context sem efeito colateral visual.

isolation.css
/* Quando você quer isolar um componente sem mexer em opacity/transform */
.componente-independente {
  isolation: isolate;
  /* Cria stacking context. Seus z-index internos não vazam
     nem são afetados pelo externo. Nenhum efeito visual. */
}

/* Uso prático: componentes reutilizáveis que gerenciam
   seus próprios layers internos (modal, dropdown, carousel)
   — você garante que eles funcionam igual em qualquer contexto. */

.card {
  isolation: isolate;
  position: relative;
}
.card .badge { position: absolute; z-index: 2; }
.card .overlay { position: absolute; z-index: 1; }
/* Não importa onde .card for inserido, os z-index internos
   sempre funcionam da mesma forma. */
Use isolation em componentes
Quando você escreve um componente reutilizável (React, Vue, Web Component), colocar isolation: isolate no contêiner raiz é uma ótima prática — evita que o z-index interno do componente "vaze" e conflite com o CSS de quem o consome. É invisível, grátis, e torna o componente mais robusto.

# Organizando z-index em projeto real

Em vez de chutar números (10, 100, 9999), defina uma escala semântica via custom properties. Isso torna a hierarquia explícita e evita a corrida armamentista de "mais um zero".

z-index-scale.css
:root {
  --z-base: 0;
  --z-dropdown: 10;
  --z-sticky: 20;
  --z-fixed: 30;
  --z-modal-backdrop: 40;
  --z-modal: 50;
  --z-popover: 60;
  --z-tooltip: 70;
  --z-notification: 80;
  --z-max: 9999;   /* exceções reais, raríssimas */
}

.dropdown { z-index: var(--z-dropdown); }
.nav-sticky { z-index: var(--z-sticky); }
.modal-bg { z-index: var(--z-modal-backdrop); }
.modal { z-index: var(--z-modal); }
.tooltip { z-index: var(--z-tooltip); }

# Top layer — o nível "acima de tudo"

Elementos renderizados no top layer do navegador sobrepõem qualquer z-index, ignoram stacking contexts, e ficam por cima de absolutamente tudo. Há duas formas nativas de chegar lá:

top-layer.html
<!-- 1. <dialog> aberto com showModal() -->
<dialog id="modal">...</dialog>
<script>
  document.getElementById("modal").showModal();
  // Ao chamar showModal(), o dialog vai para o top layer —
  // ignora stacking contexts, sempre fica no topo.
</script>

<!-- 2. Elementos com atributo popover -->
<div popover id="menu">...</div>
<button popovertarget="menu">Abrir</button>
<!-- Idem: popovers vão para o top layer automaticamente -->
Fim da guerra de z-index
Para tudo que precisa ficar "no topo absoluto" (modais, tooltips, menus), use <dialog> ou popover. O top layer é a camada acima de tudo — não há como um z-index vencê-lo. É a solução definitiva para o problema que gerou décadas de hacks.

Unidades

CSS aceita uma variedade impressionante de unidades — px, em, rem, %, vw, vh, ch, dvh, cqw... Escolher a certa é a diferença entre um layout que quebra em qualquer condição nova e um que se adapta silenciosamente a tudo. Cada unidade tem um caso de uso específico.

# Unidades absolutas

absolutas.css
/* Pixels (px) — a unidade mais usada para borders, shadows, radius */
.a {
  border: 1px solid;
  border-radius: 8px;
  box-shadow: 0 4px 12px #0002;
}

/* Nota: "pixel CSS" não é exatamente um pixel físico.
   Em telas Retina, 1px CSS = 2-3 pixels físicos. */

/* pt, pc, cm, mm, in — legados, para impressão.
   Em tela, raramente fazem sentido. Evite. */

# Unidades relativas — a base do CSS moderno

relativas.css
/* rem — relativo ao font-size do <html> (raiz).
   Se html está em 16px (padrão), 1rem = 16px. */
.a {
  font-size: 1rem;      /* = 16px padrão */
  padding: 1.5rem;      /* = 24px */
  max-width: 60rem;     /* = 960px */
}

/* em — relativo ao font-size do ELEMENTO em que é usado */
.b {
  font-size: 20px;
  padding: 1em;         /* = 20px (1x o font-size local) */
  line-height: 1.5em;   /* = 30px */
}

/* em "cascateia" — filho herda e multiplica */
.outer { font-size: 16px; }
.middle { font-size: 1.5em; }  /* 24px (16 * 1.5) */
.inner  { font-size: 1.5em; }  /* 36px! (24 * 1.5) */
/* Por isso rem é quase sempre melhor que em para tamanhos. */

/* % — relativo ao PAI, e ao CONTEXTO (width % é do width do pai,
   mas line-height % é do font-size do elemento...) */
.c { width: 50%; }      /* 50% da largura do pai */
.d { height: 100%; }    /* 100% da altura do pai (se o pai tiver altura!) */
rem como unidade padrão
Para a maioria dos tamanhos (padding, margin, font-size, max-width), prefira rem. Vantagens: (1) usuário pode aumentar o tamanho base no navegador e tudo escala proporcionalmente — acessibilidade; (2) não tem o efeito cascata do em; (3) um único ponto de ajuste global. Use px para coisas que não deveriam escalar: bordas, shadows, radius pequeno.

# Unidades tipográficas

tipograficas.css
/* ch — largura do caractere "0" na fonte atual.
   Perfeito para limitar linhas de texto para legibilidade. */
article {
  max-width: 65ch;      /* ~65 caracteres por linha — ideal para leitura */
}

/* ex — altura do 'x' minúsculo (x-height). Raramente usado. */
.a { padding-top: 0.5ex; }

/* cap — altura de uma letra maiúscula */
.b { line-height: 1cap; }

/* lh — altura de linha do elemento */
.c { margin-top: 1lh; } /* = uma linha de altura */
.d { margin-top: 2lh; }

/* rlh — line-height do elemento raiz */
.e { gap: 1rlh; }

# Unidades de viewport

Relativas ao tamanho da janela visível (viewport). Essenciais para fazer elementos ocuparem frações da tela.

viewport.css
/* Clássicas */
.a { width: 100vw; }   /* 100% da largura do viewport */
.b { height: 100vh; }  /* 100% da altura do viewport */
.c { font-size: 3vw; } /* 3% da largura */

.d {
  width: 50vmin;       /* 50% da MENOR dimensão do viewport */
  height: 50vmax;      /* 50% da MAIOR dimensão */
}

/* ⚠️ Armadilha clássica do 100vh em mobile:
   O viewport em celulares muda quando a barra de endereço
   aparece/desaparece. 100vh usa a ALTURA MÁXIMA — que é maior
   que a tela visível na maior parte do tempo.
   Resultado: sua hero section fica "cortada" quando a barra está visível. */

# Viewport dinâmico — svh, lvh, dvh

O CSS moderno resolveu o problema do 100vh em mobile com três novas unidades:

viewport-dinamico.css
/* svh — Small viewport height
   Altura do viewport ASSUMINDO que barras/UI estão visíveis (menor).
   Elemento nunca é cortado. */
.hero-segura { height: 100svh; }

/* lvh — Large viewport height
   Altura ASSUMINDO que barras estão escondidas (maior). */
.hero-grande { height: 100lvh; }

/* dvh — Dynamic viewport height ⭐
   Ajusta dinamicamente conforme as barras aparecem/somem.
   Na maioria dos casos é o que você quer. */
.hero-dinamica {
  min-height: 100dvh;   /* sempre a altura total REAL */
}

/* Todas têm versões: svw/lvw/dvw, svmin/lvmin/dvmin, etc */
Use dvh em hero sections
Em 2025, a recomendação para hero sections mobile-first é min-height: 100dvh. Suporte atual em iOS 16+, Chrome/Edge 108+, Firefox 101+. Em navegadores antigos, caí de volta para 100vh com aceitação do bug original, ou use min-height: 100vh; min-height: 100dvh (a segunda sobrescreve quando suportada).

# Unidades de container — cqw, cqh, cqi, cqb

Relativas ao container query mais próximo (não ao viewport). Essenciais para componentes realmente responsivos — vamos ver mais no capítulo 15.

container-units.css
.card-container {
  container-type: inline-size;  /* declara que é um container consultável */
}

.card-titulo {
  font-size: 5cqw;    /* 5% da largura do container mais próximo */
  /* Se o card tem 400px, font = 20px. Se tem 200px, font = 10px.
     Funciona independente do tamanho da tela! */
}

/* Unidades disponíveis:
   cqw  — % da largura do container
   cqh  — % da altura do container
   cqi  — % da inline-size (largura em LTR)
   cqb  — % da block-size (altura em LTR)
   cqmin — % do menor (inline ou block)
   cqmax — % do maior */

# calc(), min(), max(), clamp()

O CSS moderno permite expressões matemáticas misturando qualquer unidade. É um recurso poderoso — e a base do "fluid design" (capítulo 24).

calc.css
/* calc() — expressões matemáticas */
.a { width: calc(100% - 32px); }          /* 100% do pai, menos 32px */
.b { height: calc(100vh - 80px); }        /* altura total menos um header */
.c { padding: calc(1rem + 2vw); }         /* mix de unidades */

/* Precisa de espaços em volta de + e - */
.d { width: calc(100%-32px); }   /* ❌ NÃO funciona */
.e { width: calc(100% - 32px); } /* ✅ correto */

/* min() — usa o MENOR dos valores */
.f { width: min(600px, 100%); }
/* No desktop: 600px. Em tela menor que 600px: 100%.
   Substitui elegantemente: width: 600px; max-width: 100%; */

/* max() — usa o MAIOR */
.g { padding: max(1rem, 5vw); }
/* Nunca menor que 16px, mas cresce em telas grandes. */

/* clamp(min, ideal, max) — o mais poderoso */
.h { font-size: clamp(1rem, 2vw + 1rem, 2rem); }
/* No mínimo 16px, no máximo 32px, variando suavemente entre eles.
   É a fórmula mágica de fluid typography. */

Capítulo 10

Cores modernas

Durante 25 anos, CSS teve #hex, rgb() e hsl(). Desde 2023, tem oklch(), color-mix(), relative colors, e gradients em múltiplos color spaces. A diferença prática é enorme: paletas mais harmônicas, temas derivados automaticamente, e cores perceptualmente uniformes. Este é um dos capítulos mais práticos do CSS moderno.

# Notações clássicas

classicas.css
/* Nome — para prototipagem rápida */
.a { color: red; }
.b { color: cornflowerblue; }   /* > 140 nomes nomeados */
.c { color: transparent; }
.d { color: currentColor; }     /* herda a cor atual do texto */

/* Hexadecimal */
.e { color: #f0a830; }          /* 6 dígitos */
.f { color: #fa3; }             /* 3 dígitos — cada um duplica (#ffaa33) */
.g { color: #f0a83080; }        /* 8 dígitos — últimos 2 são alpha (80 = 50%) */
.h { color: #fa38; }            /* 4 dígitos com alpha */

/* RGB — valores 0-255 */
.i { color: rgb(240, 168, 48); }
.j { color: rgb(240 168 48);      }    /* sintaxe moderna — sem vírgulas */
.k { color: rgb(240 168 48 / 0.5); }   /* alpha com barra */

/* HSL — matiz, saturação, luminosidade */
.l { color: hsl(35, 87%, 56%); }
.m { color: hsl(35 87% 56% / 0.8); }   /* sintaxe moderna */
A sintaxe nova
As funções de cor modernas (Level 4) usam espaços entre os argumentos e barra para alpha. rgb(0 0 0 / 0.5) em vez de rgba(0, 0, 0, 0.5). A sintaxe antiga continua funcionando, mas a nova é mais consistente — todas as funções modernas usam o mesmo padrão.

# Por que HSL não é suficiente

HSL parece mais intuitivo que hex/rgb — "saturação 50%, luminosidade 60%". Mas tem um defeito grave: não é perceptualmente uniforme. Amarelo com L:50% parece muito mais claro que azul com L:50%, mesmo sendo "o mesmo valor".

hsl-problema.css
/* Estas duas cores têm MESMA "luminosidade" em HSL,
   mas parecem ter brilhos muito diferentes: */
.a { background: hsl(60, 100%, 50%); }   /* amarelo — quase ofuscante */
.b { background: hsl(240, 100%, 50%); }  /* azul — muito escuro */

/* Isso dificulta criar paletas harmônicas ("todos os botões
   com a mesma escuridão, só mudando matiz"). */

# oklch() — a função moderna

oklch() usa o espaço de cor OKLCH, que é perceptualmente uniforme. Mesma lightness = mesma sensação de brilho, independente da matiz.

oklch.css
/* oklch(lightness chroma hue) */

.accent { color: oklch(75% 0.15 60); }
/* L: 75% — brilho percebido (0% preto, 100% branco)
   C: 0.15 — saturação (0 = cinza, ~0.4 = máximo)
   H: 60   — matiz em graus (0=vermelho, 120=verde, 240=azul) */

/* Com alpha */
.a { background: oklch(75% 0.15 60 / 0.5); }

/* Vantagem prática — paleta harmônica:
   mesmo L e C, só mudando o H */
:root {
  --red:    oklch(65% 0.18 25);
  --orange: oklch(65% 0.18 60);
  --yellow: oklch(65% 0.18 100);
  --green:  oklch(65% 0.18 140);
  --cyan:   oklch(65% 0.18 200);
  --blue:   oklch(65% 0.18 260);
  --purple: oklch(65% 0.18 300);
}
/* As sete cores têm brilho percebido IGUAL.
   Isso não é possível de obter com HSL. */
Use oklch em projetos novos
Para projetos começando hoje, oklch é a escolha padrão. Suporte em todos os navegadores modernos desde 2023. Ferramentas como OKLCH Color Picker (oklch.com) ajudam a visualizar. A diferença qualitativa em design systems é grande: paletas que parecem "feitas por um designer" vêm quase de graça.

# Gamas de cores — P3 e Rec2020

Monitores modernos (telas OLED, M1+, iPhone 12+) mostram cores mais saturadas que o espaço sRGB padrão. oklch permite acessar esse "gamut" estendido.

gamuts.css
/* Verde mais intenso que sRGB permite */
.verde-neon { color: oklch(80% 0.3 140); }

/* Função color() — especifica espaço de cor diretamente */
.a { color: color(display-p3 0 1 0); }      /* P3 verde puro */
.b { color: color(rec2020 1 0 0); }         /* Rec2020 vermelho */
.c { color: color(srgb 0 0 1); }            /* sRGB azul */

/* Fallback para navegadores antigos */
.elem {
  color: #00ff00;                /* fallback sRGB */
  color: color(display-p3 0 1 0); /* sobrescreve se suportado */
}

# color-mix() — misturar duas cores

Um recurso extremamente prático. Permite criar variantes de uma cor (mais clara, mais escura, com tinta de outra cor) sem calculadora de cor externa.

color-mix.css
/* color-mix(in <color-space>, <cor1> <%>, <cor2> <%>) */

.a {
  background: color-mix(in oklch, red 50%, blue 50%);
  /* Mistura 50/50 de red e blue — resultado é um roxo */
}

/* Percentuais omitidos = 50/50 */
.b { background: color-mix(in oklch, red, blue); }

/* Proporção diferente */
.c { background: color-mix(in oklch, #f0a830 70%, white); }
/* 70% laranja + 30% branco = laranja clarinho */

/* Uso prático — gerar estados a partir de uma cor base */
:root {
  --primary: oklch(65% 0.18 60);
  --primary-hover: color-mix(in oklch, var(--primary), black 15%);
  --primary-soft:  color-mix(in oklch, var(--primary) 12%, transparent);
  --primary-light: color-mix(in oklch, var(--primary), white 40%);
}

.btn {
  background: var(--primary);
}
.btn:hover {
  background: var(--primary-hover);  /* 15% mais escuro automaticamente */
}
.badge {
  background: var(--primary-soft);   /* versão transparente suave */
}

# Relative colors — derivar de uma cor

Outra adição recente e poderosa: a sintaxe "from", que decompõe uma cor nos seus canais e permite modificá-los.

relative-colors.css
/* Sintaxe: cor(from <origem> <canal1> <canal2> <canal3> [/ <alpha>]) */

/* Tornar cor X com alpha 30% */
.a { background: oklch(from var(--primary) l c h / 0.3); }

/* Aumentar lightness em 10% */
.b { background: oklch(from var(--primary) calc(l + 0.1) c h); }

/* Rotacionar matiz em 180° (cor complementar) */
.c { background: oklch(from var(--primary) l c calc(h + 180)); }

/* Remover saturação (cinza com mesma luminosidade) */
.d { background: oklch(from var(--primary) l 0 h); }

/* Versão "destaque" de qualquer cor base */
.e { background: oklch(from var(--cor) calc(l * 0.85) c h); }

/* Uso real — design tokens derivados */
:root {
  --brand: oklch(60% 0.2 30);
  --brand-light: oklch(from var(--brand) calc(l + 0.2) c h);
  --brand-dark:  oklch(from var(--brand) calc(l - 0.2) c h);
  --brand-fade:  oklch(from var(--brand) l c h / 0.1);
}

# light-dark() — tema automático

Uma função pensada para dark mode: recebe dois valores e escolhe automaticamente com base em color-scheme.

light-dark.css
:root {
  color-scheme: light dark;  /* essencial — declara suporte aos dois */

  --bg:   light-dark(white, #111);
  --text: light-dark(#111, #eee);
  --border: light-dark(#ddd, #333);
}

body {
  background: var(--bg);
  color: var(--text);
  /* Em modo claro: branco/preto. Em modo escuro: preto/branco. */
}

/* ⚠️ light-dark() só funciona quando color-scheme inclui ambos.
   Override manual:
     :root { color-scheme: light; }  → sempre claro
     :root { color-scheme: dark; }   → sempre escuro
     :root { color-scheme: light dark; } → respeita preferência */
Dark mode sem media query
Com light-dark(), você não precisa mais escrever @media (prefers-color-scheme: dark) para cada cor. Declara os dois valores na mesma linha, e o navegador escolhe. É mais conciso, mais legível, e fica ainda melhor com toggle manual (veremos no cap. 23).

# Gradientes em color spaces modernos

Gradientes tradicionais interpolam em sRGB, o que gera "zonas mortas" de cinza no meio (ex: gradiente azul→amarelo passa por cinza). Agora é possível interpolar em oklch.

gradientes-modernos.css
/* Gradiente padrão — interpola em sRGB (pode ter "zona morta") */
.a {
  background: linear-gradient(to right, blue, yellow);
  /* Passa por um cinza acinzentado no meio */
}

/* Interpolação em oklch — muito mais vibrante */
.b {
  background: linear-gradient(in oklch to right, blue, yellow);
  /* Passa por cores vivas (roxo, vermelho, laranja) */
}

/* Interpolação em HSL longa (pega o outro lado do círculo) */
.c {
  background: linear-gradient(in hsl longer hue to right, red, orange);
  /* Percorre TODO o círculo de matizes antes de chegar */
}

/* Espaços disponíveis: srgb (padrão), oklch, oklab, hsl, lch, lab, p3 */

# Resumo prático

Guia de bolso
Para tokens de cor do design system → oklch().
Para variantes automáticas (hover, soft) → color-mix().
Para dark mode sem media query → light-dark().
Para gradientes vivos → in oklch ou in oklab.
Para manipulação programática de cor → relative colors (oklch(from ...)).
Para backwards compat em projetos legados → declare #hex primeiro, depois oklch.

Custom properties

Custom properties — conhecidas como "variáveis CSS" — parecem uma simples conveniência, mas são muito mais: são valores dinâmicos que vivem na cascata, acessíveis via JavaScript, reatividade gratuita, e a base de tema e design systems modernos. Diferente das variáveis do Sass (que são compiladas), essas existem em runtime e respondem a tudo.

# Sintaxe básica

basico.css
/* Declaração — qualquer nome começando com -- */
:root {
  --cor-primaria: #f0a830;
  --espaco: 8px;
  --fonte-base: "Inter", sans-serif;
}

/* Uso — função var() */
.btn {
  background: var(--cor-primaria);
  padding: var(--espaco) calc(var(--espaco) * 2);
  font-family: var(--fonte-base);
}

/* Fallback — segundo argumento de var() */
.a {
  color: var(--cor-que-talvez-nao-exista, black);
  padding: var(--espaco, 16px);  /* se --espaco não existir, usa 16px */
}

/* Fallback com outra variável */
.b {
  background: var(--cor-custom, var(--cor-primaria, gray));
  /* Tenta custom → primaria → gray */
}

# Escopo — elas participam da cascata

Essa é a diferença fundamental em relação às variáveis de preprocessadores: custom properties são herdadas, seguem a cascata, e podem ser sobrescritas em qualquer nível.

escopo.css
:root {
  --cor: blue;
}

.tema-escuro {
  --cor: white;  /* sobrescreve para os descendentes */
}

.alerta {
  --cor: red;    /* local — só dentro de .alerta e filhos */
}

p { color: var(--cor); }

<p>Azul (do :root)</p>
<div class="tema-escuro">
  <p>Branco (herda do .tema-escuro)</p>
</div>
<div class="alerta">
  <p>Vermelho (herda do .alerta)</p>
</div>
Isso é reatividade
Mudar uma custom property em runtime (via JS ou CSS) propaga automaticamente para todo filho que a usa. Toggle de tema, estados de hover, animações dinâmicas — tudo sem tocar em cada elemento individualmente. É a mesma ideia de reatividade que você tem em React, mas embutida no CSS.

# Custom properties em JavaScript

js-integration.js
// Ler uma variável (precisa estar computada)
const styles = getComputedStyle(document.documentElement);
const cor = styles.getPropertyValue("--cor-primaria").trim();

// Escrever — dispara atualização em toda a página
document.documentElement.style.setProperty("--cor-primaria", "#5eead4");

// Remover
document.documentElement.style.removeProperty("--cor-primaria");

// Em qualquer elemento (não precisa ser :root)
card.style.setProperty("--destaque", "orange");

// Uso real — toggle de tema
function alternarTema() {
  const atual = document.documentElement.getAttribute("data-tema");
  const novo = atual === "escuro" ? "claro" : "escuro";
  document.documentElement.setAttribute("data-tema", novo);
}

# @property — custom properties tipadas

Por padrão, toda custom property é tratada como "string" pelo navegador — ele não sabe se é número, cor, ou tamanho. @property muda isso, permitindo animar variáveis e validar tipos.

at-property.css
/* Sem @property, você NÃO consegue animar custom properties de cor */
:root { --cor: red; }
.elem {
  color: var(--cor);
  transition: --cor 300ms;  /* navegador não sabe interpolar — não anima */
}

/* Com @property — declara o tipo, navegador ganha o superpoder */
@property --cor {
  syntax: "<color>";          /* tipo */
  inherits: true;              /* herda para filhos? */
  initial-value: red;          /* valor inicial (obrigatório) */
}

.elem {
  color: var(--cor);
  transition: --cor 300ms;    /* AGORA anima suavemente */
}
.elem:hover { --cor: blue; }  /* transição fluida de red → blue */

/* Syntaxes aceitas: <color>, <length>, <number>, <percentage>,
   <angle>, <time>, <url>, <image>, ... ou '*' (genérico) */

/* Em JS também pode */
CSS.registerProperty({
  name: "--angulo",
  syntax: "<angle>",
  inherits: false,
  initialValue: "0deg",
});
Animações impossíveis ficam triviais
Com @property, você pode animar gradientes, cores derivadas, ângulos de transform com variáveis — coisas que antes exigiam truques com pseudo-elementos ou JavaScript. É um dos recursos mais "mágicos" do CSS moderno, ainda pouco usado mesmo por devs experientes.

# Variáveis vs Sass — quando cada um?

Em 2025
Para tokens de design, temas, valores que mudam em runtime, animações → custom properties. Para mixins, loops, funções (que custom properties não resolvem) → Sass ainda é útil, mas raramente necessário em projetos novos. Muitos times largaram Sass completamente: o CSS moderno tem nesting, layers, scope, e @property — cobre 95% do que fazia Sass essencial.
Capítulo 12

Flexbox

Flexbox é o sistema de layout unidimensional do CSS — organiza elementos em uma linha ou coluna, com controle fino sobre espaçamento, alinhamento e como eles crescem/encolhem. É provavelmente a ferramenta que você vai usar mais no dia-a-dia. Grid resolve layouts 2D complexos; Flexbox resolve 90% de tudo o resto.

# O modelo mental

Flexbox trabalha com dois conceitos: o container (pai) e os itens (filhos diretos). Tudo no flexbox acontece em relação a dois eixos:

  • Main axis (eixo principal) — a direção em que os itens fluem. Horizontal por padrão.
  • Cross axis (eixo cruzado) — perpendicular ao principal. Vertical por padrão.
eixos.css
/* flex-direction controla qual eixo é o principal */
.a { flex-direction: row; }            /* padrão — main horizontal (esquerda → direita) */
.b { flex-direction: row-reverse; }    /* main horizontal (direita → esquerda) */
.c { flex-direction: column; }         /* main vertical (cima → baixo) */
.d { flex-direction: column-reverse; } /* main vertical (baixo → cima) */

/* DICA: a maioria das propriedades flex se refere aos eixos
   (main/cross), não a horizontal/vertical. Isso é o que faz
   o mesmo código funcionar em row e column sem ajustes. */

# Ativando flex

ativar.css
/* Basta isto */
.container {
  display: flex;
  /* Filhos diretos automaticamente viram "flex items". */
}

/* Inline flex — flexbox que se comporta como inline por fora */
.inline-container { display: inline-flex; }

/* Flex com gap entre itens (nativo — use sempre, não margin) */
.container {
  display: flex;
  gap: 16px;          /* espaço entre todos os itens */
  gap: 16px 8px;      /* row-gap | column-gap */
  row-gap: 16px;
  column-gap: 8px;
}
Prefira gap a margin
Antes de gap ter suporte em flexbox (chegou em 2021 no Safari), a convenção era margin-right + :last-child { margin-right: 0 } — feio e frágil. Hoje, gap é universal em flex e grid. Nunca mais use margins entre itens de flex.

# flex-grow, flex-shrink, flex-basis

grow-shrink-basis.css
/* Os três pilares de como um item se comporta */

.item {
  flex-grow: 0;        /* padrão — não cresce além do necessário */
  flex-shrink: 1;      /* padrão — pode encolher quando falta espaço */
  flex-basis: auto;    /* padrão — tamanho inicial vem de width/height/conteúdo */
}

/* flex-grow — se sobra espaço, quanto este item "puxa" */
.cresce-igual    { flex-grow: 1; }   /* ocupa o espaço disponível */
.cresce-dobro    { flex-grow: 2; }   /* ocupa o DOBRO de quem tem grow: 1 */

/* flex-shrink — se falta espaço, quanto este item encolhe */
.nunca-encolhe   { flex-shrink: 0; } /* mantém tamanho mesmo que corte a linha */
.encolhe-dobro   { flex-shrink: 2; } /* encolhe o dobro dos outros */

/* flex-basis — tamanho IDEAL antes de grow/shrink decidirem */
.a { flex-basis: 300px; }   /* quer ser 300px */
.b { flex-basis: 50%; }     /* 50% do container */
.c { flex-basis: auto; }    /* usa width/height (ou conteúdo) */
.d { flex-basis: content; } /* sempre baseado em conteúdo, ignora width */

# A shorthand flex

flex-shorthand.css
/* flex: <grow> <shrink> <basis> */

.a { flex: 1; }            /* = flex: 1 1 0% — cresce, encolhe, basis 0 */
.b { flex: 0 1 auto; }     /* padrão — igual a "flex: initial" */
.c { flex: 1 1 auto; }     /* = flex: auto — flexível nos dois lados */
.d { flex: 0 0 auto; }     /* = flex: none — tamanho fixo do conteúdo */
.e { flex: 1 1 300px; }    /* flexível a partir de 300px de base */

/* Casos extremamente comuns */
.cresce-ocupando-sobra      { flex: 1; }
.sidebar-fixa-300           { flex: 0 0 300px; }
.encolhe-apenas-quando-fica-apertado { flex: 0 1 auto; }

/* Diferença sutil entre flex: 1 e flex: auto
   - flex: 1    → basis 0, todos dividem espaço IGUALMENTE
   - flex: auto → basis auto (conteúdo), divisão proporcional AO CONTEÚDO */
O padrão das dez mil vezes
A combinação mais usada de todo o flexbox: um item flex: 0 0 auto (tamanho fixo, sem flex) para sidebars/logos/botões, e um item flex: 1 para o conteúdo que "ocupa o resto". Memorize esses dois e você resolve a maioria dos layouts unidimensionais.

# order — reordenar visualmente

order.css
/* Padrão: ordem do DOM (cada item com order: 0) */

order só muda o visual
A ordem de tab, leitura por screen readers e scroll seguem o DOM, não o order. Se você usa order para mudar a ordem visual de elementos interativos, está quebrando acessibilidade — o usuário de teclado vai encontrar os elementos em ordem "aleatória" visualmente. Reordene no HTML sempre que possível; use order apenas quando o HTML precisa manter uma ordem semântica diferente do layout.

# A armadilha do min-width: 0

min-width.css
/* Um bug confuso: texto longo em flex item "transborda" e estica o container */

.card {
  display: flex;
  gap: 1rem;
}
.card .info {
  flex: 1;
  /* Bug: se o conteúdo tem uma palavra/URL muito longa,
     o item cresce além do flex: 1 e estoura o layout.
     Por que: o min-width padrão de um flex item é 'auto',
     que se baseia no menor conteúdo "intrincável". */
}

/* ✅ Correção */
.card .info {
  flex: 1;
  min-width: 0;       /* permite encolher até 0 se necessário */
  overflow: hidden;   /* ou esconde o que transbordar */
}

/* Mesmo problema em column: use min-height: 0 */
O truque mais importante do flexbox
Sempre que um flex item tem conteúdo que pode ser muito largo (texto, imagens sem dimensão, inputs), adicione min-width: 0 (ou min-height: 0 se for column). Isso resolve silenciosamente 80% dos bugs de "layout misterioso que estoura". É um dos primeiros passos quando algo não está encolhendo como deveria.

Grid

Grid é o sistema de layout bidimensional do CSS — linhas E colunas simultaneamente. Flexbox é excelente para uma dimensão; Grid resolve tudo que precisa de estrutura em "grade", de dashboards complexos a simples "3 colunas iguais que viram 1 em mobile". É uma das ferramentas mais poderosas da web moderna, e a menos usada em seu potencial.

# Flexbox vs Grid — quando usar cada um

SituaçãoUse
Navbar com logo e linksFlexbox
Lista de itens que quebra em várias linhasFlexbox (wrap)
Layout com sidebar fixa + conteúdoGrid
Dashboard com regiões nomeadasGrid
Galeria responsiva com colunas de tamanho igualGrid
Formulário em duas colunasGrid
Cards que crescem horizontalmenteFlexbox
"Sobrepor elementos na mesma área"Grid (1×1)
A regra do "1D vs 2D"
Pergunte-se: "estou organizando em UMA dimensão (linha OU coluna) ou em DUAS (linhas E colunas simultaneamente)?" Se é uma, Flexbox. Se é duas, Grid. Essa única decisão resolve 90% da escolha. Os dois se complementam — é comum ter Grid no layout macro e Flexbox dentro de cada célula.

# O modelo mental do Grid

Grid usa vocabulário específico que vale aprender:

  • Grid container — o elemento pai com display: grid.
  • Grid item — cada filho direto do container.
  • Track — uma linha ou coluna inteira.
  • Cell — a interseção de uma linha com uma coluna.
  • Area — um grupo retangular de células.
  • Grid lines — as "linhas invisíveis" que delimitam tracks. Numeradas a partir de 1.
  • Gap — espaço entre tracks.

# Ativando e definindo tracks

ativar-grid.css
/* Declaração básica */
.container {
  display: grid;
  grid-template-columns: 200px 1fr 200px;   /* 3 colunas */
  grid-template-rows: auto 1fr auto;        /* 3 linhas */
  gap: 1rem;
}

/* A unidade fr — "fractional unit"
   Distribui espaço DISPONÍVEL após o resto estar alocado.
   Similar a flex: 1, mas só no contexto de grid tracks. */

.layout-3col {
  display: grid;
  grid-template-columns: 1fr 1fr 1fr;   /* 3 colunas iguais */
  /* ou o atalho: */
  grid-template-columns: repeat(3, 1fr);
}

.com-sidebar {
  display: grid;
  grid-template-columns: 240px 1fr;     /* sidebar fixa + conteúdo flexível */
}

/* Mix de unidades — cada track pode ter comportamento diferente */
.misto {
  grid-template-columns: 200px 1fr 2fr auto;
  /* Primeira: 200px fixos
     Segunda: pega 1 parte do que sobra
     Terceira: pega 2 partes
     Quarta: só o necessário para o conteúdo */
}

# repeat(), minmax() e auto-fit

repeat-minmax.css
/* repeat() — replica tracks */
.a { grid-template-columns: repeat(12, 1fr); }          /* 12 colunas iguais */
.b { grid-template-columns: repeat(3, 100px 200px); }   /* padrão: 100 200 100 200 100 200 */

/* minmax(min, max) — track com limites */
.c {
  grid-template-columns: minmax(200px, 1fr) 300px;
  /* Primeira coluna: nunca menor que 200px, pode crescer flexivelmente */
}

/* auto-fit — encaixa quantos itens couberem, esticando para preencher */
.galeria-fit {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(250px, 1fr));
  gap: 1rem;
  /* Leia: "quantas colunas couberem com mínimo 250px, esticadas até 1fr".
     Em tela larga: muitas colunas. Tela estreita: uma coluna.
     Grátis e sem media query. */
}

/* auto-fill — igual mas mantém "colunas vazias" reservadas */
.galeria-fill {
  grid-template-columns: repeat(auto-fill, minmax(250px, 1fr));
  /* Diferença prática: se tem 2 itens em largura para 5 colunas,
     auto-fit → 2 itens ESTICADOS ocupando 50% cada
     auto-fill → 2 itens do tamanho mínimo, 3 "slots" vazios */
}
Uma linha que substitui 5 media queries
repeat(auto-fit, minmax(250px, 1fr)) é provavelmente a declaração mais útil do Grid — substitui vários breakpoints de media query e ainda funciona em container queries. Se você estava acostumado a fazer "4 colunas → 3 → 2 → 1" por tamanho de tela, essa linha faz tudo automaticamente.

# Grid implícito — rows e auto-flow

   O Grid cria linhas extras automaticamente — são "linhas implícitas". */

  display: grid;
  grid-template-columns: repeat(3, 1fr);

  grid-auto-flow: row;           /* padrão — preenche linha por linha */
  grid-auto-flow: column;        /* preenche coluna por coluna */
  grid-auto-flow: row dense;     /* tenta preencher "buracos" (melhor empacotamento) */
}

/* Útil quando itens têm tamanhos diferentes */

  display: grid;

# Posicionando itens — grid-column e grid-row

posicionar.css
/* Cada item pode especificar onde fica usando as grid lines */

.container {
  display: grid;
  grid-template-columns: repeat(4, 1fr);
  grid-template-rows: repeat(3, 100px);
}

/* Linhas são NUMERADAS a partir de 1.
   Em 4 colunas: lines 1, 2, 3, 4, 5 (5 é o "fim do fim") */

.destaque {
  grid-column: 1 / 3;     /* começa na linha 1, termina na 3 (ocupa colunas 1 e 2) */
  grid-row: 1 / 2;         /* primeira linha apenas */
}

/* Atalho com span */
.largo {
  grid-column: 1 / span 2;  /* começa em 1, ocupa 2 colunas */
  /* Mesma coisa que grid-column: 1 / 3 */
}

.ocupa-tudo {
  grid-column: 1 / -1;      /* primeira coluna ao ÚLTIMO limite (ocupa a linha inteira) */
}

/* Atalho para os dois eixos */
.item {
  grid-area: 1 / 1 / 3 / 3; /* row-start / column-start / row-end / column-end */
}

# Grid areas nomeadas

Um dos recursos mais elegantes do Grid: dar nomes a regiões e "desenhar" o layout em ASCII.

areas.css
.app {
  display: grid;
  grid-template-columns: 200px 1fr 200px;
  grid-template-rows: auto 1fr auto;
  grid-template-areas:
    "header header header"
    "nav    main   aside"
    "footer footer footer";
  gap: 1rem;
  min-height: 100dvh;
}

header  { grid-area: header; }
nav     { grid-area: nav; }
main    { grid-area: main; }
aside   { grid-area: aside; }
footer  { grid-area: footer; }

/* Reorganizar em mobile — só mudando template-areas */
@media (max-width: 768px) {
  .app {
    grid-template-columns: 1fr;
    grid-template-areas:
      "header"
      "nav"
      "main"
      "aside"
      "footer";
  }
}

/* Usar "." para células vazias */
.grid-especial {
  grid-template-areas:
    "a . b"
    ". c ."
    "d . e";
}
O layout em ASCII
grid-template-areas é fantástico para layouts complexos porque o código-fonte parece com o layout final. Designers e devs conseguem ler o CSS direto sem precisar renderizar. Para responsividade, só redesenhar o "mapa" em cada breakpoint — muito mais legível que recalcular grid-column em cada item.

# Alinhamento em Grid

Grid tem um sistema de alinhamento mais rico que Flexbox — você controla alinhamento do conteúdo de cada célula, do item dentro de sua área, e da grade inteira dentro do container.

alinhamento.css
/* No container — afeta TODOS os itens */
.container {
  justify-items: start | end | center | stretch;   /* horizontal, dentro da célula */
  align-items: start | end | center | stretch;     /* vertical, dentro da célula */
  place-items: center;                             /* atalho para ambos */

  /* Alinhamento do GRID INTEIRO dentro do container
     (quando ele é menor que o container) */
  justify-content: start | end | center | space-between | space-around;
  align-content: start | end | center | space-between;
  place-content: center;
}

/* Em itens específicos — sobrescreve o container */
.item-especial {
  justify-self: end;   /* este item alinha à direita da sua célula */
  align-self: center;
  place-self: center end;
}

/* Centralizar QUALQUER coisa com Grid (mais curto que flexbox!) */
.centraliza {
  display: grid;
  place-items: center;
}

# Sobrepor elementos — grid como stacking tool

}

  grid-area: 1 / 1;    /* todos os filhos na mesma célula */

  </div>
</section>

/* Sem position: absolute, sem z-index, sem quebrar o fluxo.

Capítulo 14

Subgrid e layouts complexos

Grid resolve layouts 2D — mas até 2023 tinha um limite: os tracks só se aplicavam ao próprio contêiner, não atravessavam elementos aninhados. Subgrid quebra essa barreira, permitindo que um grid filho "herde" os tracks do pai. Parece detalhe, mas é o que permite cards alinhados verticalmente, layouts de revista, e padrões que antes exigiam JavaScript.

# O problema que subgrid resolve

problema.css
/* Cenário clássico: lista de cards, cada card com header/body/footer.
   Você quer que TODOS os headers tenham a mesma altura entre si,
   mesmo que o conteúdo varie. */

.cards {
  display: grid;
  grid-template-columns: repeat(3, 1fr);
  gap: 1rem;
}

.card {
  display: grid;
  grid-template-rows: auto 1fr auto;   /* header auto, body cresce, footer auto */
}

/* ❌ Problema: cada card calcula suas próprias rows.
   Se um card tem header de 2 linhas e outro de 1 linha,
   os headers ficam com alturas diferentes entre si.
   O mesmo vale para footers. */

# A solução — subgrid

subgrid-solucao.css
.cards {
  display: grid;
  grid-template-columns: repeat(3, 1fr);
  grid-template-rows: auto 1fr auto;   /* rows definidas no PAI */
  gap: 1rem;
}

.card {
  display: grid;
  grid-row: span 3;          /* cada card ocupa 3 rows do pai */
  grid-template-rows: subgrid; /* HERDA as rows do pai! */
}

/* Agora: todos os headers compartilham a primeira row do pai.
   A altura é determinada pelo HEADER MAIS ALTO de TODOS os cards.
   Mesmo para os footers. Alinhamento automático. */

<div class="cards">
  <article class="card">
    <header>Título de 2 linhas que quebra assim</header>
    <div class="body">Texto...</div>
    <footer>Ação</footer>
  </article>
  <article class="card">
    <header>Título curto</header>  <!-- ocupa a mesma altura do título longo -->
    <div class="body">Texto...</div>
    <footer>Ação</footer>
  </article>
</div>
Suporte em 2025

# O casamento subgrid + container queries

Para o próximo capítulo, um gancho: subgrid alinha elementos através de hierarquias. Container queries (cap. 15) fazem componentes responderem ao próprio tamanho, não ao viewport. Juntos, permitem componentes que se alinham perfeitamente com vizinhos e se adaptam a qualquer contêiner onde forem colocados — o sonho do design system verdadeiramente modular.

Capítulo 15

Container queries

Durante 15 anos, o CSS perguntou "qual é o tamanho da tela?" — com @media. Container queries mudam a pergunta para "qual é o tamanho do meu contêiner?". A diferença é sutil na descrição, mas revolucionária na prática: componentes que se adaptam a onde estão, não ao tamanho da janela. É a peça que faltava para design systems realmente componentizados.

# O problema das media queries

problema-media.css
/* Um card que muda de layout conforme a tela */
.card {
  display: flex;
  flex-direction: column;   /* mobile: coluna */
}

@media (min-width: 768px) {
  .card {
    flex-direction: row;    /* desktop: linha */
  }
}

/* ❌ Problema: e se o card estiver na sidebar de 300px em desktop?
   A viewport é "grande" (1400px) mas o card tem só 300px — não cabe em row.
   O layout quebra porque decidimos pelo tamanho ERRADO. */

Media queries perguntam sobre a tela. Mas o componente não se importa com a tela — se importa com o espaço que ele tem. Container queries respondem a isso.

# Sintaxe básica

basico.css
/* Passo 1: declarar um contêiner consultável */
.card-container {
  container-type: inline-size;   /* "responde a mudanças de largura" */
  /* Ou shorthand com nome: */
  container: card / inline-size;
}

/* Passo 2: consultar o TAMANHO DO CONTAINER mais próximo */
.card {
  display: flex;
  flex-direction: column;
}

@container (min-width: 400px) {
  .card {
    flex-direction: row;    /* agora responde ao container, não à tela */
  }
}

<div class="card-container">
  <div class="card">...</div>
</div>
/* Onde quer que esse container esteja — sidebar, modal, grid item —
   ele aplica as regras com base no tamanho dele mesmo. */

# Os valores de container-type

ValorO que faz
normalPadrão — não é contêiner consultável (mas herda style queries)
inline-sizeConsultável por largura (o mais usado)
sizeConsultável por largura E altura (custa mais em performance)
Efeito colateral de container-type
Declarar container-type: size faz o elemento ignorar a altura natural do conteúdo — fica com altura 0 até você definir explicitamente. Por isso inline-size (só largura) é o padrão recomendado: resolve 95% dos casos sem efeitos colaterais.

# Nomeando contêineres

nomes.css
/* Quando você tem contêineres aninhados, pode nomear para
   escolher qual consultar */

  container-name: sidebar;
  container-type: inline-size;
  /* ou junto: */
  container: sidebar / inline-size;
}

.main {
  container: main / inline-size;
}

/* Query genérica — usa o contêiner mais próximo */
@container (min-width: 400px) { ... }

/* Query específica — usa o contêiner com aquele NOME */

}

@container main (min-width: 800px) {
  .card { padding: 2rem; }
}

     Um card de 500px → título ~20px

   cqh — % da altura

   cqmax — maior dos dois */

O fim de componentes "quebráveis"
Você já teve que descobrir onde seu componente foi colocado para saber qual CSS aplicar? Container queries acabam com isso. Um componente bem escrito carrega suas próprias decisões de layout — é reutilizável de verdade, funciona em qualquer contexto do app. É provavelmente o maior avanço prático do CSS moderno.

# Style queries — consultando custom properties

Além de tamanho, é possível consultar valores de custom properties. Isso permite temas encaixáveis e variações contextuais.

# Combinando container queries + grid

  gap: 1rem;

  container-type: inline-size;   /* cada item vira contêiner */

# Limitações e armadilhas

  • Você NÃO pode consultar o próprio elemento — o @container sempre consulta um ANCESTRAL. O elemento com container-type não se consulta.
  • container-type cria formatting context — pode ter efeitos sutis em layout.
  • container-type: size (não inline-size) afeta altura — cuidado com contêineres que dependem de altura natural.
  • Performance — container queries são eficientes, mas contêineres com size e muitos filhos podem ser mais pesados que inline-size.

# Quando usar container vs media query

Regra prática
Media queries para decisões de app: mostrar ou esconder sidebar, layout macro da página, tamanho de fonte global. O viewport inteiro importa.

Container queries para decisões de componente: layout interno de um card, tamanho de fonte de um título dentro de um bloco, direção de um media object. O espaço local importa.

Em projetos modernos, ~80% do CSS responsivo deveria ser container queries; ~20% media queries para estrutura macro.

# Suporte em 2025

Container queries têm suporte completo em todos os navegadores modernos desde outubro de 2023 (Firefox foi o último). É seguro usar em produção sem polyfill. O suporte a style queries e unidades de container também está estável em 2025.

Capítulo 16

Fontes e tipografia

A tipografia é 90% do design de qualquer site. Fontes bem escolhidas, bem carregadas e bem controladas transformam uma página em uma experiência legível. Este capítulo cobre desde carregar uma fonte personalizada até os recursos modernos de fontes variáveis, font-display e OpenType features.

# Font stack — a pirâmide de fallbacks

font-stack.css
/* Se uma fonte não carrega ou não existe, o navegador usa a próxima */

body {
  font-family: "Inter", -apple-system, BlinkMacSystemFont,
               "Segoe UI", Roboto, "Helvetica Neue", Arial, sans-serif;
}

/* Convenção: sempre termine com uma FAMÍLIA GENÉRICA:
   serif | sans-serif | monospace | cursive | fantasy | system-ui | ui-serif | ui-monospace */

/* Stack "system UI" — usa a fonte padrão do SO, super rápido */
.system {
  font-family: system-ui, -apple-system, BlinkMacSystemFont, sans-serif;
}

/* Stack monospace moderno */
.code {
  font-family: ui-monospace, "SF Mono", "JetBrains Mono",
               Consolas, "Liberation Mono", monospace;
}
system-ui é subestimada
system-ui usa a fonte nativa do sistema (San Francisco no Mac/iOS, Segoe UI no Windows, Roboto no Android). Zero requisições, zero espera, visual coerente com o SO. Para 80% dos projetos, começar com system-ui é melhor que carregar uma web font — se você precisar de identidade tipográfica específica, aí vale o custo.

# font-display — controlando a renderização

Enquanto a fonte custom baixa, o que o usuário vê? font-display decide entre "texto invisível até a fonte chegar" ou "texto com fonte fallback, troca depois". Cada opção tem trade-offs.

ValorComportamento
autoDecisão do navegador (geralmente block)
blockInvisível por ~3s, depois fallback, troca quando carregar. FOIT.
swapFallback imediato, troca quando carregar. FOUT.
fallbackInvisível por 100ms, fallback por 3s, depois congela.
optionalInvisível 100ms, fallback, nunca troca. Só usa se já cacheada.
Qual usar?
Para body text e UI, use swap — o usuário começa a ler imediatamente com fallback, troca para a fonte "certa" depois. Para headings decorativos, fallback ou optional evita que mude de layout depois. Nunca use block em conexão lenta — o "texto invisível" de 3s é pura frustração.

# Fontes variáveis — um arquivo, muitas variações

Fontes variáveis (variable fonts) contêm múltiplos "eixos" de variação em um único arquivo. Em vez de baixar Light, Regular, Bold, Heavy separadamente, você baixa um arquivo e interpola qualquer peso no meio.

variable-fonts.css
@font-face {
  font-family: "Inter";
  src: url("/fonts/inter-var.woff2") format("woff2-variations");
  font-weight: 100 900;          /* aceita qualquer valor neste range */
}

/* Agora pode usar pesos precisos */
h1 { font-weight: 750; }          /* não só 700 (bold) */
h2 { font-weight: 620; }
.destaque { font-weight: 540; }

/* font-variation-settings — controle fino de qualquer eixo */
.extra {
  font-variation-settings:
    "wght" 700,      /* weight */
    "wdth" 110,      /* width (se a fonte tiver) */
    "slnt" -10,      /* slant */
    "opsz" 14;       /* optical size */

Texto e overflow

Além da fonte, há uma coleção de propriedades que controlam como o texto se comporta dentro da sua caixa — quando quebrar, quando truncar, quando acrescentar reticências. Questões aparentemente triviais que, mal resolvidas, produzem layouts quebrados em português (onde palavras são frequentemente longas) e em URLs, hashtags, códigos.

# Quebra de linha — word-break e overflow-wrap

quebra.css
/* overflow-wrap — permite quebrar DENTRO de palavras quando necessário
   (palavras que não caberiam de jeito nenhum) */
.seguro {
  overflow-wrap: break-word;
  /* Quebra só em último caso — preserva palavras inteiras quando possível.
     Ideal para prosa em português. */
}

/* word-break — controla quando o navegador PODE quebrar palavras */
.normal     { word-break: normal; }      /* padrão — quebra em espaços e hífens */
.manter     { word-break: keep-all; }    /* nunca quebra dentro de palavras */
.quebra     { word-break: break-all; }   /* quebra em QUALQUER caractere */
.auto       { word-break: break-word; }  /* similar a overflow-wrap: break-word */

/* Cenário clássico: URL longa dentro de parágrafo */
.texto-com-url {
  overflow-wrap: break-word;
  /* "https://exemplo-muito-longo..." quebra onde couber,
     em vez de vazar pra fora da caixa. */
}
Qual usar
Para texto comum (parágrafos de prosa), overflow-wrap: break-word é o padrão seguro — só quebra palavras quando não têm alternativa. Para interfaces com dados imprevisíveis (usernames, URLs em um card estreito), combine overflow-wrap: break-word; word-break: break-word; para garantir que NADA vaze.

# Quebra inteligente — text-wrap (moderno)

Um dos recursos mais recentes e mais elegantes: text-wrap com valores balance e pretty instrui o navegador a equilibrar as linhas automaticamente.

text-wrap.css

   Ótimo para títulos que "amontoam" no fim. */
h1, h2, h3 {
  text-wrap: balance;
}

/* Antes: "Um título longo que vai amontoar
            feio"
   Com balance: "Um título longo que
                 vai amontoar feio" */

/* pretty — menos radical, só evita "órfãos" (palavra sozinha na última linha).

}

/* Suporte: Chrome/Edge desde 114, Firefox desde 121, Safari 17.4+.
   Custa um pouquinho em performance para textos muito longos —
   limite ao primeiras linhas ou títulos curtos. */
Tipografia que ajusta sozinha
text-wrap: balance em títulos é um ganho de qualidade imediato — uma linha no CSS que elimina aquela sensação de "fim quebrado" em manchetes. É um dos casos em que CSS moderno entrega no visual o que antes exigia código JavaScript rodando depois do carregamento.

Backgrounds, gradientes e filtros

Backgrounds parecem simples — uma cor, uma imagem. Na prática, background é uma das propriedades mais ricas do CSS: aceita múltiplas camadas, gradientes complexos, posicionamento preciso, e combina com filtros e blend modes para criar efeitos que antes exigiam Photoshop. Este capítulo cobre o essencial do "visual layer" do CSS.

Transforms

A propriedade transform permite mover, girar, escalar e inclinar elementos — tudo sem afetar o layout ao redor. É barata (roda na GPU), combina perfeitamente com animações e transitions, e é a base dos efeitos 3D do CSS. Saber transform bem é o que diferencia interfaces estáticas de interfaces vivas.

# As funções de transform

funcoes.css
/* translate — mover */
.a { transform: translateX(20px); }       /* eixo X */
.b { transform: translateY(-10px); }       /* eixo Y (negativo = subir) */
.c { transform: translate(20px, -10px); }  /* ambos */
.d { transform: translate(50%, 50%); }     /* % referente ao PRÓPRIO elemento */

/* scale — redimensionar */
.e { transform: scale(1.1); }              /* 110% do tamanho */
.f { transform: scaleX(1.5); }             /* só horizontal */
.g { transform: scale(1.5, 0.8); }         /* x, y separados */
.h { transform: scale(-1, 1); }            /* espelha horizontalmente */

/* rotate — girar */
.i { transform: rotate(45deg); }
.j { transform: rotate(0.5turn); }         /* 1 turn = 360deg */
.k { transform: rotate(-15deg); }

/* skew — inclinar (raramente útil) */
.l { transform: skewX(15deg); }
.m { transform: skewY(-5deg); }

/* Combinar múltiplas funções (ordem importa!) */
.n { transform: translate(20px, 10px) rotate(45deg) scale(1.2); }

# Propriedades individuais (moderno)

Desde 2022, CSS permite controlar cada tipo de transform separadamente. Isso facilita muito animações complexas e transições independentes.

individuais.css
/* Novas propriedades — cada transform em sua própria variável */
.elem {
  translate: 20px 10px;     /* = transform: translate(20px, 10px) */
  rotate: 45deg;            /* = transform: rotate(45deg) */
  scale: 1.2;               /* = transform: scale(1.2) */
}

/* Vantagem: animar um sem interferir nos outros */
.elem {
  translate: 0 0;
  rotate: 0deg;
  transition: translate 0.3s, rotate 0.5s;
}
.elem:hover {
  translate: 0 -4px;   /* move sem interferir com rotate */
}
.elem.ativo {
  rotate: 90deg;       /* gira sem interferir com translate */
}

/* Antes, você teria que usar transform único e recalcular
   TODOS os valores juntos — um pesadelo para animar. */
Use as propriedades individuais
Para projetos modernos (2023+), prefira translate, rotate, scale separados em vez de transform. Suporte universal em navegadores modernos. O código fica mais legível e animações independentes não brigam entre si.

# transform-origin — o ponto de âncora

Por padrão, transforms giram/escalam em torno do centro do elemento. transform-origin muda isso.

origin.css
.a { transform-origin: center; }        /* padrão — 50% 50% */
.b { transform-origin: top left; }       /* canto superior esquerdo */
.c { transform-origin: 0 100%; }         /* canto inferior esquerdo */
.d { transform-origin: 20px 30px; }      /* offset em px */

/* Exemplo — abrir menu "crescendo" do canto superior direito */
.menu-dropdown {
  transform-origin: top right;
  transform: scale(0);
  transition: transform 0.2s;
}
.menu-dropdown.aberto {
  transform: scale(1);
}

/* Exemplo — rotação de ponteiro (como relógio) */
.ponteiro {
  transform-origin: bottom center;   /* gira em torno da base */
  transform: rotate(var(--hora-em-graus));
}

# Por que transform é rápido

GPU vs CPU
Mudar top, left, width força o navegador a recalcular layout (rodando na CPU — caro). Mudar transform ou opacity só recompõe camadas na GPU (barato). Em animações de 60fps, sempre prefira transform a top/left. Para "mover" um elemento, translate é a forma correta em 2025 — mais rápida e mais versátil.

Transitions e animations

Transitions animam mudanças de estado — quando o usuário passa o mouse, foca, clica. Animations fazem qualquer elemento se mover sozinho a partir de keyframes. Juntas, são o que transforma interfaces estáticas em experiências responsivas. Usadas com moderação, fazem toda a diferença na percepção de qualidade.

# Transition — o básico

transition-basico.css
/* Sintaxe: property duration timing-function delay */
.btn {
  background: blue;
  transition: background 0.3s ease;
}
.btn:hover {
  background: red;
  /* A cor transiciona suavemente em 0.3s */
}

/* Múltiplas propriedades */

# Timing functions — a curva da animação

timing.css
/* Keywords pré-definidas */
.a { transition-timing-function: linear; }       /* velocidade constante */
.b { transition-timing-function: ease; }          /* começa lento, meio rápido, termina lento (padrão) */
.c { transition-timing-function: ease-in; }       /* começa lento, acelera */
.d { transition-timing-function: ease-out; }      /* começa rápido, desacelera */
.e { transition-timing-function: ease-in-out; }   /* lento nas pontas */

/* cubic-bezier — curva customizada (4 pontos de controle) */
.custom {
  transition-timing-function: cubic-bezier(0.4, 0, 0.2, 1);
  /* Esta é a curva "Material" — padrão da indústria para UIs */
}

/* Curvas populares em design systems */
:root {
  --ease-standard: cubic-bezier(0.4, 0, 0.2, 1);
  --ease-in: cubic-bezier(0.4, 0, 1, 1);
  --ease-out: cubic-bezier(0, 0, 0.2, 1);
  --ease-emphasized: cubic-bezier(0.2, 0, 0, 1);
  --ease-elastic: cubic-bezier(0.68, -0.55, 0.27, 1.55);
}

/* steps — animação em "saltos" discretos */
.discreto {
  transition-timing-function: steps(4, end);
  /* 4 passos, cada um visível por tempo igual.
     Útil para sprite animations ou efeitos retro. */
}
Para UIs, use ease-out
A maioria das interfaces se sente melhor com ease-out: elementos chegam rapidamente no começo e desaceleram no final, imitando movimento físico natural. ease-in faz exatamente o oposto e soa "pesado". Para saídas (elementos que desaparecem), ease-in faz mais sentido. Como regra: ease-out para entrar, ease-in para sair.

# O que pode ser animado

Apenas propriedades "animáveis" transicionam. Em geral:

  • Animáveis: cor, tamanho, posição, opacidade, transform, filter, box-shadow, border, background-color, blur, gap.
  • Não animáveis (historicamente): display, visibility (on/off), z-index, font-family, position values (auto → fixed).
  • Novo em 2024+: display pode ser animado com transition-behavior: allow-discrete.

# animation-fill-mode — estado antes/depois

fill-mode.css
/* Sem fill-mode (none, padrão):
   Antes da animação começar: elemento está no estado ORIGINAL do CSS
   Depois da animação terminar: volta para o estado ORIGINAL */

.caso-ruim {
  opacity: 1;                          /* estado base */
  animation: fade-in 0.3s;
}
@keyframes fade-in {
  from { opacity: 0; }
  to   { opacity: 1; }
}
/* Problema: como fill-mode é none, antes da animação o elemento
   está em opacity: 1 (base). A animação "corta" para 0 bruscamente.
   Depois volta para 1 (igual ao que a animação terminou, então tudo bem). */

/* Solução: fill-mode: backwards (assume o estado do PRIMEIRO frame antes de começar) */
.certo {
  animation: fade-in 0.3s backwards;
  /* Antes de começar: assume opacity: 0 (do 'from') */
}

/* forwards — mantém o estado do ÚLTIMO frame após terminar */
.resta-no-final {
  animation: slide-in 0.5s forwards;
}

/* both — faz os dois */
.caso-mais-comum {
  animation: slide-in 0.5s both;
}
fill-mode: both é o que você quer 90% do tempo
Animações entrando na tela (fade-in, slide-in) quase sempre devem usar animation-fill-mode: both. Sem isso, o elemento "pisca" visível brevemente antes da animação começar, ou some após terminar. Com both, a animação determina o estado antes E depois.

# Respeitando prefers-reduced-motion

A acessibilidade aqui é crítica. Pessoas com sensibilidade vestibular podem ter náusea ou crises com animações grandes. Sempre respeite a preferência do sistema.

reduced-motion.css
/* Abordagem 1: desligar TODAS animações (simples, mas agressivo) */
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}

/* Abordagem 2: desligar apenas as "grandes" (parallax, slide, rotate) */
@media (prefers-reduced-motion: reduce) {
  .parallax,
  .slide-in,
  .rotate-loop {
    animation: none;
  }
  /* Micro-interações (fade, opacity) podem ficar — são seguras */
}

/* Abordagem 3: escrever animações pensando nos dois casos */
@keyframes enter {
  from { opacity: 0; translate: 0 20px; }
  to   { opacity: 1; translate: 0 0; }
}

@media (prefers-reduced-motion: reduce) {
  @keyframes enter {
    from { opacity: 0; }       /* sem translate */
    to   { opacity: 1; }
  }
}
Animações são acessibilidade
Um site com animações bonitas mas sem respeitar prefers-reduced-motion literalmente causa náusea em algumas pessoas. Em 2025 isso não é mais opcional — é parte do básico de WCAG AA. Faça do hábito: toda vez que escrever animation ou transition grande, adicione o media query correspondente.

View Transitions

Durante anos, criar transições entre telas — o tipo de animação que apps nativos fazem naturalmente — exigia bibliotecas complexas, gerenciamento manual de estado, e frequentemente não valia o esforço. A View Transitions API mudou isso: com pouquíssimo código, a web ganha transições de nível de aplicativo nativo. É talvez o recurso mais "mágico" do CSS moderno.

# O que a API faz

A View Transitions API tira um "snapshot" da tela antes de uma mudança, outro depois, e faz o browser animar automaticamente entre os dois estados. Sem manter elementos duplicados, sem configurar transições manualmente, sem risco de dessincronia.

  • Same-document transitions — entre dois estados na mesma página (SPA). Suporte amplo desde 2023.
  • Cross-document transitions — entre páginas diferentes (MPA). Suporte em Chrome/Edge e outras desde 2024.

# Same-document — o básico

basico.js
// Qualquer mudança no DOM pode virar uma animação fluida
function trocarTema(novoTema) {
  // Verifica suporte
  if (!document.startViewTransition) {
    document.body.dataset.tema = novoTema;
    return;
  }

  // Envolve a mudança em uma view transition
  document.startViewTransition(() => {
    document.body.dataset.tema = novoTema;
  });
}

// O que acontece:
// 1. Browser tira snapshot do estado atual
// 2. Executa a função (que muda o DOM)
// 3. Tira snapshot do novo estado
// 4. Anima entre os dois automaticamente com um fade cross-fade padrão

# Customizando com CSS

O browser cria uma árvore de pseudoelementos durante a transição. Você pode customizar tudo via CSS.

customizar.css
/* Estrutura gerada automaticamente:
   ::view-transition (root)
   └─ ::view-transition-group(*)
      └─ ::view-transition-image-pair(*)
         ├─ ::view-transition-old(*)  (snapshot antigo)
         └─ ::view-transition-new(*)  (snapshot novo) */

/* Customizar a animação padrão (cross-fade) */
::view-transition-old(root),
::view-transition-new(root) {
  animation-duration: 0.5s;
  animation-timing-function: ease-in-out;
}

/* Só elemento antigo desaparecendo */
::view-transition-old(root) {
  animation: fade-out 0.3s ease-out;
}

/* Só elemento novo aparecendo */
::view-transition-new(root) {
  animation: fade-in 0.3s ease-out 0.3s;
}

@keyframes fade-out { to { opacity: 0; } }
@keyframes fade-in  { from { opacity: 0; } }

# Identificando elementos — view-transition-name

O poder real aparece quando você identifica elementos específicos. Browser anima eles individualmente — seguindo mudanças de posição, tamanho, etc.

named-transitions.css
/* Dar um "nome" a um elemento para a transição */
.hero-image {
  view-transition-name: hero-image;
}

/* Agora, quando a .hero-image muda de posição/tamanho entre estados,
   o browser anima ESSA transição individualmente — em vez de fade. */

/* Customizar só aquele elemento */
::view-transition-group(hero-image) {
  animation-duration: 0.6s;
  animation-timing-function: cubic-bezier(0.4, 0, 0.2, 1);
}

# O padrão "shared element" (Magic Move)

Um elemento que aparece em duas telas (ex: uma foto em uma lista e na página de detalhe) pode "voar" da posição antiga para a nova automaticamente. Padrão clássico de apps nativos, agora grátis no CSS.

magic-move.html
<!-- Lista -->
<ul class="lista">
  <li>
    <img src="foto1.jpg" style="view-transition-name: foto-1">
    <h3>Título 1</h3>
  </li>
</ul>

<!-- Após clicar, a lista vira a página de detalhe: -->
<article>
  <img src="foto1.jpg" style="view-transition-name: foto-1">  <!-- mesmo nome -->
  <h1>Título 1</h1>
  <p>Detalhes...</p>
</article>

<script>
  link.addEventListener("click", (e) => {
    e.preventDefault();
    document.startViewTransition(() => {
      // Trocar o HTML da lista pelo HTML do detalhe
      app.innerHTML = detalheHTML;
    });
  });
</script>

<!-- Resultado: a imagem "voa" da posição na lista para
     a posição no detalhe, suavemente. -->
Nomes únicos por elemento
Em uma lista, cada item precisa de um view-transition-name único por vez. Duas imagens com mesmo name em tela ao mesmo tempo fazem a transição falhar. Use view-transition-name: foto-{id} ou aplique o name dinamicamente só no item que está sendo aberto.

# Cross-document — entre páginas diferentes

Para apps multi-página (MPA clássico, sem framework SPA), ligue transições entre navegações inteiras.

cross-doc.css
/* Em CSS — ativa para TODA navegação same-origin */
@view-transition {
  navigation: auto;
}

/* Agora, clicar em qualquer link <a href> faz a navegação
   ter transição automática. Sem JavaScript. */

/* Elementos compartilhados entre páginas — mesmo view-transition-name
   nas duas páginas, o browser faz o "magic move" */
.logo {
  view-transition-name: logo;
}

/* Customizar como antes */
::view-transition-old(root) {
  animation: fade-slide-out 0.3s ease-in forwards;
}
::view-transition-new(root) {
  animation: fade-slide-in 0.3s 0.15s ease-out forwards;
}

@keyframes fade-slide-out {
  to { opacity: 0; transform: translateX(-20px); }
}
@keyframes fade-slide-in {
  from { opacity: 0; transform: translateX(20px); }
}

# Suporte e fallback

feature-detect.js
// Feature detection simples
if (!document.startViewTransition) {
  atualizarDOM();   // fallback: troca direta
  return;
}

document.startViewTransition(() => atualizarDOM());

// Padrão helper
function animatedSwap(updateFn) {
  if (!document.startViewTransition) {
    return updateFn();
  }
  return document.startViewTransition(updateFn);
}
Suporte em 2025
Same-document View Transitions têm suporte em Chrome/Edge (desde 2023) e Safari (desde 18, 2024). Firefox ainda está implementando. Cross-document Transitions têm suporte em Chrome/Edge e Safari 18.2+. Em navegadores sem suporte, a transição não acontece — mas a mudança no DOM ocorre normalmente. É um enhancement puro: nenhum risco para quem não tem.

# Quando View Transitions brilham

  • Troca de tema (dark/light) — cross-fade automático, sem CSS complicado.
  • Roteamento em SPA — transição entre páginas sem gerenciar estado de animação.
  • Lista → detalhe — o clássico "magic move" de apps nativos.
  • Ordenação de lista — itens se reorganizam com transições suaves automaticamente.
  • Filtros — elementos somem/aparecem com movimento coordenado.
  • MPAs tradicionais — transições entre páginas inteiras com @view-transition.
Quebrando um paradigma
Até 2023, aplicativos com transições nativas eram praticamente privilégio de apps mobile. A View Transitions API traz essa capacidade para a web com código que é muito mais simples do que alternativas anteriores (Framer Motion, GSAP, React Transition Group). É provavelmente a API mais importante do CSS moderno para experiência de usuário percebida — vale aprender e usar cedo nos projetos novos.

Media queries

Media queries são a ferramenta clássica de responsividade — o jeito de fazer o CSS responder a características da tela ou do dispositivo. Com a chegada de @container, light-dark() e unidades fluidas, muita gente pergunta se media queries ainda importam. A resposta é: sim, mas o papel delas ficou mais focado. Este capítulo cobre o uso moderno.

# Sintaxe básica

basico.css
/* Sintaxe clássica */
@media (min-width: 768px) {
  .container { padding: 2rem; }
}

/* Múltiplas condições — AND */
@media (min-width: 768px) and (orientation: landscape) {
  /* tela larga E horizontal */
}

/* OR — vírgula */
@media (max-width: 600px), (orientation: portrait) {
  /* tela pequena OU vertical */
}

/* NOT */
@media not (hover: hover) {
  /* dispositivos SEM hover (touch) */
}

/* only — para esconder de navegadores muito antigos (raramente necessário hoje) */
@media only screen and (min-width: 768px) { ... }

# Sintaxe moderna — range queries

range.css
/* Sintaxe antiga */
@media (min-width: 768px) and (max-width: 1199px) {
  /* tablets */
}

/* Sintaxe moderna com range — mais legível */
@media (768px <= width <= 1199px) {
  /* mesma coisa, mais claro */
}

/* Outros operadores */
@media (width >= 768px) { }      /* = min-width: 768px */
@media (width < 768px)  { }      /* = max-width: 767.98px (evita overlap!) */
@media (width = 1920px) { }      /* exato (raramente útil) */

/* Funciona com qualquer feature que aceita range */
@media (300dpi <= resolution <= 600dpi) { ... }
@media (aspect-ratio >= 16/9) { ... }
Use a sintaxe de range
A sintaxe (width >= 768px) é suportada em todos os navegadores modernos desde 2023. É mais legível (operadores matemáticos são mais intuitivos que "min-width"), resolve ambiguidades entre max-width: 768px e min-width: 768px (qual pega no 768px exato?), e facilita comparar ranges.

# Breakpoints — quais valores usar

breakpoints.css
/* Padrões populares (Tailwind, Bootstrap) */
:root {
  --sm: 640px;   /* phone landscape / small tablet */
  --md: 768px;   /* tablet */
  --lg: 1024px;  /* laptop */
  --xl: 1280px;  /* desktop */
  --2xl: 1536px; /* wide */
}

/* Abordagem "mobile-first" — start com estilo mobile, adicione para telas maiores */
.container {
  padding: 1rem;       /* padrão — mobile */
}

@media (width >= 768px) {
  .container { padding: 2rem; }   /* tablet+ */
}

@media (width >= 1024px) {
  .container { padding: 3rem; }   /* laptop+ */
}
Deixe de pensar em dispositivos
Breakpoints não existem para atender "celular", "tablet", "desktop" — existem para atender quando o design quebra. O melhor breakpoint é aquele que seu design pede. Abra sua página e redimensione a janela devagar; quando algo começa a ficar estranho, é um breakpoint. Os valores "mágicos" (640/768/1024/1280) são um default útil, mas não uma regra.

# Queries de capacidade — não só largura

Media queries não são só sobre tamanho. Há uma categoria de features que perguntam sobre capacidade do dispositivo — muito mais úteis que assumir por largura.

capacidades.css
/* hover — o dispositivo tem hover? */
@media (hover: hover) {
  .card:hover { transform: translateY(-2px); }
  /* Aplica só em mouse/trackpad — touch não tem hover "real" */
}

@media (hover: none) {
  /* Touch devices — desenhe tap targets maiores, sem hover */
  .btn { padding: 16px 24px; }
}

/* pointer — precisão do apontador */
@media (pointer: coarse) {
  /* touch — alvos grandes */
  .btn { min-height: 44px; min-width: 44px; }
}
@media (pointer: fine) {
  /* mouse — pode ser compacto */
  .btn { min-height: 32px; }
}

/* any-hover / any-pointer — considera TODOS os inputs disponíveis
   (laptop com touch screen, por exemplo) */
@media (any-hover: hover) {
  /* se PELO MENOS UM input suporta hover */
}

/* orientation */
@media (orientation: portrait) { }
@media (orientation: landscape) { }

/* aspect-ratio */
@media (aspect-ratio >= 16/9) { }
Mouse vs touch: use pointer, não width
Assumir "tela pequena = touch" está errado há anos (celulares com teclado bluetooth, tablets gigantes, laptops híbridos). A pergunta certa é: "este dispositivo tem um cursor preciso ou não?" — e a resposta vem de pointer: fine/coarse, não de largura da tela. Dimensione áreas clicáveis por pointer: coarse, aplique :hover apenas em hover: hover.

# Media query vs Container query — quando cada um

A regra prática
Media query para o que é global da página: mostrar/esconder sidebar, layout macro, tamanho base de fonte, tema escuro, preferências do usuário (motion, contrast).

Container query para o que é local do componente: card muda de layout, galeria adapta colunas, título aumenta de tamanho. Se o CSS só se importa com "quanto espaço EU tenho", é container.

Em um projeto moderno, 80% da responsividade deveria ser container queries e só 20% media queries.

Preferências e dark mode

O sistema operacional do usuário carrega informações importantes sobre ele — tema preferido, sensibilidade a movimento, tolerância a contraste, preferência por dados reduzidos. As queries prefers-* expõem tudo isso para o CSS. Respeitar essas preferências não é só boa prática: é o que separa sites que "funcionam para todos" dos que só funcionam para o dev que os escreveu.

# prefers-color-scheme — dark mode

color-scheme.css
/* Abordagem clássica — media query */
:root {
  --bg: white;
  --text: #111;
}

@media (prefers-color-scheme: dark) {
  :root {
    --bg: #0f0f0f;
    --text: #eee;
  }
}

body {
  background: var(--bg);
  color: var(--text);
}

/* Abordagem moderna — light-dark() (cap. 10) */
:root {
  color-scheme: light dark;   /* essencial! */
  --bg: light-dark(white, #0f0f0f);
  --text: light-dark(#111, #eee);
}
/* Mais conciso. Navegador escolhe automaticamente. */
Não esqueça de color-scheme
A propriedade color-scheme: light dark em :root faz toda a diferença. Sem ela, os controles nativos (scrollbar, inputs, select) permanecem com tema claro mesmo em dark mode. Com ela, o navegador aplica o tema do sistema a elementos nativos, tornando a página consistente. Uma linha que resolve 10 bugs visuais.

# Toggle manual — override da preferência

O usuário pode querer um tema específico, independente do sistema. Padrão moderno: data-theme + color-scheme.

toggle.css
/* Padrão — segue o sistema */
:root {
  color-scheme: light dark;
}

/* Override explícito — força um tema */
:root[data-theme="light"] { color-scheme: light; }
:root[data-theme="dark"]  { color-scheme: dark; }

/* Com light-dark() funciona automaticamente */
body {
  background: light-dark(white, #0f0f0f);
  color: light-dark(#111, #eee);
}
toggle.js
// Ler e aplicar preferência salva + do sistema
function aplicarTema() {
  const salvo = localStorage.getItem("tema");
  const doSistema = matchMedia("(prefers-color-scheme: dark)").matches;
  const tema = salvo ?? (doSistema ? "dark" : "light");
  document.documentElement.setAttribute("data-theme", tema);
}

// Alternar
function alternar() {
  const atual = document.documentElement.dataset.theme;
  const novo = atual === "dark" ? "light" : "dark";
  document.documentElement.dataset.theme = novo;
  localStorage.setItem("tema", novo);
}

// Escute mudanças no sistema em tempo real
matchMedia("(prefers-color-scheme: dark)").addEventListener("change", aplicarTema);

// ⚠️ Aplique ANTES do body renderizar (no <head>, sincronamente)
// para evitar "flash" de tema errado. É um script pequeno crítico. */

Quem realmente precisa
Pessoas com distúrbios vestibulares, enxaqueca, TDAH ou sensibilidade sensorial podem ter náusea física, tontura ou crises com animações grandes (parallax, autoplay, rotações agressivas). A preferência prefers-reduced-motion é o pedido delas. Ignorar é causar dano literal. Em 2025, é critério WCAG 2.2 AA — não negociável.

Clamp e fluid design

Durante anos, responsividade significou breakpoints: quando a tela atinge X pixels, mudamos o tamanho da fonte. Resultado: saltos bruscos em tamanhos específicos e ajuste ruim no meio. Fluid design substitui isso — valores que variam continuamente com o tamanho do viewport ou container. A ferramenta principal é clamp().

# O problema dos breakpoints

breakpoints-problema.css
/* Abordagem clássica — 3 tamanhos fixos por breakpoint */
h1 { font-size: 2rem; }

@media (width >= 768px) {
  h1 { font-size: 2.5rem; }
}

@media (width >= 1024px) {
  h1 { font-size: 3rem; }
}

/* Problemas:
   1. Saltos bruscos — de 2rem para 2.5rem ao passar 768px
   2. Em 900px (meio caminho), fica com 2.5rem — pode parecer pequeno
   3. Em 1600px (muito grande), para em 3rem
   4. Mais código = mais manutenção */

# clamp() — a função fluida

clamp-basico.css
/* clamp(mínimo, preferido, máximo) */

h1 { font-size: clamp(2rem, 4vw + 1rem, 4rem); }

/* Como ler:
   - NUNCA menor que 2rem
   - NUNCA maior que 4rem
   - Entre os dois: varia suavemente com "4vw + 1rem"

   - Em tela de 375px  (4vw = 15px):  1rem + 15px = ~32px → bate no mínimo (2rem)
   - Em tela de 768px  (4vw = 31px):  1rem + 31px = ~47px → dentro do range
   - Em tela de 1440px (4vw = 58px):  1rem + 58px = ~74px → bate no máximo (4rem)

   Resultado: título escala suavemente, sem nenhum @media. */

# Calculando o valor "preferido"

A fórmula no meio (4vw + 1rem) parece mágica, mas tem uma lógica precisa. Ela define a taxa de crescimento: quanto o valor aumenta para cada pixel a mais de viewport.

calculo.css
/* A fórmula matemática:
   preferido = a + b*vw
   Onde a e b são calculados a partir de 2 pontos:
     (minVw, minValue) e (maxVw, maxValue)

   b = (maxValue - minValue) / (maxVw - minVw)  * 100
   a = minValue - b * minVw / 100 */

/* Exemplo — font-size de 16px em 320px a 24px em 1200px */

/* Cálculo manual:
   b = (24 - 16) / (1200 - 320) * 100 = 0.909...
   a = 16 - 0.909 * 320 / 100 = 13.09

   preferido = 13.09px + 0.909vw */

h2 { font-size: clamp(1rem, 0.818rem + 0.909vw, 1.5rem); }

/* Ferramentas online fazem esse cálculo:
   - utopia.fyi (mais popular)
   - fluid.style
   - modern-fluid-typography.vercel.app */
Não calcule à mão
Use utopia.fyi. Você diz "fonte mínima X em viewport de Y, máxima W em viewport Z" e ele gera o clamp() pronto. Para projetos sérios, ele gera uma escala inteira de tamanhos (fontes, espaçamentos) todos fluidos e harmonizados. É uma das ferramentas mais úteis do ecossistema CSS moderno.

# Quando clamp NÃO é a resposta

nao-clamp.css
/* Casos em que breakpoints fixos são melhores que clamp: */

.layout {
  display: grid;
  grid-template-columns: 1fr;

@media (width >= 768px) {
  .layout { grid-template-columns: 1fr 1fr 1fr; }
}

/* 2. Comportamento binário — sidebar visível/invisível */

@media (width >= 1024px) {
  .sidebar { display: block; }
}

/* 3. Elementos com tamanhos FIXOS por razão semântica —
   um ícone de 16px continua sendo 16px em qualquer tela */

.icon-sm { width: 16px; height: 16px; }
A regra prática
Clamp para o que muda continuamente: fontes, padding, gaps, largura de containers centralizados.

Media queries para mudanças discretas: aparecer/desaparecer, 1-coluna-vs-3, abrir/fechar sidebar, substituir imagens.

Em um projeto moderno, 80-90% da "responsividade de tamanho" é clamp; 10-20% são media queries para estrutura.

O seletor :has()

Por duas décadas, o CSS teve uma limitação fundamental: seletores só olhavam para filhos ou descendentes — nunca para pais ou ancestrais. Isso forçou toda uma categoria de problemas a ser resolvida em JavaScript. Em 2023, :has() mudou isso. É provavelmente o recurso mais transformador que o CSS ganhou na última década.

# O que :has() faz

:has() seleciona um elemento com base no que está dentro dele. Literalmente: "selecione X que contém Y".

conceito.css
/* Antes de :has(), seletores só olhavam para o futuro: */
.card h2 { }          /* h2 DENTRO de .card */
.card + .card { }     /* .card APÓS outro .card */

/* Com :has(), você pode olhar o passado e o presente: */
.card:has(h2) { }              /* .card que CONTÉM um h2 */
.card:has(.featured) { }        /* .card que CONTÉM algo .featured */
article:has(> img) { }          /* article com imagem como filho DIRETO */

/* Combine com outros seletores */
.card:has(img):has(h2) { }     /* tem imagem E tem h2 */
.card:not(:has(img)) { }       /* NÃO tem imagem */

/* Use combinadores dentro do :has() */
label:has(+ input:focus) { color: blue; }  /* label seguido de input focado */
article:has(h1 + p) { }                     /* article com h1 imediatamente seguido por p */

# Performance — o mito e a realidade

Não é lento na prática
Durante anos, "parent selector" era recusado no CSS por "preocupações de performance". Com as implementações modernas em browsers atuais, :has() é suficientemente otimizado para uso normal. Em apps com DOM gigante (tabelas de 10 mil linhas, árvores infinitas), pode valer medir. Para páginas comuns — landing pages, dashboards, blogs —, use sem medo. O ganho em simplicidade supera qualquer preocupação teórica.

# A lição maior

Código que se adapta
:has() muda fundamentalmente o que CSS pode expressar. Em vez de adicionar classes condicionais no HTML (e depois JS para gerenciar), você descreve o comportamento direto no CSS: "quando este componente contém X, ele deve parecer Y". HTML fica limpo, JS diminui, CSS ganha poder. Procure refatorar com :has() — a maioria dos códigos ganha.

Anchor positioning

Posicionar um tooltip ao lado do botão que o acionou parece trivial — mas até 2024 era um dos problemas mais chatos do CSS. Exigia position: relative no pai comum, position: absolute no tooltip, e JavaScript para tudo que fugisse da regra. A CSS Anchor Positioning API resolveu isso nativamente, e é a última peça para substituir bibliotecas como Popper.js.

# O problema que resolve

problema.css
/* Cenário clássico — tooltip ao lado de botão */

/* ❌ Jeito antigo: pai comum + absolute */
.wrapper {
  position: relative;   /* tooltip precisa ser filho deste */
}
.btn { }
.tooltip {
  position: absolute;
  top: 100%;
  left: 0;
}

<div class="wrapper">
  <button class="btn">Info</button>
  <div class="tooltip">...</div>
</div>

/* Problemas:
   1. HTML fica acoplado (precisa wrapper)
   2. Impossível se o tooltip precisa ficar em outro contêiner
     (ex: em <dialog> ou <popover>, no top layer)
   3. Flip automático (se não cabe em baixo, vai em cima)? Só com JS.
   4. Ancorado a algo distante no DOM? Só com JS. */

# A nova API — sintaxe básica

basico.css
/* Passo 1: declarar a ÂNCORA */
.btn {
  anchor-name: --meu-botao;
}

/* Passo 2: posicionar ALGO em relação à âncora */
.tooltip {
  position: absolute;
  position-anchor: --meu-botao;  /* referencia a âncora nomeada */

  /* Agora as funções anchor() dão as coordenadas */
  top: anchor(bottom);    /* topo do tooltip = fundo do botão */
  left: anchor(left);     /* alinhado à esquerda do botão */
  margin-top: 4px;        /* pequeno gap */
}

<!-- O HTML pode estar em qualquer lugar — não precisa ser pai/filho -->
<button class="btn">Info</button>
<!-- ... outros elementos no meio ... -->
<div class="tooltip">Dica</div>

  left: anchor(left);         /* esquerda da âncora */

   top | bottom | left | right
   start | end (logical)

# position-try — fallbacks automáticos

A parte mais poderosa da API: o navegador pode tentar posições diferentes se a principal não couber. "Aparece em baixo; se não cabe, aparece em cima; se não cabe, aparece ao lado."

position-try.css
.tooltip {
  position: absolute;
  position-anchor: --botao;

  /* Posição principal */
  top: anchor(bottom);
  left: anchor(left);

  /* Fallbacks se a posição principal não couber na viewport */
  position-try-fallbacks:
    --top,               /* fallback 1: em cima */
    --right,             /* fallback 2: à direita */
    flip-block,          /* keyword: inverte eixo vertical */
    flip-inline;         /* keyword: inverte eixo horizontal */
}

/* Definir fallbacks customizados com @position-try */
@position-try --top {
  top: auto;
  bottom: anchor(top);   /* tooltip em cima em vez de em baixo */
  left: anchor(left);
}

@position-try --right {
  top: anchor(top);
  left: anchor(right);
}

/* Navegador tenta cada fallback em ordem, fica com o primeiro que
   couber inteiro na viewport. Zero JavaScript. */

Arquitetura CSS

Escrever CSS para uma página estática é fácil. Escrever CSS para um app que cresce ao longo de anos, com uma equipe mudando, sem que tudo desmorone — é uma disciplina completa. Metodologias, convenções e ferramentas acumuladas ao longo de 15 anos tentam resolver o mesmo problema: como manter CSS sustentável em escala. Este capítulo compara as abordagens atuais.

# O problema central

CSS tem escopo global por padrão. Qualquer regra afeta o documento inteiro. Em projetos grandes, isso gera três problemas característicos:

  • Colisão de nomes — .button definido por dois times em arquivos diferentes.
  • Especificidade crescente — para sobrescrever algo, você escreve um seletor maior; isso obriga o próximo a ser ainda maior.
  • Morte lenta — ninguém sabe se uma regra ainda é usada, então não se apaga; o arquivo só cresce.

Cada metodologia a seguir é uma tentativa de resolver alguma parte disso.

# BEM — Block, Element, Modifier

A metodologia mais usada entre 2013 e 2020. Convenção de nomes que evita colisões via disciplina de escrita.

bem.css
/* Block — componente autônomo */
.card { }

/* Element — parte DE um bloco. Dois underscores */
.card__title { }
.card__body { }
.card__footer { }

/* Modifier — variante do bloco ou elemento. Dois hífens */
.card--destaque { }
.card--horizontal { }
.card__title--grande { }

/* Estados podem ser modifiers OU classes is-/has- */
.card--ativo { }
/* ou */
.card.is-active { }
bem-uso.html
<article class="card card--destaque">
  <h2 class="card__title card__title--grande">Título</h2>
  <div class="card__body">Texto</div>
  <footer class="card__footer">Rodapé</footer>
</article>

Pontos fortes: especificidade baixa e consistente, classes são auto-documentadas, evita colisão de nomes naturalmente. Pontos fracos: classes ficam verbosas, duplicação visível no HTML, difícil manter rigor em times grandes.

# Utility-first — Tailwind e afins

A abordagem dominante na web moderna (2020+). Em vez de escrever CSS para cada componente, usa classes pequenas e reutilizáveis diretamente no HTML.

utility.html
<article class="flex flex-col gap-4 p-6 rounded-lg bg-white shadow-md">
  <h2 class="text-2xl font-bold text-gray-900">Título</h2>
  <p class="text-gray-600 line-clamp-3">Descrição...</p>
  <button class="mt-auto px-4 py-2 bg-blue-600 text-white rounded hover:bg-blue-700">
    Ação
  </button>
</article>

<!-- Cada classe = uma declaração CSS específica
     Zero CSS customizado; tudo vem do framework -->

Pontos fortes: velocidade de desenvolvimento, não precisa pensar em nomes, tree-shaking elimina CSS não usado, design system embutido (escalas de cor/spacing consistentes), praticamente impossível quebrar CSS alheio. Pontos fracos: HTML fica verboso, repetição de classes, curva de aprendizado inicial.

Por que Tailwind venceu
Apesar das críticas ("bagunça no HTML"), Tailwind resolve os três problemas do CSS em escala: zero colisão (cada classe é atômica e imutável), especificidade constante (todas as classes têm peso idêntico), morte natural (classes só ficam se o HTML que as usa fica). Na prática, em equipes grandes, a "feiura" do HTML é um preço pequeno para a manutenibilidade que ganha.

De 2017 a 2022, CSS-in-JS foi dominante em React. # Princípios universais

Independente da metodologia, algumas verdades sobrevivem:

  • Especificidade baixa e previsível é melhor que especificidade alta e imprevisível.
  • Escopo local (@scope, Modules, Tailwind) é melhor que escopo global bagunçado.
  • Design tokens centralizados são melhores que valores mágicos espalhados.

# O pipeline de renderização

Para entender o que é "rápido" em CSS, é preciso saber o que o navegador faz quando você muda um estilo:

  1. Recalculate style — calcula valores computados para elementos afetados.
  2. Layout — recalcula posições e tamanhos (todo ou em partes da árvore).
  3. Paint — pinta pixels em layers.
  4. Composite — combina layers na GPU.

Cada propriedade, ao mudar, dispara um subconjunto dessas etapas. Saber quais evitam layout e paint é a base de animações 60fps.

PropriedadeDispara
width, height, padding, margin, top, leftLayout + Paint + Composite
color, background-color, border-colorPaint + Composite
transform, opacity, filterComposite apenas (GPU)
visibility (entre visible/hidden)Paint
display (none ↔ block)Layout + Paint
A regra dos 60fps
Para animações suaves (60fps = 16.6ms por frame), evite disparar layout e paint. Use transform em vez de top/left; use opacity em vez de visibility. O maior pecado de animação: animar width ou height — é caro e força o browser a recalcular tudo por baixo.

# contain — isolamento explícito

A propriedade contain avisa o navegador: "mudanças neste elemento não afetam o resto da página". Ele pode otimizar cálculos.

contain.css
/* Valores individuais */
.a { contain: layout; }   /* mudanças internas não mexem no layout externo */
.b { contain: paint; }    /* conteúdo não pode "vazar" visualmente */
.c { contain: size; }     /* elemento tem tamanho independente do conteúdo */
.d { contain: style; }    /* contadores e quotes são independentes */

/* Combinado */
.e { contain: layout paint; }   /* as duas isolações mais comuns */
.f { contain: strict; }         /* = layout paint size style (máximo) */
.g { contain: content; }        /* = layout paint style (sem size) */

/* Uso prático — cards em listas longas */
.card {
  contain: content;
  /* Browser sabe que este card é independente.
     Scroll fica mais fluido, paint é otimizado. */
}

/* content-visibility — a evolução de contain */
.secao-distante {
  content-visibility: auto;
  contain-intrinsic-size: 500px;   /* "reserva" tamanho para scroll correto */
  /* Elementos fora da tela NÃO são pintados nem têm layout calculado.
     Ganho de performance ENORME em páginas longas. */
}
content-visibility é mágico
Em páginas com muitos blocos abaixo da dobra (blogs longos, dashboards, feeds), uma linha content-visibility: auto pode reduzir o tempo inicial de renderização em 30-50%. O navegador "pula" o trabalho de pintar o que está fora da tela até o usuário rolar para lá. Custa quase nada implementar — vale tentar em qualquer página comprida.

# Carregamento — CSS bloqueia renderização

carregamento.html
<!-- CSS no <head> BLOQUEIA a primeira renderização até baixar+processar -->
<link rel="stylesheet" href="/estilo.css">

<!-- Técnicas para não bloquear -->

<!-- 1. Inline o CSS crítico (primeiro viewport) -->
<style>
  /* Critical CSS — apenas o que aparece acima da dobra */
  body { font-family: sans-serif; margin: 0; }
  header { background: #111; padding: 1rem; }
</style>

<!-- 2. Carregue o resto sem bloquear -->
<link rel="preload" href="/estilo.css" as="style"
      onload="this.onload=null;this.rel='stylesheet'">
<noscript><link rel="stylesheet" href="/estilo.css"></noscript>

<!-- 3. CSS específico de mídia NÃO bloqueia (baixa sem prioridade) -->
<link rel="stylesheet" href="/print.css" media="print">
<link rel="stylesheet" href="/mobile.css" media="(max-width: 768px)">

# Seletores — o que é rápido?

seletores-perf.css
/* Historicamente, navegadores liam seletores da DIREITA para esquerda.
   Por isso seletores muito profundos eram "lentos".
   
   Em 2025, essa otimização ficou menos relevante — browsers são
   muito bons e a maioria dos seletores é rápida.
   
   Ainda assim, algumas regras valem: */

/* ✅ Seletores simples são melhores */
.btn { }              /* ótimo */
.btn-primary { }       /* ótimo */

/* ⚠️ Seletores profundos são OK, mas evite exageros */
nav ul li a { }        /* tudo bem */

/* ❌ Seletores universais em contextos amplos */
* + * { margin-top: 1rem; }   /* roda pra CADA par de elementos da página */

/* ❌ :has() com descendentes complexos em DOM grande */
body:has(.complex-rule .deep-rule) { }   /* pode ser pesado */

/* ✅ :has() localizado é OK */
.card:has(img) { }    /* rápido o suficiente */

# will-change — use com MUITO cuidado

will-change-real.css
/* will-change promove o elemento a uma camada de GPU PERMANENTE.
   Isso consome memória. Se aplicado em tudo, degrada performance. */

/* ❌ Abuso */
* { will-change: transform; }   /* não faça isso — prejudica tudo */

/* ❌ Também abuso */
.card { will-change: transform; }   /* 100 cards = 100 layers de GPU */

/* ✅ Aplicação pontual e temporária */
.elem-prestes-a-animar {
  will-change: transform;
}

/* ✅ Adicione via JS só ANTES da animação, remova depois */
elem.style.willChange = "transform";
elem.addEventListener("transitionend", () => {
  elem.style.willChange = "auto";
}, { once: true });

/* Na dúvida, não use. Use apenas se medir um problema real. */

# O checklist de performance CSS

Auditoria rápida
✓ CSS crítico inline no <head>, resto com preload não-bloqueante
✓ Animações usam apenas transform e opacity
✓ Blocos fora da tela têm content-visibility: auto
✓ Imagens/vídeos declaram width/height (evita CLS)
✓ will-change é temporário, não permanente
✓ @layer organiza cascata sem hacks de especificidade
✓ Web fonts têm font-display: swap + preload
✓ Prefere gap a margin entre irmãos
✓ Prefere :where() em resets para não inflar especificidade
✓ Respeita prefers-reduced-motion

# Boas práticas de escrita

1. Design tokens antes de tudo

tokens-primeiro.css
/* ❌ Valores mágicos espalhados */
.btn { padding: 11px 23px; border-radius: 7px; }
.card { padding: 14px; border-radius: 8px; }
.modal { padding: 18px; border-radius: 6px; }

/* ✅ Tokens centralizados */
:root {
  --space-2: 0.5rem;
  --space-3: 0.75rem;
  --space-4: 1rem;
  --radius-sm: 4px;
  --radius-md: 8px;
}
.btn   { padding: var(--space-3) var(--space-4); border-radius: var(--radius-md); }
.card  { padding: var(--space-4); border-radius: var(--radius-md); }
.modal { padding: var(--space-4); border-radius: var(--radius-md); }

2. Especificidade baixa sempre

baixa-espec.css
/* ❌ Especificidade desnecessariamente alta */
body .container ul.list li.item a.link { }

/* ✅ Especificidade mínima */
.list-link { }

/* Use @layer para isolar cascata sem precisar de especificidade */

3. Propriedades lógicas em vez de direcionais

logical.css
/* ❌ Presume direção */
.card {
  padding-left: 16px;
  margin-right: 8px;
  border-left: 2px solid;
}

/* ✅ Adapta-se a direção do texto (RTL, escrita vertical) */
.card {
  padding-inline-start: 16px;
  margin-inline-end: 8px;
  border-inline-start: 2px solid;
}

4. Mobile-first em breakpoints

mobile-first.css
/* ✅ Mobile first — comece com mobile, adicione para maior */
.container {
  padding: 1rem;
}
@media (width >= 768px) {
  .container { padding: 2rem; }
}

5. Comentários que explicam "por quê"

comentarios.css
/* ❌ Óbvio — não agrega */
.btn {
  /* cor de fundo azul */
  background: blue;
}

/* ✅ Explica o contexto ou a razão */
.btn {
  /* Azul específico do design system (não usar #blue direto) */
  background: var(--color-primary);
  /* Escalada leve no hover evita flicker em touch — decisão UX */
  transition: scale 0.1s;
}

# O que nunca fazer

  • Não misture metodologias no mesmo projeto — escolha uma e siga em frente.
  • Não abuse de !important — é dívida técnica instantânea.
  • Não use IDs como hook CSS — classes sempre.
  • Não estilize por tipo sem necessidade — div { ... } afeta divs em toda parte.
  • Não use * { box-sizing: border-box } como reset e depois esqueça — sempre inclua ::before, ::after.
  • Não confie em JavaScript para CSS crítico — se JS falhar, o visual ainda deve funcionar.
  • Não anime all — pode animar propriedades inesperadas.
  • Não remova o focus outline sem substituir — é acessibilidade.
  • Não use position: absolute para layout comum — é para overlays e sobreposições específicas.
  • Não escreva CSS para "todos os navegadores possíveis" — escreva para os que seu público usa. Meça.

# Palavra final

CSS é frequentemente subestimado. "É só estilo" escondem uma linguagem que tem sistema de tipos próprio (via @property), programação reativa (custom properties + cascata), layout declarativo (Grid/Flex), e uma gramática de seletores rica o suficiente para expressar lógica de interface sem JavaScript.

Em 2025, o domínio de CSS separa o desenvolvedor que "faz funcionar" do desenvolvedor que entrega interfaces com qualidade — performáticas, acessíveis, sustentáveis. Os recursos cobertos neste manual — :has(), container queries, cascade layers, anchor positioning, View Transitions, oklch, clamp, @scope, subgrid — formaram o novo patamar mínimo do profissional de web.

CSS envelhece bem
Frameworks vêm e vão. Sass dominou, depois saiu de moda. CSS-in-JS subiu e caiu. Tailwind divide opiniões. Mas display: grid vai continuar funcionando em 2030. :has() só vai melhorar. O investimento em aprender CSS bem — não só uma biblioteca — é um dos que mais rendem a longo prazo em desenvolvimento web. Este manual termina, mas a prática começa. Escreva muito CSS. Leia CSS de outros. Teste as features novas assim que chegam. O futuro da linguagem é brilhante.
